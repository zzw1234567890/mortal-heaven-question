# 卡牌效果解析引擎 (Card Effect Engine)

> **状态 (Status)**：设计中 (In Design)
> **作者 (Author)**：Claude Code + 用户
> **最后更新 (Lst Updated)**：2026-07-23
> **最后验证 (last Verified)**：—
> **实现的支柱 (Implements Pillar)**：支柱1「自由组牌，策略为王」——效果的确定性和可预测性是策略的基础

## 概述 (Overview)

卡牌效果解析引擎是游戏的运行时核心，负责解析和执行全部222张卡牌的效果逻辑。它定义了效果的类型体系、触发时机、目标选择规则、结算顺序和堆叠规则、AI评估接口和概率伪随机分布（PRD）策略。所有卡牌的行为（伤害、治疗、绑定、阵法激活、状态附加、临时费用等）都通过此引擎统一解析和结算，确保效果执行的一致性、可预测性和可扩展性。

对玩家来说，效果引擎的承诺是：**当你读一张卡牌时，发生的效果正是你期望的。** 没有歧义，没有意外。"击杀后攻击+1"就是每击杀一个敌人就永久+1攻击，不多不少。

## 玩家幻想 (Player Fantasy)

效果引擎的核心情感是**信任**——玩家相信自己读到的每一个卡牌效果都会如描述那般精确地发生。这不是"纯基础设施"冷漠的中立性，而是一个积极的承诺：引擎是玩家与游戏规则之间的契约执行者。

| 情感层 | 触发场景 | 强度 |
|--------|----------|------|
| **规则的信任** | 卡牌打出后效果如描述精确结算——伤害不多不少、状态不丢不漂 | 常在——每张卡牌打出验证 |
| **策略的自信** | 看板面能推算出效果结算顺序（新效果先触发、先发优先），敢于做多步规划 | 中频——流派成型的标志 |
| **挫败的透明** | 触发链达到上限时看到明确提示而非静默失败；目标免疫时看到"免疫"弹出 | 低频——但每次都需要清晰反馈 |
| **大师的掌控** | 利用本命加成（×1.5 floor取整）的数值锯齿——选偶数值的基础卡获得边缘收益 | 低频——可选的深度优化 |

**关键区别：** 大多数卡牌游戏的RNG是"抽卡出奇迹"的兴奋。本游戏的概率效果采用伪随机分布（PRD）——每次失败提高下次成功概率，保证在短序列内接近标示概率。这保留了概率的战术变化性，同时消除"连续失败5次30%觉得bug了」的反直觉体验。详见 §详细设计→概率效果的伪随机分布。

## 详细设计 (Detailed Design)

### 䠦心规则

#### 1. 效果类型体系

所有卡牌效果分为以下四大类：

**① 即时效果 (Instant Effects)** — 打出瞬间结算完毕，不持续存在
- 伤害（对单体/全体）
- 洽疗/回复
- 临时费用增加（修改费用系统状态，非角色状态）
- 抽牌/弃牌
- 直接移除状态（含永久状态）
- 临时属性变化（仅本回合，实际为duration=1的隐式状态）

**② 持续效果 (Persistent Effects)** — 绑定后持续存在直到条件解除
- 功法绑定（角色存活时永久生效，duration=-1）
- 法宝绑定（角觲存活时生效，可能有冷却）
- 阵法光环（阵法部署期间生效，条件满足时激活）
- 持续buff/debuff（持续N回合，由状态效果系统管理）

**③ 触发式效果 (Triggered Effects)** — 满足条件时自动触发
- 回合开始时触发（阶段0）
- 回合结束时触发（阶段6）
- 攻击时/被攻击时触发
- 击杀时触发
- 阵亡时触发（含"替代阵亡"——见下文替代效果）
- 打出时触发
- 抽牌时触发
- 延迟触发（"下回合开始时造成N点伤害"——一次性定时器）

**④ 替代/修改效果 (Replacement/Modification Effects)** — 拦截并改变其他效果的结果
- 替代阵亡：「当此角色将要阵亡时，改为治疗至1 HP」（须在阵亡序列之前拦截）
- 效果增幅：「你的下一个法术造成双倍伤害」（挂起效果修正队列）
- 效果无效化：「无效化下一次受到的减益效果」

#### 2. 触发时机映射（与战斗系统7阶段对齐）

> **⚠ 设计决策（2026-07-23）**：战斗系统的阶段模型已从8阶段修订为7阶段。原"阶段3：布阵阶段"已移除——阵位调整改为备战阶段一次性操作（见 `deployment-system.md §7）。阵法部署效果现于阶段2（出牌阶段）中作为卡牌打出效果结算。本表已与 `cobat-system.md` 的当前7阶段模型同步。

| 战斗阶段 | 可触发的效果类型 | 例子 |
|---------|-----------------|------|
| 阶段0：准备 | 回合开始触发、持续伤害/恢复结算、状态持续时间-1 | 每回合掉1血的中毒 |
| 阶段1：抽牌 | 抽牌时触发 | 抽牌时获得临时属性 |
| 阶段2：出牌 | 出牌时触发（打出卡牌自身效果）、阵法部署效果 | 所有卡牌的主效果；阵法激活光环 |
| 阶段3：攻击声明 | 攻击前触发 | 攻击前+攻击力 |
| 阶段4：攻击结算 | 命中/伤害/击杀时触发 | 击杀后攻击永久+1 |
| 阶段5：敌方行动 | 受击时触发 | 受击时反弹伤害 |
| 阶段6：结束 | 回合结束触发、费用重置 | 回合结束时回血 |

#### 3. 效果结算顺序规则

```
同一时机触发的多个效果，结算顺序为：
  第1优先级：玩家主动出牌的效果（当前回合方优先）
  第2优先级：标记了「先发」的己方持续效果（按激活时间从新到旧）
  第3优先级：未标记「先发」的己方持续效果（按激活时间从新到旧）
  第4优先级：敌方持续效果（按激活时间从新到旧）
  第5优先级：同时激活的同类效果 → 按 card_instance_id 排序（全局唯一递增ID）
```

**「先发」标记**：每个绑定位有一个可选的「先发」切换（默认为关）。标记为「先发」的效果在同一时机内优先于所有其他己方持续效果结算。两个「先发」效果之间仍按激活时间从新到旧排序。

**激活时间**：使用全局单调递增的 `uint32 activation_sequence`，在效果注册时分配。同帧内的效果共享同一个 sequence 值→通过 `card_instance_id` 决胜。确保跨平台和重播的确定性排序。

**中分辨率插入队列模型**：效果结算期间如果生成新效果 → 新效果按优先级重新插入尚未结算的队列，而非追加到末尾。这确保 A 触发 B 时 B 不会排在其他已在队列中的更高优先级效果之后。

#### 4. 效果数据模式

所有卡牌效果使用以下类型化数据模式。实现时使用 Godot 4.6 的 `Resorce` 子类作为编辑时可创作的卡牌模板，运行时的活跃效果实例使用 `RefCountd` 子类（详见 §待解决问题→开放问题#1 的架构决策）。

**即时效果模式：**
```
InstantEffect {
  template_id: String            # 效果模板标识（如 "damege_single_3"）
  type: Enum[伤害, 洽疗, 抽牌, 弃牌, 临时费用, 移除状态, 临时属性]
  base_value: int                 # 效果基础值（伤害量/洽疗量/抽牌数）
  target_selector: TargetSpec    # 目标选择规格（见 §目标选择规则）
  element: Enum[无, 火, 冰, 雷, 毒, 暗]  # 伤害/状态属性（用于免疫判定）
  animation_id: String            # 关联动画资源标识
  secondary_effects: Array[InstantEffect]  # 附加效果（如"造成3点伤害，并抽1张牌"）
  condition: ConditionSpec|null   # 条件效果（如"若目标为魔道阵营, 伤害+2"）
  condition_bonus_value: int      # 条件满足时的额外数值
}
```

**持续效果模式：**
```
PersistentEffect {
  template_id: String
  type: Enum[功法绑定, 法宝绑定, 阵法光环, buff, debuff]
  stat_modifiers: Dictionary      # { "ATK": +2, "DEF": -1, ... }
  duration: int                   # 回合数；-1=永久
  trigger_condition: TriggerSpec|null  # 触发条件（如"回合开始时"）
  trigger_effect: EffectSpec|null      # 触发后执行的效果
  stacking_rule: Enum[独立, 刷新, 叠加上限]
  max_stacks: int                 # stacking_rule=叠加上限 时有效
  cooldown: int                   # 法宝冷却（0=无冷却）
  binding_multiplier: float        # 1.0（普通）或 1.5（本命），绑定时预计算锁定
}
```

**触发式效果模式：**
```
TriggeredEffect {
  template_id: String
  trigger_event: Enum[回合开始, 回合结束, 攻击时, 受击时, 击杀时, 阵亡时, 打出时, 抽牌时, 延迟触发]
  delay_turns: int                 # 仅 delay_trigger 有效：延迟回合数
  trigger_once: bool               # true=触发一次后自动注销；false=每次事件都触发
  condition_predicate: ConditionSpec|null  # 额外触发条件（如"仅当自身HP<50%时"）
  effect_to_execute: EffectSpec    # 触发后执行的效果
  max_triggers_per_turn: int       # 每回合最大触发次数（0=无限制）
  is_replacement: bool             # true=替代效果（拦截而非响应事件）
  replacement_priority: int        # 替代效果间的优先级（更高的先拦截）
}
```

**序列化格式：** 所有卡牌模板使用 Godot `Resource (.tres)` 格式存储，便于编辑器中检查和作者。设计师可编辑的字段使用 `@export` 标注。运行时解析由 `EffectFactory` 类完成——读取 Resource 并生成轻量级的 `RefCountd` 效果实例。

#### 5. 目标选择规规

| 目标类型 | 选择方式 | 说明 |
|---------|---------|------|
| 单体己方 | 玩家点击选择 | 可选目标：场上己方角色（含未行动角色，不含未上场角色） |
| 单体敌方 | 玩家点击选择 | 符箓攻击时选择目标敌人 |
| 全体己方 | 自动选择 | 全体洽疗/群体buff |
| 全体敌方 | 自动选择 | AOE伤害 |
| 自身 | 强制绑定 | 卡牌只对自己生效 |
| 随机敌方 | 系统随机选择 | "对随机敌方造成3点害伤"；均匀分布 |
| 条件选择 | 自动匹配 | "HP最低的己方角色"、"ATK最高的敌方"等条件 |
| 事件上下文目标 | 自动引用 | "触发此效果的事件的目标"——如"攻击命中时，对该次攻击的目标附加2点伤害" |

**条件选择的决胜规则**：当多个目标满足条件时（如两个己方角色HP同为最低值）→ 按以下顺序决胜：
1. 上场位置编号（从低到高——前排优先）
2. 若仍在同位置（不可能，但作为安全防护）→ `character_instance_id` 升序

**敌方自目标**：敌方AI使用同一套目标选择逻辑。如果敌方卡牌的效果目标是"己方"→ 对于敌方而言"己方"指敌方阵容。目标解析是**对称的**——调用方的视角决定"己方"和"敌方"的含义。

#### 6. 效果堆叠规则

| 效果类型 | 堆叠规则 |
|---------|---------|
| 同名功法绑定不同角色 | 可叠加，每角色独立计算（按 `card_template_id` 判断同名） |
| 同名功法绑定同角色 | 效果不可叠加，取最高值（比较绑定实例的 `effect_value × binding_multiplier`） |
| 同名法宝绑定不同角色 | 可叠加 |
| 同名buff/debuff | 不可叠加，刷新持续时间（取 `max(旧, 新)`，见 status-system.md） |
| 不同模板的同功能效果 | 可叠加（加法叠加） |
| 本命加成（×1.5） | 逐源独乘——每个绑定的 base_value 分别乘以自己的 multiplier，再加总。不与其它加成互扰 |

**模板 vs 实例区分**：「同名」始终指 `card_template_id` 相同——两张同模板的不同副本属于「同名」。同一张卡牌的同一实例不可能被绑定两次（由绑定系统通过 `card_instance_id` 唯一性保证）。效果引擎在堆叠判断时以 `card_template_id` 为键。

#### 7. 状态效果系统接口

状态效果的完整管理（CRUD、叠和、倒计时、免逸判定）由独立的**状态效果系统** (`status-system.md`) 负责。效果引擎通过以下接口与状态系统交互：

```
# 施加状态
apply_status(target_id, status_template) → ApplyResult

# 移除状态
remove_status(status_id) → bool
remove_statuses_by_source(target_id, source_card_instance_id) → int

# 查询状态
get_active_statuses(target_id) → Array[StatusEffect]
has_status(target_id, template_id) → bool
get_accumulated_value(target_id, stat_name) → float
```

详细的状态生命周期、叠和规则和免逸判定见 `status-system.md`。

#### 8. 效果描述规范化

所有卡牌效果的文本描述必须使用统一模板：

| 模板 | 示例 |
|------|------|
| "对【目标】造成【N】点伤害" | "对单体敌方造成3点伤害" |
| "【目标】攻击+X" | "绑定角色攻击+2" |
| "【条件】时，触发【效果】" | "击杀敌方时，自身攻击永久+1" |
| "每回合【时机】，【效果】" | "每回合开始，回复1血" |
| "【概率】概率【效果】" | "30%概率冰冻目标一回合" |
| "【条件】+【效果】" | "若目标为魔道阵营，伤害+2" |
| "【效果1】，并【效果2】" | "造成3点伤害，并抽1张牌" |
| "场上每有【条件】，【效果】" | "场上每有1个正道角色，攻击+1" |
| "下【N】次【类型】效果【倍率】" | "下1次法术效果造成双倍伤害" |

描述模板与效果数据结构双向绑定——修改效果数值时，描述中的 N 必须同步更新。模板覆盖所有效果类型后文可自动生成描述；在自动生成就绪前，描述为手写存储的字符串。

#### 9. 概率效果的伪随机分布 (PRD)

概率效果（如"30%冰冻"）采用**伪随机分布 (PRD)** 而非独立随机：

- **机制**：每次失败后，内部概率累加（`P_effective += P_base × C`）。触发后重置为初始值。常数 C 调谐使长序列概率收敛到标示值。
- **怜悯保护**：连续失败次数达到 `ceil(1 / P_base)` 次后 → 下一次必然触发。30%概率 → 最多连续4次失败后强制触发。
- **粒度**：概率值以 5% 为步进（5%, 10%, 15%...95%）。设计限制确保可预测性和测试可行性。
- **种子**：PRNG 种子来自 GameStateManager 的运行种子（`meta.seed`）。同种子+同操作序列 → 确定的相同结果。支持重播和回归测试。
- **测试模式**：`set_prng_seed(int)` 允许测试代码设定已知种子以验证概率行为。

**为什么不用独立随机**：独立掷骰导致"30%概率连续5次失败"概率约16.8%——玩家会觉得游戏bug了。PRD 保证短序列体验接近标示概率，消险"随机就是不公平」的负面感知。

#### 10. AI 效果评估接口

效果引擎暴露一组只读评估 API，供 AI 系统在做出决策前对可能的卡牌进行干跑评估。这些接口不修改游戏状态——仅在不可变的 `GameStateSnapshot` 上执行纯计算。

```
# ---- 核心评估 ----

evaluate_effect(card_id, target_id, snapshot) → EffectEvaluation
  # 计算卡牌对目标施加的效果（确定性部分）。
  # 返回：{damage, healing, stat_changes, statuses_applied, is_overkill, is_overheal}

evaluate_effect_probabilistic(card_id, target_id, snapshot) → Array[ProbabilityOutcome]
  # 对于含RNG的效果，返回完整的离散概率分布。
  # 每项：{outcome: EffectEvaluation, probability: float}

simulate_chain(card_id, target_id, snapshot, max_depth=10) → ChainPreview
  # 模拟打出此卡的完整触发链。
  # 返回：{chain: [{step, source, effect, probability}], would_overflow: bool}

# ---- 辅助查询 ----

get_effect_categories(card_id) → Array[EffectCategory]
  # 返回此卡效果的类型标签：DAMAGE, HEAL, BUFF, DEBUFF, CONTROL, DRAW, BIND, FORMATION, UNEVALUABLE
  # UNEVALUABLE = 效果依赖隐藏信息（如偷牌），AI使用模板base_weight作后备

# ---- 快照 ----

create_evaluation_snapshot() → GameStateSnapshot
  # 创建当前游戏状态的不可变浅拷贝。单次AI回合只创建一次，所有评估共用。
  # 不含动画、UI、音效——仅含角色、状态、绑定、阵法等机制数据。
```

**性能预算**：单次 `evaluate_effect()` 调用 < 100μs（纯算术，无信号，无对象分配）。最坏场景——6个敌人各评估8个技能对6个目标 = 288次调用 ≈ 29ms。快照创建一次（含深拷贝约1-2ms），AI回合总评估预算约30ms——在16.6ms帧预算之外以异步方式运行。实际通过分帧评估（每帧评估一部分候选）来分摊。

### 状态与转换

效果引擎本身无状态，它处理的效果的状态流转由卡牌系统管理。活跃状态的生命周期由状态效果系统 (`status-system.md`) 管理。

### 与其他系统的交互

| 系统 | 数据流入（引擎→目标） | 数据流出（目标→引擎） |
|------|----------------------|----------------------|
| **卡牌系统** | 效果查询（根据效果ID） | 卡牌模板定义中的效果字段 |
| **战斗系统** | 效果结算结果（伤害、状态变更） | 触发信号（7阶段转换、攻击/伤害/击杀事件） |
| **绑定系统** | 绑定/解绑结果 | 功法/法宝的绑定请求、预计算的 binding_multiplier |
| **阵法系统** | 阵法激活/失效事件 | 阵法的触发条件判定 |
| **状态效果系统** | 状态施加/移除请求 | 已存在的状态查询、免疫判定结果 |
| **AI系统** | evaluate_effect() 评估结果、simulate_chain() 连锁预测 | 出牌意图（评估请求） |
| **游戏状态管理器** | 无（引擎不直接写GSM——通过各子系统写入） | 当前境界、回合数、运行种子等全避上下文 |
| **费用系统** | 无（临时费用+通过即时效果→费用系统接口结算） | 当前可用费用（仅用于出牌前的费用验证） |

## 公式 (Fomulas)

### 1. 效果有效值

```
effective_value = floor(base_value × binding_multiplier + sum(modifiers))
```

| 变量 | 类型 | 范围 | 描述 |
|------|------|------|------|
| base_value | int | [0, 100] 按卡牌类型上限 | 效果基准数值（待数据设计确定各类型精确上限） |
| binding_multiplier | float | 1.0 或 1.5 | 本命绑定系数（由绑定系统在绑定时预计算并锁定） |
| modifiers | [float] | 每项可正可负 | 来自其他来源的修正值（流派增益、阵法光环等） |
| effective_value | int | [0, ∞) | 最终效果值，floor向下取整 |

**舍入规则**：使用 `floor()`（向下取整）。3 × 1.5 = 4.5 → 4。CD指导：这创造了微妙的数值锯齿——偶数基础值在×1.5下得到整数结果，奇数基础值损失0.5。鼓励玩家注意数值整除性，这是一种隐式的系统掌握深度。

**base_value 边界**：base_value 严格非负（设计者不应创建负值效果；洽疗和伤害使用不同效果类型和正base_value）。base_value 的上限在卡牌模板数据表中按卡牌类型定义——数据管线中有作者期验证规则。详见 card-system.md §卡牌升级与数值边界。

### 2. 效果叠加公式（属性累计）

```
for each active_effect on character:
  effective_i = floor(effect.base_value × effect.binding_multiplier)
  # binding_multiplier 是每效果独立的——每个绑定的卡牌预计算自己的乘数
  # 非绑定来源的乘数 = 1.0

stacked_total = sum(effective_i for all sources i)
```

本命加成（×1.5）是逐源独立计算的——不是对总额乘1.5。一个本命绑定（×1.5）和一个普通绑定（×1.0）相加时，仅本命部分的 base_value 乘以 1.5。

**示例**：角色基础ATK=5。绑定青元剑诀（本命，base=+4 → 4×1.5=6），绑定长春功（普通，base=+3 → 3×1.0=3）。最终ATK = 5 + 6 + 3 = 14。

### 3. 同名效果去重

```
stacked = max(duplicate_sources)
  # 比较的是每个实例的 effective_value（base_value × binding_multiplier）
  # ——即升级后、乘数计算后的实例值。
  # 如果两个同名实例的 effective_value 相等，优先保留先绑定的那个。
```

### 4. 效果叠加溢出（无硬上限）

```
stacked = sum(all_sources)  # 无上限，由数值平衡约束
```

> ⚠ **数值边界**：虽无代码级硬上限，但卡牌模板的 base_value 有按类型限制的上限（见 card-system.md），且每角色绑定位有限（最高约30个活跃效果）。两者共同构成自然的边界约束。调优参数中仍有 `max_simultaneous_effects_per_phase` 作为性能安全阀。

## 边縼情况 (Edge Cases)

- **效果同时触发**：同一时机多效果触发时，按 主动出牌→先发己方→普通己方→敌方→card_instance_id 顺序结算。中分辨率插入的新效果按优先级重新排队
- **目标已死亡**：若效果结算前目标已死亡（被先手击杀），效果不结算（浪费）。与战斗系统的攻击重定向不同——卡牌效果是精确的锁定目标，错失后不会重定向。若为玩家浪费了重要卡牌，UI显示"目标死亡「飘字反馈（0.3s）
- **多目标效果中部分目标死亡**：AOE效果（3目标）中目标#2先死亡→其余目标1和3正常结算。各目标独立结算
- **多段效果的部分失败**："造成3点伤害，并抽1张牌"——若伤害部分因目标死亡无法结算，抽牌部分仍执行。效果的子组件独立结算
- **本命绑定冲突**：同一功法/法宝不可被多个角色本命绑定——第二绑定尝试自动降级为普通绑定（binding_multiplier = 1.0）
- **效果叠加溢出**：无硬上限（由绑定位数量和base_value上限自然约束）。若通过特殊组合达到>100的ATK→在调优参数中标记异常供手工审查
- **状态持续时间归零**：持续N回合的效果在N次己方回合阶段0结束后自动解除（见 status-system.md §持续时间倒计时）
- **免疫与效果**：当目标免疫某效果时，该效果不生效但不消耗卡牌——丹药/符箓仍进弃牌堆，功法/法宝仍绑定但效果不激活
- **效果触发链**：A触发B触发C时，按栈式结算（C先结算完→B→A）。最大触发链深度=10。深度定义：触发链DAG中的总节点数，从根触发（玩家打出的第一张卡）开始计数。扇出分支（A同时触发B1和B2）共享同一深度计数器。循环检测：同一card_instance_id在链中已触发过→跳过不重复触发（防止2卡无限循环）。链中第11层→终止并记录WARN日志，第1-10层正常结算完毕
- **概率效果**：使用PRD算法而非独立随机。连续失败次数达到阈值后强制触发。详见 §详细设计→概率效果的伪随机分布
- **效果描述与数据不同步**：描述是手写存储时，需人工QA验证与效果数据一致。自动生成就绪后，描述从效果数据即时生成→消除不同步
- **浮点舍入一致性**：所有乘1.5操作使用 `floor()` 统一舍入。伤害计算中战斗系统的 `max(1, ATK-DEF)` 在效果引擎传值后执行——引擎将有效值作为整数值传给战斗系统，不再二次舍入

## 验受标准 (Acceptance Criteria)

### 基础效果结算

- **GIVEN** 玩家打出伤害型符箓（base_vaue=3），AND 目标无任何伤害修正效果，AND 无本命绑定，
  **WHEN** 确认目标并结算，**THEN** 目标HP减少3点（精确匹配 effective_value）
- **GIVEN** 玩家打出伤害型符箓（base_value=3），AND 角色有本命绑定（binding_multiplier=1.5），
  **WHEN** 结算，**THEN** 目标HP减少 `floor(3×1.5)=4` 点
- **GIVEN** 玩家打出功法卡"铁布衫——绑定角色攻击+2"并选择己方角色A，
  **WHEN** 结算完成，**THEN** 角色A的ATK属性值增加2点（通过 `get_accumulated_value(target, "ATK")` 验证）
- **GIVEN** 角色有本命加成条件满足，**WHEN** 绑定对应功法/法宝，
  **THEN** `binding_multiplier` 锁定为 1.5（在绑定时预计算）且 UI 中效果数值旁显示"×1.5"标识

### 结算顺序

- **GIVEN** 己方效果A（激活时间t=3，先发标记=关）和己方效果B（激活时间t=5，先发标记=关）在同一时机触发，
  **WHEN** 结算，**THEN** B先于A执行（较新的优先），最终数值反映B先于A的效果顺序
- **GIVEN** 效果A（先发标记=开，激活时间t=1）和效果B（先发标记=关，激活时间t=10）在同一时机触发，
  **WHEN** 结算，**THEN** A先于B执行（先发无视激活时间优先）

### 目标选择

- **GIVEN** 场上敌方有角色A（HP=5）、角色B（HP=2）、角色C（HP=8），
  **WHEN** 打出"对HP最低的敌方角色造成3点伤害"，**THEN** 自动选择角色B
- **GIVEN** 场上两个己方角色HP相同均为最低值（角色A和角色B），
  **WHEN** 打出"洽疗HP最低的己方角色"，**THEN** 按上场位置编号从低到高决胜（前排优先）
- **GIVEN** 打出"对随机敌方造成3点伤害"且场上3个敌方均存活，
  **WHEN** 执行100次并固定相同的PRNG种子，**THEN** 每个敌方被选中的次数在25-42次之间（均匀分布95% CI）

### 堆叠与叠和

- **GIVEN** 角色绑定功法A（base=+4, 本命 multiplier=1.5）和功法B（base=+3, 普通 multiplier=1.0），
  **WHEN** 计算累计ATK加成，**THEN** = `flor(4×1.5) + flor(3×1.0) = 6 + 3 = 9`
- **GIVEN** 角色已绑定"铁布衫"（ATK+2），**WHEN** 再次绑定同名卡牌但数值更高（ATK+5），
  **THEN** 同名功法只取最高值，最终ATK加成保持+5
- **GIVEN** 己方角色A和角色B各自绑定同名功法"铁布衫"（ATK+2），
  **WHEN** 查询两个角色的ATK加成，**THEN** 各角色独立获得+2（不同角色可叠和）

### 状态效果交互

- **GIVEN** 目标具有"免疫冰冻"状态，**WHEN** 效果"30%概率冰冻"判定为触发，
  **THEN** 冰冻效果不应用到目标，卡牌仍正常进弃牌堆，「免疫」文字弹出
- **GIVEN** 目标在效果结算前已被其他效果击杀（HP=0），
  **WHEN** 该效果结算，**THEN** 效果不生效，播放"目标死亡」飘字，卡牌进弃牌堆

### 触发链

- **GIVEN** 效果A→B→C触发链（深度3），**WHEN** 结算，
  **THEN** C先结算完，B其次，A最后（栈式LIFO）
- **GIVEN** 同一 card_instance_id 已经在触发链中出现过，**WHEN** 该实例的效果再次被触发，
  **THEN** 跳过不重复触发（循环检测），记录DEBUG日志
- **GIVEN** 触发链达到10层深度，**WHEN** 第11层试图触发，**THEN** 第11层终止，前10层正常结算，输出WARN级别日志：
  `"[CardEffectEngine] Triger chain depth exceeded: max=10, root_card_id=<ID>, chain=<A→B→...→K>`

### 概率效果 (PRD)

- **GIVEN** 效果配置为"30%概率冰冻"，**WHEN** 连续失败4次（PRD理论连续失败上限≈4），
  **THEN** 第5次必然触发（怜悯强制触发）
- **GIVEN** 同一PRNG种子执行100次30%PRD效果，**WHEN** 统计触发次数，
  **THEN** 触发次数在24-36次之间（99% CI区间近似值；PRD收敛后该区间比独立随机更紧）

### 费用验证

- **GIVEN** 玩家打出费用=3的卡牌，AND 当前可用费用=2，
  **WHEN** 点击卡牌，**THEN** 卡牌不可打出，效果不执行，显示"费用不足」提示

### 性能

- **GIVEN** 满场6角色+3阵法+10个活跃buff状态，**WHEN** 阶段0（准备阶段）结算，
  **THEN** 所有效果的阶段0触发检查完成时间 < 2ms（Godot profiler测量）
- **GIVEN** AI评估288次 effect 调用（6敌 ×8技 ×6目标），
  **WHEN** 连续执行，**THEN** 总耗时 < 30ms（不含快照创建的一次性成本约1-2ms）

### AI评估接口

- **GIVEN** 调用 `evaluate_effect(card_id, target_id, snapshot)` 对伤害型卡牌，
  **WHEN** 检查返回值，**THEN** `EffectEvaluation.damage` 与 `et_accumulated_value()` 计算结果一致
- **GIVEN** 调用 `simulate_chain(card_id, target_id, snaphot, max_depth=5)`，
  **WHEN** 该卡会触发连锁效果，**THEN** 返回的 `ChainPreview.chain` 包含每一步的触发来源和效果评估
