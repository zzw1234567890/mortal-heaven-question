### 1.3 循环依赖检测

| 类型 | 系统对 | 详情 | 严重程度 |
|------|-------------|-------|----------|
| **直接双向** | 卡牌效果引擎 ↔ 状态效果系统 | 效果引擎将状态管理工作委托给状态系统；状态系统接收源自效果引擎的指令。采用单向模式可行（效果引擎 → 状态系统），但当前设计中状态系统 _也_ 引用 `source_card_id` 返回效果引擎。 | **低** — 可解耦为单向：效果引擎发出 `apply_status` 信号；状态系统监听但 _不_ 引用返回的效果引擎。 |
| **直接双向** | 战斗系统 ↔ 费用系统 | 战斗系统编排回合；费用系统提供每回合资源。费用系统将 `cost_per_turn` 绑定到战斗回合流程。 | **低** — 费用系统应为被动数据源；战斗系统读取当前费用值。费用系统 _绝不_ 启动战斗操作。 |
| **直接双向** | 修为养成系统 ↔ 境界系统 | 修为驱动境界；境界定义 `max_cultivation`。循环在概念上被 `max_cultivation` 定义为配置数据（非运行时逻辑）所打破，但 GDD 中明确双向命名。 | **低** — 通过有一个方向作为被动数据提供者来解决。境界系统 _读取_ 来自修为系统的当前修为；修为系统 _读取_ 来自境界系统的 `max_cultivation`。如果两者都在同一帧内发生变化才会出现循环（将 `max_cultivation` 锁定到境界切换事件上）。 |
| **传递式** | 存档系统 → 游戏状态管理器 → 轮回天赋系统 → 存档系统 | 存档系统持久化游戏状态管理器数据；游戏状态管理器持有 `player.talents`；轮回天赋系统在存档结算时读取游戏状态管理器，但写入存档系统的 `progression.dat`。 | **否** — 虚假循环。轮回天赋系统 _读取_ 游戏状态管理器进行结算，_写入_ 存档系统进行持久化。存档系统是纯粹的数据层。 |
| **架构关注点** | 剧情系统 → 探索系统 → 事件系统 → 剧情系统（`story_flags`） | 剧情系统使用探索系统进行章节触发；探索系统触发事件；事件设置剧情系统读取的 `story_flags`。三个系统读/写同一个 `story_flags` 字典（游戏状态管理器中的 `narrative.story_flags`）。 | **中** — 游戏状态管理器作为共享状态天然打破循环，但所有权不明确。三个系统 _编写_ story_flags，只有剧情系统 _权威性地读取_ 用于章节门控。约定：事件系统是 `story_flags` 的 **唯一写入者**。剧情系统 **只读**。探索系统安排节点但不编写标记。 |

### 2.3 数据流关键路径（每步操作的数据移动）

**路径 1：战斗结算**
```
战斗系统 → 战斗胜利信号
  ├→ 卡组编辑系统：触发战利品三选一流程
  ├→ 资源系统：写入灵石 + 可能灵材到游戏状态管理器
  ├→ 修为养成系统：写入修为到游戏状态管理器
  ├→ 探索系统：标记节点完成
  ├→ 存档系统：触发自动存档
  └→ HUD：刷新所有显示
```

**路径 2：境界突破**
```
修为养成系统 → 修为 >= max_cultivation 信号
  ├→ 境界系统：启用渡劫台
  ├→ 渡劫突破系统：触发渡劫战（通过战斗系统）
  │   └→ 成功：境界系统更新 realm_level → 触发连锁效应
  │       ├→ 费用系统：cost_per_turn 改变
  │       ├→ 上场阵位系统：max_deploy 改变
  │       ├→ 行动力系统：action_points 上限改变
  │       ├→ 卡牌系统：新卡池层级解锁
  │       ├→ 探索系统：新地图解锁
  │       ├→ 绑定系统：绑定位限制增加
  │       └→ HUD + 战斗UI：刷新
  └→ 失败：修为养成系统扣除 10%
```

**路径 3：死亡/轮回结算**
```
战斗系统 → 全员阵亡信号
  ├→ 游戏状态管理器：标记当前存档为死亡
  ├→ 轮回天赋系统：读取游戏状态管理器（境界、修为、回合、卡牌、成就）
  │   ├→ 计算 轮回点
  │   ├→ 检测天赋解锁
  │   └→ 写入 progression.dat（通过存档系统）
  ├→ 成就系统：检测本局触发的任何成就（通过游戏状态管理器读取）
  ├→ 剧情系统：重置章节状态
  └→ 主菜单：显示「轮回结算」界面
```

### 3.2 架构高风险边界

| 边界 | 风险 | 缓解措施 |
|----------|------|------------|
| **游戏状态管理器 ↔ 所有 35 个系统** | 如果游戏状态管理器 API 存在瓶颈，单点故障和性能问题 | 分层 API（直接属性访问用于读取，信号用于更改通知，原子化复合操作如 `apply_battle_rewards()` 作为单元进行）。性能分析以实现锁定争用。 |

## 4. 架构发现与推荐

### 4.1 游戏状态管理器 API 设计（构建前必须解决）

游戏状态管理器是单点故障。在实现任何消费系统之前：

1.  **为读取者定义三个访问层级：**
    -   **直接属性读取**（`GSM.player.realm`）——快速，无信号开销
    -   **信号订阅**（`GSM.player_changed.connect(...)`）——响应式 UI 更新
    -   **查询接口**（`GSM.query("player.realm")`）——动态键查找，适用于不适合硬编码路径的通用工具

2.  **为写入者定义原子复合操作：**
    -   `GSM.apply_battle_rewards(lingshi, cultivation, cards)`——单个原子调用而非 3 次独立写入
    -   `GSM.change_realm(new_level)`——在设置新境界 _之前_ 锁定所有消费者

### 4.2 卡牌效果引擎契约

卡牌效果引擎接触 7 个系统。每个效果类型必须与以下事物有一个定义的契约：
-   它在结算期间读取什么状态
-   它在结算期间写入什么状态
-   如果前提条件失败会发生什么（例如，目标在伤害结算前死亡）

这应该在编码开始前形成为一张与 `contracts.md` 等效的表，以 222 张卡片中的 10 张代表性卡为例。

### A.1 story_flags 写入者决议（2026-07-24）

**问题**：`narrative.story_flags` 字典有 3 个声明的写入者——事件系统、剧情系统以及对话系统——从而产生歧义并可导致竞态条件。

**决议**：**事件系统是 `story_flags` 的唯一运行时写入者。**
- 事件系统：调用 `GSM.set("narrative.story_flags.{flag}", value)` 的唯一系统。
- 剧情系统：通过 `advance_chapter()` 中间接触发——它调用事件系统来设置章节结算标记，而非直接写入。
- 对话系统：`DialogueOutcome.type = set_flag` 现在路由到 `事件系统.set_flag()`，而非直接写入 GSM。
- 结局分支系统：纯粹只读——它在通关时聚合标记，但从不写入。

- `event-system.md` L197：添加了唯一运行时写入者声明

## 附录 B：输入管理器 —— 精简架构概述

### 职责

输入管理器是场景间玩家输入路由的单一仲裁者。所有 UI 系统以及场景在消费输入事件之前都向其查询。

### 核心规则

1. **锁模型**：游戏状态管理器维护一个 `session.input_locks` 栈。输入管理器是唯一读取者 + 唯一写入者。
2. **锁类型**：
   - `transition`——场景正在加载/卸载（阻止所有输入）
   - `animation`——过场动画/效果正在播放（阻止交互输入，允许 UI 导航）
   - `dialogue`——对话正在播放（阻止战斗/探索输入，允许对话输入）
   - `modal`——模态弹窗打开（阻止除该弹窗外的所有输入）
3. **API**：`InputManager.is_input_allowed(action_type: Enum[any, ui_nav, gameplay, dialogue]) → bool`
4. **信号**：`input_lock_changed(lock_type, locked: bool)`——UI 组件监听以显示/隐藏输入提示

### 依赖

- **游戏状态管理器**：读取/写入 `session.input_locks`

### 被依赖方

- 战斗 UI、探索 UI、卡组编辑 UI、HUD、主菜单、对话系统

### 实现注意事项

- 单例自动加载，在游戏状态管理器之后、任何 UI 之前初始化
- 无持久化状态——锁是会话临时的
- ~30 行 GDScript 协调器；不得增长为通用事件系统

## 附录 C：场景管理器 —— 精简架构概述

### 职责

场景管理器编排 Godot 场景树中的场景转换。它是唯一调用 `SceneTree.change_scene_to_file()` 或 `SceneTree.change_scene_to_packed()` 的系统。

### 核心规则

1. **转换协议**：
   - 调用者通过信号请求场景变更：`scene_change_requested(from_scene, to_scene, transition_type)`
   - 场景管理器设置 `input_locks.transition = true`
   - 场景管理器在转换前触发自动存档（通过存档系统）
   - 场景管理器加载新场景
   - 加载完成后，场景管理器设置 `GSM.session.current_scene = to_scene`
   - 场景管理器设置 `input_locks.transition = false`
   - 场景管理器广播 `scene_changed(from, to)`
2. **转换类型**：`instant`（无特效）、`fade`（淡入淡出）、`slide`（滑动）
3. **场景注册表**：从 `projects.yml` 或构建配置生成的静态映射 `{scene_id: path}`

### 依赖

- **游戏状态管理器**：读取/写入 `session.current_scene`、`session.input_locks`
- **存档系统**：转换前触发自动存档
- **输入管理器**：通过输入锁协调

### 被依赖方

- 所有请求场景变更的系统（探索系统、战斗系统、主菜单、渡劫突破系统）

### 实现注意事项

- 单例自动加载，在游戏状态管理器 + 输入管理器之后初始化
- ~50 行 GDScript；场景路径从 `.tres` 配置文件加载（不可硬编码）
- 转换回调使用 Godot 4.6 的 `SceneTree.node_added` 信号或手动回调

## 附录 D：存档模式版本控制 —— 精简架构概述

### 职责

确保跨游戏版本的存档兼容性。每次写入存档时，当前模式版本会随数据一起序列化。加载时，迁移函数会将其升级到最新模式。

### 核心规则

1. **版本字段**：`meta.json` 和 `progression.dat` 均包含一个顶层的 `schema_version: int`。
2. **初始版本**：`1`（首个可玩构建版本）。
3. **迁移链**：`migrations = {1: migrate_v1_to_v2, 2: migrate_v2_to_v3, ...}`。加载时，从存档版本开始顺序运行迁移，直到等于 `CURRENT_SCHEMA_VERSION`。
4. **失败处理**：若迁移失败，则保留原始存档备份（`save.json.bak`），将错误记录至 `user://logs/migration.log`，并通知用户。
5. **空/损坏的 progression.dat**：优雅降级——从零开始（所有天赋锁定、零成就、零轮回点）。不清除——在覆盖前提示用户。

### 依赖

- **游戏状态管理器**：提供序列化/反序列化钩子
- **存档系统**：在每次 `write_save()` / `read_save()` 调用时读取/写入 `schema_version`

### 被依赖方

- 无（基础设施层）

### 实现注意事项

- 迁移在存档系统的 `read_save()` 方法内部运行
- 每迁移编写一个纯函数：`func migrate_vN_to_vN_plus_1(data: Dictionary) → Dictionary`
- 在 `tests/unit/save-load/` 下测试每个迁移的往返情况
