## 概述

修为养成系统管理玩家修为值的获取、存储和消耗逻辑——修为是游戏中核心的成长资源，相当于经验值（XP）。系统定义修为的获取途径和获取量、处理修为溢出转化、触发修为达标事件（可突破通知），并与境界系统和渡劫突破系统协作完成境界提升的核心循环。

对玩家来说，修为是衡量「这局走得有多远」的直观指标。每次战斗胜利的修为增加、每次使用丹药的修为提升、每次看到修为条接近满值时的期待感——修为是驱动玩家持续探索的核心动力。它不是金钱（灵石），不是装备（卡牌），而是角色本身的成长刻度。

暂无相关ADR。

## 玩家幻想

修为养成系统的核心情感是**积累的踏实感**——「我没有白打这一仗」。

每次战斗结束后修为条的跳动，每次丹药入腹后数值的上涨，都是对玩家投入时间的正反馈。修为满了——下一步就是渡劫——渡劫成功了就是质变。这个「攒满→突破→变强→继续攒」的循环，是Roguelike成长螺旋中最基础的节拍。

修为溢出转化为属性丹的设计，让追求修为溢出的玩家不是「在浪费」，而是在「为下一局投资」——这支撑了P2「绝不让玩家的努力白费」的承诺。

## 详细设计

### 核心规则

#### 1. 修为获取途径

| 途径 | 获取量 | 频率 | 说明 |
|------|:-----:|:----:|------|
| **战斗胜利** | 固定值（按敌方阵容） | 高频 | 主要获取来源 |
| **使用丹药** | 按丹药类型 | 中频 | 炼丹系统产出 |
| **地图事件** | 事件定义值 | 中频 | 如洞府奇遇获得修为 |
| **杀人夺宝事件** | 中~高（随机范围） | 低频 | 高风险高回报选项 |
| **修为溢出转化** | 溢出量×转化率 | 低频 | 突破后溢出转化 |
| **轮回天赋加成** | 按天赋配置 | 持续 | 跨局永久解锁效果 |

战斗胜利的修为基准值（待数值设计确认）：

| 战斗类型 | 基准修为奖励 |
|---------|:----------:|
| 普通战斗（杂兵） | 30~60 |
| 精英战斗 | 80~150 |
| Boss战斗 | 200~500 |
| 渡劫突破战 | 0（不获得修为——突破目的是突破本身） |

丹药修为（示例）：

| 丹药 | 修为值 | 稀有度 |
|------|:-----:|:------:|
| 培元丹 | +50 | 蓝色 |
| 聚气散 | +150 | 紫色 |
| 真元丹 | +500 | 金色 |

#### 2. 修为上限（max_cultivation）

修为上限由境界决定，公式在 GSM 中已定义：

```
max_cultivation(realm_level) = BASE_MAX × 1.5^(realm_level - 1)
```

| 境界 | LEVEL | max_cultivation |
|:----:|:-----:|:--------------:|
| 炼气 | 1 | 1000 |
| 筑基 | 2 | 1500 |
| 金丹 | 3 | 2250 |
| 元婴 | 4 | 3375 |

#### 3. 修为溢出转化

当修为值达到当前境界的 `max_cultivation` 后继续获取修为时：

```
overflow = cultivation + gained - max_cultivation
cultivation = max_cultivation（锁死）
overflow_pool += overflow × CONVERSION_RATE
```

- 溢出修为不丢失，存入 `player.overflow_pool`
- 溢出修为**不能**提升当前境界的战斗力（修为上限锁死）
- 突破成功后，溢出修为按比例转化为**属性丹**（自动发放到玩家背包）
- 转化率：100溢出修为 = 1属性丹（1属性丹=永久攻击+1或防御+1或HP+2，玩家选择）

#### 4. 修为满值提示

当 `cultivation >= max_cultivation` 时：
- 探索界面修为条显示"已满"状态（闪烁/特殊颜色）
- 自动弹出提示：「修为已满，可前往渡劫突破」或「修为已达瓶颈，速寻突破机缘」
- 该提示在每次进入探索地图时检查一次
- 修为满值后获得的修为全部转入溢出池

#### 5. 修为获取流程

```
gain_cultivation(amount, source):
  if amount <= 0: return
  
  available = max_cultivation - current_cultivation  # 当前还可容纳多少
  
  if amount <= available:
    current_cultivation += amount
  else:
    # 部分直接加，部分走溢出
    current_cultivation = max_cultivation
    overflow = amount - available
    overflow_pool += overflow × CONVERSION_RATE
  
  GSM.emit_signal("cultivation_changed", {delta: amount, current, max})
  
  # 检查修为是否已满
  if current_cultivation >= max_cultivation:
    GSM.emit_signal("cultivation_full")
```

#### 6. 突破后修为处理

渡劫成功后：
- 新境界的 `max_cultivation` 更新
- `current_cultivation` 不变（保留当前值）——如突破前800/1000，突破后800/1500
- `overflow_pool` 不变——等下次突破后再转化

#### 7. 溢出属性丹自动发放

突破成功后立即执行：
```
overflow_pool → 属性丹：
  pill_count = floor(overflow_pool / 100)
  if pill_count > 0:
    发放 pill_count 个属性丹到背包
    overflow_pool -= pill_count × 100
```

属性丹是消耗品，在卡组编辑界面或背包中使用，属性丹的详细设计在炼丹炼器系统中展开。本系统仅负责运算和发放。

### 状态与转换

```
修为生命周期：
  cultivation ∈ [0, max_cultivation]
  0 → 增长 → 接近max → 满值 → 继续获取→溢出转化
                                           ↓
                                    突破成功 → 新上限，继续积累
                                    突破失败 → cult损失10%（由渡劫系统处理）
```

溢出池生命周期：
```
  overflow_pool >= 0
  0 → 修为满后继续获取 → 增加
  突破成功 → 结算（转化部分→属性丹，剩余保留）
```

### 与其他系统的交互

| 系统 | 数据流入（本系统→目标） | 数据流出（目标→本系统） |
|------|----------------------|---------------------|
| **游戏状态管理器** | `add_cultivation()` 修为读写；overflow_pool 读写 | `cultivation_changed` 信号；`cultivation_full` 信号 |
| **境界系统** | 当前修为值（是否满足突破条件） | 境界等级（用于计算 max_cultivation） |
| **渡劫突破系统** | 突破后更新修为上限；转化溢出修为 | 突破成功/失败事件 |
| **战斗系统** | 战斗胜利奖励修为 | 战斗结果+敌方阵容数据 |
| **事件系统** | 事件结果增加修为 | 事件选择导致的修为变更 |
| **资源系统/炼丹系统** | 丹药增加修为 | 丹药使用事件 |
| **UI系统** | 修为值/上限/溢出值的展示 | — |

## 公式

### 1. 修为获取（含溢出）

```
gain_cultivation(current, max, overflow_pool, amount):
  new_current = min(current + amount, max)
  if current + amount > max:
    raw_overflow = current + amount - max
    overflow_pool += raw_overflow × CONVERSION_RATE
  return {cultivation: new_current, overflow_pool, actual_gain: new_current - current}
```

| 变量 | 类型 | 范围 | 描述 |
|------|------|------|------|
| CONVERSION_RATE | float | 0.0~1.0 | 溢出转化率（默认1.0=100%） |
| raw_overflow | int | [0, max] | 超出上限的原始修为值 |
| overflow_pool | int | [0, ∞) | 累计溢出修为 |

### 2. 溢出→属性丹结算

```
pill_count(overflow_pool) = floor(overflow_pool / PILL_CONVERSION_UNIT)
remaining_overflow = overflow_pool mod PILL_CONVERSION_UNIT
```

| 变量 | 类型 | 范围 | 描述 |
|------|------|------|------|
| PILL_CONVERSION_UNIT | int | 固定值 | 每枚属性丹所需溢出修为（默认100） |
| pill_count | int | [0, ∞) | 本次结算获得的属性丹数量 |

**验证：** overflow_pool=450 → pill_count=4，剩余50保留到下次结算

## 边界情况

- **修为满值后继续战斗**：修为不再增长，全部转化为溢出修为。战斗显示修为获取值时显示"(溢出)"
- **突破失败修为损失10%**：由渡劫突破系统处理，修为系统只提供当前值读取接口
- **突破成功后溢出池归零？** 不归零——溢出池中的修为只在突破时结算一次，未转化的部分保留到下次突破
- **不同境界的上限随突破动态变化**：max_cultivation 在突破后立即更新，旧上限到新上限之间的差值（如1000→1500）对已有修为无影响
- **丹药使用后修为溢出**：走标准溢出流程，正常转入溢出池
- **事件获取的修为为0**：不触发任何变化，不发射 cultivation_changed 信号（无变更）
- **溢出修为被赋予负值**：不可能——溢出池只增不减（只有结算时按整单位扣除）
- **多途径同时获取修为**：每途径独立调用 gain_cultivation，GSM 同帧去重规则保证 cultivation_changed 信号仅广播最终值
- **超额溢出池的大数问题**：overflow_pool 无上限，但实际游戏中每局不可能超过 10000（因为溢出需要满修为后继续获取，而一局时间有限）

## 调优参数

| 参数 | 默认值 | 安全范围 | 过低影响 | 过高影响 |
|------|:-----:|:--------:|---------|---------|
| BASE_MAX（修为上限基准） | 1000 | 500-3000 | 炼气期太快突破 | 炼气期太久卡住 |
| 战斗修为奖励（普通） | 30~60 | 20~100 | 练级太慢无动力 | 修为太快溢出失去意义 |
| 战斗修为奖励（Boss） | 200~500 | 100~800 | Boss战成就感低 | 一个Boss升一级 |
| 溢出转化率 | 100% | 50%-100% | 溢出收益太低不如不溢出 | — |
| 属性丹兑换比 | 100:1 | 50:1-200:1 | 溢出修为反馈太强 | 溢出修为反馈太弱 |
| 溢出属性丹类型 | 攻击/防御/HP自选 | 固定/自选 | 灵活性差 | 玩家选择困难 |
| 突破失败修为损失 | -10% | -5%~-20% | 失败无代价 | 惩罚太重不敢突破 |

## 验收标准

- **GIVEN** 炼气期修为800/1000，**WHEN** 获取+200修为，**THEN** 修为变为1000/1000，溢出100转入溢出池
- **GIVEN** 炼气期修为500/1000，**WHEN** 获取+30修为，**THEN** 修为变为530/1000，无溢出
- **GIVEN** 修为已满且溢出池=450，**WHEN** 突破成功，**THEN** 结算获得4个属性丹，溢出池剩余50
- **GIVEN** 修为已满，**WHEN** 继续获取修为，**THEN** 全部走溢出转化
- **GIVEN** 筑基期修为800/1500，**WHEN** 突破到金丹（max=2250），**THEN** 修为变为800/2250（旧值保留）
- **GIVEN** cultivation_changed 信号被监听，**WHEN** 修为变更，**THEN** 收到含 delta/current/max 的载荷
- **GIVEN** 修为满值，**WHEN** 检查 cultivation_full 信号，**THEN** 信号正确发射
- **GIVEN** 一次性获取0修为，**WHEN** 调用 gain_cultivation(0)，**THEN** 不变，不发射信号
