# 卡组编辑系统 (Deck Editing System)

> **状态 (Status)**：设计中 (In Design)
> **作者 (Author)**：Claude Code + 用户
> **最后更新 (Last Updated)**：2026-07-22
> **最后验证 (Last Verified)**：—
> **实现的支柱 (Implements Pillar)**：支柱1「自由组牌，策略为王」、支柱3「机缘巧合，意外之喜」

## 概述

卡组编辑系统管理玩家在本局运行中卡组的构建与变更全过程——每一次卡牌的获得、删除、交换都通过具体的游戏行为完成，不存在独立的「自由编辑界面」。系统覆盖四种卡牌变更渠道：**战斗战利品三选一加入卡组**、**商店购买/删除卡牌**、**事件选择获得/失去卡牌**、以及**开局身份固定初始卡组**。

对玩家来说，这套系统意味着「每一张卡牌都是一次决策的结果」——开局选择身份决定了初始工具包，战斗后的三选一要求判断「这张牌值不值得加入我的构筑」，商店中花灵石删卡的每一次点击都是对卡组精炼的投资。没有免费的优化，每一次调整都有代价或机会成本。这正是 Roguelike 卡牌的核心乐趣：在限制中做选择，在选择中形成属于自己的独特构筑。

> **⚠ 设计冲突说明**：card-system.md 原有「卡组编辑在营寨中进行」「卡组容量下限30张」「可保存多套预设」等规则——这些规则基于自由编辑模型，与本系统的 StS 式渐进构筑模型冲突。本系统以用户确认的新规则为准。

## 玩家幻想

卡组编辑在此模型下对应的不是「修者锦囊」的精挑细选，而是 **「修仙路上的积累与取舍」**——你携带的每一张卡牌都是在旅途中获得并决定留下来的。卡组不是你在出发前精心打包的行囊，而是你在历经磨难后自然而然积累的家底。

情感节奏：

| 阶段 | 情感 | 触发场景 |
|------|------|---------|
| **开局** | 简朴的出身感 | 看到初始卡组中的几张基础卡——「就靠这些开始了」 |
| **战利品三选一** | 战术判断的快感 | 三张混合奖励——「这张卡能让我流派成型，但灵石可以马上去商店删几张废牌…」 |
| **商店删卡** | 心痛的优化感 | 花灵石删掉一张曾经帮过你的卡——「是时候放手了」 |
| **核心牌入手** | 豁然开朗 | 终于拿到流派最后一块拼图——「这套齐了！」 |
| **卡组超过40张** | 被迫的精炼 | 看到提示「已达上限」——「得去商店删几张了」 |
| **Boss战前看卡组** | 审视与自信 | 打开卡组查看界面，回顾自己一路积累的成果——「这些够用了」 |

核心情感不是「精炼的自信」，而是 **「机缘的积累感」**——你卡组中的每一张卡都有来处，都是你在一局游戏中某个时刻做出的选择。卡组本身就是你的冒险日志。

## 详细设计

### 核心规则

#### 1. 初始卡组

开局选择身份时，每个身份绑定一套固定的初始卡组：

| 开局身份 | 初始卡组内容 | 角色位 |
|---------|-------------|:------:|
| 七玄门外门弟子 | 长春功×2、基础攻击×3、简易护盾×2 | 韩立（炼气） |
| 魔修遗孤 | 玄阴经×2、魔气斩×3、血祭术×1、基础攻击×2 | 魔修护卫（炼气） |
| 散修路人 | 随机蓝色卡×3、随机紫色卡×1、基础攻击×4 | 随机散修角色（炼气） |

初始卡组在开局时写入 `deck.current_deck`，开局后不可更改初始配置。

#### 2. 战斗战利品——混合三选一

每次战斗胜利后触发战利品选择流程。**三个选项中不应全部是卡牌**——灵石是混合奖励池中的常备选项：

```
battle_reward_selection(enemy_type, is_elite, is_boss, player_state):
  ① 按战斗难度和敌人类型决定选项构成模式：
     普通战斗 → 模式A「2卡+1灵石」/ 模式B「1卡+2灵石」/ 模式C「1卡+1灵石+1消耗品」
     精英战斗 → 模式A「2卡+1灵石（大量）」/ 模式B「3卡」
     Boss战斗 → 模式A「2卡（含保底紫+）+1灵石」/ 模式B「3卡（含保底金）」
  ② 卡牌选项：按卡牌系统掉落规则生成（不因同局已有卡做去重）
  ③ 灵石选项：固定数值，显示为「灵石 × N」+ 灵石图标
  ④ 消耗品选项（仅限特定模式C）：恢复血量 X 点 / 回复行动力1点 / 临时减伤等
  ⑤ UI 展示3个选项（卡牌以正面展示，灵石/消耗品以图标+数字展示）
  ⑥ 玩家选择其中1项，其余消失
  ⑦ 如果卡组已达40张上限，卡牌选项标注「卡组已满」（灰色不可选），玩家仅能在灵石/消耗品中选择
```

**灵石奖励基准值（待经济系统确认）：**

| 战斗类型 | 灵石量范围 | 说明 |
|---------|:---------:|------|
| 普通战斗（杂兵） | 10~25 | 走模式A时~15，模式B时两堆~10+~20 |
| 精英战斗 | 40~80 | 一张卡+大量灵石或2卡+少量灵石 |
| Boss战斗 | 100~200 | 高品质卡+Boss额外赏金 |

**展示规则：**
- 卡牌选项：正面展示（卡名、费用、类型、稀有度、效果简述），可悬浮查看详情
- 灵石选项：灵石堆图标 + 数字（如「灵石 × 25」），点击即确认
- 消耗品选项：对应图标 + 效果简述（如「恢复15% HP」）
- NEW标识：卡牌左上角「新卡」标签（仅首次获得的卡牌）

**没有收藏池暂存机制**——选择一项即生效，未被选的选项消失。不存在「先收着以后再用」的设计。

**不选择灵石/消耗品即永久失去该机会**——错过的灵石不会累积到下次战利品。

#### 3. 商店买卖与删卡

商店界面提供三种卡牌相关操作：

**① 购买卡牌：**
- 商店展示3-6张随机卡牌，每张标价（按卡牌系统拆解产出的 2~3 倍定价）
- 玩家选择一张→消耗灵石→卡牌加入 `deck.current_deck`
- 灵石不足时，该卡牌显示为灰色并标注「灵石不足」
- 卡组已达40张时，所有卡牌标注「卡组已满」不可购买

**② 删除卡牌：**
- 玩家支付灵石从卡组中永久移除一张卡牌
- 删卡费用 = `50 + 25 × [本局已删除次数]`
- 费用展示：「删除费用：50灵石」（首次）或「删除费用：75灵石」（第2次），依次类推
- 费用无上限（但一局游戏通常不会超过10次删卡）
- 灵石不足时不可操作
- 卡组仅剩5张以下时，删除按钮禁用并提示「卡组太少，无法继续删除」

**③ 出售卡牌：**
- 玩家可将卡组中的卡牌出售换取灵石
- 售价 = 卡牌系统拆解产出公式计算的值 × 0.8（商店抽成）
- 出售后卡牌从卡组永久移除
- 卡组仅剩5张以下时，出售按钮禁用

> **卡组最低张数保护**：卡组任何时候不得低于5张（防止删/售光卡牌导致无法战斗）。商店操作、事件销毁都受此限制保护。

#### 4. 事件增删卡牌

事件系统提供以下卡牌变更类型：

**获得卡牌：**
- 洞府奇遇：3选1奖励（其中可能包含卡牌选项）
- 斜月三星洞：固定奖励一张高品质卡牌（直接加入卡组）
- 坊市随机事件：免费获得一张随机卡牌
- 杀人夺宝胜利：从对方身上获得1-2张随机卡牌（无视卡组上限——上限检查在事件结算后执行，若超限则玩家需选择弃牌至40张以内）

**失去卡牌：**
- 杀人夺宝失败：随机失去1张卡牌（被对方夺走）
- 特定剧情事件：为推进剧情需要交出某张卡
- 陷阱事件：随机销毁一张丹药或符箓
- 渡劫准备事件：可选择「牺牲一张卡牌换取渡劫加成」

**事件导致的超限处理：**
如果事件结算后卡牌超过40张：
```
if deck.current_deck.size() > 40:
  暂停 → 弹窗展示「卡组超限，请弃牌至40张」
  玩家选择要丢弃的卡牌（多选，弃到40张为止）
  被选中的卡牌从卡组移除（不产生任何补偿）
  确认后继续游戏
```
这是一种负面反馈——玩家应尽量通过商店删卡避免此情况。超限弃牌是惩罚性的（无灵石补偿）。

#### 5. 卡组查看界面（只读）

玩家可在任何时候打开「查看卡组」界面，功能严格限制为**只读浏览**：

| 功能 | 支持 | 说明 |
|------|:----:|------|
| 浏览全部卡牌 | ✅ | 按类型分组展示（角色/功法/法宝/阵法/丹药/符箓） |
| 按稀有度筛选 | ✅ | 快速筛选白/蓝/紫/金/暗金 |
| 按费用排序 | ✅ | 升序/降序 |
| 卡牌详情弹窗 | ✅ | 点击卡牌查看大尺寸详情 |
| 总张数统计 | ✅ | 「当前卡组：35张 / 上限40张」 |
| 暗金卡计数 | ✅ | 「暗金卡：1/2」 |
| 流派标签统计 | ✅ | 展示当前卡组的流派标签分布 |
| 拖拽调整顺序 | ❌ | 纯浏览，不可调整 |
| 移除卡牌 | ❌ | 不可直接移除 |
| 添加卡牌 | ❌ | 不可直接添加 |
| 保存预设 | ❌ | 不支持多套预设 |

查看界面在以下场景可打开：
- 探索地图中（点击卡组图标）
- 商店中（在购买/删除前查看当前卡组）
- 战利品界面中（在选择前查看当前卡组）
- 战斗中（查看抽牌堆剩余卡牌）

#### 6. 角色位管理

角色位单独管理，与30-40张卡组隔离：

- 6个角色位，开局身份决定初始角色
- 角色位中的角色卡不在卡组30-40张计数内
- 获得新角色卡的方式：
  - **事件获得**：特定事件奖励角色卡（如「救下李化元→获得角色卡李化元」）
  - **商店购买**：偶有角色卡出售
  - **渡劫奖励**：突破大境界时奖励对应角色卡
- 角色位满6个后获得新角色卡时：提示选择替换——新角色替换旧角色，被替换的角色卡及所有绑定卡进入弃牌堆
- 角色位中的角色可在查看界面浏览，不可在查看界面上调整

#### 7. 卡组上限40张的硬约束

卡牌系统定义的40张上限是硬约束，在以下场景强制执行：

| 场景 | 上限规则 |
|------|---------|
| 战利品三选一 | 已达40张时不可选（仅能跳过） |
| 商店购买 | 已达40张时不可购买 |
| 事件获得 | 超限→触发弃牌流程至40张 |
| 开局初始卡组 | 不受上限约束（初始卡组通常5-10张） |

#### 8. 卡组变更历史

系统在内存中维护一个轻量的变更日志（不保存到存档，仅运行时用于展示）：

```
DeckChangeLog {
  entries: [
    {turn: int, source: enum, card_id: String, action: "add"|"remove", detail: String}
  ]
}
```

用于卡组查看界面的「历史」标签页——玩家可查看本局卡组的变更历史（什么时候获得了什么卡、什么时候删除了什么卡）。这是情感化的设计：让玩家看到卡组的形成过程。

日志每局重置，不清除到存档。如果存档读档，日志重新从空开始（不影响游戏玩法）。

### 状态与转换

```
卡组状态生命周期（一局内）：

开局 → 初始卡组确定（5-10张）
  ↓
探索循环：
  战斗胜利 → 三选一（+0~1张，受40张上限约束）
  商店 → 购买（+1张）| 删除（-1张）| 出售（-1张）
  事件 → 获得（+N张）| 失去（-N张）
  超限 → 弃牌流程（强制弃至≤40张）
  ↓
渡劫突破 → 可能有卡牌奖励/损失
  ↓
继续探索 → 循环
  ↓
结局（通关/死亡）→ 卡组数据随存档保留
```

**变更渠道汇总：**

```
三选一:    deck + [0..1] | 灵石 +N   (战利品——可选卡牌或灵石)
购买:      deck + 1        (商店)
删除:      deck - 1        (商店·付费)
出售:      deck - 1        (商店·获得灵石)
事件获得:  deck + N        (多种事件)
事件失去:  deck - N        (多种事件)
超限弃牌:  deck - N        (事件过量后的惩罚)
```

### 与其他系统的交互

| 系统 | 数据流入（本系统→目标） | 数据流出（目标→本系统） |
|------|----------------------|---------------------|
| **卡牌系统** | 卡组配置数据 `deck.current_deck` | 卡牌模板数据、掉落规则、拆解产出公式 |
| **游戏状态管理器** | 写入 `deck.current_deck`、`deck.slots` | 读取 `collection.owned_cards`（仅用于查看）；`deck_modified`信号 |
| **战斗系统** | 战斗胜利后触发三选一流程的请求 | 战利品候选卡列表 |
| **资源系统** | 购买卡消耗灵石、出售卡获得灵石、删卡消耗灵石 | 灵石余额 |
| **事件系统** | 事件结算时增删卡牌的接口 | 事件选择结果 |
| **存档系统** | 卡组数据需要被序列化保存/恢复 | 存档/读档指令 |

## 公式

### 1. 卡组上限验证

```
can_add_to_deck(deck, card_count=1) → {allowed: bool, reason: String}
  if deck.current_deck.size() + card_count > 40:
    return {allowed: false, reason: "卡组已达上限（40张）"}
  return {allowed: true, reason: ""}
```

### 2. 商店删卡费用

```
removal_cost(session_remove_count) → int
  return 50 + 25 × session_remove_count
```

| 变量 | 类型 | 范围 | 描述 |
|------|------|------|------|
| session_remove_count | int | [0, ∞) | 本局中已删卡次数（含本次） |
| 返回值 | int | [50, ∞) | 本次删卡所需灵石 |

示例：第1次删卡50灵石，第2次75，第3次100，第4次125...

### 3. 商店卡牌售价

```
sell_price(card) → int
  base = disassemble_value(card)  # 从卡牌系统获取拆解产出
  return floor(base × 0.8)
```

| 变量 | 类型 | 范围 | 描述 |
|------|------|------|------|
| base | int | [10, 2095] | 卡牌拆解基准灵石值 |
| 返回值 | int | [8, 1676] | 商店收购价（80%折价） |

### 4. 卡组最低张数保护

```
can_remove_from_deck(deck, remove_count=1) → {allowed: bool, reason: String}
  if deck.current_deck.size() - remove_count < 5:
    return {allowed: false, reason: "卡组至少保留5张"}
  return {allowed: true, reason: ""}
```

## 边界情况

- **三选一选了的卡发现后悔了**：不支持撤销——选了就是选了，卡牌已加入卡组。这是决策的权重所在
- **三选一跳过**：跳过的卡不会进入任何暂存区。不会在下一次战利品中再次出现
- **卡组刚开局就超过40张**：不可能——初始卡组固定5-10张，远低于40张上限
- **商店删卡费用过高导致玩家无法继续删卡**：设计上允许——如果玩家已经删了10+次（费用325灵石/次），删卡成本已成为战略决策的一部分。玩家可以选择出售卡牌而非删除
- **事件导致卡牌超限后强迫弃牌**：弃牌选择界面不带撤销。玩家必须在所有卡牌中选择要丢弃的，直到≤40张
- **弃牌时选择金色/暗金卡**：允许——超限弃牌不加任何保护。这是对「控制卡组规模失败」的惩罚
- **卡组只剩5张时商店操作**：删卡和出售按钮禁用。事件销毁同样受保护——如果事件要销毁卡牌但卡牌≤5张，事件文本变更为替代选项
- **角色位满6个后获得新角色**：提示「角色位已满，选择一个角色替换」。被替换角色及其绑定全部进入弃牌堆。如果玩家取消，该角色卡不获得（事件触发中则选择「放弃」）
- **战斗中查看卡组**：只显示抽牌堆剩余卡牌和已使用的弃牌堆，不显示全部卡组（避免玩家在战斗中分心查看完整卡组）
- **开局初始卡组中的同名多张卡**：允许（如长春功×2），卡牌系统已支持同名多张
- **读档后卡组变更日志清空**：预期行为——日志只是当前运行的轻量情感化功能，不影响玩法
- **同时从多个渠道获得卡牌（如战斗+事件同一帧结算）**：各渠道独立处理加入验证。如果战斗三选一已使卡组达到40张，事件结算时即使有「获得卡牌」结果，也立即触发超限弃牌流程

## 依赖关系

| 依赖系统 | 性质 | 说明 |
|----------|------|------|
| **卡牌系统** | 硬依赖 | 卡牌数据模型、稀有度枚举、掉落规则、拆解产出公式 |
| **游戏状态管理器** | 硬依赖 | 卡组状态读写（`deck.current_deck`）、信号广播（`deck_modified`） |
| **战斗系统** | 硬依赖 | 战斗胜利后触发三选一流程；提供战利品候选卡列表 |
| **资源系统** | 硬依赖 | 商店购买/出售卡牌时灵石变更；删卡时消耗灵石 |
| **事件系统** | 硬依赖 | 事件增删卡牌的接口调用 |
| **存档系统** | 硬依赖 | 卡组数据序列化/反序列化 |

### 上游依赖设计顺序

卡牌系统（已设计） → 战斗系统 + 资源系统 + 事件系统（并行设计） → 卡组编辑系统

> **⚠ 更新卡牌系统的交叉引用**：card-system.md 中「卡组构建规则」仍引用旧模型（营寨编辑、30张下限、多套预设）。这些规则已由本系统替代。需在 card-system.md 中标注「卡组构建规则由 deck-editing-system.md 定义」。

## 调优参数

| 参数 | 默认值 | 安全范围 | 过低影响 | 过高影响 |
|------|:-----:|:--------:|---------|---------|
| 卡组上限 | 40张 | 30-50 | 流派变化空间小 | 卡组松散难抽核心牌 |
| 初始卡组大小 | 5-10张 | 5-15 | 开局毫无选择 | 开局太强无收集感 |
| 战利品候选数 | 3张 | 2-4 | 选择太少 | 分析瘫痪 |
| 删卡基价 | 50灵石 | 30-100 | 删卡太便宜→随意删 | 删卡太贵→卡组臃肿 |
| 删卡递增 | +25/次 | +15~+50 | 删卡成本增长慢 | 几次后删不起 |
| 商店卡牌数量 | 3-6张 | 3-8 | 选择太少 | 商店太强成为主要来源 |
| 商店定价倍率 | 拆解价×2-3 | ×1.5-×5 | 买卡太便宜 | 买卡不如删卡 |
| 出售折价率 | 80% | 50%-90% | 出售太亏无人用 | 出售→买卡循环 |
| 卡组最低保护张数 | 5张 | 3-8 | 卡组可被清空 | 卡组太大无法处理废卡 |
| 超限弃牌惩罚 | 无补偿 | — | — | — |

## 验收标准

- **GIVEN** 开局选择七玄门外门弟子身份，**WHEN** 检查卡组，**THEN** 卡组包含长春功×2、基础攻击×3、简易护盾×2，角色位含韩立（炼气）
- **GIVEN** 战斗胜利，**WHEN** 进入战利品选择，**THEN** 展示3个混合奖励选项（卡牌/灵石/消耗品），至少包含1个灵石选项
- **GIVEN** 战利品选项包含卡牌和灵石，**WHEN** 玩家选择卡牌，**THEN** 该卡加入卡组，面板关闭
- **GIVEN** 战利品选项包含卡牌和灵石，**WHEN** 玩家选择灵石，**THEN** 灵石增加对应数值，面板关闭
- **GIVEN** 战利品选项包含灵石，**WHEN** 玩家选择灵石（如25灵石），**THEN** 面板关闭，下次战斗重新生成奖励
- **GIVEN** 卡组已达40张且战利品有卡牌选项，**WHEN** 战利品展示，**THEN** 卡牌选项标记「卡组已满」不可选，但灵石/消耗品仍可选
- **GIVEN** 卡组已达40张，**WHEN** 尝试商店购买，**THEN** 不可购买，提示「卡组已满」
- **GIVEN** 玩家在商店选择删卡，**WHEN** 支付50灵石（首次），**THEN** 卡组减少1张，灵石减少50
- **GIVEN** 玩家第2次在同一局商店删卡，**WHEN** 支付，**THEN** 费用为75灵石
- **GIVEN** 玩家在商店出售一张白色卡，**WHEN** 确认，**THEN** 卡组减少1张，灵石增加8（10×0.8）
- **GIVEN** 卡组仅有5张卡，**WHEN** 尝试商店删卡/出售，**THEN** 按钮禁用，提示「卡组至少保留5张」
- **GIVEN** 卡组35张，**WHEN** 事件获得一次增加多张卡导致超过40张，**THEN** 触发超限弃牌流程，弃至40张
- **GIVEN** 超限弃牌流程中，**WHEN** 玩家选择要弃的卡牌并确认，**THEN** 卡牌移除，无灵石补偿
- **GIVEN** 玩家在任何非战斗场景，**WHEN** 点击卡组图标，**THEN** 打开只读卡组查看界面
- **GIVEN** 卡组查看界面，**WHEN** 尝试拖拽或点击移除卡牌，**THEN** 无反应（只读）
- **GIVEN** 角色位已满6个，**WHEN** 获得新角色卡，**THEN** 弹出角色替换弹窗
- **GIVEN** 角色替换弹窗，**WHEN** 玩家选择被替换角色并确认，**THEN** 新角色加入，旧角色及绑定卡进入弃牌堆
- **GIVEN** 卡组查看界面的「历史」标签，**WHEN** 打开，**THEN** 展示本局卡牌变更时间线
- **GIVEN** 读档后打开卡组查看，**WHEN** 查看「历史」，**THEN** 日志为空（每局重置）
