## 已识别差距

### 差距 1：设计文档需要标准化迁移
两份设计文件目前位于项目根目录，标准要求放在 `design/gdd/`。同时缺少关键的结构化文档：
- **`game-concept.md`** — 一句话宣传语、核心愿景
- **`game-pillars.md`** — 三大支柱定义（自由组牌 + 苟道成长 + 原作剧情）
- **`systems-index.md`** — 系统分解与映射

**问题**：应将现有设计迁移到标准格式，确保后续技能能正确读取？推荐 `/adopt`。

### 差距 2：游戏系统尚未正式分解
设计中有丰富的子系统（探索、战斗、养成、炼丹炼器、渡劫突破），但每个系统没有独立的 GDD 文档（必需8个章节）。当前是单个大文档混合了所有系统的设计。

**问题**：是否要运行 `/map-systems` 将概念分解为独立系统，然后逐个编写 GDD？

### 差距 3：引擎未配置
这是卡牌游戏，可选引擎包括 Godot (GDScript/C#) 或 Unity (C#)。当前未配置引擎。

**问题**：是否想先配置引擎？推荐 `/setup-engine` 确定技术栈。

### 差距 4：无骨架代码或架构
`src/` 完全为空，`docs/` 下无架构决策记录。

### 差距 5：无生产管理文件
无 sprint 计划、里程碑定义、路线图。

## 推荐后续步骤

按优先级排列：

1. **`/setup-engine`** → 配置引擎（影响后续所有技术决策）
2. **`/adopt`** → 将现有设计文件标准化迁移到 `design/gdd/`，审计格式合规性
3. **`/brainstorm`**（可选）→ 如果想把现有设计完善为结构化的游戏概念文档
4. **`/map-systems`** → 将完整设计分解为各独立系统
5. **`/design-system`** × N → 为每个系统编写标准 GDD
6. **`/review-all-gdds`** → 跨系统一致性检查
7. **`/gate-check`** → 准备就绪后进入技术搭建阶段
