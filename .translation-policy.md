# Translation Policy — English → Simplified Chinese (zh-CN)

This file is the single source of truth for translating this project's
documentation into Simplified Chinese. Every translation sub-agent MUST follow
it exactly.

## Goal

Produce a faithful Chinese mirror of the English documentation under
`docs/zh-CN/`. Each source file maps to
`docs/zh-CN/<original-relative-path>` (the leading `./` is dropped).

Examples:
- `README.md` → `docs/zh-CN/README.md`
- `docs/WORKFLOW-GUIDE.md` → `docs/zh-CN/docs/WORKFLOW-GUIDE.md`
- `docs/engine-reference/godot/modules/animation.md` → `docs/zh-CN/docs/engine-reference/godot/modules/animation.md`
- `design/CLAUDE.md` → `docs/zh-CN/design/CLAUDE.md`

## What to translate

- All prose: headings, paragraphs, bullet points, table cells that contain
  natural language.
- Lists, admonitions, callouts.

## What to keep VERBATIN (never translate)

- **Code blocks** (fenced ``` blocks and inline `code`). Translate only
  comments inside code IF they are natural-language comments AND doing so does
  not risk breaking the code. When in doubt, leave the code block untouched.
- **File paths** (e.g. `docs/architecture.md`, `production/session-state/active.md`).
- **Command / skill names** that start with `/` (e.g. `/start`, `/osk-spec`,
  `/gate-check`, `/dev-story`).
- **Agent names** (e.g. `creative-director`, `technical-director`,
  `gameplay-programmer`, `godot-gdscript-specialist`).
- **Engine / API identifiers**: class names, method names, signal names,
  property names, node names, shader functions, annotation names
  (e.g. `@export`, `Node3D`, `Input.is_action_pressed`, `Mathf.lerp`,
  `UGameplayAbility`, `MonoBehaviour`, `Addressables.LoadAssetAsync`).
- **CLI / tool invocations** (e.g. `godot --headless`, `git commit -m`,
  `open-spec-kit update`, `npm install -g .`).
- **URLs** and link targets (the `(url)` part of `[text](url)` — translate the
  link text, keep the URL).
- **YAML / frontmatter keys** and their machine-read values (e.g. `model:`,
  `name:`, `haiku`, `claude-sonnet-4-6`). Translate a frontmatter *description*
  value only if it is clearly prose.
- **Version numbers, status enums** (`accepted`, `proposed`, `open`,
  `merged`), **file extensions**, **environment variables**
  (e.g. `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`).
- **Acronyms** that are universally used as-is in Chinese technical writing
  (API, CPU, GPU, UI, HUD, ADR, GDD, CI/CD, SDK, ECS, DOTS, GAS, UMG, RPC,
  WASD, FPS, LOD, etc.) — keep in English, no parentheses needed.

## Terminology rule: Chinese + (English) on first use

For **proper nouns, role names, and domain-specific technical terms**, give the
first prominent occurrence in each file (typically in a heading or first
paragraph). Afterwards, the Chinese form alone is fine.

Examples:
- 创意总监 (Creative Director)
- 技术总监 (Technical Director)
- 玩法程序员 (Gameplay Programmer)
- 架构决策记录 (Architecture Decision Record, ADR)
- 游戏设计文档 (Game Design Document, GDD)
- 垂直切片 (Vertical Slice)
- 持续集成/持续交付 (CI/CD)
- 行为树 (Behavior Tree)
- 状态机 (State Machine)
- 寻路 (Pathfinding)
- 关卡设计 (Level Design)
- 经济系统 (Economy)
- 里程碑 (Milestone)
- 冲刺 (Sprint)
- 发布经理 (Release Manager)
- 质量保证 (Quality Assurance, QA)

For agent/skill identifiers used as command names (e.g. `creative-director`,
`/start`), keep the verbatim identifier — do NOT translate the identifier
itself. You may precede it with a Chinese gloss: 运行 `/start`（首次启动流程）。

## Functional config mirrors (agent / skill / hook definitions)

These mirrored files (`.claude/agents/*.md`, `.claude/skills/*/SKILL.md`,
`.claude/rules/*.md`, hooks, `CLAUDE.md`, `CCGS Skill Testing Framework/**`)
are documentation mirrors — the originals stay in place and remain the
functional source. Translate them with these extra rules:

- **YAML frontmatter**: keep ALL keys and machine-read values verbatim
  (`name`, `model`, `tools`, `allowed-tools`, `color`, `description`'s
  structural tags, etc.). The ONLY frontmatter value you may translate is a
  purely-prose `description:` string — and even then, keep any leading
  imperative/role phrasing recognizable. When in doubt, leave `description`
  in English too.
- **Tool lists** (e.g. `Read, Glob, Grep, Write, Edit, Bash`) — verbatim.
- **Model identifiers** (`claude-haiku-4-5-20251001`, `claude-sonnet-4-6`,
  `claude-opus-4-6`, `haiku`, `sonnet`, `opus`) — verbatim.
- **Skill trigger words / slash commands** (`/start`, `/dev-story`,
  `start spec`, etc.) — verbatim.
- **File-path references** inside prose (e.g. "see `.claude/docs/...`",
  "write to `production/session-state/active.md`") — verbatim.
- Translate the body prose (the agent's role description, responsibilities,
  when-to-use guidance, phase instructions) normally, applying the Chinese +
  (English) terminology rule for role names on first use per file.

## Style

- Use Simplified Chinese (简体中文), zh-CN conventions.
- Match the original tone (professional, instructional).
- Keep markdown structure identical: heading levels, lists, tables, blockquotes,
  bold/italic, links.
- Preserve all `@`-style includes (e.g. `@.claude/docs/technical-preferences.md`)
  verbatim — they are functional references.
- Preserve HTML comments (`<!-- ... -->`) and frontmatter fences.
- Do not add, remove, or reorder sections. Do not inject opinions.
- If a file is mostly a template with placeholders (e.g. `[TO BE CONFIGURED]`),
  translate the surrounding instructions and keep placeholders verbatim.
