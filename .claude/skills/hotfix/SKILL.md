> **仅限显式调用**：本 skill 仅在用户使用 `/hotfix` 显式请求时运行。不要根据上下文匹配自动调用。

## 第一阶段：评估严重性

读取缺陷描述或 ID。使用以下标准评估严重性：

- **S1（严重）**：游戏无法运行、数据丢失、安全漏洞
- **S2（重大）**：重要功能损坏，存在临时解决方案
- **S3 或更低**：小问题——适用常规缺陷修复工作流程

使用 `AskUserQuestion` 确认：
- 提示："我已评估为 **[评估的严重性]** — [简要理由]。确认严重性以继续："
- 选项：
  - `[A] S1（严重）——游戏无法运行、数据丢失或安全问题`
  - `[B] S2（重大）——重要功能损坏，存在临时解决方案`
  - `[C] S3 或更低——重定向到常规缺陷修复工作流程`

如果选择 [C]：停止。裁决：**已重定向**——S3 及更低级别使用常规缺陷修复工作流程。

---

## 第二阶段：创建热修复记录

草拟热修复记录：

```markdown
## 热修复：[简短描述]
日期：[日期]
严重性：[S1/S2]
报告人：[谁发现的]
状态：进行中

### 问题
[清晰描述损坏的内容及其对玩家的影响]

### 根本原因
[调查期间填写]

### 修复方案
[实施期间填写]

### 测试
[测试了哪些内容以及测试方式]

### 审批
- [ ] 首席程序员审核修复方案
- [ ] 回归测试通过（QA 测试员）
- [ ] 发布已批准（制作人）

### 回滚计划
[如果修复导致新问题，如何回滚]
```

询问："我可以将此写入 `production/hotfixes/hotfix-[日期]-[简短名称].md` 吗？"

如果同意，写入文件，必要时创建目录。

---

## 第三阶段：创建热修复分支

检查是否为 git 仓库：

`Bash: git rev-parse --is-inside-work-tree 2>/dev/null`

如果此命令失败或返回空：注明"不是 git 仓库——请手动创建分支。"并跳过分支创建。

如果检查通过，在创建分支前使用 `AskUserQuestion`：
- 提示："准备从 [base-ref] 创建热修复分支 'hotfix/[简短名称]' 吗？"
- 选项：
  - `[A] 是——创建分支`
  - `[B] 使用不同的基准引用——我会指定`
  - `[C] 跳过——我会自己创建分支`

仅当用户选择 [A] 时，才运行 `git checkout -b hotfix/[简短名称] [base-ref]`。如果选择 [B]：询问用户基准引用，然后使用该引用运行命令。如果选择 [C]：跳过分支创建，进入第四阶段。

---

## 第四阶段：调查和实施

专注于解决问题所需的最小变更。不要在热修复的同时进行重构、清理或添加功能。

通过运行受影响系统的针对性测试来验证修复。检查相邻系统是否存在回归问题。

使用根本原因、修复细节和测试结果更新热修复记录。

---

## 第五阶段：收集审批

使用 Task 工具并行请求签收：

- `subagent_type: lead-programmer` — 审查修复的正确性和副作用
- `subagent_type: qa-tester` — 对受影响的系统进行针对性回归测试
- `subagent_type: producer` — 批准部署时机和沟通计划

三者都必须返回 APPROVE 才能继续。如果任何一个返回 CONCERNS 或 REJECT，不要部署——先提出问题并解决。

---

## 第五阶段 B：QA 重新进入门控

审批通过后，确定部署热修复前所需的 QA 范围。通过 Task 生成 `qa-lead`，附带：
- 热修复描述和受影响的系统
- 第五阶段的回归测试结果
- 所有触及已更改文件的系统列表（使用 Grep 查找调用者）

向 qa-lead 询问：**完整的冒烟测试是否足够，还是此修复需要针对性的 QA 团队测试？**

应用裁决：
- **冒烟测试足够**——对热修复构建运行 `/smoke-check`。如果通过，进入第六阶段。
- **需要针对性 QA 测试**——运行 `/team-qa [受影响系统]`，范围限定为已更改的系统。如果 QA 返回 APPOVED 或 APPROVED WITH CONDITIONS，进入第六阶段。
- **需要完整 QA**——触及核心系统的 S1 修复可能需要完整的 `/team-qa sprint`。这会延迟部署，但能防止错误的补丁。

不要跳过此门控。一个导致其他问题的热修复比原始缺陷更糟糕。

---

## 第六阶段：更新缺陷状态并部署

如果存在原始缺陷文件，更新它：

```markdown
## 修复记录
**修复于**：hotfix/[分支名称] — [提交哈希或描述]
**修复日期**：[日期]
**状态**：已修复——待验证
```

在缺陷文件头部设置 `**状态**：已修复——待验证`。

输出部署摘要：

```
## 热修复准备部署：[简短名称]

**严重性**：[S1/S2]
**根本原因**：[一行]
**修复方案**：[一行]
**QA 门控**：[冒烟测试通过 / QA 团队批准]
**审批**：首席程序员 ✓ / QA 测试员 ✓ / 制作人 ✓
**回滚计划**：[来自第二阶段的记录]

合并到：发布分支 AND 开发分支
下一步：部署后运行 /bug-report verify [BUG-ID] 以确认修复生效
```

### 规则
- 热修复必须是对解决问题的最小变更——不清理、不重构
- 每个热修复在部署前必须有记录在案的回滚计划
- 热修复分支同时合并到发布分支 AND 开发分支
- 所有热修复需要在 48 小时内进行事后审查
- 如果修复复杂到需要超过 4 小时，升级到 `technical-director`

---

## 第七阶段：部署后验证

部署后，运行 `/bug-report verify [BUG-ID]` 以确认修复在已部署构建中解决了问题。

如果验证为已修复：运行 `/bug-report close [BUG-ID]` 以正式关闭。
如果仍然存在：热修复失败——立即重新打开，评估回滚，并升级。

使用 `/retrospective hotfix` 在 48 小时内安排事后审查。

使用 `AskUserQuestion`：
- 提示："热修复完成。下一步是什么？"
- 选项：
  - `[A] 运行 /smoke-check 以验证修复`
  - `[B] 运行 /patch-notes 以记录此热修复`
  - `[C] 在此处停止`
