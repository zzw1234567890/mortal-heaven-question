# 快速设计

这是**轻量级设计路径**，适用于不需要完整 GDD 的变更。通过 `/design-system` 进行的完整 GDD 编写是重量级路径。在实现工作量大约 4 小时以内的任务使用此技能 — 调优调整、次要行为调整、对现有系统的小幅添加，或小到不值得编写完整文档的独立功能。

**输出：** `design/quick-specs/[name]-[date].md`

**何时运行：** 当变更太小不适合 `/design-system`，但又重要到不能在没有书面理由的情况下实现时。

---

## 1. 分类变更

首先，读取参数并确定此变更属于哪个类别：

- **调优** — 更改现有系统中的数字或平衡值，无行为变更（最轻量级路径）。例如："将跳跃高度从 5 增加到 6 个单位"、"将敌人巡逻速度降低 10%"。
- **调整** — 对现有系统的小型行为变更，不引入新的状态、分支或系统。例如："使冲刺在第 1 帧无敌"、"允许连招取消进入翻滚"。
- **添加** — 向现有系统添加小型机制，可能引入 1-2 个新状态或交互。例如："为格挡机制添加招架窗口"、"为基本攻击添加蓄力变体"。
- **新增小型系统** — 一个足够小的独立功能，没有现有的 GDD，实现工作量大约在一周以内。例如："成就弹窗系统"、"简单的昼夜视觉循环"。

如果变更**不**符合这些类别 — 它引入了一个具有显著跨系统依赖的新系统、需要超过一周的实现时间、或从根本上改变了现有系统的核心规则 — 停止并重定向到 `/design-system`。

如果没有参数，请用户描述变更（纯文本提示），然后使用上述标准进行分类。

使用 `AskUserQuestion` 呈现推断的分类：
- 提示："我已将此分类为 **[推断类型]** — [简要原因]。是否正确？"
- 选项：
  - `[A] 是 — [推断类型]正确`
  - `[B] 调优 — 仅更改数字或平衡值`
  - `[C] 调整 — 对现有系统的小型行为变更`
  - `[D] 添加 — 向现有系统添加小型机制`
  - `[E] 新增小型系统 — 独立功能，工作在一周以内`
  - `[F] 这太大了 — 重定向我到 /design-system`

如果 [F]：停止。判决：**已重定向** — 请使用 `/design-system` 进行此变更。
否则：使用所选类型继续。

---

## 2. 上下文扫描

在起草任何内容之前，读取相关上下文：

- 搜索 `design/gdd/` 查找与此变更最相关的 GDD。读取此变更会影响的章节。
- 检查 `design/gdd/systems-index.md` 是否存在。如果存在，读取它以了解此系统在依赖图中的位置及其所属层级。如果不存在，注明"未找到系统索引 — 跳过依赖层级检查。"并继续。
- 检查 `design/quick-specs/` 中是否有任何涉及此系统的先前快速规格 — 避免与其矛盾。
- 如果是调优变更，同时检查 `assets/data/` 中保存相关数值的数据文件。

报告发现的内容："在 [path] 找到 GDD。相关章节：[章节名称]。未发现冲突的快速规格。"（或注明发现的任何冲突。）

## 3. 起草快速设计规格

使用适合变更类别的规格格式。

### 对于调优变更

生成一个表格：

```markdown
# 快速设计规格：[标题]

**类型**：调优
**系统**：[系统名称]
**GDD 引用**：`design/gdd/[filename].md` — 调优参数章节
**日期**：[今天]

## 变更

| 参数 | 旧值 | 新值 | 理由 |
|-----------|-----------|-----------|-----------|
| [param]   | [old]     | [new]     | [why]     |

## 调优参数映射

映射到 GDD 调优参数：[参数名称及其记录范围]。
新值[在/在边界上/超出]记录范围。
[如果超出：解释为什么应该扩展范围。]

## 验收标准

- [ ] [参数]从 `assets/data/[file]` 读取[新值]
- [ ] 在[特定上下文]中可观察到的行为差异
- [ ] [相关行为]无回归
```

### 对于调整和添加变更

```markdown
# 快速设计规格：[标题]

**类型**：[调整 / 添加]
**系统**：[系统名称]
**GDD 引用**：`design/gdd/[filename].md`
**日期**：[今天]

## 变更摘要

[1-2 句描述变更内容及原因。]

## 动机

[为什么需要此变更？它解决什么玩家体验问题？如适用，引用相关的 MDA 美学或玩家反馈。]

## 设计增量

当前 GDD 表述（引用 `design/gdd/[filename].md`，[章节]）：

> [相关规则或描述的确切引用]

本规格将其更改为：

[新规则或描述，以 GDD 详细规则章节相同的精度编写。程序员应能仅从此文本实现。]

## 新规则 / 数值

[替代内容的完整明确陈述。如果引入了新状态，列出它们。如果引入了新参数，定义其范围。]

## 受影响系统

| 系统 | 影响 | 所需操作 |
|--------|--------|-----------------|
| [system] | [how it is affected] | [update GDD / update data file / no action] |

## 验收标准

- [ ] [具体、可测试的标准 1]
- [ ] [具体、可测试的标准 2]
- [ ] [具体、可测试的标准 3]
- [ ] 无回归：[此变更不得破坏的原始行为]

## 需要更新 GDD？

[是 / 否]
[如果是：哪个文件、哪个章节以及更新内容。]
```

### 对于新增小型系统变更

使用精简的 GDD 结构。仅包含直接必要的章节 — 除非系统特别需要，否则跳过玩家幻想、完整公式和边界情况。

```markdown
# 快速设计规格：[标题]

**类型**：新增小型系统
**范围**：[1-2 句描述此系统做什么和不做什么]
**日期**：[今天]
**预估实现时间**：[小时]

## 概览

[新团队成员能理解的一段话。此系统做什么，何时激活，产生什么输出？]

## 核心规则

[系统的明确规则。使用编号列表表示顺序行为，使用要点列表表示条件。精确到程序员无需询问即可实现的程度。]

## 调优参数

| 参数 | 默认值 | 范围 | 类别 | 理由 |
|------|---------|-------|----------|-----------|
| [name] | [value] | [min–max] | [feel/curve/gate] | [why this default] |

所有值必须放在 `assets/data/[appropriate-file].json` 中，而非硬编码。

## 验收标准

- [ ] [功能标准：做正确的事]
- [ ] [功能标准：处理边界情况]
- [ ] [体验标准：手感正确 — 试玩验证的内容]
- [ ] [回归标准：不破坏相邻系统]

## 系统索引

此系统当前不在 `design/gdd/systems-index.md` 中。
[如果应添加：建议哪个层和优先级层级。]
[如果太小无需跟踪：说明"此系统低于系统索引跟踪阈值 — 快速规格已足够。"]
```

## 4. 批准和归档

向用户完整呈现草稿。然后使用 `AskUserQuestion`：
- 提示："这是快速设计规格草稿。您希望如何进行？"
- 选项：
  - `[A] 批准 — 按所示写入`
  - `[B] 修订 — 我将描述需要更改的内容`
  - `[C] 这变得太大了 — 重定向到 /design-system`

如果 [B]：收集请求的更改，修订草稿，并重新呈现此控件。
如果 [C]：停止。判决：**已重定向** — 请使用 `/design-system` 进行此变更。

如果 [A]：询问"我可以将此快速设计规格写入 `design/quick-specs/[kebab-case-title]-[YYYY-MM-DD].md` 吗？"

文件名中使用今天的日期。标题应为变更的短横线命名法描述（例如 `jump-height-tuning-2026-03-10`、`parry-window-addition-2026-03-10`）。

如果同意，创建 `design/quick-specs/` 目录（如果不存在），然后写入文件。

如果需要更新 GDD（在规格中标记），在写入快速规格后单独询问：

"此规格修改了[系统名称]中的规则。我可以更新 `design/gdd/[filename].md` — 具体来说是[章节名称]章节吗？"

在询问前显示将要更改的确切文本（旧 vs 新）。未经明确批准不得进行 GDD 编辑。

---

## 5. 交接

写入文件后，输出：

```
快速设计规格已写入：design/quick-specs/[filename].md
类型：[调优 / 调整 / 添加 / 新增小型系统]
系统：[系统名称]
GDD 更新：[需要 — 待批准 / 已应用 / 不需要]

下一步：此规格已准备好进行 `/story-readiness` 验证，然后再实现。在 story 的 GDD 引用字段中引用此规格。
```

### 流程说明

判决：**完成** — 快速设计规格已写入，准备实现。

快速设计规格按设计**跳过** `/design-review` 和 `/review-all-gdds`。它们适用于小型、低风险、范围明确的变更，其中完整审查流程的成本超过了变更本身的风险。

如果以下任一条件成立，重定向到完整流程：
- 变更添加了一个应属于系统索引的新系统
- 变更显著改变了跨系统行为或系统与其他系统的契约
- 变更引入了影响游戏 MDA 美学平衡的面向玩家新机制
- 实现可能超过一周的工作量

在这些情况下："此变更已超出快速规格范围。我建议使用 `/design-system` 为此编写完整的 GDD。"

---

## 推荐后续步骤

- 运行 `/story-readiness [story-path]` 在实现开始前验证 story — 在 story 的 GDD 引用字段中引用此规格
- 运行 `/dev-story [story-path]` 在 story 通过就绪检查后实现
- 如果变更比预期更大，运行 `/design-system [system-name]` 编写完整的 GDD
