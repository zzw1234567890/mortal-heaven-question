# 传播设计变更

当 GDD 发生变化时，基于其编写的架构决策可能不再有效。此技能会查找所有受影响的 ADR，将 ADR 的假设与 GDD 当前的内容进行比较，并引导用户进行解决。

**用法：** `/propagate-design-change design/gdd/combat-system.md`

---

## 1. 验证参数

GDD 路径参数是**必需的**。如果缺失，则失败并返回：
> "用法：`/propagate-design-change design/gdd/[system].md`
> 提供已变更 GDD 的路径。"

验证文件是否存在。如果不存在，则失败并返回：
> "[path] 未找到。请检查路径后重试。"

---

## 2. 读取已变更的 GDD

完整读取当前的 GDD。

---

## 3. 读取上一版本

运行 git 获取上一个已提交版本：

```bash
git show HEAD:design/gdd/[filename].md
```

如果文件没有 git 历史记录（新文件），报告：
> "git 中无上一版本 — 这似乎是一个新的 GDD，而非修订。
> 无需传播。"

如果 git 返回了上一版本，进行概念性差异分析：
- 识别已变更的章节（新规则、删除的规则、修改的公式、变更的验收标准、变更的调优参数）
- 识别未变更的章节
- 生成变更摘要：

```
## 变更摘要：[GDD 文件名]
修订日期：[今天]

已变更章节：
- [章节名称]：[变更内容 — 新规则、删除规则、公式修改等]

未变更章节：
- [章节名称]

影响架构的关键变更：
- [变更 1 — 可能影响 ADR]
- [变更 2]
```

---

## 4. 加载架构输入

读取 `docs/architecture/` 中的所有 ADR：
- 对于每个 ADR，读取完整文件
- 提取"GDD 需求对应表"
- 记录每个 ADR 引用的 GDD 文档和需求 ID

读取 `docs/architecture/architecture-traceability.md`（如果存在）。

报告："已加载 [N] 个 ADR。[M] 个引用了 [gdd filename]。"

---

## 5. 影响分析

对于每个引用了已变更 GDD 的 ADR：

将 ADR 的"GDD 需求对应表"条目与 GDD 的已变更章节进行比较。对于每个引用的需求：

1. **在当前 GDD 中定位该需求** — 它是否仍然存在？
2. **比较**：ADR 编写时 GDD 的内容 vs 现在 GDD 的内容？
3. **评估 ADR 决策**：该架构决策是否仍然有效？

将每个受影响的 ADR 分类为以下之一：

| 状态 | 含义 |
|--------|---------|
| ✅ **仍然有效** | GDD 变更不影响此 ADR 的决策 |
| ⚠️ **需要审查** | GDD 变更可能影响此 ADR — 需要人工判断 |
| 🔴 **可能已被取代** | GDD 变更直接与此 ADR 的假设相矛盾 |

对于每个受影响的 ADR，生成影响条目：

```
### ADR-NNNN：[标题]
状态：[仍然有效 / 需要审查 / 可能已被取代]

ADR 对此 GDD 的假设：
  "[来自 ADR 的 GDD 需求对应表章节的相关引用]"

GDD 现在的内容：
  "[来自当前 GDD 的相关引用]"

评估：
  [解释 ADR 决策是否仍然有效及其原因]

推荐操作：
  [保持现状 / 审查并更新 / 标记为已取代并编写新的 ADR]
```

---

## 6. 呈现影响报告

在要求采取任何行动之前，向用户呈现完整的影响报告。格式：

```
## 设计变更影响报告
GDD：[文件名]
日期：[今天]
检测到的变更：[N 个章节变更]
引用此 GDD 的 ADR：[M] 个

### 未受影响
[引用此 GDD 但决策仍然有效的 ADR]

### 需要审查（[数量]）
[可能需要更新的 ADR]

### 可能已被取代（[数量]）
[其假设现已被矛盾的 ADR]
```

---

## 6b. 总监关卡 — 技术影响审查

**审查模式检查** — 在生成 TD-CHANGE-IMPACT 前应用：
- `solo` → 跳过。备注："TD-CHANGE-IMPACT 已跳过 — Solo 模式。"继续执行 Phase 7。
- `lean` → 跳过。备注："TD-CHANGE-IMPACT 已跳过 — Lean 模式。"继续执行 Phase 7。
- `full` → 正常生成。

通过 Task 使用关卡 **TD-CHANGE-IMPACT** (`.claude/docs/director-gates.md`) 生成 `technical-director`。

传入：来自 Phase 6 的完整设计变更影响报告（变更摘要、所有受影响的 ADR 及其仍然有效/需要审查/可能已被取代的分类，以及推荐操作）。

技术总监审查以下内容：
- 影响分类是否正确（没有 ADR 被低估）
- 推荐操作在架构上是否合理
- 是否遗漏了对其他 ADR 或系统的级联影响

应用判决：
- **批准** → 进入 Phase 7 解决工作流
- **关注** → 指出被标记的特定 ADR 或建议；使用 `AskUserQuestion` 提供选项：`修订影响评估` / `接受并记录关注点` / `进一步讨论`
- **拒绝** → 不进入解决阶段；在继续之前重新分析影响

---

## 7. 解决工作流

对于每个标记为"需要审查"或"可能已被取代"的 ADR，询问用户如何处理：

逐个询问每个 ADR：
> "ADR-NNNN（[标题]）— [状态]。您想怎么处理？"
> 选项：
> - "标记为已取代（我将编写新的 ADR）" — 将 ADR 状态行更新为 `Superseded by：[pending]`
> - "原地更新（小修订）" — 打开 ADR 进行编辑；注明需修改的内容
> - "保持现状（变更实际上不影响此决策）"
> - "暂时跳过（稍后重新审视）"

对于标记为**已取代**的 ADR：
- 更新 ADR 的状态字段：`Superseded by ADR-[next number]（pending — see change-impact-[date]-[system].md）`
- 询问："我可以更新 [ADR filename] 中的状态吗？"

---

## 8. 更新可追溯性索引

如果 `docs/architecture/architecture-traceability.md` 存在：
- 在"已取代的需求"表中添加已变更的 GDD 需求：

```markdown
## 已取代的需求
| 日期 | GDD | 需求 | 变更为 | 受影响的 ADR | 解决方式 |
|------|-----|-------------|------------|---------------|------------|
| [date] | [gdd] | [旧需求文本] | [新需求文本] | ADR-NNNN | [已取代/已更新/有效] |
```

询问："我可以更新可追溯性索引吗？"

---

## 9. 输出变更影响文档

询问："我可以将变更影响报告写入 `docs/architecture/change-impact-[date]-[system-slug].md` 吗？"

文档包含：
- 第 3 步的变更摘要
- 第 5 步的完整影响分析
- 第 7 步做出的解决决策
- 需要编写或更新的 ADR 列表

如果用户批准：判决：**完成** — 变更影响报告已保存。
如果用户拒绝：判决：**阻塞** — 用户拒绝写入。

---

## 10. 后续操作

根据解决决策，建议：

- **标记为已取代的 ADR**："运行 `/architecture-decision [title]` 编写替代 ADR。然后重新运行 `/propagate-design-change` 以验证覆盖范围。"
- **需要原地更新的 ADR**：列出每个 ADR 中需要更新的特定字段
- **如果许多 ADR 受影响**："在所有 ADR 更新后运行 `/architecture-review`，以验证完整的可追溯性矩阵仍然一致。"

---

## 协作协议

1. **静默读取** — 在展示任何内容之前计算完整影响
2. **先显示完整报告** — 让用户在要求采取行动之前了解范围
3. **逐条询问 ADR** — 不要批量决策；每个受影响的 ADR 可能需要不同的处理方式
4. **写入前先询问** — 在修改任何文件之前始终确认
5. **非破坏性** — 绝不删除 ADR 内容；仅添加"已取代"备注
