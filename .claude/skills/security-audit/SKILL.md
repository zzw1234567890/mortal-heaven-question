# 安全审计

安全对于任何发布的游戏来说都不是可选项。即使是单人游戏也存在存档篡改向量。多人在线游戏有作弊面、数据泄露风险和拒绝服务可能性。此技能系统地审计代码库中最常见的游戏安全漏洞，并生成优先级修复计划。

**运行此技能：**
- 在任何公开发布之前（用于完善阶段 → 发布阶段门禁）
- 在启用任何在线/多人在线功能之前
- 在实现任何从磁盘或网络读取的系统之后
- 在报告安全相关的 Bug 时

**输出：** `production/security/security-audit-[date].md`

---

## 阶段 1：解析参数和范围

**模式：**
- `full` — 所有类别（发布前推荐）
- `network` — 仅网络/多人在线
- `save` — 仅存档文件和序列化
- `input` — 仅输入验证和注入
- `quick` — 仅高严重性检查（最快，用于迭代使用）
- 无参数 — 运行 `full`

读取 `.claude/docs/technical-preferences.md` 以确定：
- 引擎和语言（影响要搜索的模式）
- 目标平台（影响适用的攻击面）
- 多人在线/网络是否在范围内

---

## 阶段 2：生成安全工程师

通过 Task 生成 `security-engineer`。传递：
- 审计范围/模式
- 来自技术偏好的引擎和语言
- 所有源目录清单：`src/`、`assets/data/`、任何配置文件

安全工程师将跨 6 个类别运行审计（见阶段 3）。在继续前收集其完整发现。

---

## 阶段 3：审计类别

安全工程师评估以下每个类别。跳过不适用于项目范围的类别。

### 类别 1：存档文件和序列化安全
- 存档文件在加载前是否经过验证？（不能盲目反序列化）
- 存档文件路径是否由用户输入构造？（路径遍历风险）
- 存档文件是否有校验和或签名？（篡改检测）
- 游戏是否信任存档文件中的数值而不进行边界检查？
- 存档加载附近是否有任何 eval() 或动态代码执行调用？

Grep 模式：`File.open`、`load`、`deserialize`、`JSON.parse`、`from_json`、`read_file` — 检查每个是否有验证。

### 类别 2：网络和多人在线安全（如果仅为单人游戏则跳过）
- 游戏状态是否由服务器权威控制，还是客户端决定结果？
- 传入的网络数据包是否经过大小、类型和数值范围验证？
- 玩家位置和状态变化是否在服务器端验证？
- 任何网络调用是否有限速？
- 认证 token 是否被正确处理（永不发送明文）？
- 游戏在发布版本中是否暴露任何调试端点？

Grep 搜索：`recv`、`receive`、`PacketPeer`、`socket`、`NetworkedMultiplayerPeer`、`rpc`、`rpc_id` — 检查每个调用点的验证。

### 类别 3：输入验证
- 是否有任何玩家提供的字符串用于文件路径？（路径遍历）
- 是否有任何玩家提供的字符串在未经清理的情况下被记录？（日志注入）
- 数字输入（例如物品数量、角色属性）在使用前是否经过边界检查？
- 成就/统计值在写入任何后端之前是否经过检查？

Grep 搜索：`get_input`、`Input.get_`、`input_map`、面向用户的文本字段 — 检查验证。

### 类别 4：数据泄露
- 是否有任何 API 密钥、凭证或密钥硬编码在 `src/` 或 `assets/` 中？
- 发布版本中是否包含调试符号或详细错误消息？
- 游戏是否将敏感玩家数据记录到磁盘或控制台？
- 是否有任何内部文件路径或系统信息暴露给玩家？

Grep 搜索：`api_key`、`secret`、`password`、`token`、`private_key`、`DEBUG`、`print(` 在面向发布的代码中。

### 类别 5：作弊和防篡改向量
- 游戏关键数值是否仅存储在内存中，而不是存储在易于编辑的文件中？
- 任何关键的游戏进度标志（例如"已支付 DLC"）是否在服务器端验证？
- 对于多人在线游戏，是否有任何对内存编辑工具（Cheat Engine 等）的保护？
- 排行榜/分数提交在接收前是否经过验证？

注意：客户端反作弊基本上是不可执行的。对于任何竞争性或营利性游戏，重点关注服务器端验证。

### 类别 6：依赖和供应链
- 是否使用了任何第三方插件或库？列出它们。
- 任何插件在使用的版本中是否有已知的 CVE？
- 插件来源是否经过验证（官方市场、经过审查的仓库）？

Glob 搜索：`addons/`、`plugins/`、`third_party/`、`vendor/` — 列出所有外部依赖。

---

## 阶段 4：分类发现

对于每个发现，分配：

**严重性：**
| 级别 | 定义 |
|-------|-----------|
| **CRITICAL** | 远程代码执行、数据泄露或可轻易利用的作弊，破坏多人在线完整性 |
| **HIGH** | 绕过进度的存档篡改、凭证暴露或服务器端权限绕过 |
| **MEDIUM** | 启用客户端作弊、信息泄露或影响有限的输入验证缺口 |
| **LOW** | 纵深防御改进——强化以减少攻击面但不存在直接利用 |

**状态：** 开放 / 已接受风险 / 超出范围

## 阶段 6：写入报告

在对话中呈现报告摘要（仅执行摘要 + 严重/高严重性发现）。

询问："我可以将完整的安全审计报告写入 `production/security/security-audit-[date].md` 吗？"

---

## 阶段 7：门禁集成

此报告是**完善 → 发布阶段门禁**的必需工件。

在修复发现后，重新运行：`/security-audit quick` 以确认严重/高严重性项已解决，然后再运行 `/gate-check release`。

如果存在严重发现：
> "⛔ 严重安全发现必须在任何公开发布前解决。在解决之前不要继续执行 `/launch-checklist`。"

如果没有严重/高严重性发现：
> "✅ 无阻塞安全发现。报告已写入 `production/security/`。在运行 `/gate-check release` 时包含此路径。"

---

## 协作协议

- **永远不要假设一个模式是安全的**——标记它并让用户决定
- **接受风险是一个有效的输出**——一些低发现对于单人团队是可接受的权衡；记录该决定
- **多人在线游戏有更高的标准**——多人在线环境中的任何高严重性发现应视为严重
- **这不是渗透测试**——此审计涵盖常见模式；对于任何竞争性或营利性多人在线发布，建议由人类安全专业人员执行真正的渗透测试
