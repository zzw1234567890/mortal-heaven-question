# 回归测试套件

此技能确保每个 bug 修复都有一个本可以捕获原始 bug 的测试作为支撑 — 并且回归测试套件随着游戏的发展保持最新。它还会检测何时添加了新功能而没有相应的回归测试覆盖。

回归测试套件不是一个新测试类别 — 它是 `tests/` 中已经存在的测试的**精选列表**，共同覆盖游戏的关键路径和已知故障点。此技能维护该列表。

**输出：** `tests/regression-suite.md`

**何时运行：**
- 修复 bug 后（确认已编写回归测试或识别差距）
- 在发布关卡之前（`/gate-check polish` 要求回归测试套件存在）
- 作为 sprint 关闭的一部分，以检测覆盖范围漂移

---

## 1. 解析参数

**模式：**
- `/regression-suite update` — 扫描本次 sprint 的新 bug 修复并检查回归测试的存在性；将新测试添加到套件清单
- `/regression-suite audit` — 对所有 GDD 关键路径与现有测试覆盖进行完整审计；标记没有回归测试的路径
- `/regression-suite report` — 只读状态报告（不写入）；适合 sprint 审查
- 无参数 — 如果 sprint 明显活跃（sprint 计划存在且有进行中的 story），运行 `update`。如果不明确或未检测到活跃 sprint，使用 `AskUserQuestion`：
  - 提示："未指定子命令。您想运行哪种模式？"
  - 选项：
    - `[A] update — 扫描本次 sprint 的新 bug 修复并添加缺失的回归测试`
    - `[B] audit — 对所有 GDD 关键路径与现有测试覆盖进行完整审计`
    - `[C] report — 只读状态报告（不写入）`

---

## 2. 加载上下文

### 步骤 2a — 加载现有回归测试套件

读取 `tests/regression-suite.md`（如果存在）。提取：
- 已注册的回归测试总数
- 最后更新日期
- 任何标记为 `STALE` 或 `QUARANTINED` 的测试

如果不存在：注明"未找到回归测试套件 — 将创建一个。"

### 步骤 2b — 加载测试清单

搜索所有测试文件：
```
tests/unit/**/*_test.*
tests/integration/**/*_test.*
tests/regression/**/*
```

对于每个文件，记录系统（来自目录路径）和文件名。除非需要名称到测试的映射，否则不读取测试文件内容。

### 步骤 2c — 加载 GDD 关键路径

对于 `audit` 模式：读取 `design/gdd/systems-index.md` 以获取所有系统。对于每个 MVP 层级系统，读取其 GDD 并提取：
- 验收标准（这些定义了关键路径）
- 公式章节（公式必须有回归测试）
- 边界情况章节（已知边界情况应有回归测试）

对于 `update` 模式：跳过完整 GDD 扫描。而是读取当前 sprint 计划和 story 文件，查找本次 sprint 中状态为 Complete 的 story。

### 步骤 2d — 加载已关闭的 bug

搜索 `production/qa/bugs/*.md` 并过滤出具有 `Status: Closed` 或 `Status: Fixed` 字段的 bug。注意：
- bug 属于哪个 story 或系统
- 修复描述中是否提到了回归测试

---

## 3. 映射覆盖范围 — 关键路径

仅适用于 `audit` 模式：

对于每个 GDD 验收标准，确定是否存在测试：

1. 在 `tests/unit/[system]/` 和 `tests/integration/[system]/` 中搜索与标准的关键名词/动词相关的文件名和函数名
2. 分配覆盖状态：

| 状态 | 含义 |
|--------|---------|
| **已覆盖** | 存在针对此标准逻辑的测试文件 |
| **部分覆盖** | 存在测试但未覆盖所有情况（例如仅快乐路径） |
| **缺失** | 未找到此关键路径的测试 |
| **豁免** | 视觉/手感或 UI 标准 — 按设计不可自动化 |

3. 将对应于公式或状态机的缺失项提升为**高优先级**差距 — 这些是最可能的回归来源。

---

## 4. 映射覆盖范围 — 已修复的 bug

对于每个已关闭的 bug：

1. 从 bug 的元数据中提取系统标识
2. 在 `tests/unit/[system]/` 和 `tests/integration/[system]/` 中搜索引用 bug ID 或特定故障场景的测试
3. 分配：
   - **有回归测试** — 找到了可以捕获此 bug 的测试
   - **缺少回归测试** — bug 已修复但没有测试防止复发

对于缺少回归测试的项目：
- 将其标记为回归差距
- 建议测试文件路径：`tests/unit/[system]/[bug-slug]_regression_test.[ext]`
- 注明："没有此测试，该 bug 可能在未来的 sprint 中悄然重现。"

---

## 5. 检测覆盖范围漂移

覆盖范围漂移发生在游戏增长但回归测试套件未同步增长时。

检查漂移指标：
- 本次 sprint 完成的 story 在 `tests/` 中没有对应的测试文件
- 自上次回归测试套件更新以来添加到 `systems-index.md` 的新系统
- 自回归测试套件上次更新以来 GDD 章节添加或修订（如果可用，使用 Grep 检查 GDD 文件修改提示，或询问用户）
- `tests/regression-suite.md` 最后更新日期 vs 当前日期 — 如果间隔 > 2 个 sprint，标记为可能过时

## 6. 生成报告和套件清单

### 报告格式（对话中）

```
## 回归测试套件状态

**模式**：[update | audit | report]
**现有已注册测试**：[N]
**已扫描测试文件**：[N]

### 关键路径覆盖（仅 audit 模式）
| 系统 | 验收标准总数 | 已覆盖 | 部分覆盖 | 缺失 | 豁免 |
|--------|-----------|---------|---------|---------|--------|
| [name] | [N] | [N] | [N] | [N] | [N] |

**覆盖率（非豁免）**：[N]%

### Bug 回归覆盖
| Bug ID | 系统 | 严重程度 | 有回归测试？ |
|--------|--------|----------|----------------------|
| BUG-NNN | [system] | S[N] | 是 / 否 ⚠ |

**无回归测试的 bug**：[N]

### 覆盖范围漂移指标
[列出没有测试覆盖的新系统或 story，或"未检测到。"]

### 推荐的新回归测试
| 优先级 | 系统 | 建议测试文件 | 覆盖 |
|----------|--------|---------------------|--------|
| 高 | [system] | `tests/unit/[system]/[slug]_regression_test.[ext]` | BUG-NNN / AC-[N] |
| 中 | [system] | `tests/unit/[system]/[slug]_test.[ext]` | [criterion] |
```

### 套件清单格式（`tests/regression-suite.md`）

清单是一个精选索引 — 不是测试本身，而是一个注册表，记录哪些测试应在发布前始终通过：

```markdown
# 回归测试套件清单

> 最后更新：[date]
> 已注册测试总数：[N]
> 覆盖率：GDD 关键路径的 [N]%

## 如何运行

[引擎特定命令，运行所有回归测试]

## 已注册的回归测试

### [系统名称]

| 测试文件 | 测试函数（如已知） | 覆盖 | 添加日期 |
|-----------|--------------------------|--------|-------|
| `tests/unit/[system]/[file]_test.[ext]` | `test_[scenario]` | AC-N / BUG-NNN | [date] |

## 已知差距

应存在但尚未存在的测试：

| 优先级 | 系统 | 建议路径 | 覆盖 | 尚未编写的原因 |
|----------|--------|----------------|--------|------------------------|
| 高 | [system] | `tests/unit/[system]/[path]` | BUG-NNN | Bug 修复时未编写测试 |

## 隔离区测试

不稳定或禁用的测试（不在 CI 中运行）：

| 测试文件 | 函数 | 原因 | 自何时隔离 |
|-----------|----------|--------|-------------------|
| （无） | | | |
```

## 7. 写入输出

询问："我可以使用当前的回归测试套件清单写入/更新 `tests/regression-suite.md` 吗？"

对于 `update` 模式：追加新条目；绝不删除现有条目（使用带有针对性插入的 `Edit`）。
对于 `audit` 模式：使用更新的覆盖数据重写完整清单。
对于 `report` 模式：不写入任何内容。

写入后（如果批准）：

- 对于每个高优先级差距："考虑在下个 sprint 之前创建缺失的回归测试。运行 `/test-helpers` 搭建测试文件框架。"
- 如果 bug 回归差距 > 0："这些 bug 在没有回归测试的情况下可能悄然重现。下个 sprint 应包含一个 story 来编写缺失的测试。"
- 如果检测到覆盖范围漂移："回归测试套件可能正在漂移。考虑在下一个 sprint 边界运行 `/regression-suite audit`。"

判决：**完成** — 回归测试套件已更新。（如果用户拒绝写入：判决：**阻塞**。）

---

## 协作协议

- **未经用户明确批准，绝不从清单中删除现有回归测试** — 删除一个有意编写的测试本身就是回归风险
- **差距是建议性的，非阻塞性的** — 清晰呈现但不要阻止其他工作继续进行（发布关卡除外，回归测试套件是必需的）
- **隔离不是删除** — 间歇性失败的测试应被隔离（在清单中注明）但不能删除；应通过 `/test-flakiness` 修复
- **写入前先询问** — 在创建或更新清单前始终确认
