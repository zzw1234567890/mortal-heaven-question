## Phase 1：解析参数

解析审查模式（一次性，本次运行所有关卡生成时使用）：
1. 如果传入了 `--review [full|lean|solo]` → 使用该值
2. 否则读取 `production/review-mode.txt` → 使用该值
3. 否则 → 默认使用 `lean`

完整检查模式请参见 `.claude/docs/director-gates.md`。

确定模式：

- `new` → 生成空白试玩报告模板
- `analyze [path]` → 读取原始笔记并用结构化发现填充模板

## Phase 2A：新建模板模式

生成此模板并输出给用户：

```markdown

- **时长**：[游戏时间]
- **测试员**：[姓名/ID]
- **平台**：[PC/主机/移动端]
- **输入方式**：[键盘+鼠标 / 手柄 / 触屏]

## 测试重点
[正在测试的特定功能或流程]

## 第一印象（前 5 分钟）
- **理解目标？** [是/否/部分]
- **理解操作？** [是/否/部分]
- **情绪反应**：[投入/困惑/无聊/沮丧/兴奋]
- **备注**：[观察记录]

## 游戏流程
### 运行良好的部分
- [观察 1]

### 痛点
- [问题 1 -- 严重程度：高/中/低]

### 困惑点
- [玩家感到困惑的地方及原因]

### 惊喜时刻
- [让玩家感到惊喜或愉悦的内容]

## 遇到的 Bug

## 特定功能反馈

- **理解用途？** [是/否]
- **觉得有趣？** [是/否]
- **建议**：[测试员建议]

## 量化数据（如可用）

## 本次会话的前 3 个优先事项

2. [第二优先]
3. [第三优先]
```

## Phase 2B：分析模式

读取指定路径下的原始笔记。与现有设计文档进行交叉参考。用结构化发现填充上述模板。标记任何与设计意图相冲突的试玩观察。

---

## Phase 3：行动分类

将所有发现分为四个类别：

- **需要设计变更** — 乐趣问题、玩家困惑、机制缺陷、与 GDD 预期体验相冲突的观察
- **平衡性调整** — 数值不合理、难度过高或过低
- **Bug 报告** — 可复现的明确实现缺陷
- **润色项目** — 不阻塞进度，但存在摩擦或手感问题，留待后续处理

呈现分类列表，然后进行路由：

- **设计变更：** "在修改前，运行 `/propagate-design-change [path]` 处理受影响的设计文档，以查找下游影响。"
- **平衡性调整：** "运行 `/balance-check [system]` 验证完整的平衡性情况，然后再调整数值。"
- **Bug：** "使用 `/bug-report` 正式跟踪这些问题。"
- **润色项目：** "当团队进入该阶段时，添加到 `production/` 中的润色积压清单。"

---

## Phase 3b：创意总监玩家体验审查

**审查模式检查** — 在生成 CD-PLAYTEST 前应用：
- `solo` → 跳过。备注："CD-PLAYTEST 已跳过 — Solo 模式。"继续执行 Phase 4（保存报告）。
- `lean` → 跳过（非阶段关卡）。备注："CD-PLAYTEST 已跳过 — Lean 模式。"继续执行 Phase 4（保存报告）。
- `full` → 正常生成。

在分类发现后，通过 Task 使用关卡 **CD-PLAYTEST** (`.claude/docs/director-gates.md`) 生成 `creative-director`。

传入：结构化报告内容、游戏支柱和核心愿景（来自 `design/gdd/game-concept.md`）、正在测试的具体假设。

在保存报告前展示创意总监的评估。如果为 CONCERNS 或 REJECT，在报告中添加一个 `## 创意总监评估` 章节，记录判决和反馈。如果为 APPROVE，在报告中注明批准。

## Phase 5：后续步骤

判决：**完成** — 试玩报告已生成。

- 首先处理优先级最高的发现类别。
- 处理完设计变更后：在更新后的 GDD 上重新运行 `/design-review`。
- 修复 Bug 后：重新运行 `/bug-triage` 更新优先级。
