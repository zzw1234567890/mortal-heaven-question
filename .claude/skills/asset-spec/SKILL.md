## 阶段 0b：实体与屏幕清单（无参数且无现有清单时运行）

此流程生成 `design/assets/entity-inventory.md` —— 游戏视觉上需要的所有内容的母清单。在资产规范工作开始前运行一次。

### 步骤 1 — 从文档收集
并行读取所有可用的源材料：
- `design/gdd/systems-index.md` —— 提取列出的每个系统
- `design/gdd/` 中的所有 GDD —— 提取：视觉/音频需求部分、提到的 UI 元素、VFX 事件、任何命名的实体（角色、敌人、建筑、物品）
- `design/art/art-bible.md` —— 提取：任何命名的视觉类别、资产类型预期
- `design/narrative/` —— 如果存在，扫描任何角色或世界实体文档（可选——非必需）

### 步骤 2 — 构建建议清单
将所有发现的内容组织成类别：

```
角色 / 主角
敌人 / 生物
建筑 / 结构
环境 / 地形
物品 / 道具
VFX / 粒子
UI 屏幕（按名称列出每个屏幕）
HUD 元素
音频（SFX、音乐——仅描述，不生成提示）
其他
```

对于每个项目，注明发现它的源文档。

### 步骤 3 — 呈现并协作
在对话中向用户呈现完整的建议清单。然后使用 `AskUserQuestion`：
- 提示："我在你的 GDD 和美术圣经中找到了 **[N] 个视觉实体和 [N] 个 UI 屏幕**。检查清单——缺少什么，哪些不需要？"
- 选项：
  - `[A] 看起来不错——保存此清单`
  - `[B] 添加我要描述的项目`
  - `[C] 删除不适用的项目`
  - `[D] 既添加也删除——让我修改`

如果 [B] 或 [D]：请用户描述额外项目。接受简短描述（"一个中世纪堡垒，用作关卡背景"）或详细描述——两种都可以。协作处理直到用户满意。

如果 [C] 或 [D]：询问要删除哪些项目及其原因。从列表中删除它们。

### 步骤 4 — 写入清单
获得用户批准后，询问："我可以将实体清单写入 `design/assets/entity-inventory.md` 吗？"

写入文件：

```markdown
# 视觉实体与屏幕清单

> 生成日期：[date]
> 来源：[已读源文档列表]

## 实体

| 编号 | 名称 | 类型 | 描述 | 来源 | 状态 |
|---|------|------|-------------|--------|--------|
| 1 | [name] | 角色 / 敌人 / 建筑 / 环境 / 物品 / 其他 | [简要描述] | [源文档] | Needed |

## UI 屏幕

| 编号 | 屏幕名称 | 描述 | 来源 | 状态 |
|---|-------------|-------------|--------|--------|
| 1 | 主菜单 | [描述] | [来源] | Needed |

## HUD 元素

| 编号 | 元素 | 描述 | 来源 | 状态 |
|---|---------|-------------|--------|--------|

## 音频

| 编号 | 名称 | 类型（SFX / 音乐 / 环境） | 描述 | 来源 | 状态 |
|---|------|------------------------------|-------------|--------|--------|
```

写入后，告诉用户：
> "实体清单已保存。后续步骤：
> - 运行 `/ux-design [screen name]` 为清单中的每个 UI 屏幕设计
> - 运行 `/asset-spec entity：[name]` 为每个视觉实体制定规范
> - 或者再次运行 `/asset-spec` 逐个处理清单中的项目"

## 阶段 0：解析参数

提取：
- **目标类型**：`system`、`level` 或 `character`
- **目标名称**：冒号后的名称（规范化为 kebab-case）
- **审查模式**：如果存在 `--review [full|lean|solo]`

**模式行为：**
- `full`（默认）：并行召唤 `art-director` 和 `technical-artist`
- `lean`：仅召唤 `art-director` —— 更快，跳过技术约束检查
- `solo`：不召唤代理 —— 主会话仅根据美术圣经规则编写规范。适用于简单资产类别或当速度比深度更重要时。

## 阶段 1：收集上下文

在询问用户任何问题之前读取所有源材料。

### 必需读取：
- **美术圣经**：读取 `design/art/art-bible.md` —— 如果缺失则失败：
  > "未找到美术圣经。请先运行 `/art-bible` —— 资产规范锚定在美术圣经的视觉规则和资产标准上。"
  提取：视觉标识陈述、色彩系统（语义颜色）、造型语言、资产标准（第 8 部分——尺寸、格式、多边形预算、纹理分辨率层级）。

- **技术偏好**：读取 `.claude/docs/technical-preferences.md` —— 提取性能预算和命名约定。

### 源文档读取（按目标类型）：
- **system**：读取 `design/gdd/[target-name].md`。提取**视觉/音频需求**部分。如果它不存在或显示 `[To be designed]`：
  > "`design/gdd/[target-name].md` 的视觉/音频部分为空。请运行 `/design-system [target-name]` 完成 GDD，或手动描述视觉需求。"
  使用 `AskUserQuestion`：`[A] 手动描述需求` / `[B] 停止——先完成 GDD`
- **level**：读取 `design/levels/[target-name].md`。提取美术需求、资产列表、VFX 需求、以及步骤 4 中艺术总监的生产概念规范。
- **character** 或 **entity**：读取 `design/narrative/characters/[target-name].md` 或在 `design/narrative/` 和 `design/assets/entity-inventory.md` 中搜索匹配条目。提取视觉描述、角色和任何指定的区分特征。
  - **如果不存在源文档**：不要失败。而是使用 `AskUserQuestion`：
    - 提示："未找到 **[name]** 的档案。请简要描述——一两句话就足够了。"
    - 选项：`[A] 立即描述` / `[B] 跳过此实体` / `[C] 在此停止`
    - 如果 [A]：用户的描述成为来源。简短的答案产生简洁的规范；详细的答案产生详细的规范。接受用户提供的任何细节级别并据此工作。

### 可选读取：
- **现有清单**：读取 `design/assets/asset-manifest.md`（如果存在）——提取此目标已规范的资产以避免重复。
- **相关规范**：Glob `design/assets/specs/*.md` —— 扫描可共享的资产（例如，为一个系统规范了的通用 UI 元素也可能适用于这里）。

### 呈现上下文摘要：
> **资产规范：[目标类型] — [目标名称]**
> - 源文档：[path] — 识别了 [N] 个资产类型
> - 美术圣经：已找到 — 第 8 节的资产标准
> - 此目标的现有规范：[N 个已规范 / 无]
> - 在其他规范中找到的共享资产：[列出或"无"]

## 阶段 2：资产识别

从源文档中提取提到的每个资产类型——显式和隐式。

**对于系统**：查找 VFX 事件、精灵引用、UI 元素、音频触发器、粒子效果、图标需求和任何"视觉反馈"语言。

**对于关卡**：查找独特的环境道具、氛围 VFX、光照设置、环境音频、天空盒/背景以及任何区域特定的材质。

**对于角色**：查找精灵表（空闲、行走、攻击、死亡）、头像/肖像、附着在能力上的 VFX、UI 表现（图标、生命条皮肤）。

将资产分组到类别：
- **精灵 / 2D 美术** —— 角色精灵、UI 图标、图块表
- **VFX / 粒子** —— 命中效果、环境粒子、屏幕效果
- **环境** —— 道具、图块、背景、天空盒
- **UI** —— HUD 元素、菜单美术、字体（如果是自定义的）
- **音频** —— SFX、音乐曲目、环境循环 *（注意：音频规范仅为描述——不生成提示）*
- **3D 资产** —— 网格、材质（如果适用）

将完整的识别列表呈现给用户。使用 `AskUserQuestion`：
- 提示："我识别了 **[target]** 的 **[N] 个资产，涵盖 [N] 个类别**。在规范前检查："
- 先在对话文本中显示分组列表
- 选项：`[A] 继续——规范所有这些` / `[B] 删除一些资产` / `[C] 添加我没有捕获到的资产` / `[D] 调整类别`

未经用户确认资产列表，不要继续到阶段 3。

## 阶段 3：规范生成

基于审查模式召唤专家代理。**同时发出所有 Task 调用——不要等一个完成后才开始下一个。**

### 完整模式 —— 并行召唤：

**`art-director`** 通过 Task：
- 提供：阶段 2 的完整资产列表、美术圣经视觉标识陈述、色彩系统、造型语言、源文档的视觉需求以及美术圣经第 9 节中提到的任何参考游戏/艺术作品
- 询问："对于此列表中的每个资产，生成：（1）一段 2-3 句的视觉描述，锚定在美术圣经的造型语言和色彩系统上——要足够具体，以便两个不同的美术师能产生一致的结果；（2）一个可直接用于 AI 图像工具（Midjourney/Stable Diffusion 风格）的生成提示——包括风格关键词、构图、调色板锚点、反向提示；（3）直接约束此资产的美术圣经规则（按部分引用）。对于音频资产，描述声音特征而不是生成提示。"

**`technical-artist`** 通过 Task：
- 提供：完整资产列表、美术圣经资产标准（第 8 部分）、technical-preferences.md 性能预算、引擎名称和版本
- 询问："对于此列表中的每个资产，指定：（1）精确尺寸或多边形数量（匹配美术圣经资产标准层级——不要发明新尺寸）；（2）文件格式和导出设置；（3）命名约定（来自 technical-preferences.md）；（4）此资产类型必须遵守的任何引擎特定约束；（5）如果适用，LOD 需求。标记美术圣经的偏好标准与引擎约束冲突的任何资产类型。"

### Lean 模式 —— 仅召唤 art-director（跳过 technical-artist）。

### Solo 模式 —— 两者都跳过。仅根据美术圣经规则推导规范，注明技术约束未经验证。

**在阶段 4 之前收集两个响应。** 如果 art-director 和 technical-artist 之间存在任何冲突（例如 art-director 指定 4K 纹理但 technical-artist 标记引擎预算要求 512px），明确呈现——不要静默解决。

## 阶段 4：编译和审查

将代理输出合并为每个资产的草案规范。在对话文本中使用以下格式呈现所有规范：

```
## ASSET-[NNN] — [资产名称]

| 字段 | 值 |
|-------|-------|
| 类别 | [精灵 / VFX / 环境 / UI / 音频 / 3D] |
| 尺寸 | [例如 256×256px、4 帧精灵表] |
| 格式 | [PNG / SVG / WAV / 等] |
| 命名 | [例如 vfx_frost_hit_01.png] |
| 多边形数 | [如果是 3D —— 例如 <800 tris] |
| 纹理分辨率 | [例如 512px —— 匹配美术圣经第 8 节第 2 层] |

**视觉描述：**
[2-3 句话。足够具体，使两个美术师能产生一致的结果。]

**美术圣经锚点：**
- 第 3 节 造型语言：[应用的规则]
- 第 4 节 色彩系统：[颜色角色 —— 例如"根据语义颜色规则使用威胁蓝"]

**生成提示：**
[可直接使用的提示。包括：风格关键词、构图说明、调色板锚点、光照方向、反向提示。]

**状态：** Needed
```

在呈现所有规范后，使用 `AskUserQuestion`：
- 提示："**[target]** 的资产规范 —— [N] 个资产。审查完成？"
- 选项：`[A] 全部批准——写入文件` / `[B] 修改特定资产` / `[C] 用不同的方向重新生成`

如果 [B]：询问哪个资产以及要更改什么。内联修改并重新呈现。不要为小的文本修改重新召唤代理——仅当视觉方向本身需要更改时才重新召唤。

如果 [C]：询问要更改的方向。使用更新后的简报重新召唤相关代理。

## 资产 ID 分配

资产 ID 在整个项目中顺序分配——不是按上下文分配。在分配 ID 之前读取清单以找到当前最高编号：

```
Grep pattern="ASSET-" path="design/assets/asset-manifest.md"
```

从 `ASSET-[highest + 1]` 开始新资产。这确保 ID 在整个项目中稳定且唯一。

如果尚不存在清单，从 `ASSET-001` 开始。

---

## 共享资产协议

在规范资产之前，检查等效资产是否已存在于另一个上下文的规范中：

- 常见的 UI 元素（生命条、分数显示）通常跨系统共享
- 通用环境道具可能出现在多个关卡中
- 角色 VFX（命中火花、死亡效果）可能重用基础规范并带有颜色变体

如果找到匹配：引用现有 ASSET-ID 而不是创建重复。在清单的 referenced-by 列中注明共享使用。

> "ASSET-012（通用命中火花）已为战斗系统规范。塔防系统重用它——向 referenced-by 添加 tower-defense。"

---

## 错误恢复协议

如果任何被召唤的代理返回阻塞或无法完成：

1. 立即呈现："[AgentName]：阻塞 — [reason]"
2. 在 `lean` 模式或如果 `technical-artist` 阻塞：仅使用 art-director 输出继续——注明技术约束未经验证
3. 在 `solo` 模式或如果 `art-director` 阻塞：从美术圣经规则推导描述——标记为"未咨询艺术总监——在生产前对照美术圣经验证"
4. 始终生成部分规范——绝不因为一个代理阻塞而丢弃工作

---

## 协作协议

每个阶段遵循：**识别 → 确认 → 生成 → 审查 → 批准 → 写入**

- 未经用户确认资产列表，绝不规范资产
- 始终将规范锚定到美术圣经——与美术圣经矛盾的规范是错误的
- 呈现所有代理分歧——不要静默选择其一
- 仅在明确批准后写入规范文件
- 写入规范后立即更新清单

---

## 推荐的后续步骤

- 运行 `/asset-spec [next-context]` 继续规范剩余的系统、关卡或角色
- 运行 `/asset-audit` 验证交付的资产是否符合写入的规范并识别差距或不匹配
