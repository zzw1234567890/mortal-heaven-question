当本 skill 被调用时：

## 1. 解析参数并确定模式

根据参数存在三种撰写模式：

| 参数 | 模式 | 输出文件 |
|----------|------|-------------|
| `hud` | HUD设计 | `design/ux/hud.md` |
| `patterns` | 交互模式库 | `design/ux/interaction-patterns.md` |
| 任何其他值（例如 `main-menu`、`inventory`） | 屏幕或流程的UX规范 | `design/ux/[参数].md` |
| 无参数 | 询问用户 | （见下方） |

**如果未提供参数**，不要失败——改为询问。使用 `AskUserQuestion`：
- "我们今天要设计什么？"
  - 选项："某个特定屏幕或流程（我来命名）"、"游戏HUD"、"交互模式库"、"我不确定——帮我弄清楚"

如果用户选择"我来命名"或输入屏幕名称，将其规范化为 kebab-case
作为文件名（例如，"Main Menu"变为 `main-menu`）。

---

## 2. 收集上下文（读取阶段）

在向用户询问任何问题之前，先**读取**所有相关上下文。本 skill 的价值
来源于充分知情的状态。

### 2a：必须读取的内容

- **游戏概念**：读取 `design/gdd/game-concept.md` —— 如果缺失，警告：
  > "未找到游戏概念。请先运行 `/brainstorm` 来确定游戏的
  > 基础，然后再设计UX。"
  > 如果用户要求，仍然继续执行。

### 2b：玩家旅程

读取 `design/player-journey.md`（如果存在）。针对每个相关部分，提取：
- 此屏幕出现在哪个旅程阶段？
- 玩家到达此屏幕时的情绪状态是什么？
- 此屏幕在旅程中满足了什么玩家需求？
- 此屏幕交付了哪些（来自旅程地图的）关键时刻？

如果玩家旅程文件不存在，记录此缺口并继续执行：
> "在 `design/player-journey.md` 未找到玩家旅程地图。在没有它的前提下进行设计
> 意味着我们将对玩家上下文做出假设。考虑在起草此规范后
> 进行一次玩家旅程会议。"

同时将以下内容添加到UX规范的待解决问题部分：
> "玩家旅程地图尚未创建。模板位于 `.claude/docs/templates/player-journey.md`。运行 `/ux-design` 阶段 2b 或手动创建它以建立此屏幕的玩家上下文。"

### 2c：GDD UI需求

使用 Glob 搜索 `design/gdd/*.md` 并用 grep 查找 `UI Requirements` 部分。读取任何
其 UI Requirements 部分按名称或类别引用此屏幕的 GDD。

这些 GDD UI需求是本规范的**需求输入**。将它们收集为
规范必须满足的一系列约束。

如果正在设计 HUD，请读取所有 GDD UI Requirements 部分——HUD聚合了
来自每个系统的需求。

### 2d：现有UX规范

使用 Glob 搜索 `design/ux/*.md` 并记录哪些屏幕已有规范。针对那些将链接到
当前屏幕或从当前屏幕链接出去的屏幕，读取它们的导航/流程部分以
找到此规范必须匹配的进入点和退出点。

### 2e：交互模式库

如果 `design/ux/interaction-patterns.md` 存在，读取模式目录索引
（模式名称列表及其一行描述）。不要读取完整的
模式细节——只需阅读目录。这告诉你哪些模式已经存在，
因此你可以引用它们而不是重新发明。

### 2f：美术圣经

检查是否存在 `design/art/art-bible.md`。如果找到，读取视觉方向
部分。UX布局必须与已做出的美学承诺保持一致。

### 2g：无障碍需求

检查是否存在 `design/accessibility-requirements.md`。如果找到，读取它。规范
必须满足其中承诺的无障碍等级。

### 2h：输入方式（来自项目配置）

读取 `.claude/docs/technical-preferences.md` 并提取 `## Input & Platform`
部分。在整个 skill 中存储并使用这些值——它们驱动着
交互地图并影响无障碍需求：

- **输入方式**——例如 键盘/鼠标、手柄、触屏、混合
- **主要输入**——此游戏的主导输入方式
- **手柄支持**——完整 / 部分 / 无
- **触屏支持**——完整 / 部分 / 无
- **目标平台**——用于安全区域和宽高比决策

如果该部分未配置（`[TO BE CONFIGURED]`），询问一次：
> "输入方式尚未配置。此游戏的目标是什么？"
> 选项："仅键盘/鼠标"、"仅手柄"、"两者皆有（PC + 主机）"、"触屏（移动端）"、"以上所有"
>
> （运行 `/setup-engine` 永久保存此设置，这样就不会再次被问到。）

在本会话的剩余部分存储答案。不要在每部分
或每屏幕中**再次**询问。

### 2i：呈现上下文摘要

在任何设计工作之前，向用户呈现简要摘要：

> **正在设计：[屏幕/流程名称]**
> - 模式：[UX规范 / HUD设计 / 模式库]
> - 旅程阶段：[来自 player-journey.md，或"未知——无旅程地图"]
> - 提供给此规范的GDD需求：[数量和名称，或"未找到"]
> - 已有规范的相关屏幕：[列表，或"尚无"]
> - 已知可用模式：[数量，或"尚无模式库"]
> - 无障碍等级：[来自需求文档，或"尚未定义"]
> - 输入方式：[来自 technical-preferences.md，或"已在上方询问"]

然后询问："在我们开始之前，还有其他需要我读取的内容吗，或者我们可以继续了？"

## 2b. 改造模式检测

在创建骨架之前，检查目标输出文件是否已存在。

使用 Glob 搜索 `design/ux/[filename].md`（其中 `[filename]` 是阶段 1 中解析出的输出路径）。

**如果文件存在——改造模式：**
- 完整读取文件
- 针对每个预期部分，检查正文是否有实际内容（不仅仅是 `[To be designed]` 占位符）还是为空/占位符
- 向用户呈现部分状态摘要：

> "在 `design/ux/[filename].md` 找到现有UX规范。以下是已完成的内容：
>
> | 部分 | 状态 |
> |---------|--------|
> | 概述与上下文 | [已完成 / 空 / 占位符] |
> | 玩家旅程集成 | ... |
> | 屏幕布局与信息架构 | ... |
> | 交互模型 | ... |
> | 反馈与状态通信 | ... |
> | 无障碍 | ... |
> | 边界情况与错误状态 | ... |
> | 待解决问题 | ... |
>
> 我将只处理 [N] 个未完成的部分——现有内容不会被覆盖。"

- 跳过第 3 部分（骨架创建）——文件已存在
- 在阶段 4（部分撰写）中，只处理状态为"空"或"占位符"的部分
- 使用 `Edit` 原地填充占位符，而不是创建新的骨架

**如果文件不存在——全新撰写模式：**
按正常流程进入阶段 3（创建文件骨架）。

## 4. 逐节撰写

按顺序遍历每个部分。针对**每个部分**，遵循以下循环：

```
上下文  ->  问题  ->  选项  ->  决策  ->  草稿  ->  批准  ->  写入
```

1. **上下文**：说明此部分需要包含的内容，并呈现从阶段 2 收集的上下文中相关的任何约束。
2. **问题**：询问撰写此部分所需的内容。对于有限选择使用 `AskUserQuestion`，对于开放式探索使用对话文本。
3. **选项**：在设计选项存在的地方，呈现 2-4 种方法及其优缺点。在对话中解释理由，然后使用 `AskUserQuestion` 捕获决策。
4. **决策**：用户选择一种方法或提供自定义方向。
5. **草稿**：在对话中撰写部分内容以供审查。明确标记临时假设。
6. **批准**：使用 `AskUserQuestion`：
   - "这是否正确捕获了 [部分名称]？"
   - 选项："是的——写入文件"、"需要小的修改（在下方描述）"、"需要重大重做"
   - 在用户选择"是的"之前，不要继续到步骤 7。
7. **写入**：使用 `AskUserQuestion`："我可以将 [部分名称] 部分写入 `[filepath]` 吗？"
   - 选项："是的，写入"、"等等——还有一处修改"
   - 确认后，使用 `Edit` 将 `[To be designed]` 占位符替换为已批准的内容。

在写入每个部分后，更新 `production/session-state/active.md`。

### 部分指导：UX规范模式

#### 部分A：目的与玩家需求

本部分是基础。所有其他决策都源于此。

**要问的问题**：
- "此屏幕服务于什么玩家目标？玩家在这里试图做什么？"
- "如果此屏幕不存在或难以使用，会出什么问题？"
- "完成这句话：'玩家到达此屏幕时想要___。'"

交叉参考阶段 2 中收集的玩家旅程上下文。所述目的
必须与旅程阶段和情绪状态一致。

---

#### 部分B：到达时的玩家上下文

**要问的问题**：
- "在游戏中，玩家第一次遇到此屏幕是在什么时候？"
- "在到达此屏幕之前，他们刚刚在做什么？"
- "设计应假设什么情绪状态？（平静、紧张、好奇、时间紧迫）"
- "玩家是自愿到达此屏幕，还是被游戏送到这里？"

如果玩家旅程文档存在，主动提议将其映射到旅程阶段。

---

#### 部分B2：导航位置

此屏幕位于游戏导航层级中的什么位置？这是一段描述方向的段落——不是完整的流程图。

**要问的问题**：
- "此屏幕是从主菜单、暂停界面、游戏玩法内部还是从另一个屏幕进入的？"
- "它是一个顶级目的地（始终可达）还是上下文相关的（仅在特定状态下可进入）？"
- "玩家可以从游戏中的多个位置到达此屏幕吗？"

呈现为："此屏幕位于：[根] → [父级] → [此屏幕]" 加上任何替代进入路径。

---

#### 部分B3：进入点与退出点

映射玩家到达和离开此屏幕的每种方式。

**要问的问题**：
- "玩家到达此屏幕的所有方式有哪些？"（列出每个触发器：按钮按下、游戏事件、从另一个屏幕重定向等）
- "玩家可以做什么来退出？当他们这样做时会发生什么？"（返回按钮、确认操作、超时、游戏事件）
- "是否有任何退出是不可逆的——玩家无法在不重新开始的情况下返回此屏幕？"

呈现为两个表格：

| 进入来源 | 触发器 | 玩家携带的上下文 |
|---|---|---|
| [屏幕/事件] | [方式] | [他们到达时携带的状态/数据] |

| 退出目的地 | 触发器 | 备注 |
|---|---|---|
| [屏幕/事件] | [方式] | [任何不可逆的状态变更] |

---

#### 部分C：布局规范

这是最大且交互性最强的部分。在子部分中逐步完成：

**子部分 1 —— 信息层级**（在任何布局之前建立此项）：
- 请用户列出此屏幕必须传达的每一条信息。
- 然后请他们排列项目："玩家需要首先看到的最重要的东西是什么？第二重要的是什么？什么是可以被发现而不是立即可见的？"
- 在进入区域布局之前，呈现结果层级以供批准。

**子部分 2 —— 布局区域**：
- 基于信息层级，提出粗略的屏幕区域（标题、内容区、操作栏、侧边栏等）。
- 提供 2-3 种区域安排，附上各自的理由。引用从游戏概念中收集的平台和输入上下文。
- 使用 `AskUserQuestion` 捕获选择：
  - "哪种区域安排最合适？"
  - 选项：[你刚刚呈现的 2-3 种命名安排] + "都不合适——构建自定义安排"

**子部分 3 —— 组件清单**：
- 针对每个区域，列出其包含的UI组件。针对每个组件，注明：
  - 组件类型（按钮、列表、卡片、状态显示、输入字段等）
  - 它显示的内容
  - 是否可交互
  - 是否使用库中的现有模式（通过模式名称引用）
  - 是否引入了新模式（标记为后续添加到库中）

**子部分 4 —— ASCII线框**：
- 主动提议基于区域布局和组件列表生成 ASCII 线框。
- 使用 `AskUserQuestion`："想要在此规范中包含 ASCII 线框吗？"
  - 选项："是的，包含一个"、"不，我会附加一个单独的文件"
- 如果是，先在对话中生成线框。在写入文件之前征求反馈。

---

#### 部分D：状态与变体

引导用户思考超出理想路径的情况。

**要问的问题**（一次处理一个）：
- "玩家第一次看到此屏幕时，当还没有数据时，它是什么样子的？（空状态）"
- "当出现问题时——错误、操作失败、资源缺失——会发生什么？（错误状态）"
- "此屏幕上是否有加载等待？如果有，它显示什么？（加载状态）"
- "是否有任何玩家进度状态会改变此屏幕显示的内容？例如，锁定内容、高级内容或教程模式覆盖层？"
- "此屏幕在任何支持的平台上表现是否不同？（平台变体）"

将收集到的状态以表格形式呈现以供批准：

| 状态 / 变体 | 触发器 | 变动的部分 |
|-----------------|---------|--------------|
| 默认 | 正常加载 | — |
| 空 | 无可用数据 | [内容区域描述] |
| [等] | [触发器] | [变动] |

---

#### 部分E：交互地图

针对布局规范中识别的每个可交互组件，定义：
- 动作（轻触、点击、按压、长按、滚动、拖拽）
- 触发它的平台输入（鼠标点击、手柄A键、键盘Enter键）
- 即时反馈（视觉、音频、触觉）
- 结果（导航目标、状态变更、数据写入）

使用阶段 2h 中从 `technical-preferences.md` 加载的输入方式——不要
再次询问用户。提前声明："正在映射交互：[
来自技术偏好的输入方式]。涵盖 [手柄支持] 手柄支持。"

一次处理一个组件，而不是一口气全部问完。
对于导航动作（前往另一个屏幕），验证目标是否匹配
现有的UX规范，或将其标记为规范依赖。

---

#### 部分E2：触发的事件

针对交互地图中的每个玩家动作，记录游戏或分析系统应触发的相应事件——如果没有任何事件适用，则明确注明"无事件"。

**要问的问题**：
- "对于每个动作，游戏应该触发分析事件、触发游戏状态变更，还是两者都做？"
- "是否有任何动作不应该触发事件——这是有意的选择吗？"

与交互地图一起以表格形式呈现：

| 玩家动作 | 触发的事件 | 载荷 / 数据 |
|---|---|---|
| [动作] | [EventName] 或 无 | [随事件传递的数据] |

标记任何修改持久游戏状态（存档数据、进度、经济系统）的动作——这些需要架构团队的特别关注。

---

#### 部分E3：过渡与动画

规定屏幕如何进入和退出，以及它如何响应状态变更。

**要问的问题**：
- "此屏幕如何出现？（淡入、从右侧滑入、立即弹出、从按钮缩放）"
- "它如何关闭？（淡出、滑回、切出）"
- "是否有任何屏幕内状态转换需要动画？（加载旋转器、成功状态、错误闪烁）"
- "是否有任何动画可能导致晕动症——游戏是否有减少动态选项？"

最低要求：
- 屏幕进入过渡
- 屏幕退出过渡
- 如果屏幕有多个状态，至少一个状态变更动画

---

#### 部分F：数据需求

交叉参考阶段 2 中收集的 GDD UI Requirements 部分。

针对屏幕显示的每条信息，询问：
- "这些数据来自哪里？哪个系统拥有它？"
- "此屏幕需要写回数据，还是只读？"
- "是否有任何数据是时间敏感的或实时的？（血条、冷却计时器）"

标记任何UI需要拥有或管理游戏状态的情况，作为架构
关注点。UX规范定义UI需要什么；它们不规定数据如何
交付。那是一个架构决策。

将数据需求以表格形式呈现：

| 数据 | 源系统 | 读 / 写 | 备注 |
|------|--------------|--------------|-------|
| [项目] | [系统] | 读 | — |
| [项目] | [系统] | 写 | [如有任何关注点] |

---

#### 部分G：无障碍

交叉参考 `design/accessibility-requirements.md`（如果存在）。

针对此屏幕，逐步执行 ux-designer 代理的标准检查清单：
- 通过所有可交互元素的纯键盘导航路径
- 手柄导航顺序（如适用）
- 文本对比度和最小可读字体大小
- 独立于颜色的传达（没有仅通过颜色传达的信息）
- 任何非文本元素的屏幕阅读器考虑事项
- 任何需要减少动态替代方案的动态效果或动画

如果此项目尚未定义无障碍等级，在UX规范的待解决问题部分记录此缺口：
> "无障碍等级尚未定义——考虑将 WCAG-AA 作为基线。运行 `/gate-check` 查看这是否阻塞任何阶段关口。"
然后继续到下一部分而不停止。

---

#### 部分H：本地化考虑事项

记录影响此屏幕在文本被翻译时行为的约束。

**要问的问题**：
- "此屏幕上哪些文本元素最长？适合布局的最大字符数是多少？"
- "是否有任何元素文本长度对布局至关重要——例如，必须保持在一行上的按钮标签？"
- "是否有任何显示数字、日期或货币的元素需要特定于区域设置的格式？"

注意：旨在标记任何文本膨胀 40%（常见于从英语翻译到德语或法语）会破坏布局的元素。将这些标记为本地化工程师的**高优先级**。

---

#### 部分I：验收标准

编写至少 5 条具体的、可测试的标准，QA测试人员可以在不阅读任何其他设计文档的情况下进行验证。这些成为 `/story-done` 的通过/失败条件。

**格式**：使用复选框。每条标准必须可由人工测试人员验证：

```
- [ ] 屏幕在 [X]ms 内从 [触发器] 打开
- [ ] [元素] 在 [最小] 和 [最大] 值下正确显示
- [ ] [导航动作] 正确路由到 [目标屏幕]
- [ ] 当 [条件] 时出现错误状态，并显示 [特定消息或图标]
- [ ] 键盘/手柄导航按逻辑顺序到达所有可交互元素
- [ ] [无障碍需求] 已满足——例如，"所有可交互元素都有焦点指示器"
```

**最低要求**：
- 1 条性能标准（加载/打开时间）
- 1 条导航标准（至少验证一条进入或退出路径）
- 1 条错误/空状态标准
- 1 条无障碍标准（根据承诺的等级）
- 1 条特定于此屏幕核心目的的标准

使用 `AskUserQuestion` 确认：
- "这些验收标准是否涵盖了让此屏幕在你的QA流程中'完成'的要求？"
- 选项："是的——这些很扎实"、"再加一条标准"、"移除或改写一条"

---

### 部分指导：HUD设计模式

HUD设计遵循与UX规范模式不同的顺序。从理念开始；
在信息架构完成之前不要触及布局。

#### 部分A：HUD理念

请用户用 1-2 句话描述游戏与屏幕上信息的关系。

提供框架示例以帮助理解：
- "几乎无HUD——氛围需要无阻碍的沉浸感（例如，Hollow Knight、Firewatch）"
- "简洁但存在——只有关键信息可见，其他一切按上下文显示（例如，Dark Souls）"
- "信息密集——所有决策相关数据始终可见（例如，Diablo IV、StarCraft II）"
- "自适应——HUD密度响应战斗状态、探索模式、菜单（例如，God of War）"

此理念成为每个后续HUD决策的设计约束。
如果提议的元素与所述理念冲突，提出该冲突。

---

#### 部分B：信息架构

在任何布局工作之前完成此项。不要跳过。

**步骤 1 —— 完整信息清单**：
提取阶段 2 中收集的所有 GDD UI Requirements 部分的信息。
呈现完整列表："这些是你的游戏系统说它们需要
在屏幕上向玩家传达的所有内容。"

**步骤 2 —— 分类**：
针对每个项目，请用户对其进行分类：

| 类别 | 描述 |
|----------|-------------|
| **必须显示** | 始终可见，玩家核心决策需要 |
| **上下文相关** | 仅在相关时可见（战斗中、靠近可交互物等） |
| **按需显示** | 玩家必须主动请求（切换、按住按钮） |
| **隐藏** | 通过世界/音频传达，屏幕上从不使用文本 |

使用 `AskUserQuestion` 以 3-4 个为一组逐步处理项目，不要一次性处理全部。
这是HUD中最具决定性影响的设计决策——不要草率。

**冲突检查**：如果信息理念（部分A）说"几乎无HUD"
但必须显示列表越来越长，明确提出冲突：
> "当前的必须显示列表有 [N] 个项目。这可能与无HUD
> 理念冲突。选项：减少必须显示列表、修改理念、或定义
> 一个混合方法，在探索时无HUD、在战斗中显示HUD。"

---

#### 部分C：布局区域

只有在信息架构获得批准后，才设计布局区域。

布局基于：
- 哪些项目是必须显示的（它们驱动永久的区域决策）
- 玩家注意力在游戏过程中自然投向哪里（动作游戏为屏幕中心、
  策略游戏为角落）
- 平台和宽高比目标

提供 2-3 种区域安排。包括基于HUD理念和
部分B分类的理由。

---

#### 部分D：HUD元素

针对布局中的每个元素，规定：
- 元素名称和类别（必须显示 / 上下文相关 / 按需显示）
- 显示的内容
- 视觉形式（条、数字、图标、计数器、地图）
- 更新行为（实时、事件驱动、玩家查询）
- 上下文触发器（如果不是始终可见）
- 动画行为（低时长时闪烁？淡入？弹入？）

逐个元素处理。如果状态显示、资源条或冷却指示器存在相关模式，
引用交互模式库。

---

#### 部分E、F、G：动态行为、平台变体、无障碍

这些遵循与UX规范对应部分相同的结构。参见UX规范部分
关于D（状态/变体）、E（交互）和G（无障碍）的指导。

针对HUD特别强调：
- 动态行为：什么导致HUD在游戏进行中改变密度？
- 平台变体：移动端/主机是否需要不同的元素大小或位置？

---

### 部分指导：交互模式库模式

模式库撰写是增量和目录驱动的，而非线性的。

#### 阶段 1：编目现有模式

使用 Glob 搜索 `design/ux/*.md`（排除 `interaction-patterns.md`）并读取每个规范的组件清单和交互地图部分。提取所使用的每个交互模式。

呈现提取的列表："基于现有UX规范，这些模式已经在游戏中使用了："
- [模式名称]：用于 [屏幕]、[屏幕]
- [等]

询问："是否有哪些你知道存在但尚未出现在现有规范中的模式？现在列出任何额外的模式。"

---

#### 阶段 2：规范化每个模式

针对每个模式（现有或新增），记录：

```markdown
### [Pattern Name]

**Category**: Navigation / Input / Feedback / Data Display / Modal / Overlay / [other]
**Used In**: [list of screens]

**Description**: [One paragraph explaining what this pattern is and when to use it]

**Specification**:
- [Component behavior]
- [Input mapping]
- [Visual/audio feedback]
- [Accessibility requirements for this pattern]

**When to Use**: [Conditions where this pattern is appropriate]
**When NOT to Use**: [Conditions where another pattern is more appropriate]

**Reference**: [Screenshot path or ASCII example, if available]
```

分组处理模式。使用 `AskUserQuestion`：
- "你想如何处理这些模式？"
- 选项："从现有规范起草第一批（更快）"、"逐个定义（更多控制）"、"从最常用的模式开始"

---

#### 阶段 3：识别缺口

在编目已知模式后，询问：
- "是否计划了尚未在此库中拥有模式的屏幕或交互？"
- "现有规范中是否有任何模式之间感觉不一致，应予以整合？"

将缺口记录在缺口部分以供后续跟进。

---

## 5. 交叉引用检查

在将规范标记为可审查之前，运行以下检查：

**1. GDD需求覆盖率**：每个引用此屏幕的 GDD UI Requirement 是否在此规范中都有对应的元素？呈现任何缺口。

**2. 模式库对齐**：此规范中使用的所有交互模式是否按名称引用？如果在此规范会话期间发明了新模式，标记它以便添加到模式库：
使用 `AskUserQuestion`：
- "此规范使用了 [模式名称]，该模式尚未在模式库中。我们该怎么办？"
- 选项："立即将其添加到模式库"、"标记为缺口并继续"、"跳过——此模式是一次性的"

**3. 导航一致性**：此规范中的进入/退出点是否与任何相关规范中的导航图匹配？标记不匹配项。

**4. 无障碍覆盖**：规范是否解决了 `design/accessibility-requirements.md` 中承诺的无障碍等级？如果没有，标记为待解决问题。

**5. 空状态**：每个依赖数据的元素是否有定义的空状态？标记任何没有的。

呈现检查结果：
> **交叉引用检查：[屏幕名称]**
> - GDD需求：[已覆盖 N/ 个 / 已全部覆盖]
> - 需添加到库的新模式：[列表或"无"]
> - 导航不匹配：[列表或"无"]
> - 无障碍缺口：[列表或"无"]
> - 缺失的空状态：[列表或"无"]

如果会话中断（压缩、崩溃、新会话）：

## 8. 专家代理路由

本 skill 使用 `ux-designer` 作为主要代理（在 frontmatter 中设置）。对于特定的子主题，可能需要额外的上下文或协调：

| 主题 | 协调对象 |
|-------|----------------|
| 视觉美学、颜色、布局感觉 | `art-director` —— UX规范定义区域；美术定义它们的外观 |
| 实施可行性（引擎约束） | `ui-programmer` —— 在最终确定组件清单之前 |
| 游戏数据需求 | `game-designer` —— 当数据所有权不明确时 |
| 在UI中可见的叙事/世界观 | `narrative-director` —— 用于风味文本、物品名称、世界观面板 |
| 无障碍等级决策 | 由此会话处理 —— 由 ux-designer 拥有 |

当通过 Task 工具委托给另一个代理时：
- 提供：屏幕名称、游戏概念摘要、需要专家输入的具体问题
- 代理将分析结果返回到此会话
- 此会话将代理的输出呈现给用户
- 用户决定；此会话写入文件
- 代理不直接写入文件——此会话拥有所有文件写入操作

## 协作协议

本 skill 在每一步都遵循协作设计原则：

1. **问题 -> 选项 -> 决策 -> 草稿 -> 批准** 针对每个部分
2. **AskUserQuestion** 在每个决策点（解释 -> 捕获模式）：
   - 阶段 2："准备好开始了，还是需要更多上下文？"
   - 阶段 3："我可以创建骨架吗？"
   - 阶段 4（每个部分）：设计问题、方法选项、草稿批准
   - 阶段 5："运行交叉引用检查？接下来做什么？"
3. **"我可以写入 [filepath] 吗？"** 在骨架之前和每个部分写入之前
4. **增量写入**：每个部分在批准后立即写入文件
5. **会话状态更新**：在每次部分写入之后

**美学尊重**：当布局或视觉选择归结为个人品味时，
呈现选项并询问。不要因为某个布局是"标准"就选择它——始终
确认。用户是创意总监。

**冲突呈现**：当GDD需求与可用屏幕空间
冲突时，呈现冲突并提供解决选项。永远不要默默地放弃
需求。永远不要在不标记的情况下默默地扩展布局。

**绝不要**自动生成完整规范并将其作为既成事实呈现。
**绝不要**在未经用户批准的情况下写入部分。
**绝不要**在不标记冲突的情况下与现有已批准的UX规范矛盾。
**始终**展示决策的来源（GDD需求、玩家旅程、用户选择）。

结论：**完成**——UX规范已逐节编写并获得批准。
