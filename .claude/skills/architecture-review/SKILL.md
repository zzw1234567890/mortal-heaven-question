# 架构审查


架构审查验证完整的架构决策体系是否覆盖所有游戏设计需求、内部一致，并正确针对项目锁定的引擎版本。它是技术设置与预生产之间的质量关卡。

**参数模式：**
- **无参数 / `full`**：完整审查——所有阶段
- **`coverage`**：仅可追溯性——哪些 GDD 需求没有 ADR
- **`consistency`**：仅跨 ADR 冲突检测
- **`engine`**：仅引擎兼容性审计
- **`single-gdd [path]`**：审查一个特定 GDD 的架构覆盖
- **`rtm`**：需求可追溯性矩阵——将标准矩阵扩展到包括故事文件路径和测试文件路径；输出 `docs/architecture/requirements-traceability.md`，包含完整的 GDD 需求 → ADR → 故事 → 测试链条。在生产阶段当故事和测试已存在时使用。

## 阶段 1：加载所有内容

### 阶段 1a — L0：摘要扫描（快速、低 token）

在读取任何完整文档之前，使用 Grep 从所有 GDD 和 ADR 中提取 `## Summary` 部分：

```
Grep pattern="## Summary" glob="design/gdd/*.md" output_mode="content" -A 4
Grep pattern="## Summary" glob="docs/architecture/adr-*.md" output_mode="content" -A 3
```

对于 `single-gdd [path]` 模式：使用目标 GDD 的摘要来识别哪些 ADR 引用了相同的系统（对 ADR 进行系统名称的 Grep），然后仅完整读取那些 ADR。完全跳过完整读取无关的 GDD。

对于 `engine` 模式：仅完整读取 ADR —— GDD 对引擎检查不需要。

对于 `coverage` 或 `full` 模式：继续完整读取以下所有内容。

### 阶段 1b — L1/L2：完整文档加载

读取适合模式的所有输入：

### 设计文档
- `design/gdd/` 中所有范围内的 GDD —— 完整读取每个文件
- `design/gdd/systems-index.md` —— 权威系统列表

### 架构文档
- `docs/architecture/` 中所有范围内的 ADR —— 完整读取每个文件

### 引擎参考

**同时读取 `docs/consistency-failures.md`**（如果存在）。提取领域与被审查系统匹配的条目（架构、引擎或任何被覆盖的 GDD 领域）。在阶段 4 冲突检测输出的顶部，将重复出现的模式作为"已知冲突易发区域"说明呈现。

## 阶段 2：从每个 GDD 提取技术需求

### 预加载 TR 注册表

在提取任何需求之前，读取 `docs/architecture/tr-registry.yaml`（如果存在）。按 `id` 和规范化后的 `requirement` 文本（小写、修剪）索引现有条目。这防止跨审查运行的 ID 重新编号。

对于提取的每个需求，匹配规则为：
1. **完全/近似匹配**同一系统的现有注册表条目 → 不变地重用该条目的 TR-ID。仅当 GDD 措辞发生变化时更新注册表中的 `requirement` 文本（相同意图，更清晰的表述）——添加 `revised：[date]` 字段。
2. **无匹配** → 分配新 ID：该系统的下一个可用 `TR-[system]-NNN`，从现有最高序列 +1 开始。
3. **不明确**（部分匹配，意图不清晰）→ 询问用户：
   > "'[新需求文本]' 是否与 `TR-[system]-NNN：[现有文本]'` 是同一个需求，还是一个新的需求？"
   用户回答："同一需求"（重用 ID）或"新需求"（新 ID）。

对于注册表中 `status：deprecated` 的任何需求 —— 跳过它。它已被有意从 GDD 中移除。

对于每个 GDD，读取它并提取所有**技术需求**——架构必须提供以便系统工作的内容。技术需求是任何暗示特定架构决策的陈述。

要提取的类别：

| 类别 | 示例 |
|----------|---------|
| **数据结构** | "每个实体有生命值、最大生命值、状态效果" → 需要组件/数据模式 |
| **性能约束** | "碰撞检测必须在 200 个实体时以 60fps 运行" → 物理预算 ADR |
| **引擎能力** | "角色动画的逆向运动学" → IK 系统 ADR |
| **跨系统通信** | "伤害系统同时通知 UI 和音频" → 事件/信号架构 ADR |
| **状态持久化** | "玩家进度在会话间持久化" → 保存系统 ADR |
| **线程/时序** | "AI 决策在主线程外进行" → 并发 ADR |
| **平台需求** | "支持键盘、手柄、触屏" → 输入系统 ADR |

对于每个 GDD，生成结构化列表：

```
GDD：[filename]
系统：[system name]
技术需求：
  TR-[GDD]-001：[需求文本] → 领域：[物理/渲染/等]
  TR-[GDD]-002：[需求文本] → 领域：[...]
```

这成为**需求基线**——架构必须覆盖的完整集合。

## 阶段 3：构建可追溯性矩阵

对于阶段 2 中提取的每个技术需求，搜索 ADR：

1. 读取每个 ADR 的"解决的 GDD 需求"部分
2. 检查它是否明确引用了该需求或其 GDD
3. 检查 ADR 的决策文本是否隐含地涵盖了该需求
4. 标记覆盖状态：

| 状态 | 含义 |
|--------|---------|
| ✅ **已覆盖** | 某个 ADR 明确解决了此需求 |
| ⚠️ **部分覆盖** | 某个 ADR 部分覆盖了此需求，或覆盖不明确 |
| ❌ **缺口** | 没有 ADR 解决此需求 |

构建完整矩阵：

```
## 可追溯性矩阵

| 需求 ID | GDD | 系统 | 需求 | ADR 覆盖 | 状态 |
|---------------|-----|--------|-------------|--------------|--------|
| TR-combat-001 | combat.md | 战斗 | 碰撞箱检测 < 1 帧 | ADR-0003 | ✅ |
| TR-combat-002 | combat.md | 战斗 | 连击窗口时序 | — | ❌ 缺口 |
| TR-inventory-001 | inventory.md | 库存 | 持久物品存储 | ADR-0005 | ✅ |
```

统计总数：X 已覆盖、Y 部分覆盖、Z 缺口。

## 阶段 3b：故事和测试关联（仅 RTM 模式）

*除非参数为 `rtm` 或 `full` 且故事存在，否则跳过此阶段。*

此阶段将阶段 3 矩阵扩展到包括实现每个需求的故事和验证它的测试——生成完整的需求可追溯性矩阵（RTM）。

### 步骤 3b-1 — 加载故事

Glob `production/epics/**/*.md`（排除 EPIC.md 索引文件）。对于每个故事文件：
- 从故事的 Context 部分提取 `TR-ID`
- 提取故事文件路径、标题、状态
- 提取 `## 测试证据` 部分——声明的测试文件路径

### 步骤 3b-2 — 加载测试文件

Glob `tests/unit/**/*_test.*` 和 `tests/integration/**/*_test.*`。
构建索引：系统 → [测试文件路径]。

对于步骤 3b-1 中的每个测试文件路径，通过 Glob 确认文件是否实际存在。如果声明的路径不存在，标记为缺失（MISSING）。

### 步骤 3b-3 — 构建扩展 RTM

对于阶段 3 矩阵中的每个 TR-ID，添加：
- **故事**：引用此 TR-ID 的故事文件路径（可能有多个）
- **测试文件**：故事测试证据部分中声明的测试文件路径
- **测试状态**：已覆盖（测试文件存在）/ 缺失（声明了路径但未找到）/ 无（未声明测试路径，故事类型可能为视觉/感觉/UI）/ 无故事（需求还没有故事——预生产缺口）

扩展矩阵格式：

```
## 需求可追溯性矩阵（RTM）

| TR-ID | GDD | 需求 | ADR | 故事 | 测试文件 | 测试状态 |
|-------|-----|-------------|-----|-------|-----------|-------------|
| TR-combat-001 | combat.md | 碰撞箱 < 1 帧 | ADR-0003 | story-001-hitbox.md | tests/unit/combat/hitbox_test.gd | 已覆盖 |
| TR-combat-002 | combat.md | 连击窗口 | — | story-002-combo.md | — | 无（视觉/感觉） |
| TR-inventory-001 | inventory.md | 持久存储 | ADR-0005 | — | — | 无故事 |
```

RTM 覆盖摘要：
- 已覆盖：[N] — 具有 ADR + 故事 + 通过的测试的需求
- 测试缺失：[N] — 故事存在但测试文件未找到
- 无故事：[N] — 具有 ADR 但还没有故事的需求
- 无 ADR：[N] — 没有架构覆盖的需求（来自阶段 3 缺口）
- 完整链条完成（已覆盖）：[N/总数]（[%]）

## 阶段 4：跨 ADR 冲突检测

将每个 ADR 与所有其他 ADR 进行比较以检测矛盾。当以下情况时存在冲突：

- **数据所有权冲突**：两个 ADR 声称排他性地拥有相同的数据
- **集成合同冲突**：ADR-A 假设系统 X 有接口 Y，但 ADR-B 用不同的接口定义了系统 X
- **性能预算冲突**：ADR-A 分配 N ms 给物理，ADR-B 分配 N ms 给 AI，合计超过总帧预算
- **依赖循环**：ADR-A 说系统 X 在 Y 之前初始化；ADR-B 说 Y 在 X 之前初始化
- **架构模式冲突**：ADR-A 对子系统使用事件驱动通信；ADR-B 对同一子系统使用直接函数调用
- **状态管理冲突**：两个 ADR 定义了相同游戏状态的权威（例如战斗 ADR 和角色 ADR 都声称拥有生命值）

对于找到的每个冲突：

```
## 冲突：[ADR-NNNN] vs [ADR-MMMM]
类型：[数据所有权 / 集成 / 性能 / 依赖 / 模式 / 状态]
ADR-NNNN 声称：[...]
ADR-MMMM 声称：[...]
影响：[如果两者都按编写的实现，什么会出问题]
解决方案选项：
  1. [选项 A]
  2. [选项 B]
```

### ADR 依赖排序

在冲突检测之后，分析所有 ADR 的依赖图：

1. **收集所有 `Depends On` 字段**从每个 ADR 的"ADR 依赖关系"部分
2. **拓扑排序**：确定正确的实现顺序——没有依赖的 ADR 排在前面（基础层），依赖这些的 ADR 排在后面，依此类推。
3. **标记未解决的依赖**：如果 ADR-A 的"Depends On"字段引用了一个仍为 `Proposed` 或不存在的 ADR，标记它：
   ```
   ⚠️  ADR-0005 依赖于 ADR-0002 —— 但 ADR-0002 仍为 Proposed。
       ADR-0005 在 ADR-0002 被接受之前无法安全实现。
   ```
4. **循环检测**：如果 ADR-A 依赖于 ADR-B 且 ADR-B 依赖于 ADR-A（直接或传递），标记为 `依赖循环`：
   ```
   🔴 依赖循环：ADR-0003 → ADR-0006 → ADR-0003
      必须打破此循环后两者才能实现。
   ```
5. **输出推荐的实现顺序**：
   ```
   ### 推荐的 ADR 实现顺序（拓扑排序）
   基础层（无依赖）：
     1. ADR-0001：[标题]
     2. ADR-0003：[标题]
   依赖基础层：
     3. ADR-0002：[标题]（需要 ADR-0001）
     4. ADR-0005：[标题]（需要 ADR-0003）
   功能层：
     5. ADR-0004：[标题]（需要 ADR-0002、ADR-0005）
   ```

## 阶段 5：引擎兼容性交叉检查

在所有 ADR 中，检查引擎一致性：

### 版本一致性
- 所有提及引擎版本的 ADR 是否同意同一版本？
- 如果任何 ADR 是为较旧的引擎版本编写的，标记为可能过时

### 截止后 API 一致性
- 收集所有 ADR 的"Post-Cutoff APIs Used"字段
- 对每个字段，对照相关模块参考文档进行验证
- 检查没有两个 ADR 对同一截止后 API 做出矛盾假设

### 已弃用 API 检查
- 在所有 ADR 中 Grep `deprecated-apis.md` 中列出的 API 名称
- 标记任何引用已弃用 API 的 ADR

### 缺少引擎兼容性部分
- 列出所有完全缺少引擎兼容性部分的 ADR
- 这些是盲点——它们的引擎假设未知

输出格式：
```
### 引擎审计结果
引擎：[name + version]
具有引擎兼容性部分的 ADR：X / Y 总数

已弃用 API 引用：
  - ADR-0002：使用 [deprecated API] —— 自 [version] 起已弃用

过时版本引用：
  - ADR-0001：为 [older version] 编写 —— 当前项目版本为 [version]

截止后 API 冲突：
  - ADR-0004 和 ADR-0007 都使用 [API]，假设不兼容
```

### 引擎专家咨询

在完成上述引擎审计后，通过 Task 召唤**主要引擎专家**进行领域专家第二意见：
- 读取 `.claude/docs/technical-preferences.md` 的 `Engine Specialists` 部分以获取主要专家
- 如果没有配置引擎，跳过此咨询
- 召唤 `subagent_type：[primary specialist]`，提供：所有包含引擎特定决策或 `Post-Cutoff APIs Used` 字段的 ADR、引擎参考文档以及阶段 5 审计结果。请他们：
  1. 确认或质疑每个审计发现——专家可能知道参考文档中未包含的引擎细微差别
  2. 识别 ADR 中审计可能遗漏的引擎特定反模式（例如使用错误的 Godot 节点类型、Unity 组件耦合、Unreal 子系统误用）
  3. 标记对引擎行为做出与实际锁定版本不同假设的 ADR

在阶段 5 输出下的 `### 引擎专家发现` 中纳入额外发现。这些结果影响最终裁决——专家识别的问题与审计识别的问题具有同等权重。

## 阶段 5b：设计修订标志（架构 → GDD 反馈）

对于阶段 5 中每个**高风险引擎发现**，检查是否有任何 GDD 做出了与已验证的引擎现实相矛盾的假设。

要检查的具体情况：

1. **截止后 API 行为与训练数据假设不同**：如果 ADR 记录了与默认 LLM 假设不同的已验证 API 行为，检查所有引用相关系统的 GDD。寻找基于旧（假设的）行为编写的设计规则。

2. **ADR 中已知的引擎限制**：如果 ADR 记录了已知的引擎限制（例如"Jolt 忽略 HingeJoint3D damp"、"D3D12 现在是默认后端"），检查围绕受影响功能设计机制的那些 GDD。

3. **已弃用 API 冲突**：如果阶段 5 标记了 ADR 中使用的已弃用 API，检查是否有任何 GDD 包含假设已弃用 API 行为的机制。

对于找到的每个冲突，记录在 GDD 修订标志表中：

```
### GDD 修订标志（架构 → 设计反馈）
这些 GDD 假设与已验证的引擎行为或已接受的 ADR 冲突。
GDD 应在其实施前修订。

| GDD | 假设 | 现实（来自 ADR/引擎参考） | 操作 |
|-----|-----------|--------------------------------------|--------|
| combat.md | "使用 HingeJoint3D damp 实现武器后坐力" | Jolt 忽略 damp —— ADR-0003 | 修订 GDD |
```

如果未找到修订标志，写入："无 GDD 修订标志——所有 GDD 假设与已验证的引擎行为一致。"

在询问之前，内联显示提议的更改——并排显示每个标记 GDD 的当前 systems-index 行和提议的更新行，以便用户确切看到将更改什么。

然后使用 `AskUserQuestion`：
- "我发现了 [N] 个 GDD 修订标志。我可以更新系统索引吗？"
  - [A] 是——立即应用所有 [N] 个更新到系统索引
  - [B] 先显示完整差异，然后再次询问
  - [C] 不——暂时保持系统索引不变

如果 [A]：应用更新。状态字段必须精确为 `Needs Revision` —— 不加括号（其他技能匹配该精确字符串，括号会破坏匹配）。
如果 [B]：显示完整的提议系统索引部分，然后使用 `AskUserQuestion` 再次询问。

## 错误恢复协议

如果任何被召唤的代理返回阻塞、错误或未能完成：

1. **立即呈现**："[AgentName]：阻塞 — [reason]"
2. **评估依赖关系**：如果阻塞代理的输出是后面阶段所需的，没有用户输入不要通过该阶段
3. **通过 AskUserQuestion 提供选项**，包含三个选择：
   - 跳过此代理并在最终报告中注明缺口
   - 以更窄的范围重试（更少的 GDD、单系统聚焦）
   - 在此处暂停并先解决阻塞
4. **始终生成部分报告**——输出已完成的内容，这样工作不会丢失

## 协作协议

1. **静默读取**——不要叙述每个文件读取
2. **显示矩阵**——在询问任何内容之前呈现完整的可追溯性矩阵；让用户看到状态
3. **不要猜测**——如果需求不明确，询问："[X] 是技术需求还是设计偏好？"
4. **在批准前草拟**——始终在对话中内联显示将要写入的内容（报告、更新的 ADR 部分、systems-index 行），然后再请求批准。从不要求写入用户尚未看到的内容。
5. **使用 `AskUserQuestion` 进行写入批准**——纯文本的"可以吗？"是不够的。使用带有标记选项 [A]/[B]/[C] 的结构化工具，以便用户可以在"立即写入"、"先显示完整草案"和"还不"之间选择。多文件变更集必须列出每个文件及其更改内容，然后一次性询问分组选项——而不是每个文件单独问一个纯文本问题。
6. **非阻塞性**——裁决是建议性的；用户决定是否即使面临关切甚至失败结果也继续
