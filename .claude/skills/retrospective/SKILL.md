## 阶段 1：解析参数

确定这是冲刺回顾（`sprint-N`）还是里程碑回顾（`milestone-name`）。

---

## 阶段 1b：检查现有回顾

在加载任何数据之前，使用 Glob 搜索现有的回顾文件：

- 对于冲刺回顾：`production/retrospectives/retro-[sprint-slug]-*.md`
  （同时检查 `production/sprints/sprint-[N]-retrospective.md` 作为备选位置）
- 对于里程碑回顾：`production/retrospectives/retro-[milestone-name]-*.md`

如果找到匹配的文件，使用 `AskUserQuestion`：
- 提示："发现了一个现有的回顾文件：[文件名]。您想如何处理？"
- 选项：
  - `[A] 更新现有文件 — 加载并添加/修订部分内容（使用新数据）`
  - `[B] 重新开始 — 生成新的回顾（将旧文件归档）`

如果选择 [A]：读取现有文件并保留其内容前进，用新数据修订各部分。
如果选择 [B]：以空白状态继续阶段 2。在写入新文件前，将现有文件重命名并添加 `-archived-[date]` 后缀。

---

## 阶段 2：加载冲刺或里程碑数据

从相应位置读取冲刺或里程碑计划：

- 冲刺计划：`production/sprints/`
- 里程碑定义：`production/milestones/`

**同时检查 `production/sprint-status.yaml`**：如果存在，与冲刺计划一起读取。它是实际故事完成状态的权威来源（状态：已完成、完成日期、阻塞项）。在阶段 3 中将其作为完成指标的主要来源。仅在 yaml 不存在时回退到 Markdown 扫描。注意 yaml 与冲刺计划之间的差异（例如，yaml 中有但计划中没有的故事，或反之）。

**如果文件不存在或为空**，输出：

> "未找到 [sprint/milestone] 的冲刺数据。请先运行 `/sprint-status` 生成冲刺数据，或手动提供冲刺详情。"

然后使用 `AskUserQuestion` 呈现两个选项：

- **[A] 手动提供数据** — 要求用户粘贴或描述冲刺任务、日期和结果；将其作为回顾的真实来源。
- **[B] 停止** — 中止技能。裁决：**BLOCKED** — 无冲刺数据可用。

如果用户选择 [A]，收集数据并继续到阶段 3，使用他们提供的内容。
如果用户选择 [B]，在此停止。

提取：计划任务、预估工作量、负责人和目标。

运行冲刺期间的 git 日志以了解实际提交内容和时间。使用 Bash 工具（在 Windows 上使用 Git Bash——`2>/dev/null` 是 bash 语法，不是 PowerShell）：

```
Bash: git log --oneline --since="4 weeks ago" 2>/dev/null || git log --oneline -20
```

如果从冲刺计划中知道冲刺持续时间，将 `--since` 日期调整为匹配。

---

## 阶段 3：分析完成情况和趋势

通过将计划与实际交付物进行比较，扫描已完成和未完成的任务。检查：

- 按计划完成的任务
- 已完成但与计划相比有修改的任务
- 结转（未完成）的任务
- 冲刺中期添加的任务（计划外工作）
- 已移除或缩减范围的任务

扫描代码库中的 TODO/FIXME 趋势：

- 统计当前的 TODO/FIXME/HACK 注释数量
- 与之前的冲刺数量比较（如有可用的之前的回顾）
- 注意技术债务是增长还是减少

读取之前的回顾（如有）来自 `production/retrospectives/`，检查：

- 之前的行动项是否已处理？
- 相同的问题是否重复出现？
- 速度趋势如何？

---

## 阶段 4：生成回顾

```markdown
## 回顾：[冲刺 N / 里程碑名称]
周期：[开始日期] -- [结束日期]
生成日期：[日期]

### 指标

| 指标 | 计划 | 实际 | 偏差 |
|--------|---------|--------|-------|
| 任务数 | [X] | [Y] | [+/- Z] |
| 完成率 | -- | [Z%] | -- |
| 故事点/工作天数 | [X] | [Y] | [+/- Z] |
| 发现的 Bug | -- | [N] | -- |
| 修复的 Bug | -- | [N] | -- |
| 新增计划外任务 | -- | [N] | -- |
| 提交次数 | -- | [N] | -- |

### 速度趋势

| 冲刺 | 计划 | 完成 | 完成率 |
|--------|---------|-----------|------|
| [N-2] | [X] | [Y] | [Z%] |
| [N-1] | [X] | [Y] | [Z%] |
| [N]（当前） | [X] | [Y] | [Z%] |

**趋势**：[上升 / 稳定 / 下降]
[一句话解释趋势]

### 做得好的方面
- [基于具体数据或示例的观察]
- [另一个正面观察]
- [认可带来回报的具体贡献或决策]

### 做得不好的方面
- [有可衡量的影响的具体问题——例如："功能 X 花费了 5 天而不是预估的 2 天，阻塞了任务 Y 和 Z"]
- [另一个有影响的问题]
- [不要归咎于个人——关注系统性原因]

### 遇到的阻塞项

| 阻塞项 | 持续时间 | 解决方案 | 预防措施 |
|---------|----------|------------|------------|
| [什么阻塞了进展] | [多长时间] | [如何解决的] | [如何防止再次发生] |

### 预估准确性

| 任务 | 预估 | 实际 | 偏差 | 可能原因 |
|------|-----------|--------|----------|--------------|
| [最被高估的任务] | [X] | [Y] | [+Z] | [原因] |
| [最被低估的任务] | [X] | [Y] | [-Z] | [原因] |

**总体预估准确性**：[X%] 的任务偏差在预估的 +/- 20% 以内

[分析：我们是否持续高估或低估？针对哪种类型的任务？应该应用什么调整？]

### 结转分析

| 任务 | 原始冲刺 | 结转次数 | 原因 | 处理方式 |
|------|----------------|---------------|--------|--------|
| [未完成的任务] | [冲刺 N-X] | [N] | [原因] | [完成 / 缩减范围 / 重新设计] |

### 技术债务状态
- 当前 TODO 数量：[N]（之前：[N]）
- 当前 FIXME 数量：[N]（之前：[N]）
- 当前 HACK 数量：[N]（之前：[N]）
- 趋势：[增长 / 稳定 / 减少]
- [注意任何关注领域]

### 先前行动项跟进

| 行动项（来自冲刺 N-1） | 状态 | 备注 |
|-------------------------------|--------|-------|
| [之前的行动] | [已完成 / 进行中 / 未开始] | [上下文] |

### 下个迭代的行动项

| # | 行动 | 负责人 | 优先级 | 截止日期 |
|---|--------|-------|----------|----------|
| 1 | [具体、可衡量的行动] | [谁] | [高/中/低] | [何时] |
| 2 | [另一个行动] | [谁] | [优先级] | [何时] |

### 流程改进
- [工作方式的具体变更，以及预期收益]
- [另一个改进——保持在 2-3 个可操作项，不要变成愿望清单]

### 总结
[2-3 句话的整体评估：这是一个好的冲刺/里程碑吗？未来最需要改变的一件事是什么？]
```

## 阶段 5：保存回顾

向用户展示回顾和主要发现（完成率、速度趋势、主要阻塞项、最重要的行动项）。

询问："我可以将此文件写入 `production/retrospectives/retro-sprint-[N]-[date].md` 吗？"（里程碑回顾使用 `production/retrospectives/retro-[milestone-name]-[date].md`）

如果同意，写入文件，并在需要时创建 `production/retrospectives/` 目录。裁决：**COMPLETE** — 回顾已保存。

如果不同意，在此停止。裁决：**BLOCKED** — 用户拒绝写入。

---

## 阶段 6：后续步骤

使用 `AskUserQuestion`：
- 提示："回顾已完成。行动项和速度数据已就绪。是否现在开始冲刺规划并预加载这些数据？"
- 选项：
  - `[A] 是 — 打开冲刺规划，预填充回顾行动项和速度变化`
  - `[B] 否 — 我准备好时会手动参考回顾文件`

如果用户选择 [A]：继续调用 `/sprint-plan new`，传递回顾文件路径以及行动项和速度变化的摘要，以便冲刺规划器可以引用它们。

- 如果是里程碑回顾，运行 `/gate-check` 以正式评估下一阶段的就绪状态。

### 指导原则

- 要诚实和具体。模糊的回顾（"沟通可以更好"）会产生模糊的改进。使用数据和示例。
- 关注系统性问题，而非个人责任。
- 行动项限制在 3-5 个。超过此数会分散注意力。
- 每个行动项必须有负责人和截止日期。
- 检查之前的行动项是否已完成。反复出现未解决的项目是流程问题的信号。
- 如果是里程碑回顾，还要评估里程碑目标是否实现，以及这对整个项目时间表意味着什么。
