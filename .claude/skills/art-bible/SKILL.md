## 阶段 2：视觉标识基础（第 1-4 部分）

这四个部分定义了核心视觉语言。**所有其他部分都衍生自它们。** 逐个编写并写入文件，然后才进入下一个。

### 第 1 部分：视觉标识陈述

**目标**：一行视觉规则加上 2-3 个支持性原则，用于解决视觉歧义。

如果 game-concept.md 中存在视觉锚点：呈现它并询问：
- "直接以此锚点为基础构建？"
- "在扩展之前先修改它？"
- "从新选项重新开始？"

**代理委派（强制）**：通过 Task 召唤 `art-director`：
- 提供：游戏概念（电梯游说、核心幻想）、完整支柱集、平台目标、阶段 1 框架中提到的任何参考游戏/艺术作品、视觉锚点（如果存在）
- 询问："为此游戏草拟一份视觉标识陈述。提供：（1）一行可以解决任何视觉决策歧义的视觉规则，（2）2-3 个支持性视觉原则，每个附带一句话设计测试（'当 X 有歧义时，此原则说选择 Y'）。所有原则直接锚定在已陈述的支柱上——每个原则必须服务于一个特定的支柱。"

向用户展示艺术总监的草稿。使用 `AskUserQuestion`：
- 选项：`[A] 锁定这个` / `[B] 修改一行规则` / `[C] 修改支持性原则` / `[D] 描述我自己的方向`

立即将批准的部分写入文件。

### 第 2 部分：情绪与氛围

**目标**：按游戏状态划分的情感目标——具体到光照美术师可以据此工作。

对于每个主要游戏状态（例如探索、战斗、胜利、失败、菜单——根据该游戏的状态进行调整），定义：
- 主要情绪/氛围目标
- 光照特征（一天中的时间、色温、对比度级别）
- 氛围描述词（3-5 个形容词）
- 能量水平（狂热 / 适中 / 沉思 / 等）

**代理委派**：通过 Task 召唤 `art-director`，提供视觉标识陈述和支柱集。询问："为此游戏定义每个主要游戏状态的情绪和氛围目标。要具体——'黑暗且不祥'不够。指明确切的情感目标、光照特征（暖/冷、高/低对比度、一天中的时间方向）以及至少一个承载该情绪的视觉元素。每个游戏状态必须在视觉上与其他状态有明显区别。"

立即将批准的部分写入文件。

### 第 3 部分：造型语言

**目标**：使该游戏世界视觉上连贯且可区分的几何词汇。

涵盖：
- 角色剪影哲学（缩略图尺寸下的可读性如何？每个原型的区分特征？）
- 环境几何（棱角分明/曲线流畅/有机/几何——哪个占主导地位及其原因？）
- UI 造型语法（UI 是否呼应世界美学，还是独立的 HUD 语言？）
- 主角形状 vs. 配角形状（什么吸引眼球，什么退居其次？）

**代理委派**：通过 Task 召唤 `art-director`，提供视觉标识陈述和情绪目标。询问："为此游戏定义造型语言。将每个造型原则连接回视觉标识陈述和特定的游戏支柱。解释这些形状选择在情感上向玩家传达了什么。"

立即将批准的部分写入文件。

### 第 4 部分：色彩系统

**目标**：一个完整、可生产的调色板系统，服务于美学和沟通需求。

涵盖：
- 主色调色板（5-7 种具有角色的颜色——不仅是十六进制代码，还有每种颜色在此世界中的含义）
- 语义色彩用法（红色传达什么？金色？蓝色？白色？建立色彩词汇）
- 每个生物群系或区域的色温规则（如果游戏有不同区域）
- UI 调色板（可能与世界调色板不同——明确界定差异）
- 色盲安全性：哪些语义颜色需要形状/图标/声音备份

**代理委派**：通过 Task 召唤 `art-director`，提供视觉标识陈述和情绪目标。询问："为此游戏设计色彩系统。每个语义色彩分配必须解释——为什么在这个世界中这种颜色表示危险/安全/奖励？识别哪些颜色对可能对色盲玩家不友好，并指定需要哪些后备提示。"

立即将批准的部分写入文件。

## 阶段 3：生产指南（第 5-8 部分）

这些部分将视觉标识转化为具体生产规则。它们应该足够具体，以至于外包团队可以在没有额外简报的情况下遵循。

### 第 5 部分：角色设计方向

**代理委派**：通过 Task 召唤 `art-director`，提供第 1-4 部分。询问："为此游戏定义角色设计方向。涵盖：玩家角色的视觉原型（如果有）、每角色类型的区分特征规则（玩家如何一眼区分敌人/NPC/盟友？）、表情/姿势风格目标（僵硬/表现力/写实/夸张）以及 LOD 理念（在游戏摄像机距离下保留多少细节？）。"

立即将批准的部分写入文件。

### 第 6 部分：环境设计语言

**代理委派**：通过 Task 召唤 `art-director`，提供第 1-4 部分。询问："为此游戏定义环境设计语言。涵盖：建筑风格及其与世界文化/历史的关系、纹理理念（手绘 vs. PBR vs. 风格化——为什么为此游戏做此选择？）、道具密度规则（稀疏/密集——每区域类型的驱动因素是什么？）以及环境叙事指南（哪些视觉细节应该无需文字讲述故事？）。"

立即将批准的部分写入文件。

### 第 7 部分：UI/HUD 视觉方向

**代理委派**：并行召唤：
- **`art-director`**：UI 的视觉风格——直通式 vs. 屏幕空间 HUD、排版方向（字体个性、粗细、大小层级）、图标风格（扁平/轮廓/插图/照片写实）、UI 元素的动画感觉
- **`ux-designer`**：UX 对齐检查——视觉方向是否支持该游戏所需的交互模式？标记美术方向与可读性/可访问性需求之间的任何冲突。

收集两者。如果它们冲突（例如艺术总监想要精致的直通式 UI 但 UX 设计师标记这会降低战斗可读性），明确呈现冲突，附带双方立场。不要静默解决——使用 `AskUserQuestion` 让用户决定。

立即将批准的部分写入文件。

### 第 8 部分：资产标准

**代理委派**：并行召唤：
- **`art-director`**：文件格式偏好、命名约定方向、纹理分辨率层级、LOD 级别期望、导出设置理念
- **`technical-artist`**：引擎特定的硬约束——每资产类别的多边形数量预算、纹理内存限制、材质插槽数量、导入器约束、`.claude/docs/technical-preferences.md` 中性能预算的任何内容

如果任何艺术偏好与技术约束冲突（例如艺术总监想要 4K 纹理但性能预算要求移动端 2K），明确解决冲突——同时说明理想标准和受限标准，并解释权衡。资产标准中的歧义是生产成本产生的地方。

立即将批准的部分写入文件。

## 阶段 4：参考方向（第 9 部分）

**目标**：一个精心策划的参考集，明确说明从每个来源中吸取什么以及避免什么。

**代理委派**：通过 Task 召唤 `art-director`，提供已完成第 1-8 部分。询问："为此游戏编制参考方向。提供 3-5 个参考来源（游戏、电影、艺术风格或特定艺术家）。对于每个：命名它，指定要从中汲取的确切视觉元素（不只是'整体美学'——特定的技术、色彩选择或构图规则），并指定要明确避免或偏离的内容（以防止'试图复制 X'的观感）。参考应是增量的——没有两个参考应指向完全相同的方向。"

立即将批准的部分写入文件。

- GDD 是否存在（如上检查）？→ 包含 /consistency-check 选项

**选项池 —— 仅在尚未完成时包含：**
- `[_] 运行 /map-systems —— 在编写 GDD 之前将概念分解为系统`（如果 systems-index.md 已存在则跳过）
- `[_] 运行 /setup-engine —— 配置引擎（设置引擎后资产标准可能需要重新审视）`（如果引擎已配置则跳过）

- `[_] 运行 /review-all-gdds —— 跨 GDD 一致性检查（技术支持关卡前必需）`（如果 gdd-cross-review-*.md 已存在则跳过）

## 协作协议

每个部分遵循：**问题 → 选项 → 决定 → 草案（来自 art-director 代理） → 批准 → 写入文件**

- 在召唤相关代理之前绝不草拟部分
- 每个部分批准后立即写入文件——不要批量处理
- 将所有代理分歧呈现给用户——绝不在 art-director 和 technical-artist 之间静默解决冲突
- 美术圣经是一份约束文档：它以视觉连贯性换取未来决策的限制。每个部分应感觉它有效地缩小了解决方案空间。
