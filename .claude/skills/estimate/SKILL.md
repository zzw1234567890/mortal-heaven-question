## 阶段 1：理解任务

从参数中读取任务描述（task description）。如果描述过于模糊而无法进行有意义的估算，请先请求澄清。

读取 CLAUDE.md 以获取项目上下文：技术栈、编码标准、架构模式以及任何估算指南。

如果任务涉及已记录的功能或系统，请从 `design/gdd/` 读取相关设计文档。

---

## 阶段 2：扫描受影响的代码

识别需要修改的文件和模块：

- 评估复杂度（规模、依赖数量、圈复杂度）
- 识别与其他系统的集成点
- 检查受影响区域的现有测试覆盖率
- 从 `production/sprints/` 读取过去的冲刺数据，查找类似已完成任务和历史速度

---

## 阶段 3：分析复杂度因素

**代码复杂度（Code Complexity）：**
- 受影响文件中的代码行数
- 依赖数量及耦合程度
- 是否触及核心/引擎代码（core/engine code） vs 叶子/功能代码（leaf/feature code）
- 是否可以遵循现有模式，或需要创建新模式

**范围（Scope）：**
- 涉及的系统数量
- 新代码 vs 修改现有代码
- 需要的新测试覆盖量
- 所需的数据迁移或配置变更

**风险（Risk）：**
- 新技术或不熟悉的库
- 不清晰或模糊的需求
- 对未完成工作的依赖
- 跨系统集成复杂度
- 性能敏感性

## 阶段 4：生成估算

```markdown
## 任务估算：[任务名称]
生成日期：[日期]

### 任务描述
[用 1-2 句话清晰重述任务]

### 复杂度评估

| 因素 | 评估 | 备注 |
|--------|-----------|-------|
| 受影响系统 | [列举] | [核心、玩法、UI 等] |
| 可能修改的文件 | [数量] | [关键文件见下方] |
| 新代码 vs 修改 | [比例] | |
| 集成点 | [数量] | [哪些系统交互] |
| 需要的测试覆盖 | [低 / 中 / 高] | |
| 现有模式可用 | [是 / 部分 / 否] | |

**可能受影响的关键文件：**
- `[path/to/file1]` -- [此处更改内容]

### 工作量估算

| 场景 | 天数 | 假设 |
|----------|------|------------|
| 乐观 | [X] | 一切顺利，无意外 |
| 预期 | [Y] | 正常节奏，小问题，一轮评审 |
| 悲观 | [Z] | 出现重大未知情况，受阻一天 |

**建议预算：[Y 天]**

### 置信度：[高 / 中 / 低]

[解释哪些因素驱动此特定任务的置信度等级。]

### 风险因素

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|-----------|--------|------------|

### 依赖关系

| 依赖项 | 状态 | 延迟影响 |
|-----------|--------|-------------------|

### 建议分解

| # | 子任务 | 估算 | 备注 |
|---|----------|----------|-------|
| 1 | [研究 / 技术预研] | [X 天] | |
| 2 | [核心实现] | [X 天] | |
| 3 | [测试与验证] | [X 天] | |
| | **总计** | **[Y 天]** | |

### 备注与假设
- [影响估算的关键假设]
- [关于范围边界的任何说明]
```

输出估算结果并附上简要摘要：建议预算、置信度等级以及最大的单一风险因素。

本技能为只读——不写入任何文件。结论：**完成**——估算已生成。

---

## 阶段 5：后续步骤

- 如果置信度为低：建议在承诺执行之前先进行限时技术预研（`/prototype`）。
- 如果任务超过 10 天：建议通过 `/create-stories` 将其拆分为更小的故事。
- 要安排任务：运行 `/sprint-plan update` 将其添加到下一个冲刺。

### 指南

- 始终给出范围（乐观 / 预期 / 悲观），绝不使用单一数值
- 建议预算应为预期估算，而非乐观估算
- 以半天为增量进行四舍五入——对于超过一天的任务，以小时估算意味着虚假精度
- 不要默默地为估算加余量——明确指出风险，以便团队做出决策
