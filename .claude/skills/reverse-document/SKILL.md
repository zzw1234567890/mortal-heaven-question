# 逆向文档

此技能分析现有实现（代码、原型、系统）并生成适当的设计或架构文档。在以下情况下使用：
- 你在没有先编写设计文档的情况下构建了功能
- 你接手了一个没有文档的代码库
- 你制作了一个机制原型并需要将其正式化
- 你需要为现有代码记录"为什么"

---

## 工作流程

## 阶段 1：解析参数

**格式**：`/reverse-document <type> <path>`

**类型选项**：
- `design` → 生成游戏设计文档（GDD 章节）
- `architecture` → 生成架构决策记录（ADR）
- `concept` → 从原型生成概念文档

**路径**：要分析的目录或文件
- `src/gameplay/combat/` → 所有战斗相关代码
- `src/core/event-system.cpp` → 特定文件
- `prototypes/stealth-mech/` → 原型目录

**示例**：
```bash
/reverse-document design src/gameplay/magic-system
/reverse-document architecture src/core/entity-component
/reverse-document concept prototypes/vehicle-combat
```

## 阶段 2：分析实现

**阅读并理解代码/原型**：

**对于设计文档（GDD）：**
- 识别机制、规则、公式
- 提取游戏数值（伤害、冷却时间、范围）
- 查找状态机、能力系统、进度系统
- 检测代码中处理的边界情况
- 映射依赖关系（哪些系统交互）

**对于架构文档（ADR）：**
- 识别模式（ECS、单例、观察者等）
- 理解技术决策（线程、序列化等）
- 映射依赖关系和耦合度
- 评估性能特征
- 寻找约束条件和权衡

**对于概念文档（原型分析）：**
- 识别核心机制
- 提取涌现式游戏模式
- 记录有效和无效的部分
- 寻找技术可行性见解
- 记录玩家幻想/感受

## 阶段 3：提出澄清性问题

**不要**仅仅描述代码。**询问**意图：

**设计问题**：
- "我看到一个[资源]系统在[活动]期间消耗。这是为了：
  - 节奏控制（防止滥用）？
  - 资源管理（策略深度）？
  - 还是其他目的？"
- "[机制]似乎是核心。这是核心支柱还是辅助功能？"
- "[数值]随[因素]呈指数级增长。这是有意设计还是需要重新平衡？"

**架构问题**：
- "你使用了服务定位器模式。选择这个是因为：
  - 可测试性（模拟依赖）？
  - 解耦（减少硬引用）？
  - 还是从现有代码继承的？"
- "我看到手动内存管理而不是智能指针。是性能要求还是遗留代码？"

**概念问题**：
- "这个原型强调潜行而非战斗。这是预期的支柱吗？"
- "玩家似乎利用抓钩加速。是功能还是 Bug？"

## 阶段 4：呈现发现

在起草之前，展示你发现的内容：

```
我已分析了[路径]。以下是我的发现：

已实现的机制：
- [机制A] 具有[属性]（例如，时机窗口、冷却时间）
- [机制B]（例如，两个状态之间的交互）
- [资源]系统（在[动作]时消耗，在[条件]下恢复）
- [状态]系统（累积，触发[效果]）

发现的公式：
- [输出] = [使用发现变量的公式]
- [次要输出] = [公式]

意图不明确的领域：
1. [资源]系统——节奏控制还是资源管理？
2. [机制]——核心支柱还是辅助功能？
3. [数值]缩放——故意设计还是需要调整？

在起草设计文档前，您能否澄清这些要点？
```

等待用户澄清意图后再起草。

## 阶段 5：使用模板起草文档

根据类型，使用适当的模板：

| 类型 | 模板 | 输出路径 |
|------|----------|-------------|
| `design` | `templates/design-doc-from-implementation.md` | `design/gdd/[system-name].md` |
| `architecture` | `templates/architecture-doc-from-code.md` | `docs/architecture/[decision-name].md` |
| `concept` | `templates/concept-doc-from-prototype.md` | `prototypes/[name]/CONCEPT.md` 或 `design/concepts/[name].md` |

**起草结构**：
- 捕捉**现有内容**（机制、模式、实现）
- 记录**存在的原因**（与用户明确的意图）
- 识别**缺失的内容**（未处理的边界情况、设计缺口）
- 标记**后续工作**（平衡调整、缺失功能）

## 阶段 6：展示草稿并请求批准

**协作协议**：
```
我已根据您的代码和澄清说明起草了[系统名称]设计文档。

[显示关键章节：概述、机制、公式、设计意图]

我做的补充：
- 根据您的说明，将[机制]记录为"[意图]"
- 添加了代码中未包含的边界情况（例如，如果[资源]在[动作]中降至 0 会怎样？）
- 标记了平衡问题：[缩放类型]在[边界条件]下的缩放

标记为不完整的章节：
- "[系统]与[其他系统]的交互"（尚未完全实现）
- "[变体或功能]"（目前仅实现了[子集]）

我可以将其写入 design/gdd/[system-name].md 吗？
```

等待批准。用户可能在写入前要求更改。

## 阶段 7：写入文档并添加元数据

批准后，使用特殊标记写入文件：

```markdown
---
status: reverse-documented
source: [path/]
date: [today]
verified-by: [用户名称]
---

# [系统名称] 设计

> **注意**：本文档是从现有实现逆向推导的。
> 它记录了当前行为和明确的设计意图。某些章节可能
> 在实现不完整或意图不明确时标记为不完整。

[文档其余部分...]
```

## 阶段 8：标记后续工作

写入后，建议后续步骤：

```
✅ 已写入 design/gdd/combat-system.md

推荐的后续工作：
1. 运行 /balance-check 检查战斗公式（指数缩放问题）
2. 为耐力系统架构决策创建 ADR
3. 实现缺失的边界情况：
   - 连击中途耐力消耗行为
   - 闪避期间的硬直状态
4. 实现武器多样性时扩展设计文档

是否需要我现在处理其中任何一项？
```

## 模板选择逻辑

| 如果分析... | 使用模板... | 原因... |
|----------------|-----------------|------------|
| `src/gameplay/*` | design-doc-from-implementation.md | 游戏机制 → GDD |
| `src/core/*`, `src/ai/*` | architecture-doc-from-code.md | 核心系统 → ADR |
| `prototypes/*` | concept-doc-from-prototype.md | 实验 → 概念文档 |
| `src/networking/*` | architecture-doc-from-code.md | 技术系统 → ADR |
| `src/ui/*` | design-doc-from-implementation.md | UI/UX → 设计规范 |

## 协作协议

此技能遵循协作设计原则：

1. **先分析**：读取代码，理解实现
2. **询问意图**：问"为什么"，而不仅仅是"是什么"
3. **呈现发现**：展示发现，突出不明确领域
4. **用户澄清**：区分意图与偶然
5. **起草文档**：基于现实 + 意图创建文档
6. **展示草稿**：显示关键章节，解释补充内容
7. **获取批准**："我可以写入 [filepath] 吗？"批准后：裁决：**COMPLETE** — 文档已生成。拒绝后：裁决：**BLOCKED** — 用户拒绝写入。
8. **标记后续工作**：建议相关工作，不自动执行

**永远不要假设意图。在记录"为什么"之前务必询问。**
