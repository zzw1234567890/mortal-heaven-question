如果未提供参数，直接输出使用指南并退出，不生成任何子代理：
> 用法：`/team-narrative [叙事内容描述]` — 描述要创作的故事内容、场景或叙事区域（例如 `boss 遭遇过场动画`、`派系介绍对话`、`教程叙事`）。请勿在此处使用 `AskUserQuestion`；直接输出指南。

当本 skill 被调用并带有参数时，通过结构化管线编排叙事团队。

**决策节点：** 在每个阶段过渡时，使用 `AskUserQuestion` 向用户展示子代理的提案作为可选选项。在对话中完整记录代理的分析，然后以简洁的标签记录决策。用户必须批准后才能进入下一阶段。

## 阶段 0：解析审查模式

1. 如果 `--review [mode]` 作为参数传入，使用该模式。
2. 否则读取 `production/review-mode.txt` — 使用其中写入的内容。
3. 否则默认为 `lean`。

模式：
- `full` — 按描述生成所有总监和主管关卡
- `lean` — 跳过总监关卡，除非是阶段关卡类型（CD-PHASE-GATE、TD-PHASE-GATE、PR-PHASE-GATE、AD-PHASE-GATE）
- `solo` — 完全跳过所有总监关卡生成；在没有任何代理关卡的情况下运行 skill

将解析后的模式存储起来，用于所有后续阶段。

## 团队组成
- **narrative-director（叙事总监）** — 故事弧、角色设计、对话策略、叙事愿景
- **writer（编剧）** — 对话撰写、背景条目、物品描述、游戏内文本
- **world-builder（世界构建师）** — 世界规则、派系设计、历史、地理、环境叙事
- **art-director（艺术总监）** — 角色视觉设计、环境视觉叙事、过场动画/电影质感
- **level-designer（关卡设计师）** — 服务于叙事的关卡布局、节奏、环境叙事节拍
- **localization-lead（本地化负责人）** — 本地化就绪性——标记不可本地化的字符串、文化假设和 i18n 缺口

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: narrative-director` — 故事弧、角色设计、叙事愿景
- `subagent_type: writer` — 对话撰写、背景条目、游戏内文本
- `subagent_type: world-builder` — 世界规则、派系设计、历史、地理
- `subagent_type: art-director` — 角色视觉资料、环境视觉叙事、电影质感
- `subagent_type: level-designer` — 服务于叙事的关卡布局、节奏
- `subagent_type: localization-lead` — 本地化就绪性——标记不可本地化的字符串、文化假设和 i18n 缺口

始终为每个代理提供完整的上下文（叙事简报、背景依赖、角色资料）。在管线允许的情况下并行启动独立代理（例如阶段 2 的代理可以同时运行）。

## 管线

### 阶段 1：叙事方向
委派给 **narrative-director**：
- 定义此内容的叙事目的：它服务于哪个故事节拍？
- 确定所涉及的角色、他们的动机以及这如何融入整体故事弧
- 设定情感基调和节奏目标
- 指定任何背景依赖或由此引入的新背景
- 输出：包含故事需求的叙事简报

### 阶段 2：世界基础（并行）
并行委派——同时发出所有三个 Task 调用，无需等待任何结果：
- **world-builder**：创建或更新与内容相关的派系、地点和历史的背景条目。对照现有背景检查矛盾之处。为新条目设定正典等级。
- **writer**：使用角色声线草拟角色对话。确保所有台词不超过 120 个字符，使用命名占位符表示变量，并具备本地化就绪性。
- **art-director**：为此内容中出现的关键角色定义角色视觉设计方向（轮廓、视觉原型、区分特征）。为每个关键场景指定环境视觉叙事元素（道具构成、光照说明、空间布置）。为任何过场动画或脚本序列定义基调调色板和电影方向。

### 阶段 3：关卡叙事整合
委派给 **level-designer**：
- 审查叙事简报和背景基础
- 在关卡中设计环境叙事元素
- 放置叙事触发器、对话区域和发现点
- 确保节奏既服务于游戏性又服务于故事

### 阶段 4：审查与一致性
委派给 **narrative-director**：
- 对照角色声线审查所有对话
- 验证新旧条目之间的背景一致性
- 确认叙事节奏与关卡设计保持一致
- 检查所有谜团是否有文档记录的"真实答案"

### 阶段 5：打磨（并行）
并行委派：
- **writer**：最终自我审查——验证没有台词超出对话框限制，所有文本使用字符串键（非原始字符串），占位符变量名保持一致
- **localization-lead**：验证 i18n 合规性——检查字符串键命名约定，标记任何包含硬编码格式的字符串（这些格式在翻译后无法保留），验证对扩展语言（德语/芬兰语通常 +30%）的字符限制余量，确认文本中不存在需要特定语言环境变体的文化假设
- **world-builder**：为所有新背景条目最终确定正典等级

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、出错或无法完成：

1. **立即上报**：在继续依赖阶段之前向用户报告"[代理名称]：BLOCKED — [原因]"
2. **评估依赖关系**：检查被阻塞代理的输出是否为后续阶段所需。如果是，在没有用户输入的情况下不要越过该依赖点继续。
3. **通过 AskUserQuestion 提供选项**，包含以下选择：
   - 跳过此代理并在最终报告中记录缺口
   - 以更窄的范围重试
   - 在此停止，先解决阻塞因素
4. **始终生成部分报告** — 输出已完成的内容。绝不要因为一个代理阻塞而丢弃已完成的工作。

常见阻塞因素：
- 输入文件缺失（故事未找到，GDD 不存在）→ 重定向到创建该文件的 skill
- ADR 状态为 Proposed → 不实施；先运行 `/architecture-decision`
- 范围过大 → 通过 `/create-stories` 拆分为两个 story
- ADR 与 story 之间的指令冲突 → 上报冲突，不要猜测

## 文件写入协议

所有文件写入（叙事文档、对话文件、背景条目）都委派给通过 Task 生成的子代理。每个子代理执行"May I write to [path]?"协议。此编排器不直接写入文件。

## 输出

一份总结报告，涵盖：叙事简报状态、创建/更新的背景条目、已撰写的对话台词、关卡叙事整合点、一致性审查结果以及任何未解决的矛盾。

裁决：**COMPLETE** — 叙事内容已交付。

如果管线因依赖关系未解决而停止（例如背景矛盾或缺失前提条件未被用户解决）：

裁决：**BLOCKED** — [原因]

## 后续步骤

- 运行 `/design-review` 对叙事文档进行一致性验证。
- 在对话最终确定后，运行 `/localize extract` 提取新字符串进行翻译。
- 运行 `/dev-story` 在引擎中实现对话触发器和叙事事件。
