# 项目阶段检测

该技能扫描您的项目，以确定其当前开发阶段、工件的完整性以及需要注意的差距。在以下情况下尤其有用：
- 开始处理现有项目时
- 新加入代码库时
- 检查里程碑前缺少什么时
- 了解"我们在哪个阶段？"时

---

## 工作流程

### 1. 扫描关键目录

分析项目结构和内容：

**设计文档**（`design/`）：
- 统计 `design/gdd/*.md` 中的 GDD 文件数量
- 检查 game-concept.md、game-pillars.md、systems-index.md
- 如果 systems-index.md 存在，统计总系统数与已设计系统数
- 分析完整性（概览、详细设计、边界情况等）
- 统计 `design/narrative/` 中的叙事文档数量
- 统计 `design/levels/` 中的关卡设计数量

**源代码**（`src/`）：
- 统计源文件数量（语言无关）
- 识别主要系统（包含 5 个以上文件的目录）
- 检查 core/、gameplay/、ai/、networking/、ui/ 目录
- 估算代码行数（粗略规模）

**生产工件**（`production/`）：
- 检查是否有活跃的 sprint 计划
- 查找里程碑定义
- 寻找路线图文档

**原型**（`prototypes/`）：
- 统计原型目录数量
- 检查 README（已记录 vs 未记录）
- 评估原型是已归档还是活跃中

**架构文档**（`docs/architecture/`）：
- 统计 ADR（架构决策记录）数量
- 检查概览/索引文档

**测试**（`tests/`）：
- 统计测试文件数量
- 估算测试覆盖率（粗略启发式）

### 2. 分类项目阶段

根据扫描到的工件确定阶段。首先检查 `production/stage.txt` — 如果存在，使用其值（来自 `/gate-check` 的显式覆盖）。否则，使用以下启发式自动检测（从最先进的阶段向后检查）：

| 阶段 | 指标 |
|-------|-----------|
| **构思** | 无游戏概念文档，头脑风暴阶段 |
| **系统设计** | 游戏概念存在，系统索引缺失或不完整 |
| **技术搭建** | 系统索引存在，引擎未配置 |
| **预制作** | 引擎已配置，`src/` 中源代码文件 < 10 个 |
| **制作** | `src/` 中源代码文件 ≥ 10 个，活跃开发中 |
| **润色** | 仅显式设置（由 `/gate-check` 的 Production → Polish 关卡设置） |
| **发布** | 仅显式设置（由 `/gate-check` 的 Polish → Release 关卡设置） |

### 3. 协作式差距识别

**不要**仅仅列出缺失的文件。相反，**提出澄清性问题**：

- "我看到了战斗代码（`src/gameplay/combat/`），但没有 `design/gdd/combat-system.md`。这是先制作了原型，还是我们应该反向文档化？"
- "你有 15 个 ADR，但没有架构概览。我应该创建一个来帮助新贡献者吗？"
- "`production/` 中没有 sprint 计划。你们是否在其他地方（Jira、Trello 等）跟踪工作？"
- "我找到了游戏概念，但没有系统索引。你们是否已将概念分解为各个系统，还是应该运行 `/map-systems`？"
- "原型目录中有 3 个项目没有 README。这些是实验性的，还是需要编写文档？"

### 4. 生成阶段报告

使用模板：`.claude/docs/templates/project-stage-report.md`

**报告结构**：
```markdown
# 项目阶段分析

**日期**：[日期]
**阶段**：[构思/系统设计/技术搭建/预制作/制作/润色/发布]
**阶段可信度**：[通过 — 明确检测到 / 关注 — 信号模糊 / 失败 — 关键差距阻碍进展]

## 完整性概览
- 设计：[X%]（[N] 个文档，[差距]）
- 代码：[X%]（[N] 个文件，[系统]）
- 架构：[X%]（[N] 个 ADR，[差距]）
- 生产：[X%]（[状态]）
- 测试：[X%]（[覆盖率估算]）

## 已识别差距
1. [差距描述 + 澄清性问题]
2. [差距描述 + 澄清性问题]

## 推荐后续步骤
[基于阶段和角色的优先级排序列表]
```

### 5. 按角色筛选建议（可选）

如果用户提供了角色参数（例如 `/project-stage-detect programmer`）：

**程序员**：
- 关注架构文档、测试覆盖率、缺失的 ADR
- 代码与文档之间的差距

**设计师**：
- 关注 GDD 完整性、缺失的设计章节
- 原型文档

**制作人**：
- 关注 sprint 计划、里程碑跟踪、路线图
- 跨团队协调文档

**通用**（无角色）：
- 所有差距的整体视图
- 跨领域最高优先级的项目

### 6. 写入前请求批准

**协作协议**：
```
我已分析您的项目。以下是我的发现：

[显示概要]

已识别差距：
1. [差距 1 + 问题]
2. [差距 2 + 问题]

推荐后续步骤：
- [优先级 1]
- [优先级 2]
- [优先级 3]

我可以将完整的阶段分析写入 production/project-stage-report.md 吗？
```

在创建文件前等待用户批准。

## 后续操作

生成报告后，建议相关后续步骤：

- **已有概念但无系统索引？** → `/map-systems` 分解为系统
- **缺少设计文档？** → `/reverse-document design src/[system]`
- **缺少架构文档？** → `/architecture-decision` 或 `/reverse-document architecture`
- **原型需要文档？** → `/reverse-document concept prototypes/[name]`
- **没有 sprint 计划？** → `/sprint-plan`
- **临近里程碑？** → `/milestone-review`

---

## 协作协议

该技能遵循协作设计原则：

1. **先提问**：询问差距，不要假设
2. **提供选项**："我应该创建 X，还是在其他地方跟踪？"
3. **用户决定**：等待指示
4. **显示草稿**：展示报告概要
5. **获取批准**："我可以写入 production/project-stage-report.md 吗？"

**绝不**静默写入文件。**始终**展示发现并在创建工件前询问。
