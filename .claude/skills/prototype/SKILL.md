## 目的

这是**概念原型** — 一个快速、一次性的构建，回答一个问题：
*"这个核心创意实际玩起来有趣吗？"*

**默认用途** — 在 `/brainstorm` 和 `/setup-engine` 之后、编写 GDD 或架构文档之前立即运行。其判决决定了该概念是否值得投入完整的规划设计文档。

**制作中期？** 您也可以在任意阶段运行此功能，以测试特定机制、设计变更或技术问题。传入 `--spike` 激活突击模式：一个轻量级 ~4 小时的构建，无需 GDD 前置条件，无阶段关卡影响。

**已有完整的 GDD 和架构？** 要在投入制作之前验证完整的游戏循环，请运行 `/vertical-slice`。

## Phase 1：定义问题

解析审查模式（一次性，本次运行所有关卡生成时使用）：
1. 如果传入了 `--review [full|lean|solo]` → 使用该值
2. 否则读取 `production/review-mode.txt` → 使用该值
3. 否则 → 默认使用 `lean`

**检查突击模式：** 如果传入了 `--spike`，跳转到本技能底部的**突击模式**章节。

否则，使用 `AskUserQuestion` 确认意图后再继续：

- **提示**："您希望如何使用本次原型会期？"
- **选项**：
  - `Prototype this concept` — 构建一次性原型，在编写 GDD 前验证核心创意是否有趣（1–3 天）
  - `Skip — concept already proven` — 我有足够证据证明此方案可行；记录下来，直接进入设计阶段
  - `Mid-production spike` — 我已在制作阶段，想快速测试特定机制或技术问题（~4 小时，无阶段关卡影响）

**如果选择"跳过 — 概念已证明"：**
询问（纯文本，非控件）："您有哪些证据证明该概念可行？"
记录一行回答，然后停止。备注："概念原型已跳过 — 证据：[回答]。"建议下一步：`/map-systems` 或 `/design-system [mechanic]`。

**如果选择"制作中期突击"**：跳转到下面的**突击模式**章节。

**如果选择"Prototype this concept"**：继续执行下面的 Phase 1。

---

**关于原型策略的说明：** 关于成功独立游戏开发的研究一致表明，构建 2-3 个概念变体并让最好的那个胜出，远比迭代一个概念直到它可用更有可能成功。这是您的第一个原型，不一定是唯一的一个。如果此原型产生 PIVOT 判决，请考虑是优化此概念，还是以不同角度重新开始同一个游戏创意并制作原型。

**游戏 jAm 作为原型载体：** 如果您已经在计划概念原型，考虑将其安排在游戏 jAm 期间（Ludum Dare、GMTK Game Jam、Global Game Jam）。jAm 提供强制时间盒（48-72 小时）、立即分发给成千上万个评价和评论早期构建的玩家，以及一个从设计上防止范围蔓延的截止日期。许多已发布的游戏（Celeste、VVVVVV）最初就是 jAm 原型。非必需 — 但如果时间合适，值得考虑。

从参数中读取概念描述。在构建任何内容之前，定义此原型必须回答的**可证伪假设**：

> *"如果玩家[做 X]，他们会感受到[Y] — 如果我们观察到[可测量信号 Z]，我们就知道这是真的。"*

好的示例："如果玩家使用抓钩摆荡，移动体验会流畅 — 如果玩家在拿起抓钩后 2 分钟内连续完成 3 次以上摆荡而不停止，我们就知道这一点。"

糟糕的示例："这玩起来有趣吗？" ← 不可测试，不可证伪。

**如果概念太模糊无法形成假设，在此停止。** 请用户在继续之前缩小问题范围。没有明确问题的原型就是在浪费时间。

同时询问：**"这个概念中最冒险的假设是什么？"** 这是原型应该首先测试的内容 — 不是最容易的部分，而是最冒险的部分。

## Phase 3：选择原型路径

选择原型路径。如果传入了 `--path [html|engine|paper]`，使用该值。否则，先使用此快速参考，然后阅读下面的完整路径详情：

| 游戏类型 | 推荐路径 | 关键原因 |
|-------|-----------------|------------|
| 平台游戏 / 动作 / 格斗 | **引擎** | 手感就是假设；浏览器延迟会产生错误结果 |
| 竞速 / 体育 | **引擎** | 同样 — 时机和物理反馈是关键 |
| 俯视角射击 / 双摇杆 | **引擎** | 瞄准手感对时机敏感 |
| 益智（逻辑） | **HTML** 或 **纸面** | 时机不是关键；逻辑和清晰度才是 |
| 卡牌游戏 | **纸面** 优先 | 用手工迭代最快，之后再接触代码 |
| 叙事 / 视觉小说 | **纸面**（Twine / Ink / Yarn Spinner） | 故事就是机制 — 无需代码开销即可测试 |
| 策略 / 4X / 城市建造 | **纸面**（电子表格模拟） | 在构建之前验证经济和发展规则 |
| Roguelike（系统密集型） | **纸面** → 引擎 | 在构建之前验证规则集是否有趣 |
| 放置 / 点击 / 增量 | **HTML** | 回合制逻辑，无需手感敏感性 |
| 节奏游戏 | **纸面** 优先（在音频中设计关卡） | 在引擎存在之前设计关卡 |
| RPG / 开放世界 | **纸面** → 引擎 | 系统复杂性：先验证规则，再验证手感 |
| 恐怖 / 氛围 | **引擎** | 氛围需要真实渲染 |

**经验法则：** "手感对吗？" → 引擎。"这些规则有趣吗？" → 纸面。"这个逻辑正确吗？" → HTML 或纸面。

### 路径：HTML（浏览器可玩）

**最适合：** 益智游戏、卡牌游戏、回合制策略、文字游戏、放置游戏、俯视角逻辑游戏。任何时机精度不重要的类型。

**可靠性：** ~85–90% 一次成功。智能体编写一个自包含的 HTML 文件，用户在浏览器中打开 — 无需安装。

**局限性 — 浏览器延迟会扭曲游戏手感。** 浏览器引入 50–133ms 的渲染差异。这使得 HTML 原型对于动作游戏、平台游戏、格斗游戏或任何需要测试输入时机、跳跃弧线或碰撞手感的游戏从根本上不可靠。如果手感是假设，请使用引擎路径。

**此路径的替代工具：** PICO-8（极端约束，适合复古街机概念，一个命令即可导出网页）、Phaser.js（能力更强的浏览器游戏框架，仍无需安装）、或 Twine（叙事/选择类游戏）。这些工具在其各自类型中比原生 HTML 更快 — 如果合适请推荐它们。

**输出：** 单个 `prototype.html`（或等效的 PICO-8/Phaser 文件），用户在任何浏览器中打开即可。

**分发 — HTML 路径的最大优势：** 与引擎原型不同，此构建可在几分钟内触达全球真实玩家。积极利用这一点：
- **itch.io** — 上传文件，分享链接，数小时内即可获得游戏次数和书面反馈。免费。独立游戏社区在这里游玩粗糙的构建，不期望润色。这是零成本的真实外部验证。
- **Loom + 文件分享** — 通过 Google Drive/Dropbox 分享，请某人用 Loom 录制他们的屏幕和音频。您可以获得真实的第一印象反应和困惑的视频，无需同步安排。
- **r/playmygame 或 r/WebGames**（Reddit） — 活跃的社区，专门测试早期构建并给出自发的诚实反馈。
- **游戏开发 Discord 服务器**（GMTK、Brackeys、GameDev.tv） — 成员经常互相测试原型；HTML 文件是最低门槛的请求。

---

### 路径：引擎（引擎项目）

**最适合：** 动作游戏、平台游戏、物理密集型游戏、任何时时刻刻手感就是假设的游戏。在 HTML 延迟会扭曲结果时使用。

**可靠性：** ~50–60% 一次成功。预计需要 2–4 轮迭代 — 这很正常，不是失败。

**局限性 — 需要引擎已安装并运行。** 此路径是一个多轮协作循环：
1. 智能体编写代码
2. 用户在引擎中运行
3. 用户报告错误或观察
4. 智能体修复并迭代

**沉没成本规则：** 如果用户已经迭代超过 2 小时仍未达到可玩状态，停止。范围太大或问题本身有误。重新定义假设并大力简化，或切换到纸面路径。

**输出：** `prototypes/[name]-concept/` 下的最小可运行引擎项目。

**更轻量的替代方案 — Love2D（Lua）：** 如果项目引擎（Godot、Unity、Unreal）感觉太重而不适合一次性构建，考虑 Love2D — 一个最小的 2D 框架，几分钟即可安装，无需项目脚手架，原生渲染无浏览器延迟。许多独立开发者用于快速 2D 动作和平台游戏原型（Balatro 在 Love2D 中制作原型；Nuclear Throne 的早期构建使用过它）。它介于 HTML 开销和完整引擎开销之间：比打开浏览器重，比搭建完整引擎项目轻。最适合在项目引擎偏向 3D 或需要大量时间配置时进行 2D 动作/平台游戏手感验证。

---

### 路径：纸面（规则文档 + 游戏日志）

**最适合：** 策略游戏、卡牌游戏、桌游式机制、经济系统、进度循环、任何可以手动模拟逻辑的游戏。在需要验证规则而非手感时适用于任何类型。

**可靠性：** 100%。无需代码、无需引擎、无需安装。

**局限性 — 无法验证时时刻刻的手感。** 纸面原型证明规则内部一致且决策有趣。无法告诉您跳跃手感是否正确或爆炸是否令人满意。

**纸面试玩观察协议（与 5+ 人一起运行）：**
1. 说明规则一次。交给他们规则总结表。然后退后。
2. 不要再解释。不要帮忙。不要澄清。困惑就是数据。
3. 静默观察。记录他们放慢速度、重读或提问的每一个时刻。
4. 会话结束后，只问一个问题："什么让你感到困惑？" — 不是"你喜欢吗？"
5. 每次迭代使用新的测试员。同一个人无法提供新的第一印象数据。
6. 如果 3+ 个测试员遇到同一个困惑点，该规则有问题 — 在重新测试前重新设计它。

**输出：** 一份可打印的规则文档 + 一份显示一次模拟会话的完整游戏日志。

**此路径的叙事工具：** 对于对话密集型和故事驱动的游戏，跳过通用规则文档 — 使用专用的叙事脚本工具：
- **Twine** — 零代码超文本小说；适合分支结构实验和选择影响测试
- **Ink**（Inkle） — 纯文本脚本语言，用于 *80 Days*、*Heaven's Vault* 和 *Overboard*；可直接导出到 Unity 和 Godot
- **Yarn Spinner** — 对话脚本，用于 *A Short Hike*、*DREDGE* 和 *Night in the Woods*；原生集成 Unity 和 Godot

三者都让您在几分钟内编写和试玩分支对话。叙事原型的关键指标：**到达第一个情感节拍的时间** — 需要多少轮对话玩家才能感受到什么？如果需要超过 3-4 轮，开场太慢。

---

评估哪种路径最适合假设，然后使用 `AskUserQuestion` 并预先说明您的建议：

- **提示**："您想使用哪条原型路径？（根据您的概念，我推荐 [路径] — [一句话理由]。）"
- **选项**：
  - `HTML — 浏览器原型` — 益智、卡牌、回合制、策略、放置。双击打开，无需安装。85–90% 可靠。**不适用于动作游戏** — 浏览器延迟会扭曲手感。
  - `Engine — 原生原型` — 动作、平台、物理、或任何手感就是假设的游戏。50–60% 一次成功；2–4 轮迭代正常。需要引擎已安装。
  - `Paper — 规则文档 + 游戏日志` — 策略、经济、逻辑、桌游式机制。100% 可靠。无法验证手感。

## Phase 4：规划原型

用 3–5 个要点定义最小可行原型：

- 可证伪假设是什么？
- 最冒险的假设是什么 — 此原型如何首先测试它？
- 回答问题所需的最小内容是什么？
- 明确排除什么？（菜单、存档系统、错误处理、润色、架构 — 全部排除）

**范围约束：** 概念原型测试一个机制 — 不是整个游戏。如果范围涵盖多个机制，缩减它。有疑问时，多砍一些。

在构建之前向用户展示此计划。获得确认后再继续。

一旦确认，将会话检查点写入 `production/session-state/active.md`（如果 `production/session-state/` 不存在则创建）。包括：概念名称、假设、所选路径、范围要点和当前阶段（"Phase 5 — 实现"）。这样，如果会话在构建中途结束，下一个会话可以恢复而无需重新开始 — 对于多天的引擎路径工作尤为重要。

## Phase 5：实现

询问："我可以创建原型目录 `prototypes/[concept-name]-concept/` 并开始实现吗？"

如果同意，创建目录。每个文件必须以以下内容开头：

```
// 原型 — 不得用于生产
// 问题：[正在测试的核心问题]
// 日期：[当前日期]
```

标准有意放宽：

- 随意硬编码数值
- 使用占位资源（彩色矩形、调试形状）
- 完全跳过错误处理
- 使用最简单有效的方法
- 复制代码而非从生产代码导入
- 无需架构、无需模式、无需抽象

**不要添加润色。** 没有菜单、没有游戏结束画面、没有音乐、没有教程文本 — 除非教程正是正在测试的机制。超出假设的每一个添加都是浪费。

**试玩建议：** 如果您能接触到任何没见过该游戏的人 — 朋友、家人、网上的陌生人 — 在不做解释的情况下观看他们游玩，比您自己测试能获得更好的信号。静默观察；不要引导他们。困惑就是数据。之后只问一个问题："什么让你感到困惑？"不是"你喜欢吗？"

**没有外部测试员？** 使用轮换制：如果您构建了系统 A，您就是系统 B 的天然测试员。两人团队中这很有效。独立开发者？离开 2-3 天再以全新状态游玩 — 不会有完美的第一印象信号，但能发现最大障碍。另一个选项：将您自己的原型作为速通玩（强迫自己在 5 分钟内完成，不停下来修复） — 您感受到的摩擦就是陌生人会遇到的问题。

**想要更精细的 UX 数据？** 让测试员在游玩时**出声思考** — 实时叙述他们的想法："我按了空格...什么都没发生...那是跳跃键吗？"这样能在困惑发生的当下就发现它，而不是等待事后复盘。最适合 UI/UX 和上手清晰度。静默观察仍然更适合测试纯粹手感；出声思考会稍微改变人们的游玩方式，但能提供更丰富的关于他们为何困惑的数据。

**HTML 原型？** itch.io、Reddit（r/playmygame）和 Discord（GMTK、Brackeys）让您今天就能零成本触达陌生人 — 请参见上方 HTML 路径章节的分发选项。

**在编写代码之前测试 AI、NPC 或复杂系统行为？** 使用**绿野仙踪**技术：一个人正常游玩，同时第二个人秘密实时控制 NPC、敌人或系统行为 — 做出人类会做出的决策，而非算法。玩家相信它是自动化的。这让您可以在编写一行寻路或决策树代码之前验证 AI 设计是否*手感正确*。当您观察人类控制器自然产生的响应时，您就确切地知道了 AI 需要做什么。

### 引擎路径：多轮循环

编写初始代码后：

> "原型文件已编写。现在在您的引擎中运行项目。
> 如果有错误，请粘贴在这里，我会修复。如果运行成功，
> 描述您看到的内容以及它是否在回答问题。"

迭代直到原型可玩。每个循环：
1. 用户运行 → 报告错误或观察
2. 智能体修复错误或调整机制
3. 重复直到可玩或触发沉没成本规则

### HTML 路径：单一输出

将单个 `prototype.html` 写入 `prototypes/[concept-name]-concept/`。内联包含所有样式、逻辑和资源。文件必须可以双击打开，无需服务器。

### 纸面路径：文档 + 日志

写入 `prototypes/[concept-name]-concept/rules.md`（游戏规则）和 `prototypes/[concept-name]-concept/play-log.md`（一次模拟会话，逐步走过一个完整的游戏周期，包含掷骰、决策和结果叙述）。

## Phase 6：试玩汇报

原型已构建。现在交给用户，捕捉他们实际体验的内容。不要跳到报告生成 — 报告的质量取决于您在此收集的观察。

**对于 HTML 路径：** 确切地说：
> "原型已准备就绪。在浏览器中打开 `prototypes/[name]-concept/prototype.html` 并进行游玩。请充分体验。不要匆忙通过 — 尝试像新玩家一样接近它。完成后回到这里。"

**对于引擎路径：** 多轮迭代循环已经捕捉了错误和行为。现在询问整体评估：
> "现在已经运行了 — 以玩家而非开发者的身份多玩几次。当您对其有了感觉后回到这里。"

**对于纸面路径：** 确切地说：
> "通读 `prototypes/[name]-concept/rules.md` 并像第一次游玩一样走过 `play-log.md`。如果您附近有人，尝试和他们一起运行规则。当您至少看到一整个游戏周期后回到这里。"

用户回来后，**逐个**问这些问题 — 等待每个答案后再问下一个：

1. **假设检查：**
   > "假设是：[重述 Phase 1 的假设]。它是否成立 — 确认、部分确认还是反驳？告诉我您看到了什么。"

2. **最佳时刻：**
   > "哪个时刻 — 如果有的话 — 让您感觉它有效？请具体说明。"

3. **最差时刻：**
   > "哪个时刻最令人沮丧、困惑或糟糕？请具体说明 — 不是'感觉慢'而是'跳跃大约需要半秒才响应，感觉像是在和操作较劲'。"

4. **意外：**
   > "有没有发生任何您没有预料到的事情 — 好的或坏的？"

5. **判决：**
   > "PROCEED、PIVOT 还是 KILL — 并附上一句话原因。"

在进入报告生成之前收集所有答案。如果任何答案模糊（"感觉还行"、"相当不错"），追问："能说得更具体些吗？到底什么感觉还行？"精确的观察让报告有用。模糊的观察让报告毫无价值。

---

## Phase 7：生成原型报告

读取 `.claude/docs/templates/prototype-report.md` 获取报告结构。根据本次会期观察到的内容填写每个章节。将所有占位文本替换为真实观察 — 不使用通用的填充内容。

询问："我可以将此报告写入 `prototypes/[concept-name]-concept/REPORT.md` 吗？"

如果同意，写入文件。然后更新 `prototypes/index.md`（如果不存在则创建）— 在概念原型表中附加一行：概念名称、日期、使用的路径、判决（PROCEED/PIVOT/KILL）以及 REPORT.md 的链接。如果存在 PIVOT 链条（相关概念文件夹中的先前 PIVOT-NOTE.md），记录该链条。此文件是项目尝试过什么和学到了什么的完整历史记录。

## Phase 8：创意总监审查

**审查模式检查：**
- `solo` → 跳过。备注："CD-PLAYTEST 已跳过 — Solo 模式。"
- `lean` → 跳过。备注："CD-PLAYTEST 已跳过 — Lean 模式。"
- `full` → 如果 `design/gdd/game-concept.md` 存在且定义了游戏支柱，通过 Task 使用关卡 **CD-PLAYTEST** 生成 `creative-director`。如果支柱尚未定义，备注："CD-PLAYTEST 已跳过 — 概念原型阶段尚未定义游戏支柱。"

传入：完整的 REPORT.md 内容、原始假设以及来自 `design/gdd/game-concept.md` 的游戏支柱/核心幻想。

创意总监根据游戏的创意愿景评估结果，并确认、修改或覆盖推荐。他们的判决是最终的。如果判决不同，更新 REPORT.md。

## Phase 9：总结与后续步骤

输出总结：假设、结果和最终推荐。链接到 `prototypes/[concept-name]-concept/REPORT.md`。

**如果 PROCEED：**
您的概念原型验证了核心创意。现在根据您刚刚学到的内容进行适当设计。

推荐路径（按顺序）：
1. `/design-review design/gdd/game-concept.md` — 根据原型揭示的内容验证概念文档
2. `/gate-check` — 确认准备进入系统设计阶段
3. `/art-bible` — 定义视觉风格（可选，但值得在 GDD 之前完成）
4. `/map-systems` — 将概念分解为所有游戏系统
5. `/design-system [mechanic]` — 每个 MVP 系统的 GDD；在调优参数和公式章节中使用原型学习的经验
6. `/review-all-gdds` — 跨系统一致性检查

**注意：** 如果您使用了 HTML 路径而手感仍不确定，考虑在编写 GDD 之前运行一个快速引擎路径原型来测试手感。

**如果 PIVOT：**

在路由到下一个原型之前，捕获向前传递的笔记。按顺序问这两个问题（纯文本，逐个）：

1. "这个原型中哪些具体部分有效，我们应该在下一个版本中保留？"
2. "需要改变的单一最重要的事情是什么？"

询问："我可以将此写入 `prototypes/[concept-name]-concept/PIVOT-NOTE.md` 吗？"

如果同意，写入文件包含：原始假设、保留什么、改变什么、以及下一个原型的修订假设。下次运行 `/prototype` 时，检查 `prototypes/` 中是否有任何 `PIVOT-NOTE.md` 文件 — 如果找到，读取它们并使用修订假设作为起点，而不是从头形成。

- 运行 `/prototype [revised-concept]` 测试调整后的方向
- 或者如果概念需要更根本的重构，运行 `/brainstorm [hint]`

**如果 KILL：**

在继续之前，运行此检查以确认判决是合理的而非暂时的挫折：

- [ ] 经过 2+ 次试玩后，核心机制对测试者来说仍然不清晰？
- [ ] 在任何会期中未观察到"有趣时刻"（微笑、大笑或自愿重试）？
- [ ] 同一概念上进行了 3+ 次 PIVOT 迭代，无明显改进？
- [ ] 概念只有在大量解释或开发者引导玩家时才有效？
- [ ] 构建此内容感觉像是义务而非兴奋？

如果 2+ 个条件适用 → KILL 判决合理。如果 0–1 个条件适用 → 在放弃之前考虑再进行一次有针对性的 PIVOT。

**在 `prototypes/GRAVEYARD.md` 中记录失败**（如果不存在则创建）。询问："我可以将此概念追加到 `prototypes/GRAVEYARD.md` 吗？" 如果同意，添加一个条目：

```
## [概念名称] — YYYY-MM-DD
- **失败原因：** [具体障碍 — 不是"太无聊"而是"玩家从未理解核心动作"]
- **有效的内容：** [2-3 件值得在未来概念中保留的事情]
- **失败的内容：** [具体的机制、设计决策或范围问题]
- **下次尝试：** [对类似概念进行不同尝试的一个明确行动]
```

此文件的存在是为了确保同样的错误不会在下一个概念上重犯。

- 运行 `/brainstorm open` 或 `/brainstorm [new-hint]` 探索不同的概念
- 原型报告是交付物 — 无需进一步操作

## 突击模式

**触发方式：** `--spike` 标志或在 Phase 1 中选择"制作中期突击"条目。

**目的：** 在制作中期测试特定的技术或设计问题，无需完整概念原型工作流的开销。无需 GDD 前置条件。无阶段关卡影响。硬上限：~4 小时。

**何时使用：**
- 您在制作阶段，想测试是否应添加一个新机制
- 您不确定某个技术方案在正式构建前是否可行
- 正在考虑一个设计变更，想要快速的修改前/后对比
- 一个 GDD 系统比预期更难，您想对困难部分进行原型测试
- 在编写游戏逻辑之前，您需要确认目标硬件能够维持所需的帧率（**性能突击** — 见下文）

**突击模式工作流（替代 Phase 1–9）：**

1. **定义突击问题**（纯文本，非控件）："这次突击回答什么具体问题？请给我一句话：'我们能使用[方法 Y]实现[任务 X]吗？'"

2. **选择路径** — 与 Phase 3 相同的 AskUserQuestion 控件（HTML / 引擎 / 纸面）。

3. **范围** — 最多 2-3 个要点。一个机制，一个技术问题，别无其他。

4. **构建** — 与概念原型相同的放宽标准。硬上限：4 小时。如果 4 小时内无法展示，问题太大。拆分它。

5. **观察并决定** — 无需正式试玩汇报。问："突击是否回答了问题？是或否，并在一句话中说明原因。"

6. **编写突击笔记**（非完整报告）到 `prototypes/[concept-name]-spike-[date]/SPIKE-NOTE.md`：
   - 测试的问题
   - 结果（是 — 有效 / 否 — 无效 / 部分 — 需要进一步调查）
   - 下一步做什么（添加到当前 sprint / 进一步调查 / 放弃该想法）

7. **更新 `production/session-state/active.md`** 以清除突击状态并返回到当前的 sprint 状态。

**无创意总监关卡。无阶段关卡。无 PROCEED/PIVOT/KILL。** 突击结果为决策提供信息；它们不做决策。由开发者根据突击结果决定是否将机制/方法添加到 sprint 积压中。

**性能突击（特殊情况）：** 如果游戏涉及高要求的渲染 — 大型开放世界、数百个同时物理对象、重型粒子系统、复杂着色器 — 在编写游戏逻辑之前运行性能突击，以确认目标硬件能够维持所需的帧率。这与其他突击在两个方面不同：
- 问题是"引擎能在[最低规格硬件]上以 60fps 渲染[场景 X]吗？"而不是"这个机制手感好吗？"
- 输出是基准测试数字，而非手感判决
- 不需要游戏逻辑 — 只需同时运行的最大预期场景负载（地形、绘制调用、物理对象、粒子）
- 构建时间保持在 ~4 小时上限内；突击是建立渲染负载，而非游戏本身
- 如果在此范围内答案是"否"，这是一个影响下游所有内容的架构或范围约束 — 最好在 Sprint 8 期间就发现它

---

### 重要约束

- 原型代码绝不能从生产源文件中导入
- 生产代码绝不能从原型目录中导入
- 如果推荐是 PROCEED，生产实现从头开始编写 — 原型代码绝不会被重构进入生产
- 总工作量硬上限为 1 天（概念原型测试一个机制）
- 测试一个机制 — 如果范围扩大，停止并简化问题
- 无润色。没有菜单、没有游戏结束、没有音乐、没有 UI — 除非它正是机制本身
- 如果经过 2 小时引擎迭代后卡住，重新定义问题或切换路径
- **3 次 PIVOT 迭代 → 强制做出 KILL 决策。** 如果同一概念第三次产生 PIVOT 判决，该概念很可能不成立。问："这是正确的想法，还是我陷入了沉没成本陷阱？"一个全新制作的原型几乎总是会击败苦苦挣扎的第四次迭代。
- 构建 2-3 个不同的概念变体并选择最佳的一个，比把一个概念迭代到死更健康。原型之间的自然选择胜过意志力。
- **网络/多人游戏：** 本地原型无法验证网络机制的手感。延迟从根本上改变战斗、移动和预测的手感 — 在 0ms 本地运行的原型与 80ms 网络延迟下的感觉完全不同。使用本地原型验证机制是否*有趣*。不要将其作为在真实网络条件下*手感良好*的证据。网络手感需要真实的对等连接或模拟延迟（例如限速工具、网络条件模拟器）。
