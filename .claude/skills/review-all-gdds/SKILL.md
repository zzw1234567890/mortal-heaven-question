# 审查所有 GDD

此技能同时读取每个系统 GDD 并进行两项互补的审查，
这是单个 GDD 独立审查无法做到的：

1. **跨 GDD 一致性**——文档之间的矛盾、过时引用和所有权冲突
2. **游戏设计整体性**——只有同时看到所有系统才会出现的问题：
   主导策略、崩溃的经济、认知超载、支柱漂移、相互竞争的进度循环

**这与 `/design-review` 不同**，后者审查单个 GDD 的内部完整性。
此技能审查所有 GDD 之间的*关系*。

**何时运行：**
- 所有 MVP 级别的 GDD 单独批准后
- 任何 GDD 在生产中期进行重大修订后
- 在 `/create-architecture` 开始之前（基于不一致的 GDD 构建的架构会继承这些不一致）

**参数模式：**

**焦点：** `$ARGUMENTS[0]`（空白 = `full`）

- **无参数 / `full`**：一致性和设计理论双重检查
- **`consistency`**：仅跨 GDD 一致性检查（更快）
- **`design-theory`**：仅游戏设计整体性检查
- **`since-last-review`**：仅自上次审查报告以来修改过的 GDD（基于 git）

## 阶段 1：加载所有内容

### 阶段 1a — L0：摘要扫描（快速，低 token 消耗）

在读取任何完整文档之前，使用 Grep 从所有 GDD 文件中提取 `## Summary` 章节：

```
Grep pattern="## Summary" glob="design/gdd/*.md" output_mode="content" -A 5
```

向用户显示清单：
```
找到 [N] 个 GDD。摘要：
  • combat.md — [摘要文本]
  • inventory.md — [摘要文本]
  ...
```

对于 `since-last-review` 模式：运行 `git log --name-only` 识别自上次审查报告文件写入以来修改过的 GDD。在进行完整读取前，向用户显示基于摘要的范围内的 GDD。仅对这些 GDD 及其"关键依赖"中列出的任何 GDD 继续到 L1。

### 阶段 1b — 注册表预加载（快速基准线）

在完整读取任何 GDD 之前，检查实体注册表：

```
Read path="design/registry/entities.yaml"
```

如果注册表存在且有条目，将其用作**预构建的冲突基准线**：已知实体、物品、公式和常量及其权威值和来源 GDD。在阶段 2 中，首先使用 Grep 在 GDD 中搜索已注册的名称——这比在知道要找什么之前完整读取所有 GDD 更快。

如果注册表为空或不存在：继续而不使用。在报告中注明："实体注册表为空——一致性检查仅依赖完整 GDD 读取。本次审查后运行 `/consistency-check` 以填充注册表。"

### 阶段 1c — L1/L2：完整文档加载

完整读取范围内文档：

1. `design/gdd/game-concept.md` — 游戏愿景、核心循环、MVP 定义
2. `design/gdd/game-pillars.md`（如果存在）— 设计支柱和反支柱
3. `design/gdd/systems-index.md` — 权威系统列表、层级、依赖关系、状态
4. **`design/gdd/` 中每个范围内的系统 GDD** — 完整读取（跳过上面已读取的 game-concept.md 和 systems-index.md）

报告："已加载 [N] 个系统 GDD，覆盖 [M] 个系统。支柱：[列表]。反支柱：[列表]。"

如果系统 GDD 少于 2 个，停止：
> "跨 GDD 审查需要至少 2 个系统 GDD。请先编写更多 GDD，然后重新运行 `/review-all-gdds`。"

### 并行执行

阶段 2（一致性）和阶段 3（设计理论）是独立的——它们读取相同的 GDD 输入，但生成独立的报告。同时生成两个并行 Task 代理，而不是等待阶段 2 完成后再开始阶段 3。在写入合并报告前收集两个结果。

**生成阶段 2 和阶段 3 的并行 Task 代理时，始终传递：**
- 阶段 1 中加载的完整 GDD 文件路径列表（显式路径，而不仅仅是计数）
- 阶段 1b 中加载的完整 TR 注册表内容（粘贴注册表文本，而不仅仅是文件路径）
- 分配给该代理阶段的特定检查项列表（阶段 2 得到 2a–2f；阶段 3 得到 3a–3g）
- 来自 `.claude/docs/technical-preferences.md` 和 `docs/engine-reference/[engine]/VERSION.md` 的引擎名称和版本

不要依赖子代理重新读取这些文件——它有自己独立的上下文窗口，除非在 Task 提示中显式传递，否则无法访问阶段 1 的结果。

## 阶段 2：跨 GDD 一致性

遍历每一对和每组 GDD，找出矛盾点和缺口。

### 2a：依赖双向性

对于每个 GDD 的依赖章节，检查每个列出的依赖都是双向的：
- 如果 GDD-A 列出"依赖 GDD-B"，检查 GDD-B 是否将 GDD-A 列为依赖者
- 如果 GDD-A 列出"被 GDD-C 依赖"，检查 GDD-C 是否将 GDD-A 列为依赖项
- 将任何单向依赖标记为一致性问题

```
⚠️  依赖不对称
[system-a].md 列出：依赖对象 → [system-b].md
[system-b].md 没有将 [system-a].md 列为依赖者
→ 其中一个文档的依赖章节已经过时
```

### 2b：规则矛盾

对于任何 GDD 中定义的每个游戏规则、机制或约束，检查是否有其他 GDD 针对相同情况定义了矛盾的规则：

要扫描的类别：
- **下限/上限规则**：是否有 GDD 定义了输出的最小值？是否有其他 GDD 说不同的系统可以绕过该下限？这些是矛盾的。
- **资源所有权**：如果两个 GDD 都定义了共享资源的累积或消耗方式，它们是否一致？
- **状态转换**：如果 GDD-A 描述了角色死亡时会发生什么，GDD-B 对同一事件的描述是否一致？
- **时序**：如果 GDD-A 说"X 在同一帧发生"，GDD-B 是否假定它是异步发生的？
- **叠加规则**：如果 GDD-A 说状态效果叠加，GDD-B 是否假定它们不叠加？

```
🔴 规则矛盾
[system-a].md："减少后的最小[输出]为[下限值]"
[system-b].md："[机制]绕过[system-a]的规则，可以将[输出]降至 0"
→ 这些规则直接矛盾。哪个 GDD 是权威的？
```

### 2c：过时引用

对于每个跨文档引用（GDD-A 提到来自 GDD-B 的机制、数值或系统名称），验证被引用的元素在 GDD-B 中仍然存在且名称和行为相同：

- 如果 GDD-A 说"来自战斗系统的连击乘数输入到分数中"，检查战斗 GDD 是否确实定义了一个输出到分数的连击乘数
- 如果 GDD-A 引用"在 [system].md 中定义的进度曲线"，检查 [system].md 是否确实有该曲线，而不是不同的进度模型
- 如果 GDD-A 是在 GDD-B 之前编写的，并且假设了 GDD-B 后来以不同方式设计的机制，将 GDD-A 标记为包含过时引用

```
⚠️  过时引用
inventory.md（先编写）："物品重量使用 movement.md 中的负重公式"
movement.md（后编写）：未定义负重公式——使用扁平负重上限替代
→ inventory.md 引用了一个不存在的公式
```

### 2d：数据和调整参数所有权冲突

两个 GDD 不应同时声称拥有相同的数据或调整参数。扫描所有 GDD 中的所有调整参数章节，标记重复项：

```
⚠️  所有权冲突
[system-a].md 调整参数："[乘数名称] — 控制[输出]缩放"
[system-b].md 调整参数："[乘数名称] — 用[因子]缩放[输出]"
→ 两个 GDD 在相同输出上定义了乘数。谁拥有最终值？
  这将导致双重应用 Bug 或设计冲突。
```

### 2e：公式兼容性

对于公式相连的 GDD（一个的输出作为另一个的输入），检查上游公式的输出范围是否在下游公式的预期输入范围内：

- 如果 [system-a].md 输出值在 [min]–[max] 之间，而 [system-b].md 设计为接收 [min2]–[max2] 之间的值，不匹配是有意的吗？
- 如果经济 GDD 预期的资源获取范围是 X，而进度 GDD 生成的资源范围是 Y，那么经济将变得过于简单或无法访问——这是有意的吗？

将不兼容标记为关切项（需要设计判断，不一定错误）：

```
⚠️  公式范围不匹配
[system-a].md：最大[输出] = [value_a]（在最大[条件]时）
[system-b].md：基础[输入] = [value_b]，最大[输入] = [value_c]
→ 后期[阶段]的[场景]可以在单个[事件]中解决。
  这是有意的吗？如果不是，要么[system-a]的上限要么[system-b]的上限需要调整。
```

### 2f：验收标准交叉检查

扫描所有 GDD 中的验收标准章节，查找矛盾：

- GDD-A 标准："玩家不能一击死亡"
- GDD-B 标准："首领攻击造成玩家最大生命值的 150%"
这些验收标准无法同时通过。

## 阶段 3：游戏设计整体性

通过游戏设计理论和玩家心理学的视角，一起审查所有 GDD。这些是单个 GDD 审查无法捕捉的问题，因为它们需要同时看到所有系统。

### 3a：进度循环竞争

一个游戏应该有一个主导的进度循环，让玩家感觉是游戏的"核心目标"，辅以支撑它的辅助循环。当多个系统同等地竞争作为主要进度驱动力时，玩家不知道游戏是关于什么的。

扫描所有 GDD 中那些：
- 授予玩家主要资源（经验值、等级、声望、解锁）的系统
- 将自己定义为"核心"或"主要"循环的系统
- 与其他做同样事的系统具有相当的深度和时间投入

```
⚠️  竞争的进度循环
combat.md：授予经验值，解锁能力，被描述为"核心循环"
crafting.md：授予经验值，解锁配方，被描述为"主要活动"
exploration.md：授予经验值，解锁地图区域，被描述为"主要驱动力"
→ 三个系统都声称是主要进度循环，并且都授予相同的主要货币。
  玩家将优化其中一个并忽略其他。
  建议：一个主要循环，其余作为辅助系统。
```

### 3b：玩家注意力预算

统计在典型游戏过程中需要同时主动关注多少个系统。每个主动管理的系统消耗注意力：

- 主动 = 玩家必须在游戏过程中经常对该系统做出决策
- 被动 = 系统自动运行，玩家看到结果但不管理它

超过 3-4 个同时主动系统会造成大多数玩家的认知超载。呈现计数，如果超过 4 个并发主动系统则标记：

```
⚠️  认知负荷风险
[核心循环时刻]期间同时主动的系统：
  1. [system-a].md — [决策类型]（主动）
  2. [system-b].md — [资源管理]（主动）
  3. [system-c].md — [追踪]（主动）
  4. [system-d].md — [物品/动作使用]（主动）
  5. [system-e].md — [冷却/计时器管理]（主动）
  6. [system-f].md — [协调决策]（主动）
→ 核心循环期间有 6 个同时主动的系统。
  研究表明 3-4 是大多数玩家的舒适限制。
  建议：哪些可以变为被动或简化？
```

### 3c：主导策略检测

主导策略使其他策略变得无关紧要——玩家发现它，独占使用它，然后发现游戏的其余部分变得无聊。寻找：

- **资源垄断**：一种策略比其他所有策略显著更快地生成资源
- **无风险力量**：一个既高回报又低风险的策略（如果存在高风险策略，它们需要相应更高的回报）
- **无权衡**：一个在所有维度上优于所有其他选项的选项
- **明显的最优路径**：如果任何进度选择是"明显正确的"，那么其他选择不是真正的选择

```
⚠️  潜在主导策略
combat.md：远程攻击造成近战伤害的 80%，没有风险
combat.md：近战攻击造成 100% 伤害，但需要近距离
→ 除非近战具有显著的补偿优势（AOE、硬直、资源回复），
  远程是主导的——更高的安全性，只少 20% 伤害。
  考虑近战能提供什么远程不能的。
```

### 3d：经济循环分析

识别所有 GDD 中的所有资源（金币、经验值、制作材料、耐力、生命值、法力值等）。对于每种资源，映射其**来源**（玩家如何获得）和**消耗**（玩家如何花费）。

标记危险的经济条件：

| 条件 | 迹象 | 风险 |
|-----------|------|------|
| **无限来源，无消耗** | 资源无限累积 | 游戏后期变得过于容易 |
| **有消耗，无来源** | 资源降至零 | 系统变得不可用 |
| **来源 >> 消耗** | 积累过剩 | 资源变得无意义 |
| **消耗 >> 来源** | 持续稀缺 | 挫败感和卡关 |
| **正反馈循环** | 更多资源 → 更容易获得更多 | 滚雪球式领先 |
| **无追赶机制** | 落后加速赤字 | 不可恢复的状态 |

```
🔴 经济失衡：无界正反馈
金币经济：
  来源：怪物掉落（随玩家力量缩放），商人出售（无限）
  消耗：装备购买（一次性），能力升级（有限数量）
→ 装备和能力购买完成后，金币没有消耗。
  无限过剩。金币在游戏中后期变得无意义。
  增加持续金币消耗（维护费、消耗品、外观、赌博）。
```

### 3e：难度曲线一致性

当多个系统随玩家进度缩放时，它们必须在兼容的方向和兼容的速率上缩放。不匹配的缩放曲线会造成意外的难度峰值或软化。

对于每个随时间缩放的系统，提取：
- 什么在缩放（敌人生命值、玩家伤害、资源成本、区域大小）
- 如何缩放（线性、指数、阶梯式）
- 何时缩放（等级、时间、区域）

比较所有缩放曲线。标记不匹配：

```
⚠️  难度曲线不匹配
combat.md：敌人生命值按区域指数缩放（每区域 ×2）
progression.md：玩家伤害按等级线性缩放（每级 +10%）
→ 到区域 5，敌人有 32 倍基础生命值；玩家造成约 1.5 倍基础伤害。
  差距无限扩大。后期区域将变得无法到达地困难，除非曲线被调和。
```

### 3f：支柱对齐

每个系统应明确服务于至少一个设计支柱。不服务于任何支柱的系统是"设计上的范围蔓延"——它在游戏中但不服务于游戏试图成为的目标。

对于每个 GDD 系统，检查其玩家幻想章节与设计支柱。标记任何其所述幻想不映射到任何支柱的系统：

```
⚠️  支柱漂移
fishing-system.md：玩家幻想——"平和、冥想般的活动"
支柱："残酷战斗"、"紧张生存"、"涌现故事"
→ 钓鱼系统不服务于三个支柱中的任何一个。
  要么添加一个覆盖它的支柱，要么重新设计以服务于现有支柱，要么删掉它。
```

同时检查反支柱——标记任何做了反支柱明确说明游戏不会做的事情的系统：

```
🔴 反支柱违规
反支柱："我们不会采用线性故事推进——玩家定义自己的路径"
main-quest.md：定义了 12 章节的线性故事，具有强制顺序
→ 此系统直接违反了定义的反支柱。
```

### 3g：玩家幻想整体性

所有系统的玩家幻想应该是兼容的——它们应该强化一个一致的玩家身份认同。冲突的玩家幻想会造成身份混淆。

```
⚠️  玩家幻想冲突
combat.md："你是一个冷酷、精准的战士——每一次击杀都是挣来的"
dialogue.md："你是一个有魅力的外交官——暴力总是可以避免的"
exploration.md："你是一个鲁莽的冒险家——不制定计划就冲进去"
→ 三个系统呈现了不相容的身份。玩家会感觉游戏
  不知道希望他们成为什么。考虑：这些幻想是从不同角度
  服务于同一个核心身份，还是它们真实地冲突？
```

## 阶段 4：跨系统场景演练

从玩家的视角走过游戏，发现只有在多个系统交互边界才出现的问题——单个 GDD 的静态分析无法揭示的问题。

### 4a：识别关键多系统时刻

扫描所有 GDD，识别 3-5 个最重要的玩家面对的时刻，其中多个系统同时激活。特别寻找：

- **战斗 + 经济重叠**：击杀掉落资源的敌人，在战斗中花费资源，死亡/重生与经济状态交互
- **进度 + 难度重叠**：战斗中升级，能力解锁改变战斗可行性，难度在进度里程碑处缩放
- **叙事 + 游戏玩法重叠**：对话选择锁定/解锁机制，故事节点打断资源循环，任务完成触发系统状态变化
- **3+ 系统链**：任何触发系统 A 的玩家动作，系统 A 输入系统 B，系统 B 触发系统 C（这些是最高风险的交互路径）

在继续之前，用一行描述列出每个识别的场景。

### 4b：演练每个场景

对于每个场景，明确地逐步执行：

1. **触发器** — 什么玩家动作或游戏事件启动这个？
2. **激活顺序** — 哪些系统激活，按什么顺序？
3. **数据流** — 每个系统输出什么，该输出是否是链中下一个系统的有效输入？
4. **玩家体验** — 在每个步骤中，玩家看到、听到或感觉到什么？
5. **故障模式** — 是否存在以下任何情况？
   - **竞态条件**：两个系统试图同时修改同一状态
   - **反馈循环**：系统 A 放大系统 B，系统 B 再次放大系统 A，没有上限或缓冲
   - **损坏的状态转换**：一个系统假设了前一个系统可能已更改的状态（例如，在可能导致死亡的战斗步骤之后的"玩家存活"假设）
   - **矛盾的消息传递**：玩家从对同一事件做出反应的两个系统接收到冲突的反馈（例如，"成功"音效 + "失败"UI）
   - **复合难度峰值**：两个系统在同一进度点同时上升，成倍增加预期的难度提升
   - **奖励冲突**：两个系统都对同一触发器做出反应并授予奖励，总和超出预期价值（双重获利）
   - **未定义行为**：GDD 未指定在这种组合状态下会发生什么（两个系统的规则都不涵盖）

```
演练示例：
场景：玩家在活跃任务期间在升级阈值处击杀精英敌人

触发器：玩家对精英敌人进行致命一击
→ combat.md：授予击杀经验值（100 点）
→ progression.md：经验值总计超过等级阈值 → 触发升级
  输出：新等级、属性增加、能力解锁弹窗
→ quest.md：击杀计数标准满足 → 触发任务完成事件
  输出：任务奖励经验值（500 点）、完成庆祝动画
→ progression.md（再次）：任务经验值添加 → 触发同一帧中的第二次升级
  ⚠️  数据流问题：quest.md 授予经验值而不检查升级是否
  正在进行中。progression.md 没有防止并发升级事件的保护。
  未定义行为：玩家升级一次还是两次？
  能力弹窗弹出两次？第二次升级使用更新前还是更新后的属性基准？
```

### 4c：标记场景问题

对于演练中发现的每个问题，分类严重程度：

- **BLOCKER**：未定义行为、损坏的状态转换或矛盾的玩家消息传递——此场景中的体验已损坏或不连贯
- **WARNING**：复合峰值、无上限的反馈循环、奖励冲突——体验有效但产生意外结果
- **INFO**：轻微顺序歧义或消息传递重叠——值得注意但不太可能引起玩家可见的问题

在输出报告中添加所有发现，放在**"跨系统场景问题"**下。每个发现必须引用：场景名称、涉及的具体系统、问题发生的步骤以及故障模式的性质。

## 错误恢复协议

如果任何生成的代理返回阻塞、出错或未能完成：

1. **立即上报**：在继续前报告"[AgentName]：BLOCKED — [原因]"
2. **评估依赖关系**：如果被阻塞代理的输出是后续阶段所需的，在没有用户输入的情况下不要继续进行该阶段
3. **通过 AskUserQuestion 提供选项**，包括三个选择：
   - 跳过此代理并在最终报告中注明缺口
   - 以更窄的范围重试（更少的 GDD，单系统焦点）
   - 在此停止并先解决阻塞项
4. **始终生成部分报告**——输出已完成的内容，以免工作丢失

## 协作协议

1. **静默阅读**——在呈现任何内容之前加载所有 GDD
2. **展示一切**——在要求任何行动之前呈现完整的一致性和设计理论分析
3. **区分阻塞和建议**——不是每个问题都需要阻塞架构；清楚哪些是
4. **不做设计决策**——标记矛盾点和选项，但从不单方面决定哪个 GDD 是"正确的"
5. **先问再写**——在写入报告或更新系统索引之前确认
6. **具体明确**——每个问题必须引用确切的 GDD、章节和文本；没有模糊的警告
