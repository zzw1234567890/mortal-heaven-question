1. **解析参数**以获取可选的类型/主题提示（例如 `roguelike`、`space survival`、`cozy farming`）。如果是 `open` 或无参数，从头开始。同时解析审查模式（一次性完成，为本运行的所有关卡生成存储）：
   1. 如果传入了 `--review [full|lean|solo]` → 使用该值
   2. 否则读取 `production/review-mode.txt` → 使用该值
   3. 否则 → 默认为 `lean`

   完整的检查模式请参见 `.claude/docs/director-gates.md`。

2. **检查现有概念工作**：
   - 如果存在，读取 `design/gdd/game-concept.md`（继续，不要重新开始）
   - 如果存在，读取 `design/gdd/game-pillars.md`（建立在已建立的支柱上）

3. **交互式地运行构思阶段**，在每个阶段向用户提问。不要静默地生成所有内容——目标是**协作探索**，AI 作为创意促进者，而不是人类愿景的替代品。

   在头脑风暴的关键决策点使用 `AskUserQuestion`：
   - 受限品味问题（类型偏好、范围、团队规模）
   - 概念选择（"哪 2-3 个概念引起共鸣？"）呈现选项后
   - 方向选择（"进一步发展、探索更多还是制作原型？"）
   - 概念优化后的支柱排序
   先在对话文本中写入完整的创意分析，然后使用 `AskUserQuestion` 以简洁标签捕获决策。

   遵循的专业工作室头脑风暴原则：
   - 暂缓判断——在探索期间没有坏主意
   - 鼓励不寻常的想法——跳出框架的思维激发更好的概念
   - 相互构建——"是的，而且..."的回应，而不是"但是..."
   - 将约束用作创意燃料——限制往往产生最好的想法
   - 给每个阶段设定时间盒——保持动力，不要在早期过度推敲

### 阶段 1：创意发现

首先了解人，而不是游戏。以对话方式询问这些问题（不作为清单）：

**情感锚点**：
- 在游戏中有什么时刻真正打动过你、让你兴奋或让你忘记了时间？具体是什么创造了那种感觉？
- 有没有你一直想在游戏中体验但从未真正找到的幻想或力量之旅？

**品味档案**：
- 哪 3 个游戏你花的时间最多？是什么让你回来继续玩？
  *（以纯文本形式询问——用户必须能够自由输入具体的游戏名称。不要将其放在有预设选项的 AskUserQuestion 中。）*
- 你有喜欢的类型吗？你回避的类型？为什么？
- 你喜欢挑战你、让你放松、给你讲故事，还是让你表达自己的游戏？*（使用 `AskUserQuestion` 询问——受限制选择。）*

**实际约束**（在头脑风暴前塑造沙盒）。
将这些打包成单个多标签 `AskUserQuestion`，使用这些确切的标签名称：
- 标签"体验"——"你最想让玩家拥有什么样的体验？"（挑战与精通 / 故事与发现 / 表达与创造 / 放松与心流）
- 标签"时间线"——"你现实的开发时间线是什么？"（数周 / 数月 / 1-2 年 / 多年）
- 标签"开发水平"——"你在开发旅程的哪个阶段？"（第一款游戏 / 曾发布过 / 专业背景）

使用确切这些标签名称——不要重命名或重复它们。

**综合**答案成一个**创意简报**——一段 3-5 句话的摘要，说明该人的情感目标、品味档案和约束。回读简报并确认它捕捉了他们的意图。

### 阶段 2：概念生成

以创意简报为基础，生成**3 个不同的概念**，每个走不同的创意方向。使用这些构思技巧：

**技巧 1：动词优先设计**
从核心玩家动词（建造、战斗、探索、解决、生存、创造、管理、发现）开始，并由此向外构建。动词就是游戏本身。

**技巧 2：混搭法**
组合两个意想不到的元素：[类型 A] + [主题 B]。两者之间的张力创造了独特的卖点。（例如，"模拟农场 + 宇宙恐怖"、"roguelike + 约会模拟"、"城市建造者 + 实时战斗"）

**技巧 3：体验优先设计（逆向 MDA）**
从期望的玩家情感开始（MDA 框架中的审美目标：感觉、幻想、叙事、挑战、 fellowship、发现、表达、顺从），然后反向推导产生该情感的动力和机制。

对于每个概念，呈现：
- **工作标题**
- **电梯游说**（1-2 句——必须通过"10 秒测试"）
- **核心动词**（最常见的单个玩家动作）
- **核心幻想**（情感承诺）
- **独特卖点**（通过"而且还有"测试："像 X，而且还有 Y"）
- **主要 MDA 审美**（哪种情感占主导地位？）
- **预估范围**（小 / 中 / 大）
- **它为什么可能成功**（1 句关于市场/受众契合度）
- **最大风险**（1 句关于最难回答的问题）

呈现全部三个。然后使用 `AskUserQuestion` 捕获选择。

**关键**：这必须是一个纯列表调用——没有标签，没有表单字段。使用确切此结构：

```
AskUserQuestion(
  prompt: "哪个概念与你产生共鸣？你可以选择一个、组合元素，或要求全新的方向。",
  options: [
    "概念 1 — [标题]",
    "概念 2 — [标题]",
    "概念 3 — [标题]",
    "跨概念组合元素",
    "生成全新的方向"
  ]
)
```

不要在这里使用 `tabs` 字段。`tabs` 表单仅用于多字段输入——在这里使用会导致"无效工具参数"错误。这是一个纯 `prompt` + `options` 调用。

永远不要施压选择——让他们慢慢来。

### 阶段 3：核心循环设计

对于所选概念，使用结构化提问构建核心循环。
核心循环是游戏跳动的心脏——如果在孤立状态下不好玩，再多的内容或打磨也救不了游戏。

**30 秒循环**（瞬间到瞬间）：

使用 `AskUserQuestion` 调用询问这些问题——从所选概念中推导选项，不要硬编码：

1. **核心动作感觉**——提示："核心动作的主要感觉是什么？"生成适合概念类型和基调的 3-4 个选项，加上一个自由文本退出选项（`我来描述`）。

2. **关键设计维度**——识别该特定概念最重要的设计变量（例如世界反应性、节奏、玩家能动性）并询问。生成匹配概念的选项。始终包含自由文本退出选项。

在捕获答案后，分析：这个动作本身是否令人满足？什么使它感觉良好？（音频反馈、视觉效果、时机满足感、战术深度？）

**5 分钟循环**（短期目标）：
- 什么将瞬间到瞬间的游戏组织成循环？
- "再来一回合"/"再来一次"的心理从何时开始发挥作用？
- 玩家在这个层面做什么选择？

**会话循环**（30-120 分钟）：
- 一个完整的会话是什么样的？
- 自然的停止点在哪里？
- 什么"钩子"让玩家在不玩游戏时也在思考游戏？

**进度循环**（数天/数周）：
- 玩家如何成长？（能力？知识？选项？故事？）
- 长期目标是什么？游戏何时"完成"？

**玩家动机分析**（基于自我决定理论）：
- **自主性**：玩家拥有多少有意义的选择？
- **胜任感**：玩家如何感觉自己的技能在增长？
- **关联性**：玩家如何感觉与（角色、其他玩家或世界的）连接？

### 阶段 4：支柱和边界

游戏支柱被真正的 AAA 工作室（战神、哈迪斯、最后生还者）用来让数百名团队成员做决策时都指向同一方向。即使对于独立开发者，支柱也能防止范围蔓延并保持愿景清晰。

协作定义 **3-5 个支柱**：
- 每个支柱有**名称**和**一句话定义**
- 每个支柱有**设计测试**："如果我们在 X 和 Y 之间争论，这个支柱说我们选择 __"
- 支柱应该感觉彼此之间有张力——如果所有支柱指向同一方向，它们就不够努力

然后定义 **3 个以上的反支柱**（这个游戏不是什么）：
- 反支柱防止最常见的范围蔓延形式："如果也加入...岂不是很酷"的不服务于核心愿景的功能
- 表述为："我们不会做 [thing]，因为它会损害 [pillar]"

**支柱确认**：呈现完整的支柱集后，使用 `AskUserQuestion`：
- 提示："这些支柱对你的游戏合适吗？"
- 选项：`[A] 锁定这些` / `[B] 重命名或重新定义其中一个` / `[C] 替换一个支柱` / `[D] 其他`

如果用户选择 B、C 或 D，进行修改，然后再次使用 `AskUserQuestion`：
- 提示："支柱已更新。准备好锁定这些了吗？"
- 选项：`[A] 锁定这些` / `[B] 修改另一个支柱` / `[C] 其他`

重复直到用户选择 [A] 锁定这些。

**审查模式检查** —— 在召唤 CD-PILLARS 和 AD-CONCEPT-VISUAL 之前应用：
- `solo` → 两者都跳过。注意："CD-PILLARS 已跳过——Solo 模式。AD-CONCEPT-VISUAL 已跳过——Solo 模式。" 继续阶段 5。
- `lean` → 两者都跳过（不是 PHASE-GATE）。注意："CD-PILLARS 已跳过——Lean 模式。AD-CONCEPT-VISUAL 已跳过——Lean 模式。" 继续阶段 5。
- `full` → 正常召唤。

**在支柱和反支柱达成一致后，在进入阶段 5 之前通过 Task 并行召唤 `creative-director` 和 `art-director`。同时发出两个 Task 调用——不要等一个完成后才开始另一个。**

- **`creative-director`** — 关卡 **CD-PILLARS**（`.claude/docs/director-gates.md`）
  传递：完整的支柱集及其设计测试、反支柱、核心幻想、独特卖点。

- **`art-director`** — 关卡 **AD-CONCEPT-VISUAL**（`.claude/docs/director-gates.md`）
  传递：游戏概念电梯游说、完整的支柱集及其设计测试、目标平台（如果已知）、用户提到的任何参考游戏或视觉基石。

收集两个裁决，然后使用双标签 `AskUserQuestion` 一起呈现：
- 标签 **"支柱"**：呈现创意总监反馈。选项反映标准的 CD-PILLARS 处理——`按现状锁定` / `修改 [specific pillar]` / `进一步讨论`。
- 标签 **"视觉锚点"**：呈现艺术总监的 2-3 个命名视觉方向选项。选项：每个命名方向（每个一个选项）+ `跨方向组合元素` + `描述我自己的方向`。

用户选择的视觉锚点（命名的方向或他们的自定义描述）被存储为**视觉标识锚点**——它将被写入游戏概念文档，并成为美术圣经的基础。

如果创意总监对支柱返回关切或拒绝，在询问视觉锚点选择之前解决支柱问题——视觉方向应来自已确认的支柱。

### 阶段 5：玩家类型验证

使用 Bartle 分类法和 Quantic Foundry 动机模型，验证这个游戏实际适合谁：

- **主要玩家类型**：谁会*爱上*这个游戏？（成就者、探索者、社交者、竞争者、创造者、故事爱好者）
- **次要吸引力**：还有谁可能喜欢它？
- **这不适合谁**：清楚谁不会喜欢这个游戏与知道谁会喜欢同样重要
- **市场验证**：是否有成功的游戏服务于类似的玩家类型？我们能从他们的受众规模中学到什么？

### 阶段 6：范围和可行性

将概念落实到现实中：

- **目标平台**：使用 `AskUserQuestion` —— "你针对哪些平台开发此游戏？"
  选项：`PC（Steam / Epic）` / `移动端（iOS / Android）` / `主机` / `网页/浏览器` / `多平台`
  记录答案——它直接影响引擎推荐，并将传递给 `/setup-engine`。
  如果相关，说明平台影响（例如，移动端强烈推荐 Unity；主机意味着 Godot 有限制；网页意味着 Godot 导出干净）。

- **引擎经验**：使用 `AskUserQuestion` —— "你已经有使用的引擎了吗？"
  选项：`Godot` / `Unity` / `Unreal Engine 5` / `无偏好——帮我决定`
  - 如果他们选了一个引擎 → 记录为他们的偏好并继续。不要质疑它。
  - 如果"无偏好" → 告诉他们："此会话后运行 `/setup-engine`——它将根据你的概念和平台目标引导你完成完整决策。" 不要在这里做推荐。
- **美术管线**：美术风格是什么，劳动强度如何？
- **内容范围**：估计关卡/区域数量、物品数量、游戏时长
- **MVP 定义**：测试"核心循环有趣吗？"的绝对最小构建是什么？
- **最大风险**：技术风险、设计风险、市场风险
- **范围层级**：完整愿景 vs. 时间耗尽时发布什么？

**审查模式检查** —— 在召唤 TD-FEASIBILITY 之前应用：
- `solo` → 跳过。注意："TD-FEASIBILITY 已跳过——Solo 模式。" 直接进入范围层级定义。
- `lean` → 跳过（不是 PHASE-GATE）。注意："TD-FEASIBILITY 已跳过——Lean 模式。" 直接进入范围层级定义。
- `full` → 正常召唤。

**在识别出最大技术风险后，在范围层级定义之前通过 Task 使用关卡 TD-FEASIBILITY（`.claude/docs/director-gates.md`）召唤 `technical-director`。**

传递：核心循环描述、平台目标、引擎选择（或"未决定"）、已识别的技术风险列表。

向用户呈现评估。如果高风险（HIGH RISK），提供在最终确定前重新审视范围的选项。如果关切（CONCERNS），注明并继续。

**审查模式检查** —— 在召唤 PR-SCOPE 之前应用：
- `solo` → 跳过。注意："PR-SCOPE 已跳过——Solo 模式。" 继续文档生成。
- `lean` → 跳过（不是 PHASE-GATE）。注意："PR-SCOPE 已跳过——Lean 模式。" 继续文档生成。
- `full` → 正常召唤。

**在范围层级定义后，通过 Task 使用关卡 PR-SCOPE（`.claude/docs/director-gates.md`）召唤 `producer`。**

传递：完整愿景范围、MVP 定义、时间线估计、团队规模。

向用户呈现评估。如果不切实际（UNREALISTIC），提供在写入文档前调整 MVP 定义或范围层级的选项。
