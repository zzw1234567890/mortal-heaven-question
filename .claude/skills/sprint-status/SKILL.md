# 冲刺状态 (Sprint Status)

这是一个快速态势感知检查，而非冲刺评审。它读取当前的冲刺计划和故事文件，扫描状态标记，并生成一个少于 30 行的简洁快照。如需详细的冲刺管理，请使用 `/sprint-plan update` 或 `/milestone-review`。

**此技能为只读。** 它从不提议更改，从不请求写入文件，并且最多提出一个具体建议。

---

## 1. 查找冲刺

**参数：** `$ARGUMENTS[0]`（空白 = 使用当前冲刺）

- 如果给出了参数（例如 `/sprint-status 3`），在 `production/sprints/` 中搜索与 `sprint-03.md`、`sprint-3.md` 等匹配的文件。报告找到的文件。
- 如果未给出参数，找到 `production/sprints/` 中最近修改的文件，并将其视为当前冲刺。
- 如果 `production/sprints/` 不存在或为空，报告："未找到冲刺文件。使用 `/sprint-plan new` 开始一个冲刺。"然后停止。

完整读取冲刺文件。提取：
- 冲刺编号和目标
- 开始日期和结束日期
- 所有故事或任务条目及其优先级（必须完成/应该完成/可以完成）、负责人和预估

---

## 2. 计算剩余天数

使用今天的日期和冲刺文件中的结束日期，计算：
- 冲刺总天数（结束减开始）
- 已过去天数
- 剩余天数
- 已消耗时间百分比

如果冲刺文件未包含明确日期，注明"未找到冲刺日期——跳过燃尽评估。"

---

## 3. 扫描故事状态

**首先：检查 `production/sprint-status.yaml`。**

如果存在，直接读取——它是权威的真理来源。
从 `status` 字段提取每个故事的状态。无需 markdown 扫描。
使用其 `sprint`、`goal`、`start`、`end` 字段，而不是重新解析冲刺计划。

**如果 `sprint-status.yaml` 不存在**（旧版冲刺或首次设置），回退到 markdown 扫描：

1. 如果条目引用了故事文件路径，检查该文件是否存在。读取文件并扫描状态标记：DONE、COMPLETE、IN PROGRESS、BLOCKED、NOT STARTED（不区分大小写）。
2. 如果条目没有文件路径（冲刺计划中的内联任务），在冲刺计划本身中扫描该条目旁边的状态标记。
3. 如果未找到状态标记，归类为 NOT STARTED。
4. 如果引用了文件但该文件不存在，归类为 MISSING 并注明。

使用回退时，在输出底部添加一条备注："⚠ 未找到 `sprint-status.yaml`——状态从 markdown 推断。运行 `/sprint-plan update` 生成一个。"

可选（仅快速检查——不做深度扫描）：在 `src/` 中 grep 与故事系统简称匹配的目录或文件名，以检查实现证据。这仅是提示，而非确定状态。

### 陈旧故事检测

在收集所有故事的状态后，检查每个 IN PROGRESS 的故事是否存在陈旧情况：

- 对于每个有引用文件的故事，读取文件并查找 frontmatter 或标题中的 `Last Updated:` 字段（例如 `Last Updated: 2026-04-01` 或 `updated: 2026-04-01`）。接受任何合理的日期字段名：`Last Updated`、`Updated`、`last-updated`、`updated_at`。
- 使用今天的日期计算自该日期以来的天数。
- 如果日期距今超过 4 天，将该故事标记为 **STALE（陈旧）**。（4 天阈值考虑了周末——最后在周五处理的故事直到周三才会显示为陈旧。）
- 如果故事文件中未找到日期字段，注明"无时间戳——无法检查陈旧性。"
- 如果故事没有引用文件（内联任务），注明"内联任务——无法检查陈旧性。"

STALE 故事包含在输出表中，并汇总到"需要注意"部分（见阶段 5 输出格式）。

**陈旧故事升级**：如果任何 IN PROGRESS 故事被标记为 STALE（超过 4 天无进展），燃尽判定至少升级为 **At Risk（有风险）**——即使完成百分比在正常的按计划窗口内。记录此升级原因："At Risk — [N] 个故事在 [N] 天内无进展。"

---

## 4. 燃尽评估

计算：
- 已完成任务（DONE 或 COMPLETE）
- 进行中任务（IN PROGRESS）
- 已阻塞任务（BLOCKED）
- 未开始任务（NOT STARTED 或 MISSING）
- 完成百分比：（完成数 / 总数）* 100

通过比较完成百分比与已消耗时间百分比来评估燃尽情况：

- **按计划 (On Track)**：完成百分比在已消耗时间百分比的 10 个百分点以内或超前
- **有风险 (At Risk)**：完成百分比落后已消耗时间百分比 10-25 个百分点
- **落后 (Behind)**：完成百分比落后已消耗时间百分比超过 25 个百分点

如果日期不可用，跳过燃尽评估并报告"按计划 / 有风险 / 落后：未知——未找到冲刺日期。"

---

## 5. 输出

保持输出简洁。故事状态表是强制性的——不要截断。总行数控制在 50 行以内；如果未发现值得注意的内容，省略新出现风险部分。使用以下格式：

```markdown
## Sprint [N] 状态 — [今天的日期]
**冲刺目标**：[来自冲刺计划]
**剩余天数**：[N] / [总天数]（[已消耗时间 %]）

### 进度：[已完成/总任务数]（[%]）

| 故事 / 任务         | 优先级   | 状态        | 负责人 | 阻塞项        |
|----------------------|------------|-------------|---------|----------------|
| [标题]              | Must Have  | DONE        | [负责人] |                |
| [标题]              | Must Have  | IN PROGRESS | [负责人] |                |
| [标题]              | Must Have  | BLOCKED     | [负责人] | [简要原因] |
| [标题]              | Should Have| NOT STARTED | [负责人] |                |

### 需要注意
| 故事 / 任务         | 状态      | 最后更新   | 陈旧天数 | 备注           |
|----------------------|-------------|----------------|------------|----------------|
| [标题]              | IN PROGRESS | [日期或 N/A]  | [N 天]   | [STALE / 无时间戳 — 无法检查陈旧性 / 内联任务 — 无法检查陈旧性] |

*（如果没有 IN PROGRESS 故事存在陈旧性或时间戳问题，完全省略此部分。）*

### 燃尽：[按计划 / 有风险 / 落后]
[1-2 句话。如果落后：哪些必须完成项有风险。如果按计划：确认并指出团队可以拉取的应该完成项。]

### 有风险的必须完成项
[列出任何状态为 BLOCKED 或 NOT STARTED 且冲刺时间剩余不足 40% 的必须完成故事。如果没有，写"无。"]

### 新出现风险
[从故事扫描中可见的任何风险：缺失文件、级联阻塞项、无负责人的故事。如果没有，写"未发现。"]

### 建议
[一个具体行动，或"冲刺按计划进行——无需操作。"]
```

---

## 6. 快速升级规则

在输出前应用这些规则，如果触发，将标记放在输出的顶部（状态表上方）：

**关键标记**——如果必须完成故事被 BLOCKED 或 NOT STARTED，且冲刺时间剩余不足 40%：

```
冲刺有风险：[N] 个必须完成故事在剩余 [X]% 的冲刺时间内未完成。建议使用 `/sprint-plan update` 重新规划。
```

**完成标记**——如果所有必须完成故事都是 DONE：

```
所有必须完成项已完成。团队可以从应该完成积压中拉取任务。
```

**缺失故事标记**——如果任何引用的故事文件不存在：

```
注意：冲刺计划中引用的 [N] 个故事文件缺失。
运行 `/story-readiness sprint` 以验证故事文件覆盖范围。
```

---

## 协作协议

此技能为只读。它报告从磁盘文件观察到的事实。

- 它不会更新冲刺计划
- 它不会更改故事状态
- 它不会提议范围削减（这是 `/sprint-plan update` 的职责）
- 它每次运行最多提出一个建议

如需了解特定故事的更多详情，用户可以直接读取故事文件或运行 `/story-readiness [path]`。

如需重新规划冲刺，请使用 `/sprint-plan update`。
如需冲刺结束后的回顾，请使用 `/retrospective`。
