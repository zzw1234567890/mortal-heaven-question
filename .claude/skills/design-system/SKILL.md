当此技能被调用时：

## 1. 解析参数与验证

解析审查模式（一次完成，在此次运行的所有关卡生成中存储使用）：
1. 如果传入了 `--review [full|lean|solo]` → 使用该模式
2. 否则读取 `production/review-mode.txt` → 使用该值
3. 否则 → 默认为 `lean`

完整检查模式请参见 `.claude/docs/director-gates.md`。

系统名称或改造路径为**必填项**。如果缺失：

1. 检查 `design/gdd/systems-index.md` 是否存在。
2. 如果存在：读取它，找到优先级最高的状态为"未开始"或同等的系统，并使用 `AskUserQuestion`：
   - 提示："你的设计顺序中的下一个系统是 **[系统名称]**（[优先级] | [层级]）。开始设计它吗？"
   - 选项：`[A] 是——设计 [系统名称]` / `[B] 选择其他系统` / `[C] 在此停止`
   - 如果 [A]：继续使用该系统名称。如果是 [B]：询问要设计哪个系统（纯文本）。如果是 [C]：退出。
3. 如果不存在系统索引，则报错：
   > "用法：`/design-system <系统名称>`——例如，`/design-system movement`
   > 或者要填补现有游戏设计文档(GDD)的空白：`/design-system retrofit design/gdd/[系统名称].md`
   > 未找到系统索引。请先运行 `/map-systems` 来映射系统并获取设计顺序。"

**检测改造模式：**
如果参数以 `retrofit` 开头，或者参数是 `design/gdd/` 中现有 `.md` 文件的路径，则进入**改造模式**：

1. 读取现有的游戏设计文档(GDD)文件。
2. 识别哪些必需的8个章节存在（扫描章节标题）。
   必需章节：概述(Overview)、玩家幻想(Player Fantasy)、详细设计/规则(Detailed Design/Rules)、
   公式(Formulas)、边界情况(Edge Cases)、依赖关系(Dependencies)、调优参数(Tuning Knobs)、验收标准(Acceptance Criteria)。
3. 识别哪些章节仅包含占位文本（`[To be designed]` 或类似的空白、单行、明显不完整的内容）。
4. 在执行任何操作之前向用户展示：
   ```
   ## 改造：[系统名称]
   文件：design/gdd/[文件名].md

   已编写完成的章节（不会触碰）：
   ✓ [章节名称]
   ✓ [章节名称]

   缺失或不完整的章节（将重新编写）：
   ✗ [章节名称] — 缺失
   ✗ [章节名称] — 仅有占位符
   ```
5. 询问："是否要填补 [N] 个缺失的章节？我不会修改任何现有内容。"
6. 如果是：正常进入**阶段 2（收集上下文）**，但在**阶段 3**中跳过创建骨架（文件已存在），在**阶段 4**中跳过已完成章节。仅对缺失/不完整的章节执行章节循环。
7. **切勿覆盖现有章节内容。** 使用编辑(Edit)工具仅替换 `[To be designed]` 占位符或空的章节体。

如果不在改造模式中，将系统名称规范化为短横线命名法(kebab-case)用于文件名（例如，"combat system" 变为 `combat-system`）。

## 2. 收集上下文（读取阶段）

**在向用户询问任何问题之前**读取所有相关上下文。这是本技能相对于临时设计的核心优势——它在了解信息后才开始。

### 2a：必需读取

- **游戏概念**：读取 `design/gdd/game-concept.md`——如果缺失则报错：
  > "未找到游戏概念。请先运行 `/brainstorm`。"
- **系统索引**：读取 `design/gdd/systems-index.md`——如果缺失则报错：
  > "未找到系统索引。请先运行 `/map-systems` 来映射你的系统。"
- **目标系统**：在索引中找到该系统。如果未列出，则警告：
  > "[系统名称] 不在系统索引中。是否要添加它，或者将其设计为索引外系统？"
- **实体注册表**：读取 `design/registry/entities.yaml`（如果存在）。
  提取与该系统相关或被其引用的所有条目（通过 grep 查找
  `referenced_by.*[系统名称]` 和 `source.*[系统名称]`）。将这些保留在
  上下文中作为**已知事实**——其他游戏设计文档(GDD)已经确立的、本游戏设计文档(GDD)不得矛盾的值。
- **反思日志**：读取 `docs/consistency-failures.md`（如果存在）。
  提取其领域(Domain)匹配本系统分类的条目。这些是
  重复出现的冲突模式——在阶段 2d 的上下文摘要的"过去失败模式"下展示它们，
  让用户了解在此领域中之前出现过的错误。

### 2b：依赖读取

从系统索引中，识别：
- **上游依赖**：本系统依赖的系统。如果存在则读取它们的游戏设计文档(GDD)
  （这些包含本系统必须尊重的决策）。
- **下游依赖者**：依赖本系统的系统。如果存在则读取它们的游戏设计文档(GDD)
  （这些包含本系统必须满足的期望）。

对于每个存在的依赖游戏设计文档(GDD)，提取并在上下文中保留：
- 关键接口（系统之间流动的数据）
- 引用本系统输出的公式
- 假设本系统行为的边界情况
- 输入本系统的调优参数

### 2c：可选读取

- **游戏支柱**：读取 `design/gdd/game-pillars.md`（如果存在）
- **现有游戏设计文档(GDD)**：读取 `design/gdd/[系统名称].md`（如果存在）（继续，不要从头开始）
- **相关游戏设计文档(GDD)**：通过 Glob 匹配 `design/gdd/*.md` 并读取任何主题上相关的文档
  （例如，如果设计的系统在范围上与另一个重叠，即使不是正式的依赖关系也要读取相关游戏设计文档(GDD)）

### 2d：展示上下文摘要

在开始设计工作之前，向用户展示简要摘要：

> **设计：[系统名称]**
> - 优先级：[来自索引] | 层级：[来自索引]
> - 依赖：[列表，标注哪些已有游戏设计文档(GDD)，哪些尚未设计]
> - 被依赖：[列表，标注哪些已有游戏设计文档(GDD)，哪些尚未设计]
> - 需要尊重的现有决策：[来自依赖游戏设计文档(GDD)的关键约束]
> - 支柱对齐：[本系统主要服务的支柱]
> - **已知跨系统事实（来自注册表）：**
>   - [实体名称]: [属性]=[值], [属性]=[值]（归属 [来源游戏设计文档(GDD)]）
>   - [物品名称]: [属性]=[值], [属性]=[值]（归属 [来源游戏设计文档(GDD)]）
>   - [公式名称]: 变量=[列表], 输出=[最小值–最大值]（归属 [来源游戏设计文档(GDD)]）
>   - [常量名称]: [值] [单位]（归属 [来源游戏设计文档(GDD)]）
>   *(这些值已锁定——如果本游戏设计文档(GDD)需要不同的值，请在编写前指出冲突。不要静默使用不同的数字。)*
>
> 如果没有相关的注册表条目：省略"已知跨系统事实"章节。

如果有任何上游依赖尚未设计，则警告：
> "[依赖] 还没有游戏设计文档(GDD)。我们需要对其接口做出假设。建议先设计它，或者我们可以定义预期的契约并将其标记为临时性。"

### 2e：技术可行性预检

在要求用户开始设计之前，加载引擎上下文并展示任何将影响设计的约束或知识空白。

**步骤 1——确定本系统的引擎领域：**
将系统的分类（来自 systems-index.md）映射到引擎领域：

| 系统分类 | 引擎领域 |
|----------------|--------------|
| 战斗、物理、碰撞 | 物理(Physics) |
| 渲染、视觉效果、着色器 | 渲染(Rendering) |
| 界面(UI)、抬头显示(HUD)、菜单 | 界面(UI) |
| 音频、音效、音乐 | 音频(Audio) |
| 人工智能(AI)、寻路、行为树 | 导航/脚本(Navigation / Scripting) |
| 动画、反向动力学(IK)、骨架 | 动画(Animation) |
| 网络、多人、同步 | 网络(Networking) |
| 输入、控制、按键绑定 | 输入(Input) |
| 存档/读档、持久化、数据 | 核心(Core) |
| 对话、任务、叙事 | 脚本(Scripting) |

**步骤 2——读取引擎上下文（如果可用）：**
- 读取 `.claude/docs/technical-preferences.md` 以确定引擎和版本
- 如果引擎已配置，读取 `docs/engine-reference/[引擎]/VERSION.md`
- 读取 `docs/engine-reference/[引擎]/modules/[领域].md`（如果存在）
- 读取 `docs/engine-reference/[引擎]/breaking-changes.md` 中领域相关的条目
- 通过 Glob 匹配 `docs/architecture/adr-*.md` 并读取任何领域匹配的架构决策记录(ADR)
  （检查引擎兼容性表的"领域"字段）

**步骤 3——展示可行性简报：**

如果存在引擎参考文档，在开始设计前展示：

```
## 技术可行性简报：[系统名称]
引擎：[名称 + 版本]
领域：[领域]

### 已知引擎能力（已验证于 [版本]）
- [与本系统相关的能力]
- [能力 2]

### 将影响本设计的引擎约束
- [来自引擎参考或现有架构决策记录(ADR)的约束]

### 知识空白（在承诺使用前验证）
- [本设计可能依赖的、模型训练截止日期后的特性——标记为高/中风险]

### 约束本系统的现有架构决策记录(ADR)
- 架构决策记录(ADR)-XXXX: [决策摘要]——意味着 [对本游戏设计文档(GDD)的影响]
  （或"暂无"）
```

如果不存在引擎参考文档（引擎尚未配置），显示简短提示：
> "尚未配置引擎——跳过技术可行性检查。如果尚未完成，请在进入架构阶段前运行 `/setup-engine`。"

**步骤 4——询问是否继续：**

使用 `AskUserQuestion`：
- "在开始之前有任何要添加的约束，或者我们按以上标注继续？"
  - 选项："按以上标注继续"，"先添加约束"，"我需要检查引擎文档——在此暂停"

---

使用 `AskUserQuestion`：
- "准备好开始设计 [系统名称] 了吗？"
  - 选项："是的，开始吧"，"先展示更多上下文"，"先设计一个依赖"

## 4. 逐节设计

按顺序遍历每个章节。对于**每个章节**，遵循以下循环：

### 章节循环

```
上下文  ->  问题  ->  选项  ->  决策  ->  草稿  ->  批准  ->  写入
```

1. **上下文**：说明本章节需要包含的内容，并展示来自依赖游戏设计文档(GDD)中约束它的相关决策。

2. **问题**：针对本章节提出澄清性问题。对限定性问题使用 `AskUserQuestion`，对开放式探索使用对话文本。

3. **选项**：当章节涉及设计选择（而不仅仅是文档记录）时，提供 2-4 种方法并附上优缺点。在对话文本中说明理由，然后使用 `AskUserQuestion` 记录决策。

4. **决策**：用户选择一种方法或提供自定义方向。

5. **草稿**：在对话文本中编写章节内容以供审查。标记任何关于尚未设计的依赖的临时性假设。

6. **批准**：在草稿之后——在**同一响应**中——使用 `AskUserQuestion`。**切勿使用纯文本。切勿跳过此步骤。**
   - 提示："批准 [章节名称] 章节？"
   - 选项：`[A] 批准——写入文件` / `[B] 修改——描述需要修复的内容` / `[C] 重新开始`

   **草稿和批准小工具必须在同一响应中一起出现。
   如果草稿出现时没有小工具，用户将面对空白提示符，无路可走——这是协议违规。**

7. **写入**：使用编辑(Edit)工具将占位符替换为已批准的内容。
   **关键**：始终在 `old_string` 中包含章节标题以确保唯一性——切勿单独匹配 `[To be designed]`，因为多个章节使用相同的占位符，而编辑(Edit)工具需要唯一匹配。使用此模式：
   ```
   old_string: "## [章节名称]\n\n[To be designed]"
   new_string: "## [章节名称]\n\n[已批准的内容]"
   ```
   确认写入。

8. **注册表冲突检查**（仅限章节 C 和 D——详细设计和公式）：
   写入后，扫描章节内容中出现在注册表中的实体名称、物品名称、公式名称和数值常量。对于每个匹配：
   - 比较刚写入的值与注册表条目。
   - 如果不同：在开始下一章节之前**立即指出冲突**。不要静默继续。
     > "注册表冲突：[名称] 在 [来源游戏设计文档(GDD)] 中注册为 [注册表值]。
     > 本章节刚写入了 [新值]。哪个是正确的？"
   - 如果是新值（不在注册表中）：将其标记为注册表注册的候选对象（将在阶段 5 处理）。

在写入每个章节后，用已完成的章节名称更新 `production/session-state/active.md`。使用 Glob 检查文件是否存在——使用写入(Write)创建文件（若不存在），使用编辑(Edit)更新文件（若已存在）。

### 章节特定指南

每个章节都有独特的设计考量，可能受益于专业代理：

---

### 章节 A：概述

**目标**：一个陌生人能读懂并理解的段落。

**在构建小工具之前推导推荐选项**：从系统索引读取系统的分类和层级（已在阶段 2 的上下文中），然后确定每个标签的推荐选项：
- **框架标签**：基础/基础设施层 → 推荐 `[A]`。面向玩家的分类（战斗、界面、对话、角色、动画、视觉效果、音频）→ 推荐 `[C] 两者兼顾`。
- **架构决策记录(ADR)引用标签**：通过 Glob 匹配 `docs/architecture/adr-*.md` 并在任何架构决策记录(ADR)的游戏设计文档(GDD)需求章节中 grep 系统名称。如果找到匹配的架构决策记录(ADR) → 推荐 `[A] 是——引用架构决策记录(ADR)`。如果未找到 → 推荐 `[B] 否`。
- **幻想标签**：基础/基础设施层 → 推荐 `[B] 否`。所有其他分类 → 推荐 `[A] 是`。

在相应选项文本后追加 `（推荐）`。

**框架性问题（在起草前询问）**：使用多标签小工具的 `AskUserQuestion`：
- 标签"框架"——"概述应如何构建本系统？" 选项：`[A] 作为数据/基础设施层（技术框架）` / `[B] 通过其面向玩家的效果（设计框架）` / `[C] 两者兼顾——描述数据层及其对玩家的影响`
- 标签"架构决策记录(ADR)引用"——"概述是否应引用本系统的现有架构决策记录(ADR)？" 选项：`[A] 是——引用架构决策记录(ADR)了解实现细节` / `[B] 否——保持游戏设计文档(GDD)在纯设计层面`
- 标签"幻想"——"本系统是否有值得陈述的玩家幻想？" 选项：`[A] 是——玩家可以直接感受到` / `[B] 否——纯基础设施，玩家感受到的是它带来的可能性`

使用用户的答案来塑造草稿。不要自行回答这些问题并自动起草。

**要问的问题**：
- 用一句话来说，这个系统是什么？
- 玩家如何与之互动？（主动/被动/自动）
- 这个系统为什么存在——没有它游戏会缺少什么？

**交叉引用**：检查描述是否与系统索引的描述方式一致。标记差异。

**设计与实现边界**：概述问题必须停留在行为层面——系统**做什么**，而不是**如何构建**。如果在概述中出现实现问题（例如，"这应该使用自动加载单例还是信号总线？"），将其标记为"→ 转为架构决策记录(ADR)"并继续。实现模式属于 `/architecture-decision`，不属于游戏设计文档(GDD)。游戏设计文档(GDD)描述行为；架构决策记录(ADR)描述用于实现该行为的技术方法。

### 章节 B：玩家幻想

**目标**：情感目标——玩家应该*感受到*什么。

**在构建小工具之前推导推荐选项**：从阶段 2 上下文读取系统的分类和层级：
- 面向玩家的分类（战斗、界面、对话、角色、动画、音频、关卡/世界）→ 推荐 `[A] 直接`
- 基础/基础设施层 → 推荐 `[B] 间接`
- 混合分类（摄像机/输入、经济、具有可见玩家效果的人工智能(AI)）→ 推荐 `[C] 两者兼顾`

在相应选项文本后追加 `（推荐）`。

**框架性问题（在起草前询问）**：使用 `AskUserQuestion`：
- 提示："这个系统是玩家直接参与的系统，还是玩家间接体验的基础设施？"
- 选项：`[A] 直接——玩家主动使用或感受此系统` / `[B] 间接——玩家感受到的是效果，而非系统本身` / `[C] 两者兼顾——有直接的交互层和底层基础设施`

使用答案来适当构建玩家幻想章节。不要假设答案。

**要问的问题**：
- 这服务于什么样的情感或力量幻想？
- 哪些参考游戏完美实现了这种感觉？具体是什么创造了它？
- 这是"你乐于参与的系统"还是"你不会注意到的底层设施"？

**交叉引用**：必须与游戏支柱对齐。如果系统服务于某个支柱，引用相关的支柱文本。

**审查模式检查**（在生成之前应用）：
- `solo` → 跳过此代理的生成。在没有专家的情况下起草该章节。添加备注："未咨询 `creative-director`——单人模式。请在生产前手动审查。"
- `lean` → 跳过，除非这是具有高风险实现的章节（仅限章节 D 和 H）。对于其他章节，在没有代理的情况下起草。
- `full` → 按下文描述生成。

**代理委派（强制）**：在给出框架性答案后但在起草之前，通过任务(Task)生成 `creative-director`：
- 提供：系统名称、框架性答案（直接/间接/两者兼顾）、游戏支柱、用户提到的任何参考游戏、游戏概念摘要
- 询问："塑造本系统的玩家幻想。它应服务于什么样的情感或力量幻想？我们应该锚定到什么样的玩家时刻？什么样的语气和语言符合游戏已建立的风格？具体一些——给我 2-3 个候选框架。"
- 收集创意总监(creative-director)的框架，并与草稿一起呈现给用户。

**未经事先咨询 `creative-director` 不得起草章节 B。** 框架性答案告诉我们它是什么*类型*的幻想；创意总监负责塑造*如何描述它*——语气、语言、要锚定的具体玩家时刻。

### 章节 C：详细设计（核心规则、状态、交互）

**目标**：程序员无需问题即可实现的无歧义规范。

这通常是最大的章节。将其拆分为子章节：

1. **核心规则**：基本机制。顺序流程使用编号规则，属性使用项目符号。
2. **状态与转换**：如果系统有状态，映射每个状态和每个有效转换。使用表格。
3. **与其他系统的交互**：对于每个依赖（上游和下游），指定哪些数据流入、哪些流出以及谁拥有接口。

**要问的问题**：
- 请逐步引导我完成此系统的典型用法
- 玩家面临哪些决策点？
- 玩家**不能**做什么？（约束和能力一样重要）

**审查模式检查**（在生成之前应用）：
- `solo` → 跳过此代理的生成。在没有专家的情况下起草该章节。添加备注："未咨询专业代理——单人模式。请在生产前手动审查。"
- `lean` → 跳过，除非这是具有高风险实现的章节（仅限章节 D 和 H）。对于其他章节，在没有代理的情况下起草。
- `full` → 按下文描述生成。

**代理委派（强制）**：在起草章节 C 之前，通过任务(Task)并行生成专业代理：
- 在路由表中查找系统分类（本技能的第 6 节）
- 生成本分类列出的主要代理和辅助代理
- 向每个代理提供：系统名称、游戏概念摘要、支柱集、依赖游戏设计文档(GDD)摘录、正在处理的具体章节
- 在起草前收集他们的发现
- 通过 `AskUserQuestion` 向用户展示代理之间的任何分歧
- 仅在收到专业输入后起草

**未经事先咨询相关专家不得起草章节 C。** 审查规则和机制的系统设计师将发现主会话无法发现的设计缺陷。

**交叉引用**：对于列出的每个交互，验证它是否与依赖游戏设计文档(GDD)指定的匹配。如果依赖定义了一个值或公式而本系统期望不同的值，标记冲突。

### 章节 D：公式

**目标**：每个数学公式，带有定义的变量、指定的范围和标注的边界情况。

**完成引导——始终以此精确结构开始每个公式：**

```
[公式名称] 公式定义为：

`[公式名称] = [表达式]`

**变量：**
| 变量 | 符号 | 类型 | 范围 | 描述 |
|----------|--------|------|-------|-------------|
| [名称] | [符号] | 浮点数/整数 | [最小值–最大值] | [表示的内容] |

**输出范围：** 正常游戏下 [最小值] 到 [最大值]；[极值时的行为]
**示例：** [带实际数字的示例]
```

不要写 `[公式待定]` 或在没有变量表的情况下用散文化文字描述公式。没有定义变量的公式无法在没有猜测的情况下实现。

**要问的问题**：
- 此系统执行哪些核心计算？
- 缩放应该是线性的、对数的还是阶梯式的？
- 在游戏早期/中期/后期，输出范围应该是什么样的？

**审查模式检查**（在生成之前应用）：
- `solo` → 跳过此代理的生成。在没有专家的情况下起草该章节。添加备注："未咨询 `systems-designer`——单人模式。请在生产前手动审查。"
- `lean` → 跳过，除非这是具有高风险实现的章节（仅限章节 D 和 H）。对于其他章节，在没有代理的情况下起草。
- `full` → 按下文描述生成。

**代理委派（强制）**：在提出任何公式或平衡值之前，通过任务(Task)并行生成专业代理：
- **始终生成 `systems-designer`**：提供来自章节 C 的核心规则、用户的调优目标、来自依赖游戏设计文档(GDD)的平衡上下文。请他们提出带有变量表和输出范围的公式。
- **对于经济/成本系统，还要生成 `economy-designer`**：提供布局成本、升级成本意图和进度目标。请他们验证成本曲线和比率。
- 通过 `AskUserQuestion` 将专家的提案呈现给用户审查
- 用户决定；主会话写入文件
- **未经专家输入，不得自行发明公式值或平衡数值。** 没有平衡设计专业知识的用户无法评估原始数字——他们需要专家的推理过程。

**交叉引用**：如果依赖游戏设计文档(GDD)定义了一个输出输入本系统的公式，显式引用它。不要重新发明——进行连接。

### 章节 E：边界情况

**目标**：显式处理异常情况，使它们不会变成错误。

**完成引导——将每个边界情况格式化为：**
- **如果 [条件]**：[精确结果]。[如果不明显的理由]

示例（根据游戏领域调整术语）：
- **如果 [资源] 在 [保护性条件] 激活时达到 0**：保持在最小值直到条件结束，然后应用后果。
- **如果两个 [触发器/事件] 同时触发**：按照 [定义的优先级顺序] 解决；平局使用 [定义的平局决胜规则]。

不要写"适当处理"之类的模糊条目——每个条目必须命名确切的条件和确切的解决方案。没有解决方案的边界情况是开放的设计问题，而不是规范。

**要问的问题**：
- 在零时会发生什么？在最大值时？在超出范围的值时？
- 当两条规则同时适用时会发生什么？
- 如果玩家发现了意外的交互会发生什么？（识别退化策略）

**审查模式检查**（在生成之前应用）：
- `solo` → 跳过此代理的生成。在没有专家的情况下起草该章节。添加备注："未咨询 `systems-designer`——单人模式。请在生产前手动审查。"
- `lean` → 跳过，除非这是具有高风险实现的章节（仅限章节 D 和 H）。对于其他章节，在没有代理的情况下起草。
- `full` → 按下文描述生成。

**代理委派（强制）**：在最终确定边界情况之前通过任务(Task)生成 `systems-designer`。提供：已完成的章节 C 和 D，请他们从公式和规则空间中识别主会话可能遗漏的边界情况。对于叙事系统，还要生成 `narrative-director`。将他们的发现呈现给用户并询问要包含哪些内容。

**交叉引用**：对照依赖游戏设计文档(GDD)检查边界情况。如果依赖定义了本系统可能违反的底线、上限或解决规则，标记它。

---

### 章节 F：依赖关系

**目标**：映射每个系统连接的方向和性质。

此章节部分预先填充了来自上下文收集阶段的信息。展示来自系统索引的已知依赖并询问：
- 是否有我遗漏的依赖？
- 对于每个依赖，具体的数据接口是什么？
- 哪些依赖是硬性的（系统没有它就无法运行）vs. 软性的（有它更好但没有也能工作）？

**交叉引用**：本章节必须是双向一致的。如果本系统列出"依赖战斗"，那么战斗游戏设计文档(GDD)应列出"被 [本系统] 依赖"。标记任何单向依赖以供纠正。

---

### 章节 G：调优参数

**目标**：每个设计者可调整的值，附有安全范围和极端行为。

**要问的问题**：
- 哪些值应该是设计者无需修改代码即可调整的？
- 对于每个参数，如果设置得过高会出什么问题？过低呢？
- 哪些参数相互影响？（更改 A 使 B 变得无关紧要）

**代理委派**：如果公式复杂，委派给 `systems-designer` 以从公式变量推导调优参数。

**交叉引用**：如果依赖游戏设计文档(GDD)列出了影响本系统的调优参数，在此引用它们。不要创建重复的参数——指向事实来源。

---

### 章节 H：验收标准

**目标**：可测试的条件，证明系统按设计工作。

**完成引导——将每个标准格式化为 Given-When-Then（给定-当-则）：**
- **GIVEN** [初始状态]，**WHEN** [动作或触发器]，**THEN** [可衡量的结果]

示例（根据游戏领域调整术语）：
- **GIVEN** [初始状态]，**WHEN** [玩家动作或系统触发器]，**THEN** [具体的可衡量结果]。
- **GIVEN** [约束激活]，**WHEN** [玩家尝试动作]，**THEN** [显示反馈和动作结果]。

至少包括：来自章节 C 的每个核心规则一个标准，以及来自章节 D 的每个公式一个标准。不要写"系统按设计工作"——每个标准必须能由质量保证(QA)测试人员在无需阅读游戏设计文档(GDD)的情况下独立验证。

**审查模式检查**（在生成之前应用）：
- `solo` → 跳过此代理的生成。在没有专家的情况下起草该章节。添加备注："未咨询 `qa-lead`——单人模式。请在生产前手动审查。"
- `lean` → 跳过，除非这是具有高风险实现的章节（仅限章节 D 和 H）。对于其他章节，在没有代理的情况下起草。
- `full` → 按下文描述生成。

**代理委派（强制）**：在最终确定验收标准之前通过任务(Task)生成 `qa-lead`。提供：已完成的游戏设计文档(GDD)章节 C、D、E，并请他们验证这些标准是否可独立测试并覆盖所有核心规则和公式。向用户展示任何空白或不可测试的标准。

**要问的问题**：
- 证明此系统有效的最少测试集是什么？
- 此系统获得多少性能预算？（帧时间、内存）
- 质量保证(QA)测试人员会首先检查什么？

**交叉引用**：包括验证跨系统交互工作的标准，而不仅仅是孤立的此系统。

对于**视觉/音频**（非必需系统）：如果需要细节，与 `art-director` 和 `audio-director` 协调。在游戏设计文档(GDD)阶段，简短的说明通常就足够了。

> **资源规格标记**：在视觉/音频章节写入实际内容后，输出此通知：

对于**用户界面需求**：对于复杂的用户界面系统，与 `ux-designer` 协调。

> 如果更新系统索引，请在此系统中记下这一点。

对于**待解决问题**：记录设计期间出现的任何未完全解决的问题。每个问题应有负责人和目标解决日期。

  已注册（将更新 referenced_by）：

## 7. 恢复与继续

如果会话中断（上下文压缩、崩溃、新会话）：

1. 读取 `production/session-state/active.md`——它记录了当前系统和哪些章节已完成
2. 读取 `design/gdd/[系统名称].md`——有实际内容的章节已完成；有 `[To be designed]` 的章节仍需工作
3. 从下一个不完整的章节继续——无需重新讨论已完成的章节

这就是增量写入重要的原因：每个已批准的章节能经受任何中断。

## 协作协议

此技能在每一步都遵循协作设计原则：

1. 每个章节的**问题 -> 选项 -> 决策 -> 草稿 -> 批准**
2. 在每个决策点使用 **AskUserQuestion**（解释 -> 记录模式）：
   - 阶段 2："准备好开始，还是需要更多上下文？"
   - 阶段 3："我可以创建骨架吗？"
   - 阶段 4（每个章节）：设计问题、方法选项、草稿批准
   - 阶段 5："运行设计审查？更新系统索引？接下来做什么？"
3. 在骨架之前和每个章节写入之前的 **"我可以写入 [文件路径] 吗？"**
4. **增量写入**：每个章节在批准后立即写入文件
5. **会话状态更新**：每次章节写入后
6. **交叉引用**：每个章节检查现有游戏设计文档(GDD)是否存在冲突
7. **专业路由**：复杂章节获得专家代理输入，呈现给用户决策——从不静默写入

**切勿**自动生成完整的游戏设计文档(GDD)并将其作为既成事实呈现。
**切勿**在未经用户批准的情况下写入章节。
**切勿**在未标记冲突的情况下与现有已批准的游戏设计文档(GDD)矛盾。
**始终**展示决策的来源（依赖游戏设计文档(GDD)、支柱、用户选择）。

## 上下文窗口意识

这是一个长时间运行的技能。在写入每个章节后，检查状态行是否显示上下文已使用 70% 或更多。如果是，在响应中追加此通知：

> **上下文正在接近限制（≥70%）。** 你的进度已保存——所有已批准的章节已写入 `design/gdd/[系统名称].md`。准备好继续时，打开一个新的 Claude Code 会话并运行 `/design-system [系统名称]`——它将检测哪些章节已完成并从下一个章节继续。
