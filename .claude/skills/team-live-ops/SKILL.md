**参数检查：** 如果未提供赛季名称或赛事描述，输出：
> "用法：`/team-live-ops [season name or event description]` — 提供要规划的赛季或线上赛事的名称或描述。"
然后立即停止，不派生任何子 agent，也不读取任何文件。

当此技能以有效参数调用时，通过结构化规划管线编排线上运营团队。

**决策点：** 在每个阶段转换时，使用 `AskUserQuestion` 将子 agent 的
提案作为可选选项呈现给用户。在对话中写出 agent 的完整分析，然后以简洁的
标签捕获决策。用户必须批准后才能进入下一阶段。

## 阶段 0：解析评审模式

1. 如果参数中传入了 `--review [mode]`，使用该模式。
2. 否则读取 `production/review-mode.txt` — 使用其中所写内容。
3. 否则默认为 `lean`。

模式：
- `full` — 按描述派生所有 director 和 lead 关卡
- `lean` — 跳过 director 关卡，除非是 PHASE-GATE 类型（CD-PHASE-GATE、TD-PHASE-GATE、PR-PHASE-GATE、AD-PHASE-GATE）
- `solo` — 完全跳过所有 director 关卡的派生；运行技能时不带任何 agent 关卡

保存解析出的模式，供所有后续阶段使用。

## 团队构成
- **live-ops-designer** — 赛季结构、赛事节奏、留存机制、战斗通行证
- **economy-designer** — 线上经济平衡、商店轮换、货币定价、保底计时器
- **analytics-engineer** — 成功指标、A/B 测试设计、赛事追踪、仪表盘规格
- **community-manager** — 面向玩家的公告、赛事描述、赛季消息
- **narrative-director** — 赛季叙事主题、故事弧线、世界事件框架
- **writer** — 赛事描述、奖励物品名称、赛季风味文本、公告文案

## 如何委派

使用 Task 工具将每位团队成员派生为子 agent：
- `subagent_type: live-ops-designer` — 赛季/赛事结构和留存机制
- `subagent_type: economy-designer` — 线上经济平衡和奖励定价
- `subagent_type: analytics-engineer` — 成功指标、A/B 测试、赛事埋点
- `subagent_type: community-manager` — 面向玩家的沟通和消息传递
- `subagent_type: narrative-director` — 赛季主题和叙事框架
- `subagent_type: writer` — 所有面向玩家的文本：赛事描述、物品名称、文案

始终在每个 agent 的提示中提供完整上下文（游戏概念路径、现有赛季文档、伦理政策路径、当前经济状态）。在管线允许的情况下并行启动独立 agent（阶段 3 和 4 可以同时运行）。

## 管线

### 阶段 1：赛季/赛事范围界定
委派给 **live-ops-designer**：
- 定义赛季或赛事：类型（季节性、限时赛事、挑战）、持续时间、主题方向
- 概述内容清单：新增内容（模式、物品、挑战、故事节拍）
- 定义留存钩子：是什么让玩家在本赛季期间每日/每周回归
- 确定资源预算：需要创建多少新内容 vs 复用
- 输出：包含范围、内容清单和留存机制概览的赛季简报

### 阶段 2：叙事主题
委派给 **narrative-director**：
- 读取阶段 1 的赛季简报
- 设计赛季叙事主题：该赛事如何连接到游戏世界？
- 定义玩家在赛事期间将发现的中心故事钩子
- 确定本赛季可以推进哪些现有的世界观线索
- 输出：叙事框架文档（主题、故事钩子、世界观连接）

### 阶段 3：经济设计（如果主题清晰，可与阶段 2 并行）
委派给 **economy-designer**：
- 读取赛季简报和 `design/live-ops/economy-rules.md` 中的现有经济规则
- 设计奖励轨道：免费层进度、高级层价值主张
- 规划赛季内经济：赛季货币、商店轮换、定价
- 定义任何随机元素的保底计时器和坏运气保护机制
- 验证高级轨道中不存在"付费赢"物品
- 输出：包含奖励表、定价和货币流动的经济设计文档

### 阶段 4：分析与成功指标（与阶段 3 并行）
委派给 **analytics-engineer**：
- 读取赛季简报
- 定义成功指标：参与率目标、留存提升目标、战斗通行证完成率
- 设计赛季期间运行的任何 A/B 测试（例如不同的奖励节奏）
- 指定本赛季内容所需的新遥测事件
- 输出：包含成功标准和埋点需求的分析计划

### 阶段 5：内容写作（并行）
并行委派：
- **narrative-director**（如需要）：为本赛季编写游戏内叙事文本（过场脚本、NPC 对话、世界赛事描述）
- **writer**：编写所有面向玩家的文本 — 赛事名称、奖励物品描述、挑战目标文本、赛季风味文本
- 两者都应读取阶段 2 的叙事框架文档

### 阶段 6：玩家沟通计划
委派给 **community-manager**：
- 读取赛季简报、经济设计和叙事框架
- 起草赛季上线公告（语气、关键亮点、平台特定版本）
- 规划沟通节奏：上线前预告、上线日帖子、季中提醒、最后一周 FOMO 推送
- 起草已知问题部分的占位内容，供首日补丁说明使用
- 输出：包含每个接触点草稿文案的沟通日历

### 阶段 7：审查与签收
收集所有阶段的输出，呈现整合的赛季计划：
- 赛季简报（阶段 1）
- 叙事框架（阶段 2）
- 经济设计和奖励表（阶段 3）
- 分析计划和成功指标（阶段 4）
- 已写内容清单（阶段 5）
- 沟通日历（阶段 6）

向用户呈现摘要，包含：
- **内容范围**：正在创建什么
- **经济健康检查**：奖励轨道是否公平且非掠夺性？
- **分析就绪度**：成功标准是否已定义并埋点？
- **伦理审查**：对照 `design/live-ops/ethics-policy.md` 检查阶段 3 的经济设计
  - 如果文件不存在：标记"伦理审查已跳过：`design/live-ops/ethics-policy.md` 未找到。经济设计未经伦理政策审查。建议在生产开始前创建。"在赛季设计输出文档中包含此标记。添加到后续步骤：创建 `design/live-ops/ethics-policy.md`。
  - 如果文件存在且发现违规：标记"伦理警告：[元素] 在阶段 3 经济设计中违反了 [政策规则]。在该问题解决前批准被阻塞。"不要发出 COMPLETE 判定或编写输出文档。使用 `AskUserQuestion` 提供选项：修改经济设计 / 覆盖并记录理由 / 取消。如果用户选择修改：重新派生 economy-designer 以生成更正后的设计，然后返回阶段 7 审查。如果用户选择取消：以判定 BLOCKED 结束 — "因未解决的伦理违规，线上运营设计已取消。解决标记的问题后重新运行 /team-live-ops。"
- **未解决问题**：在生产开始前仍需做出的决策

在将赛季计划委派给生产团队之前，要求用户批准。仅在用户批准且不存在未解决的伦理违规后，发出 COMPLETE 判定。如果伦理违规未解决，以判定 **BLOCKED** 结束。

## 输出文档

所有文档保存到 `design/live-ops/`：
- `seasons/S[N]_[name].md` — 赛季设计文档（来自阶段 1-3）
- `seasons/S[N]_[name]_analytics.md` — 分析计划（来自阶段 4）
- `seasons/S[N]_[name]_comms.md` — 沟通日历（来自阶段 6）

## 错误恢复协议

如果任何派生的 agent（通过 Task）返回 BLOCKED、出错或无法完成：

1. **立即暴露**：在继续依赖阶段之前向用户报告 "[AgentName]: BLOCKED — [原因]"
2. **评估依赖**：检查受阻 agent 的输出是否为后续阶段所需。如果是，在没有用户输入的情况下不要越过该依赖点。
3. **通过 AskUserQuestion 提供选项**：
   - 跳过此 agent 并在最终报告中记录缺口
   - 缩小范围后重试
   - 就此停止，先解决阻塞问题
4. **始终产出部分报告** — 输出已完成的内容。永远不要因一个 agent 受阻而丢弃工作成果。

如果 BLOCKED 状态无法解决，以判定 **BLOCKED** 结束，而非 COMPLETE。

## 文件写入协议

所有文件写入（赛季设计文档、分析计划、沟通日历）均委派给
通过 Task 派生的子 agent。每个子 agent 执行
"May I write to [path]?" 协议。本编排器不直接写文件。

## 输出

摘要，包括：赛季主题和范围、经济设计亮点、成功指标、内容清单、沟通计划以及任何在生产前需要用户输入待定决策。

判定：**COMPLETE** — 赛季计划已产出并移交生产。
