## 目的

**垂直切片（vertical slice）** 回答的是一个与概念原型（concept prototype）不同的问题：
*"我们能否按照生产质量、按时交付完整的游戏循环？"*

**默认用途** —— 在预生产后期运行，即 GDD、架构和 UX 规范完成后。它展示一个接近生产质量的构建，演示一个完整的 [开始 → 挑战 → 解决] 循环。

**转型后重试？** 如果早期垂直切片的 PIVOT 判定要求你回去修改 GDD 和架构，那么在修改完成后再次运行本流程以重新验证。可以根据需要多次运行，直到达成 PROCEED 或 KILL 判定。

它验证以下内容：

1. **管线**（团队能否真正产出这种质量的内容？）
2. **执行可行性**（针对该游戏，架构决策是否正确？）
3. **乐趣存续性**（概念原型中的乐趣在完整设计中是否仍然存在？）
4. **开发速度**（花了多长时间？这将是你的真实生产效率估算。）

**项目早期？** 如果你尚未编写 GDD，只是想验证核心创意是否值得设计，请运行 `/prototype`（概念原型）。

## 阶段 2：定义切片范围与验证问题

在构建之前，定义**可证伪的验证问题**：

> *"一个从零开始的玩家，是否能在 [N] 分钟内、无需开发者引导地体验到 [来自 game-concept.md 的核心愿景]，而我们能否在 [X] 天内以代表性质量构建这样一个循环？"*

两部分都很重要：玩家体验 **和** 构建可行性。

**范围纪律：**
- 包含所有核心循环系统（最低限度）。如果一个系统是完成一个 [开始 → 挑战 → 解决] 循环所必需的，它就必须纳入切片中。
- **目标范围：3–5 分钟的精致、连续游戏体验。** 这是行业标准的垂直切片时长 —— 足够长以展示机制和基调，足够短以代表性能构建。如果你的切片需要超过 5 分钟才能玩完，请削减内容，而非质量。
- **在削减质量之前先削减范围。** 一个低质量的切片如果看起来与目标游戏完全不同，就无法验证生产可行性。
- 如果感觉范围太大，无法在 1-3 周内完成，说明切片范围有问题 —— 不是太大而无法构建，而是切片试图一次性证明太多东西。

**范围蔓延警告：** 垂直切片是预生产阶段范围蔓延风险最高的时刻。功能感觉"差不多了"，很容易想添加"再多一个系统"。请抵制这种冲动。削减，而非扩展。

在构建前向用户展示范围并获得确认。

## 阶段 3：规划构建

用要点定义：
- 实现的系统（正在演练哪些 GDD 章节）
- 完整的游戏循环（准确的 [开始] → [挑战] → [解决]）
- 美术和音频质量等级（可接受占位资产，优先使用代表性资产）
- 针对验证问题的具体、可衡量的成功标准
- 硬时间限制：[X] 天。如果超出，说明范围有问题 —— 停止并重新评估。

要求用户在构建前确认范围。

确认后，将当前会话检查点写入 `production/session-state/active.md`（如果 `production/session-state/` 不存在则创建）。内容包括：概念名称、验证问题、范围内系统、美术质量等级和当前阶段（"阶段 4 —— 实施"）。在每个构建日结束时，更新该文件以记录完成情况。这是若会话在中途结束时的主要恢复机制 —— 跨多周的引擎构建将跨越许多会话。

## 阶段 4：实施

询问："我是否可以在 `prototypes/[概念名称]-vertical-slice/` 下创建垂直切片目录并开始实施？"

如果同意，创建该目录。每个文件必须以以下内容开头：

```
// VERTICAL SLICE - NOT FOR PRODUCTION
// Validation Question: [本构建所要证明的内容]
// Date: [当前日期]
```

**质量标准** —— 高于概念原型，但未达到完整生产水平：
- 遵循 `docs/architecture/control-manifest.md` 中的架构层级
- 遵循 `.claude/docs/technical-preferences.md` 中的命名约定
- 不使用硬编码的游戏数值 —— 使用常量或配置文件
- 关键路径上要有基本错误处理
- 可接受占位美术；优先使用代表性美术

**多轮循环：** 写入初始文件后，要求用户运行构建并报告观察到的情况。持续迭代，直到完整的游戏循环可演示。每轮：
1. 用户运行 → 报告错误或观察结果
2. 代理修复错误或调整系统
3. 重复直到完整的 [开始 → 挑战 → 解决] 循环可玩

**沉没成本检查点（计划时间线的第 3 天）：** 如果完整的游戏循环仍不可演示，停止并重新评估。要么范围过大，要么架构假设有误。明确指出阻塞因素，而非继续迭代。

一旦循环可演示，至少进行 1 次试玩会议。

**试玩技巧：** 如果你能找到任何未接触过该游戏的人来试玩 —— 朋友、家人或线上社区 —— 静静地观察他们，不做任何解释。不要引导他们。他们的困惑会揭示游戏本身未能传达的内容。这比自测提供的信号要好得多。

**没有外部测试人员？** 在团队内部轮换：开发者 A 构建了系统 X，那么开发者 A 对系统 Y 而言就是一个天真的测试者。即使两人团队也能有效轮换。独自一人？离开 2-3 天，然后以新玩家的身份试玩 —— 你不会有完美的第一印象信号，但会发现关键的阻塞问题。也可以尝试"静默走查"：一次性玩完你的切片而不停下来修复任何问题，并记录每一个你犹豫的时刻。

**想要更丰富的观察数据？** 要求测试者在游玩时**出声思考** —— 实时叙述他们在做什么以及为什么这么做。"我在试图弄清楚怎么攻击……我按了 E……没反应……是不是要点击？"这能在困惑发生的那一刻就暴露出来，而不是事后回顾。这对于新手引导和 UI 清晰度验证最为有效。对于手感测试，静默观察仍然更好；出声思考会轻微改变体验，但能产生远更细粒度的 UX 数据。

**异步远程选项：** 录制 Loom 或 OBS 会话 —— 把构建发给某人，请他们录制屏幕+音频，然后把视频发给你。你可以获得真实的第一印象数据，而无需同步安排时间。跨时区也适用。

**在 AI、NPC 或复杂系统行为完全实现之前进行测试？** 使用**绿野仙踪（Wizard of Oz）** 技术：一个人正常游玩，另一个人秘密实时控制 NPC 或系统行为。玩家认为这是自动的。这可以在实现完成之前验证 AI 或经济系统的*设计意图* —— 并准确揭示系统需要产生哪些行为才能让人感觉正确。对于垂直切片特别有用，当 AI 系统在范围内但尚未足够完善以进行无引导测试时。

## 阶段 5：试玩汇报

循环已可演示。在撰写报告之前，从实际游玩中收集结构化观察。**不要跳过这一步直接生成报告** —— 报告的质量取决于你在此捕获的观察。

请准确说出以下内容：
> "请从头到尾完整地玩一遍 [开始 → 挑战 → 解决] 循环，就像你是一个新玩家，对它的构建方式一无所知。不要跳过或使用开发者快捷方式。完成完整循环后回来 —— 或者当你遇到让你停下来的问题时回来。"

用户回来后，**逐个**提出以下问题：

1. **循环完成情况：**
   > "你是否在没有我引导或对构建有预先知识的情况下，独立完成了完整的 [开始 → 挑战 → 解决] 循环？"

2. **时间检查：**
   > "到达第一个有意义的操作 —— 你第一次感觉自己真正在玩游戏的那一刻 —— 花了多长时间？"

3. **核心愿景：**
   > "这款游戏应该让你感受到 [来自 game-concept.md 的核心愿景]。它做到了吗？请诚实回答 —— 不要'差不多'，而是具体说明你的感受和当时的情景。"

4. **障碍：**
   > "有什么让你停下来、困惑或脱离体验的地方？是否有不确定该做什么的时刻，或者有东西坏了？"

5. **管线检查：**
   > "作为开发者 —— 而非玩家 —— 你觉得以这种质量为完整游戏构建可行吗？你在构建时间长短上有哪些意外之处？"

6. **判定：**
   > "PROCEED、PIVOT 还是 KILL —— 以及具体原因。"

如果任何答案模糊，追问："你能说出具体发生时的那个时刻吗？" 精确的观察才能填充报告。模糊的观察只会产生无用的报告。

## 阶段 6：生成垂直切片报告

在整个构建过程中跟踪开发速度。记录：
- 第 1 天：构建了什么
- 第 2 天：构建了什么
- 依此类推

这将是你拥有的关于生产效率最真实的数据。不要跳过它。它会直接输入到冲刺规划中。

读取 `.claude/docs/templates/vertical-slice-report.md` 以获取报告结构。如果模板文件未找到，使用以下后备结构：
- `## 垂直切片报告 —— [游戏标题] —— [日期]`
- `### 执行摘要`（PROCEED / PIVOT / STOP 判定 + 2 句话理由）
- `### 核心循环验证`（测试了什么，哪些通过，哪些失败）
- `### 手感评估`（动画、控制、反馈 —— 主观记录）
- `### 技术发现`（性能、引擎问题、架构风险）
- `### 速度日志`（逐日实际进度 —— 不要跳过）
- `### 推荐后续步骤`

根据本次会话中的观察和构建内容填写每个部分。速度日志必须反映逐日实际进度，而非估算 —— 这是你拥有的关于生产效率最真实的数据。将所有占位文本替换为真实观察。

### 经验教训
- 哪些假设在实际上手接近生产质量的构建后被打破？
- 管线或架构方面有哪些意外发现？
- 如果再运行一次，我们会如何改变切片范围？

询问："我是否可以将此报告写入 `prototypes/[概念名称]-vertical-slice/REPORT.md`？"

如果同意，写入文件。然后更新 `prototypes/index.md`（如果不存在则创建）—— 在垂直切片表中追加一行：概念名称、日期、判定以及 REPORT.md 的链接。注明这是首次切片还是在 PIVOT 后重新运行。本报告中的速度日志是项目中一些最有价值的数据 —— 将其与冲刺估算进行交叉验证。

## 阶段 8：总结与后续步骤

输出总结：验证问题、速度数据和最终推荐。链接到 `prototypes/[概念名称]-vertical-slice/REPORT.md`。

**如果判定为 PROCEED：**
你的垂直切片已验证了完整的游戏循环。项目已准备好进入生产阶段。

推荐的后续步骤：
- `/create-epics layer:foundation` —— 规划基础层 epic
- `/create-epics layer:core` —— 规划核心层 epic
- `/create-stories [epic-slug]` —— 将每个 epic 分解为可实现的 story
- `/sprint-plan` —— 使用切片中的速度数据规划第一个冲刺
- `/gate-check pre-production` —— 正式将阶段推进到生产

**试玩备注：** `/gate-check` 会查找已记录的试玩证据。至少需要 1 次已记录的会议，并附带显示 PROCEED 的 REPORT.md 才能通过门控。更多会议提供更可靠的信号 —— 建议在让整个团队投入生产前进行 3 次以上，但这不是硬性门控。

**如果判定为 PIVOT：**

在返回到 GDD 修改之前，捕获可延续的备忘。逐个（纯文本，一次一个）询问以下两个问题：

1. "哪些系统或机制在这个质量水平上是有效的，应在修订版设计中保留？"
2. "具体哪些方面失败了 —— 核心循环、架构、管线还是乐趣？"

询问："我是否可以将此写入 `prototypes/[概念名称]-vertical-slice/PIVOT-NOTE.md`？"

如果同意，写入文件，内容包括：哪些有效、哪些失败、需要修改的具体系统或架构决策，以及下一次切片应如何不同地证明。当 PIVOT 后再次运行 `/vertical-slice` 时，检查 `prototypes/` 目录中是否有 `PIVOT-NOTE.md` —— 用它来构建新的验证问题并为范围决策提供信息。

- 使用 `/design-system [机制]` 修改受影响的 GDD
- 通过 `/architecture-decision` 处理架构问题
- 然后重新运行 `/vertical-slice` 以验证修改后的方向

**如果判定为 KILL：**

在放弃这个概念之前，确认判定是合理的：

- [ ] 即使是有经验的玩家，完整游戏循环也需要超过 5 分钟？
- [ ] 在任何试玩会议中未观察到情感高潮（喜悦、惊喜、满足）？
- [ ] 在 2 次以上切片尝试后，超过 50% 的测试者在同一点感到困惑或卡住？
- [ ] 架构问题需要重写超过 50% 的已构建内容？
- [ ] 这是对同一概念的第三次垂直切片尝试？

如果 2 项以上适用 → KILL 判定合理。如果 0-1 项适用 → 一次有针对性的 PIVOT 可能挽救这个概念。

**在 `prototypes/GRAVEYARD.md` 中记录此次终止**（如果不存在则创建）。询问："我是否可以将此追加到 `prototypes/GRAVEYARD.md`？" 如果同意，添加一条记录：

```
## [概念名称] 垂直切片 —— YYYY-MM-DD
- **终止原因：** [具体是什么阻止了玩家体验核心愿景]
- **在切片质量下有效的部分：** [经得起检验的系统或机制]
- **失败的部分：** [核心循环问题、架构决策或管线阻塞]
- **下次建议：** [下次尝试类似概念时应做的一项具体改变]
```

- 带着你的收获回到 `/brainstorm`
- 或者运行 `/prototype [新概念]` 先低成本测试新方向

### 重要约束

- 垂直切片代码**绝不能**重构到生产代码中 —— 它仅作为参考
- 生产代码**绝不能**从 `prototypes/` 导入
- 如果推荐结果为 PROCEED，生产实现将从头开始编写，仅将切片作为设计参考
- 允许削减范围；不允许削减质量 —— 低质量的切片证明不了什么
- 总工作量：1-3 周。如果更长，说明范围过大 —— 削减切片，而非质量
- 第 3 天沉没成本规则：如果到那时完整的游戏循环仍不可演示，停止并揭示阻塞因素
- **网络/多人游戏：** 本地垂直切片无法验证网络机制的手感。延迟从根本上改变了战斗、移动和预测的体验 —— 在本地 0ms 测试与 80ms 网络延迟下的感受完全不同。切片可以验证游戏循环是否有趣且完整；它无法验证网络机制在真实条件下的手感。网络手感需要真实对等节点或模拟延迟。
