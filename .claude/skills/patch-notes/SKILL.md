## Phase 1：解析参数

- `version`：要为其生成说明的发布版本（例如 `1.2.0`）
- `--style`：输出风格 — `brief`（要点列表）、`detailed`（附带上下文）、`full`（附带开发者评注）。默认值：`detailed`。

如果未提供版本号，请先询问用户再继续。

---

## Phase 2：收集变更数据

- 读取内部更新日志 `production/releases/[version]/changelog.md`（如果存在）
- 同时检查 `docs/CHANGELOG.md` 中相关的版本条目
- 作为后备方案，运行 `git log` 查看上一个发布标签到当前标签/HEAD 之间的提交
- 读取 `production/sprints/` 中的 sprint 回顾以获取上下文
- 读取 `design/balance/` 中的平衡性变更文档
- 如果可用，读取 QA 的 bug 修复记录

**如果没有可用更新日志数据**（既不存在 `production/releases/[version]/changelog.md`，也没有该版本的 `docs/CHANGELOG.md` 条目，且 git log 为空或不可用）：

> "未找到 [version] 的更新日志数据。请先运行 `/changelog [version]` 生成内部更新日志，然后重新运行 `/patch-notes [version]`。"

判决：**阻塞** — 在此停止，不生成补丁说明。

---

## Phase 2b：检测语气指南和模板

**语气指南检测** — 在起草补丁说明之前，检查是否有写作风格指导：

1. 检查 `.claude/docs/technical-preferences.md` 中是否有任何"tone"、"voice"或"style"字段或章节。
2. 检查 `docs/PATCH-NOTES-STYLE.md`（如果存在）。
3. 检查 `design/community/tone-guide.md`（如果存在）。
4. 如果任何来源包含语气/声音/风格说明，提取并应用到生成说明的语言和框架中。
5. 如果未找到任何语气指南，默认使用：玩家友好、非技术性语言；热情但不夸张；聚焦于玩家体验的内容，而非开发者修改的内容。

**模板检测** — 检查是否存在补丁说明模板：

1. 搜索 `docs/patch-notes-template.md` 和 `.claude/docs/templates/patch-notes-template.md`。
2. 如果在任一位置找到，读取它并用其作为 Phase 4 的输出结构，替代内置风格模板（Brief / Detailed / Full）。用分类后的数据填充模板的各章节。
3. 如果未找到，使用 Phase 4 中定义的内置风格模板。

---

## Phase 3：分类与翻译

将所有变更分类为面向玩家的类别：

- **新内容**：新功能、地图、角色、物品、模式
- **玩法变更**：平衡性调整、机制修改、进度系统变更
- **体验优化**：UI 改进、便利功能、无障碍
- **Bug 修复**：按系统分组（战斗、UI、网络等）
- **性能优化**：玩家可能注意到的优化改进
- **已知问题**：对未解决问题的透明说明

将开发者语言翻译为玩家语言：

- "重构了伤害计算管线" → "提升了命中检测精度"
- "修复了库存管理器中的空引用" → "修复了打开库存时的崩溃问题"
- "减少了战斗循环中的 GC 分配" → "提升了战斗性能"
- 移除不影响玩家的纯内部变更
- 保留平衡性调整的具体数值（伤害：50 → 45）

## Phase 4：生成补丁说明

### 简洁风格
```markdown
# 补丁 [版本号] — [标题]

**新增**
- [功能 1]
- [功能 2]

**变更**
- [平衡性/机制变更，附修改前 → 修改后数值]

**修复**
- [Bug 修复 1]
- [Bug 修复 2]

**已知问题**
- [问题 1]
```

### 详细风格
```markdown
# 补丁 [版本号] — [标题]
*[日期]*

## 亮点
[1-2 句描述最令人兴奋的变更]

## 新内容
### [功能名称]
[2-3 句描述该功能及玩家应期待的原因]

## 玩法变更
### 平衡性
| 变更项 | 修改前 | 修改后 | 原因 |
| ---- | ---- | ---- | ---- |
| [物品/能力] | [旧值] | [新值] | [简要说明] |

### 机制
- **[变更]**：[变更内容及原因说明]

## 体验优化
- [带有上下文的改进说明]

## Bug 修复
### 战斗
- 修复了 [玩家遇到的问题描述]

### UI
- 修复了 [描述]

### 网络
- 修复了 [描述]

## 性能优化
- [玩家将注意到的改进]

## 已知问题
- [问题及可用的临时解决方案]
```

### 完整风格
包含详细风格的所有内容，外加：
```markdown
## 开发者评注
### [主题]
> [开发者对重大变更的见解——为何做出此变更、考虑过哪些方案、团队学到了什么。以第一人称团队口吻撰写。]
```

---

## Phase 5：审查输出

检查生成的说明是否符合以下要求：

- 无内部行话（用玩家友好的语言替代技术术语）
- 无对内部系统、工单或 sprint 编号的引用
- 平衡性调整包含修改前/后的数值
- Bug 修复描述的是玩家体验，而非技术原因
- 语气符合游戏风格（根据游戏类型调整正式程度）

---

## Phase 6：保存补丁说明

向用户展示完整的补丁说明，同时提供：按类别分类的变更数量，以及任何被排除的内部变更（供审查）。

询问："我可以将这些补丁说明写入 `docs/patch-notes/[version].md` 吗？"

如果同意，将文件写入 `docs/patch-notes/[version].md`，必要时创建目录。同时将副本写入 `production/releases/[version]/patch-notes.md` 作为内部存档。

---

## Phase 7：后续步骤

判决：**完成** — 补丁说明已生成并保存。

- 在公开发布前，运行 `/release-checklist` 确认所有其他发布关卡已通过。
- 将补丁说明草稿分享给社区经理进行语气审查，然后再公开发布。
