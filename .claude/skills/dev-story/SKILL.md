# 开发故事 (Dev Story)

本技能桥接规划与代码。它完整读取故事文件，汇集程序员所需的所有上下文，路由到正确的专业代理，并驱动实现至完成——包括编写测试。

**每个故事的循环：**
```
/qa-plan sprint             ← 在冲刺开始前定义测试需求
/story-readiness [路径]     ← 在开始前验证
/dev-story [路径]           ← 实现故事（本技能）
/code-review [文件]         ← 审查代码
/story-done [路径]          ← 验证并关闭故事
```

**所有冲刺故事完成后：** 运行 `/team-qa sprint` 以执行完整的质量保证(QA)周期并在推进项目阶段前获得签收判定。

**输出：** 项目 `src/` 和 `tests/` 目录中的源代码 + 测试文件。

---

## 阶段 1：查找故事

**如果提供了路径**：直接读取该文件。

**如果没有参数**：检查 `production/session-state/active.md` 中的活动故事。如果找到，确认："继续 [故事标题] 的工作——是否正确？" 如果未找到，询问："我们要实现哪个故事？" 通过 Glob 匹配 `production/epics/**/*.md` 并列出状态为"就绪(Ready)"的故事。

---

## 阶段 2：加载完整上下文

**在加载任何上下文之前，验证必需文件存在。** 从故事的"实施治理架构决策记录(ADR Governing Implementation)"字段提取架构决策记录(ADR)路径，然后检查：

| 文件 | 路径 | 如果缺失 |
|------|------|------------|
| 技术需求(TR)注册表 | `docs/architecture/tr-registry.yaml` | **停止**——"在 `docs/architecture/tr-registry.yaml` 未找到技术需求(TR)注册表。运行 `/architecture-review` 以从你的游戏设计文档(GDD)和架构决策记录(ADR)初始化注册表。" |
| 治理架构决策记录(ADR) | 故事架构决策记录(ADR)字段中的路径 | **停止**——"未找到架构决策记录(ADR)文件 [路径]。运行 `/architecture-decision` 以创建它，或更正故事架构决策记录(ADR)字段中的文件名。" |
| 控制清单 | `docs/architecture/control-manifest.md` | **警告并继续**——"未找到控制清单——无法检查层级规则。运行 `/create-control-manifest`。" |

如果技术需求(TR)注册表或治理架构决策记录(ADR)缺失，将故事状态设置为**已阻塞(BLOCKED)** 在会话状态中，并且不生成任何程序员代理。

同时读取以下所有内容——这些是独立的读取操作。在所有上下文加载完毕之前不要开始实现：

### 故事文件
提取并保留：
- **故事标题、ID、层级、类型**（逻辑(Logic) / 集成(Integration) / 视觉/感觉(Visual/Feel) / 界面(UI) / 配置/数据(Config/Data)）
- **技术需求(TR)-ID**——游戏设计文档(GDD)需求标识符
- **治理架构决策记录(ADR)** 引用
- **嵌入在故事标题中的清单版本**
- **验收标准**——每个复选框项，逐字逐句
- **实现说明**——故事中的架构决策记录(ADR)指南章节
- **范围外(Out of Scope)** 边界
- **测试证据**——所需的测试文件路径
- **依赖关系**——在此故事之前必须完成的内容

### 技术需求(TR)注册表
读取 `docs/architecture/tr-registry.yaml`。查找故事的 TR-ID。
读取当前的 `requirement` 文本——这是游戏设计文档(GDD)当前需求的真实来源。不要依赖故事文件中的任何内联文本（可能已过时）。

### 治理架构决策记录(ADR)
读取 `docs/architecture/[adr-file].md`。提取：
- 完整的决策(Decision)章节
- 实现指南(Implementation Guidelines)章节（这是程序员遵循的内容）
- 引擎兼容性(Engine Compatibility)章节（训练截止日期后的应用程序接口、已知风险）
- 架构决策记录(ADR)依赖关系章节

### 控制清单
读取 `docs/architecture/control-manifest.md`。提取适用于故事层级的规则：
- 必需的模式
- 禁止的模式
- 性能护栏

检查：故事中嵌入的清单版本是否与当前清单标头日期匹配？
如果不同，在继续之前使用 `AskUserQuestion`：
- 提示："故事是根据清单 v[故事日期] 编写的。当前清单是 v[当前日期]。可能有新规则适用。你希望如何处理？"
- 选项：
  - `[A] 更新故事清单版本并按当前规则实现（推荐）`
  - `[B] 按旧规则实现——我接受不合规的风险`
  - `[C] 在此停止——我想先查看清单差异`

如果是 [A]：在生成程序员之前编辑故事文件的"清单版本(Manifest Version):"字段为当前清单日期。然后仔细阅读清单中的新规则。
如果是 [B]：编辑故事文件的"清单版本(Manifest Version):"字段为当前清单日期，并在故事标头中添加 `Manifest-Note: 已在 [日期] 使用旧清单规则进行——不合规风险已接受。` 行。无论如何阅读清单中的新规则。在阶段 6 摘要的"偏差(Deviations)"下记录此决策。`/story-done` 将清单备注(Manifest-Note)包含在其偏差章节中，而无需重新检查过时性。
如果是 [C]：停止。不生成任何代理。让用户审查并重新运行 `/dev-story`。

### 依赖验证

从故事文件中提取**依赖关系**列表后，验证每个：

1. 通过 Glob 匹配 `production/epics/**/*.md` 以查找每个依赖故事文件。
2. 读取其 `状态(Status):` 字段。
3. 如果任何依赖的状态不是"完成(Complete)"或"已完成(Done)"：
   - 使用 `AskUserQuestion`：
     - 提示："故事'[当前故事]'依赖于'[依赖标题]'，其当前状态是 [状态]，而非"完成"。你希望如何处理？"
     - 选项：
       - `[A] 无论如何继续——我接受依赖风险`
       - `[B] 停止——我先完成依赖`
       - `[C] 依赖已完成但状态未更新——将其标记为"完成"并继续`
   - 如果是 [B]：在会话状态中将故事状态设置为**已阻塞(BLOCKED)** 并停止。不生成任何程序员代理。
   - 如果是 [C]：在继续之前询问"我可以更新 [依赖路径] 的状态为"完成"吗？"
   - 如果是 [A]：在阶段 6 摘要的"偏差(Deviations)"下备注："以未完成的依赖实现：[依赖标题]——[状态]。"

如果找不到依赖文件：警告"未找到依赖故事：[路径]。请验证路径或创建故事文件。"

---

### 引擎参考
读取 `.claude/docs/technical-preferences.md`：
- `引擎(Engine):` 值——确定使用哪些程序员代理
- 命名约定（类名、文件名、信号/事件名）
- 性能预算（帧预算、内存上限）
- 禁止的模式

### 将故事标记为进行中

在生成任何代理之前静默更新两项内容：

1. **`production/sprint-status.yaml`**（如果存在）：查找匹配此故事文件路径的条目，并将 `status` 设置为 `in_progress`。更新顶层的 `updated` 字段为今天的日期。如果文件不存在，静默跳过。

2. **故事文件本身**：编辑故事标头中的"上次更新(Last Updated):"字段为今天的日期（格式：`YYYY-MM-DD`）。如果故事标头中不存在该字段，在"状态(Status):"行之后添加它。这使得冲刺状态能够检测此故事的过时情况。

## 阶段 3：路由到正确的程序员

根据故事的**层级**、**类型**和**系统名称**，确定要通过任务(Task)生成哪个专家。

**配置/数据(Config/Data)故事——完全跳过代理生成：**
如果故事类型是"配置/数据(Config/Data)"，则不需要程序员代理或引擎专家。直接跳转到阶段 4（配置/数据说明）。实现是数据文件编辑——无需路由表评估，无需引擎专家。

### 主要代理路由表

| 故事上下文 | 主要代理 |
|---|---|
| 基础层——任何类型 | `engine-programmer` |
| 任何层——类型：界面(UI) | `ui-programmer` |
| 任何层——类型：视觉/感觉(Visual/Feel) | `gameplay-programmer`（实现） |
| 核心或功能——游戏玩法机制 | `gameplay-programmer` |
| 核心或功能——人工智能行为、寻路 | `ai-programmer` |
| 核心或功能——网络、复制 | `network-programmer` |
| 配置/数据——无需代码 | 无需代理（参见阶段 4 配置说明） |

### 引擎专家——始终作为代码故事的次要代理生成

读取 `.claude/docs/technical-preferences.md` 的"引擎专家(Engine Specialists)"章节以获取配置的主要专家。当故事涉及引擎特定的应用编程接口、模式或架构决策记录(ADR)具有高风险时，与主要代理一起生成引擎专家。

| 引擎 | 可用的专家代理 |
|--------|----------------------------|
| Godot 4 | `godot-specialist`、`godot-gdscript-specialist`、`godot-shader-specialist` |
| Unity | `unity-specialist`、`unity-ui-specialist`、`unity-shader-specialist` |
| Unreal Engine | `unreal-specialist`、`ue-gas-specialist`、`ue-blueprint-specialist`、`ue-umg-specialist`、`ue-replication-specialist` |

**当引擎风险为高(HIGH)时**（来自架构决策记录(ADR)或 VERSION.md）：即使对于非面向引擎的故事，也始终生成引擎专家。高风险意味着架构决策记录(ADR)记录了关于训练截止日期后引擎应用编程接口的假设，需要专家验证。

## 阶段 4：实现

通过任务(Task)生成选定的程序员代理，附带完整的上下文包：

使用文件路径和定向读取说明向代理做简报——不要将文档内容序列化到任务提示中。代理直接读取所需内容：

1. **故事文件**：`[故事路径]`——完整读取
2. **游戏设计文档(GDD)需求**：在 `docs/architecture/tr-registry.yaml` 中查找技术需求(TR)-ID `[TR-XXX-NNN]`——使用 `requirement` 字段作为真实来源
3. **架构决策记录(ADR)**：`docs/architecture/[adr-file].md`——仅读取**决策(Decision)**和**实现指南(Implementation Guidelines)**章节
4. **控制清单**：`docs/architecture/control-manifest.md`——仅读取**[层级]**层级的规则
5. **引擎偏好**：`.claude/docs/technical-preferences.md`——读取命名约定和性能预算
6. **测试文件路径**：`[故事测试证据章节中的路径]`——此文件必须作为实现的一部分创建
7. **测试要求**（仅逻辑(Logic)和集成(Integration)故事）：测试文件**必须**在[故事测试证据章节中的路径]处创建。将测试与实现一起编写——不要推迟。没有此文件，无法通过 `/story-done` 关闭故事。每个验收标准必须至少有一个测试函数覆盖它。测试文件命名：`[系统]_[功能]_test.[扩展名]`。函数命名：`test_[场景]_[预期结果]`。无随机种子、无时间相关断言、无外部输入输出。
8. **明确指令**：按照架构决策记录(ADR)指南实现此故事，遵守清单规则，保持在故事的范围外(Out of Scope)边界内。编写带有文档注释的干净公共应用编程接口。

代理应：
- 按照架构决策记录(ADR)指南在 `src/` 中创建或修改文件
- 遵守控制清单中的所有必需和禁止模式
- 保持在故事的范围外边界内（不触碰无关文件）
- 编写带有文档注释的干净公共应用编程接口

### 配置/数据故事（无需代理）

对于类型为"配置/数据(Config/Data)"的故事，不需要程序员代理。实现是编辑数据文件。直接读取故事的验收标准并对数据文件进行指定的更改。记录哪些值发生了更改以及更改前后的值。

### 视觉/感觉故事

生成 `gameplay-programmer` 来实现代码/动画调用。请注意，视觉/感觉验收标准无法自动验证——"感觉对吗？"检查在 `/story-done` 中通过手动确认完成。

## 阶段 5：测试证据要求

测试要求已包含在阶段 4 的程序员代理简报中（第 7 项）。本阶段总结每个故事类型所需的证据——用于收集阶段 6 摘要。

| 故事类型 | 所需证据 | 备注 |
|---|---|---|
| **逻辑(Logic)** | 位于故事测试证据章节路径的自动化单元测试 | 阻塞性——已包含在阶段 4 代理简报中 |
| **集成(Integration)** | 集成测试或记录的试玩记录 | 阻塞性——已包含在阶段 4 代理简报中 |
| **视觉/感觉(Visual/Feel)** | 位于 `production/qa/evidence/[slug]-evidence.md` 的证据文档 | 建议性——在阶段 6 摘要中备注 |
| **界面(UI)** | 手动演练文档或交互测试 | 建议性——在阶段 6 摘要中备注 |
| **配置/数据(Config/Data)** | 无——烟雾测试作为证据 | 不适用 |

对于视觉/感觉和界面(UI)故事，在阶段 6 摘要中包含："在 `production/qa/evidence/[slug]-evidence.md` 处需要手动证据，然后才能完全关闭此故事。"

## 阶段 6：收集与总结

在程序员代理完成后，收集：

- 已创建或修改的文件（带路径）
- 已创建的测试文件（路径和编写的测试函数数量）
- 任何偏离故事范围外边界的偏差（标记这些）
- 代理提出的任何问题或阻塞项
- 专家标记的任何引擎特定风险

展示简洁的实现摘要：

```
## 实现完成：[故事标题]

**更改的文件**：
- `src/[路径]`——已创建 / 已修改（[简要描述]）
- `tests/[路径]`——测试文件（[N] 个测试函数）

**覆盖的验收标准**：
- [x] [标准]——已在 [文件:函数] 中实现
- [x] [标准]——已由测试 [测试名称] 覆盖
- [ ] [标准]——已推迟：需要试玩（视觉/感觉）

**范围偏差**：[无] 或 [列出在故事边界外触碰的文件]
**标记的引擎风险**：[无] 或 [专家发现]
**阻塞项**：[无] 或 [描述]

**在运行 `/story-done` 之前：** 在本地运行测试套件并确认你编写的测试通过。`/story-done` 将自动重新运行它们，但如果在那时发现失败的测试，意味着需要回到实现上下文。

准备就绪：`/code-review [文件1] [文件2]` 然后 `/story-done [故事路径]`
```

## 错误恢复协议

如果任何生成的代理（通过任务(Task)）返回已阻塞、错误或无法完成：

1. **立即展示**：在继续到依赖阶段之前向用户报告"[代理名称]：已阻塞——[原因]"
2. **评估依赖**：检查阻塞代理的输出是否为后续阶段所需。如果是，在没有用户输入之前不要越过该依赖点。
3. **通过 AskUserQuestion 提供选项**：
   - 跳过此代理并在最终报告中备注缺口
   - 以更窄的范围重试
   - 在此停止并先解决阻塞项
4. **始终生成部分报告**——输出任何已完成的内容。永远不要因为一个代理阻塞而丢弃工作。

常见阻塞项：
- 输入文件缺失（未找到故事、游戏设计文档(GDD)缺失）→ 重定向到创建它的技能
- 架构决策记录(ADR)状态为"已提议(Proposed)"→ 不实现；先运行 `/architecture-decision`
- 范围过大 → 通过 `/create-stories` 拆分为两个故事
- 架构决策记录(ADR)和故事之间的指令冲突 → 展示冲突，不要猜测
- 清单版本不匹配 → 向用户显示差异，询问是按旧规则进行还是先更新故事

## 协作协议

- **文件写入是委派的**——所有源代码、测试文件和证据文档都由通过任务(Task)生成的子代理写入。每个子代理单独执行"我可以写入 [路径] 吗？"协议。此编排器不直接写入文件。
- **先加载再实现**——在所有上下文加载完毕之前不要开始编码
  （故事、技术需求(TR)-ID、架构决策记录(ADR)、清单、引擎偏好）。不完整的上下文会产生偏离设计的代码。
- **架构决策记录(ADR)是准则**——实现必须遵循架构决策记录(ADR)的实现指南。如果指南与"更好的做法"冲突，在摘要中标记而不是静默偏离。
- **保持在范围内**——范围外章节是一个契约。如果实现故事需要触碰范围外的文件，停止并展示：
  "实现 [标准] 需要修改 [文件]，该文件在范围外。我应该继续还是创建单独的故事？"
- **对于逻辑/集成故事，测试不是可选项**——没有测试文件存在之前，不要标记实现完成
- **视觉/感觉标准是推迟而非跳过**——在摘要中将其标记为已推迟；它们将在 `/story-done` 中手动验证
- **在重大的结构决策前询问**——如果故事需要架构决策记录(ADR)未涵盖的架构模式，在实现前展示：
  "架构决策记录(ADR)未指定如何处理 [情况]。我的计划是 [X]。继续吗？"

## 建议的后续步骤
