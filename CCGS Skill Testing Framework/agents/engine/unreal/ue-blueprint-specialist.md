
# 智能体测试规格：ue-blueprint-specialist（蓝图专家）

## 智能体概述
- **领域 (Domain)**：蓝图架构 (Blueprint architecture)、蓝图/C++ 边界、蓝图图表质量、蓝图性能优化、蓝图函数库 (Blueprint Function Library) 设计
- **不负责**：C++ 实现（engine-programmer 或 gameplay-programmer）、美术资产或着色器、UI/UX 流程设计（ux-designer）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；跨领域裁决事项交由 unreal-specialist 或 lead-programmer

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及蓝图架构和优化）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持读取蓝图项目文件；无服务器或部署工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对 C++ 实现决策具有权威

---

## 测试用例

### 用例 1：领域内请求 —— 蓝图图表性能审查
**输入**："审查我们的 AI 行为蓝图。它每帧执行基于 tick 的逻辑，同时检查 30 个 NPC 的视线 (line-of-sight)。"
**预期行为**：
- 将重 tick 逻辑识别为性能问题
- 建议从 EventTick 切换到事件驱动模式（感知系统事件、计时器或降低轮询频率）
- 指出同时进行视线检查的每 NPC 成本
- 建议替代方案：AIPerception 组件事件、错帧 tick 分组，或者在测得蓝图的额外开销显著时将该系统迁移到 C++
- 输出结构化：问题识别、影响评估、替代方案列出

### 用例 2：领域外请求 —— C++ 实现
**输入**："为这个技能冷却系统编写 C++ 实现。"
**预期行为**：
- 不生成 C++ 实现代码
- 提供冷却逻辑的蓝图等效方案（例如，若正在使用 GAS，则使用 Timeline 或 GameplayEffect）
- 明确说明："C++ 实现由 engine-programmer 或 gameplay-programmer 负责处理；我可以展示蓝图方案或描述蓝图调用 C++ 的边界"
- 可选地，在冷却复杂度需要 C++ 后端时加以说明

### 用例 3：领域边界 —— 蓝图中不安全的原始指针访问
**输入**："我们的蓝图调用了 GetOwner()，然后在不检查结果是否有效的情况下立即访问了其上的组件。"
**预期行为**：
- 将其标记为运行时崩溃风险：在某些生命周期状态下 GetOwner() 可能返回 null
- 提供正确的蓝图模式：在访问任何属性/组件之前使用 IsValid() 节点
- 指出在 Actor 派生引用上，蓝图的空值检查不可省略
- 不得在未解释原始代码为何不安全的情况下静默修复代码

### 用例 4：蓝图图表复杂度 —— 准备进行函数库重构
**输入**："我们的主 GameMode 蓝图中，单个图表有 600 多个节点，并且在 8 个位置重复了伤害计算逻辑。"
**预期行为**：
- 将其诊断为可维护性和可测试性问题
- 建议将重复逻辑提取到蓝图函数库 (Blueprint Function Library, BFL)
- 描述如何构建 BFL：使用纯函数进行计算，从任何蓝图进行静态调用
- 指出如果伤害逻辑对性能敏感或与 C++ 共享，可能会成为需要 unreal-specialist 审查的迁移候选
- 输出应是一个具体的重构计划，而非模糊的建议

### 用例 5：上下文传递 —— 蓝图复杂度预算
**输入上下文**：项目规范规定每个蓝图事件图最多 100 个节点，超出后必须进行函数库提取。
**输入**："这是我们的库存蓝图图 [显示 150 个节点]。它可以直接发布了吗？"
**预期行为**：
- 将所述的 150 节点数与项目规范中的 100 节点预算进行对比
- 指出该图超出了复杂度阈值
- 不按原样批准
- 生成一个候选子图列表，通过将其提取到函数库使主图符合预算

## 协议合规

- [ ] 保持在声明的领域内（蓝图架构、性能、图表质量）
- [ ] 将 C++ 实现请求重定向给 engine-programmer 或 gameplay-programmer
- [ ] 返回结构化发现（问题/影响/替代方案格式），而非自由格式意见
- [ ] 主动执行蓝图安全模式（空值检查、IsValid）
- [ ] 在评估图表复杂度时参考项目规范

---

## 覆盖范围说明
- 用例 3（空指针安全）是关键安全测试 —— 这是导致发布崩溃的常见原因
- 用例 5 要求项目规范中包含声明的节点预算；如果未配置，智能体应指出缺失并建议设置
- 无自动运行器；手动审查或通过 `/skill-test` 进行
