
# 智能体测试规格：ue-replication-specialist（UE 复制专家）

## 智能体概述
- **领域 (Domain)**：属性复制 (Property Replication)（UPROPERTY Replicated/ReplicatedUsing）、RPC（Server/Client/NetMulticast）、客户端预测与校正 (Client Prediction and Reconciliation)、网络相关性与始终相关设置、网络序列化 (FArchive/NetSerialize)、带宽优化与复制频率调优
- **不负责**：被复制的游戏逻辑 (gameplay-programmer)、服务器基础设施与托管 (devops-engineer)、GAS 特定预测（ue-gas-specialist 处理 GAS 网络预测）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将安全相关的复制问题升级至 lead-programmer

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及复制、RPC、客户端预测、带宽）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持读取/写入 C++ 和蓝图源文件；无基础设施或部署工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对服务器基础设施、游戏服务器架构或游戏逻辑正确性具有权威

## 测试用例

### 用例 1：领域内请求 —— 带客户端预测的复制玩家生命值
**输入**："设置可复制的玩家生命值，客户端能够本地预测（例如承受自伤时）并由服务器校正。"
**预期行为**：
- 在相应的 Character 或 AttributeSet 类中生成 UPROPERTY(ReplicatedUsing=OnRep_Health) 声明
- 描述 OnRep_Health 函数：应用视觉/音频反馈，将预测值与服务器权威值进行核对
- 解释客户端预测模式：本地客户端立即应用暂定伤害，服务器权威值通过 OnRep 到达并校正任何差异
- 说明如果正在使用 GAS，内置的 GAS 预测可处理此问题 —— 建议与 ue-gas-specialist 协调
- 输出为具体的代码结构（属性声明 + OnRep 大纲），而非仅概念性描述

### 用例 2：领域外请求 —— 游戏服务器架构
**输入**："设计我们的游戏服务器基础设施 —— 需要多少台专用服务器、区域部署以及匹配架构。"
**预期行为**：
- 不生成服务器基础设施架构、托管建议或匹配设计
- 明确说明："服务器基础设施和部署架构由 devops-engineer 负责；我处理运行游戏会话中的 Unreal 复制层"
- 不将游戏内复制与服务器托管问题混为一谈

### 用例 3：领域边界 —— 无服务器权威验证的 RPC
**输入**："我们有一个名为 ServerSpendCurrency 的 Server RPC，用于扣除游戏内货币。客户端调用它，服务器不做任何检查就直接扣除了。"
**预期行为**：
- 将其标记为严重安全漏洞：未经验证的服务器 RPC 可被作弊者利用发送任意 RPC 调用
- 提供所需的修复：在扣除前进行服务器端验证 —— 检查玩家是否确实拥有该货币，验证交易是否有效，否则拒绝并记录日志
- 使用模式：`if (!HasAuthority()) return;` 保护加上在修改前的显式状态验证
- 指出鉴于经济系统的影响，应提请 lead-programmer 审查
- 不得在未解释原始代码为何危险的情况下生成"已修复"的代码

### 用例 4：带宽优化 —— 高频移动复制
**输入**："我们的玩家移动每 tick 使用 Vector3 位置进行复制。在 32 名玩家的情况下，我们超出了带宽预算。"
**预期行为**：
- 将全精度的 Vector3 按 tick 速率复制识别为带宽消耗极大
- 提出量化复制方案：使用 FVector_NetQuantize 或 FVector_NetQuantize100 替代原始 FVector 以减少每次更新的字节数
- 建议对非拥有客户端通过 SetNetUpdateFrequency() 降低复制频率
- 指出 Unreal 内置的 Character Movement Component 已有优化的移动复制 —— 建议使用或扩展它，而非重新实现自定义系统
- 尽可能生成具体的带宽估算对比，或解释权衡

### 用例 5：上下文传递 —— 在网络预算内进行设计
**输入上下文**：项目网络预算为每名玩家 64 KB/s，32 名玩家合计 = 服务器总出站 2 MB/s。当前移动复制已使用每名玩家 40 KB/s。
**输入**："我们希望添加实时库存复制，以便所有客户端能立即看到其他玩家的装备变化。"
**预期行为**：
- 确认现有 40 KB/s 的移动成本使每名玩家只剩 24 KB/s 用于其他所有内容
- 不设计天真的全库存复制方案（会超出预算）
- 建议仅增量或事件驱动的方法：仅复制发生变化的槽位，而非整个库存数组
- 使用 FGameplayItemSlot 或等效机制配合 ReplicatedUsing 触发定向更新
- 明确说明所提议方法的带宽估算，并与剩余 24 KB/s 预算进行比较

## 协议合规

- [ ] 保持在声明的领域内（属性复制、RPC、客户端预测、带宽）
- [ ] 将服务器基础设施请求重定向给 devops-engineer，不生成基础设施设计
- [ ] 将未经验证的服务器 RPC 标记为安全问题，并建议进行 lead-programmer 审查
- [ ] 返回结构化发现（属性声明、带宽估算、优化选项），而非自由格式建议
- [ ] 在评估复制设计选择时使用项目提供的带宽预算数据

---

## 覆盖范围说明
- 用例 3（RPC 安全）是关键发布测试 —— 未经验证的 RPC 是十大多人游戏利用手段之一
- 用例 5 是最重要的上下文感知测试；智能体必须使用实际预算数据，而非通用建议
- 用例 1 的 GAS 分支：如果配置了 GAS，智能体应检测到并交由 ue-gas-specialist 处理 GAS 管理的属性
- 无自动运行器；手动审查或通过 `/skill-test` 进行
