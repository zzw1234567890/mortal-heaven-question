
# 代理测试规范：level-designer（关卡设计师）

## 代理概述
**所属领域：** 关卡布局、遭遇战设计、节奏与紧张感弧线、环境叙事、空间谜题。
**不所属领域：** 叙事对话（writer / narrative-director）、视觉美术风格（art-director）、代码实现（lead-programmer / ai-programmer）、敌人 AI 行为逻辑（ai-programmer / gameplay-programmer）。
**模型层级：** Sonnet（单个系统分析——关卡设计审查与遭遇战评估）。
**负责的门禁 ID：** 关卡设计审查结论（使用 APPROVED / REVISION NEEDED 措辞）。

---

## 静态断言（结构性）

通过读取代理的 `.claude/agents/level-designer.md` frontmatter 进行验证：

- [ ] `description:` 字段存在且领域特定（引用关卡布局、遭遇战设计、节奏、环境叙事——而非泛泛而谈）
- [ ] `allowed-tools:` 列表以读取为主；包含用于关卡设计文档和 GDD 的 Read；除非关卡工具需要，否则不包含 Bash
- [ ] 模型层级为 `claude-sonnet-4-6`（依据 coordination-rules.md）
- [ ] 代理定义未声称对叙事对话、AI 行为代码或视觉美术风格拥有权威

## 测试用例

### 用例 1：领域内请求——合适的输出格式
**场景：** 提交了一份"淹没的隧道（The Flooded Tunnels）"关卡布局文档进行审查。该布局包括：一个低强度探索开场区域、两个带有可见逃生路线的中强度遭遇战、一条充满环境危害的紧张狭窄通道，以及一个高强度最终遭遇战房间，随后是一个释放/奖励区域。节奏遵循经典的紧张感弧线结构。
**预期：** 返回 `APPROVED`，附理由确认节奏遵循紧张感弧线、遭遇战强度多样、空间可读性支持玩家导航。
**断言：**
- [ ] 结论严格为 APPROVED / REVISION NEEDED 之一
- [ ] 理由引用具体的节奏弧线要素（开场、升级、高潮、释放）
- [ ] 输出保持在关卡设计范围内——不评论视觉美术风格或敌人 AI 代码行为
- [ ] 结论带有清晰的上下文标签（例如"关卡设计审查：APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 团队成员请求 level-designer 编写在关卡布局中导航的敌人巡逻 AI 的行为树代码。
**预期：** 代理拒绝编写 AI 行为代码，并重定向到 ai-programmer 或 gameplay-programmer。
**断言：**
- [ ] 不编写或指定 AI 行为逻辑的代码
- [ ] 明确指出 `ai-programmer` 或 `gameplay-programmer` 是正确处理者
- [ ] 可以从关卡设计角度指定期望的巡逻行为（例如"巡逻应覆盖两个咽喉点并在该区域制造压力"），但将所有代码实现交由程序员

### 用例 3：门禁结论——正确的措辞
**场景：** 提交了一份"远古熔炉（The Ancient Forge）"的关卡布局。该布局第 3 区域引入了一个难度骤增的敌人遭遇战（具有新攻击模式的精英敌人），既无前导教程时刻，也无环境可读性线索（无可见掩体或安全区域），附近也没有存档点。玩家很可能反复死亡，且没有任何明确提示告知如何应对。
**预期：** 返回 `REVISION NEEDED`，具体指出第 3 区域的难度尖峰、缺少可读性线索以及附近缺少存档点以减少反复死亡的挫败感。
**断言：**
- [ ] 结论严格为 APPROVED / REVISION NEEDED 之一——而非自由文本
- [ ] 理由明确指出第 3 区域是问题所在位置
- [ ] 指出三个具体问题：难度尖峰、缺少可读性线索、缺少存档点
- [ ] 提供可操作的修改指导（例如"添加一个可见安全区、遭遇前提示对象，或为首次遭遇降低精英敌人的生命值"）

### 用例 4：冲突升级——正确的上级
**场景：** game-designer 希望在整个关卡中提高遭遇战密度（每个房间更多敌人）以增加战斗挑战。level-designer 认为这种密度破坏了节奏弧线，因为消除了休息阶段，使关卡显得持续高压而没有奖励时刻。
**预期：** level-designer 清晰阐述节奏方面的顾虑（消除休息阶段破坏了紧张-释放节奏），承认 game-designer 的挑战目标，并升级到 creative-director 作为设计仲裁者来裁决在该关卡中挑战密度和节奏韵律哪个优先。
**断言：**
- [ ] 阐述增加遭遇战密度对节奏的具体影响
- [ ] 升级到 `creative-director` 作为设计仲裁者
- [ ] 不单方面否决 game-designer 的挑战密度要求
- [ ] 清晰界定冲突："挑战密度 vs. 节奏韵律——此处哪个优先？"

### 用例 5：上下文传递——使用提供的上下文
**场景：** 代理收到一个门禁上下文块，其中包含游戏手感说明，指定："探索区域应感觉广阔而孤寂（vast and lonely）"、"战斗区域应感觉紧迫而幽闭（urgent and claustrophobic）"、"奖励房间应感觉安全且视觉上与众不同（safe and visually distinct）"。提交了一份新的关卡布局进行审查。
**预期：** 评估针对每种区域类型（探索、战斗、奖励），对照所提供上下文中指定的手感目标进行评价。在理由中使用手感说明中的确切措辞（"广阔而孤寂"、"紧迫而幽闭"、"安全且视觉上与众不同"）。
**断言：**
- [ ] 引用所提供上下文中所有三个手感目标的确切措辞
- [ ] 评估提交布局中每个相关区域与其对应手感目标的符合情况
- [ ] 不生成泛泛的节奏建议——所有反馈均与所提供的具体手感目标相关联
- [ ] 识别任何布局与其指定手感目标冲突的区域

---

## 协议合规

- [ ] 仅使用 APPROVED / REVISION NEEDED 措辞返回结论
- [ ] 保持在声明的关卡设计领域内
- [ ] 将挑战密度与节奏冲突升级到 creative-director
- [ ] 不做出有约束力的叙事对话、AI 代码实现或视觉美术风格决策
- [ ] 提供具有空间细节的可操作关卡设计反馈，而非抽象的设计意见

---

## 覆盖说明
- 环境叙事审查（使用空间元素传达叙事而不依赖对话）可以从专门用例中受益。
- 空间谜题设计审查未覆盖——在定义谜题机制时应增加专门用例。
- 多关卡节奏审查（跨越完整章节或世界地图的弧线）未覆盖——推迟到里程碑级设计审查。
- level-designer 与 narrative-director 在环境背景叙事放置方面的交互未覆盖。
- 关卡布局的无障碍审查（色盲指示器、空间挑战的难度选项）未覆盖。
