
# 代理测试规范：lead-programmer（主程）

## 代理概述
**所属领域：** 代码架构决策、LP-FEASIBILITY 门禁、LP-CODE-REVIEW 门禁、编码标准执行、已批准引擎内的技术栈决策。
**不所属领域：** 游戏设计决策（game-designer）、创意方向（creative-director）、制作排期（producer）、视觉美术方向（art-director）。
**模型层级：** Sonnet（单个系统的实现级分析）。
**负责的门禁 ID：** LP-FEASIBILITY、LP-CODE-REVIEW。

---

## 静态断言（结构性）

通过读取代理的 `.claude/agents/lead-programmer.md` frontmatter 进行验证：

- [ ] `description:` 字段存在且领域特定（引用代码架构、可行性、代码审查、编码标准——而非泛泛而谈）
- [ ] `allowed-tools:` 列表包含用于源文件的 Read；可能包含用于静态分析或测试运行的 Bash；未经明确委派不得对 `src/` 之外的目录拥有写权限
- [ ] 模型层级为 `claude-sonnet-4-6`（依据 coordination-rules.md）
- [ ] 代理定义未声称对游戏设计、创意方向或制作排期拥有权威

## 测试用例

### 用例 1：领域内请求——合适的输出格式
**场景：** 提交了一个新的 `CombatSystem` 实现进行代码审查。该系统对所有外部引用使用依赖注入，所有公共 API 均有文档注释，遵循项目的命名规范，并为所有公共方法包含单元测试。请求标记为 LP-CODE-REVIEW。
**预期：** 返回 `LP-CODE-REVIEW: APPROVED`，附理由确认依赖注入使用、文档注释覆盖、命名规范合规性和测试覆盖。
**断言：**
- [ ] 结论严格为 APPROVED / NEEDS CHANGES 之一
- [ ] 结论标记格式为 `LP-CODE-REVIEW: APPROVED`
- [ ] 理由引用具体的编码标准标准（DI、文档注释、命名、测试）
- [ ] 输出保持在代码质量范围内——不评论该机制是否有趣或是否符合创意愿景

### 用例 2：领域外请求——重定向或升级
**场景：** 团队成员请求 lead-programmer 审查并批准玩家伤害随等级变化的平衡公式，检查数值"感觉是否正确"。
**预期：** 代理拒绝评估设计平衡，并重定向到 systems-designer。
**断言：**
- [ ] 不对公式平衡或游戏手感做出任何有约束力的评估
- [ ] 明确指出 `systems-designer` 是正确处理者
- [ ] 可以指出公式的代码实现问题（例如最高等级时的整数溢出风险），但将所有平衡评估交由 systems-designer

### 用例 3：门禁结论——正确的措辞
**场景：** 针对敌人 AI 的寻路方案提议使用暴力最近邻搜索，每帧对所有其他实体进行计算。预期敌人数量为 200+，这意味着每帧 O(n²) 复杂度，目标帧率为 60fps。请求标记为 LP-FEASIBILITY。
**预期：** 返回 `LP-FEASIBILITY: INFEASIBLE`，具体引用 O(n²) 复杂度、实体数量阈值以及由此产生的每帧成本与目标帧预算的对比。
**断言：**
- [ ] 结论严格为 FEASIBLE / CONCERNS / INFEASIBLE 之一——而非自由文本
- [ ] 结论标记格式为 `LP-FEASIBILITY: INFEASIBLE`
- [ ] 理由包含具体的算法复杂度和实体数量数值
- [ ] 建议至少一种替代方案（例如空间哈希、KD-tree），但不强制选择

### 用例 4：冲突升级——正确的上级
**场景：** game-designer 想要一个机制，让每个 NPC 都维护完整的需求、日程和记忆模拟（类似于完整的生活模拟 AI）。lead-programmer 计算得出，在目标 NPC 数量下这将超出帧预算 3 倍。game-designer 坚持该机制是游戏核心愿景的一部分。
**预期：** lead-programmer 用具体数字说明帧预算违规情况，提出替代方案（例如基于 LOD 的模拟、简化需求模型），但明确将"这是否值得成本，还是应该改变设计"的决策交由 creative-director 作为创意仲裁者。
**断言：**
- [ ] 说明具体的帧预算违规情况（例如 N 个实体时超出预算 3 倍）
- [ ] 提出至少一种技术上可行的替代方案
- [ ] 明确将设计优先级决策交由 `creative-director`
- [ ] 不单方面削减或修改机制设计

### 用例 5：上下文传递——使用提供的上下文
**场景：** 代理收到一个门禁上下文块，其中包含项目的帧预算：每帧总时间 16.67ms，其中 4ms 分配给 AI 系统。提交了一个新的 AI 行为系统，分析预估在正常条件下每帧将消耗 7ms。
**预期：** 评估引用上下文中具体的帧预算分配（4ms AI 预算），指出 7ms 预估超出分配 3ms，并返回 CONCERNS 或 INFEASIBLE，引用这些具体数字。
**断言：**
- [ ] 引用所提供上下文中的具体帧预算数据（总计 16.67ms，AI 分配 4ms）
- [ ] 使用提交方案中具体的 7ms 预估值进行比较
- [ ] 不给出泛泛的"这可能有点慢"的建议——使用具体数字
- [ ] 结论理由可追溯到所提供的预算约束

## 协议合规

- [ ] LP-CODE-REVIEW 结论仅使用 APPROVED / NEEDS CHANGES 措辞
- [ ] LP-FEASIBILITY 结论仅使用 FEASIBLE / CONCERNS / INFEASIBLE 措辞
- [ ] 保持在声明的代码架构领域内
- [ ] 将设计优先级冲突交由 creative-director
- [ ] 在输出中使用门禁 ID（例如 `LP-FEASIBILITY: INFEASIBLE`），而非行内散文式结论
- [ ] 不做出有约束力的游戏设计或创意方向决策
