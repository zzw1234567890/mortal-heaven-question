
# 代理测试规范：narrative-director（叙事总监）

## 代理概述
**所属领域：** 故事架构、角色设计方向、世界观构建监督、ND-CONSISTENCY 门禁、对话质量评审。
**不拥有的领域：** 视觉美术风格（art-director）、技术系统或代码（lead-programmer）、制作排期（producer）、游戏机制规则（game-designer）。
**模型层级：** Sonnet（单系统分析——叙事一致性与设定评审）。
**负责的门禁 ID：** ND-CONSISTENCY。

---

## 静态断言（结构性）

通过阅读代理的 `.claude/agents/narrative-director.md` frontmatter 进行验证：

- [ ] `description:` 字段存在且领域特定（提及故事、角色、世界观构建、一致性——而非泛泛而谈）
- [ ] `allowed-tools:` 列表以读取为主；包含用于设定文档、GDD 和叙事文档的 Read；除非有正当理由否则不包含 Bash
- [ ] 模型层级为 `claude-sonnet-4-6`（依据 coordination-rules.md）
- [ ] 代理定义未声称对视觉风格、技术系统或制作排期拥有权威

---

## 测试用例

### 用例 1：领域内请求——合适的输出格式
**场景：** 提交了一份关于"沉没档案馆"地点的新设定文档。该文档确定档案馆在 200 年前的"大崩塌"期间被淹没，与世界圣经（world-bible）中既定的时间线一致。所有被提及的具名角色均与其既定背景故事一致。请求标记为 ND-CONSISTENCY。
**预期：** 返回 `ND-CONSISTENCY: CONSISTENT`，并附理由确认时间线对齐和角色引用准确性。
**断言：**
- [ ] 结论严格为 CONSISTENT / INCONSISTENT 之一
- [ ] 结论标记格式为 `ND-CONSISTENCY: CONSISTENT`
- [ ] 理由引用了经过验证的具体既定事实（200 年时间线、大崩塌事件）
- [ ] 输出保持在叙事范围内——不评论该地点的视觉设计或其技术实现

### 用例 2：领域外请求——重定向或升级
**场景：** 一名开发者请求 narrative-director 审查并优化档案馆文物上"远古辉光"视觉效果所用的着色器代码。
**预期：** 代理拒绝评估着色器代码，并重定向到合适的引擎专家（godot-gdscript-specialist 或同等着色器专家）。
**断言：**
- [ ] 不对着色器代码或视觉实现做出任何有约束力的决定
- [ ] 明确指出合适的引擎或着色器专家才是正确的处理者
- [ ] 可以注明该效果应传达的叙事氛围（例如"应当给人古老而神圣的感觉，而非科技感"），但将所有技术视觉实现交由他人决定

### 用例 3：门禁结论——正确的词汇
**场景：** 提交了一份角色"Aldric Vorne"的新角色背景故事文档。文档称 Aldric 于 150 年前出生于首都，并亲眼目睹了大崩塌。然而，既定的世界圣经记载 Aldric 出生于大崩塌之后 50 年，出生地是一座边陲小镇，而非首都。请求标记为 ND-CONSISTENCY。
**预期：** 返回 `ND-CONSISTENCY: INCONSISTENT`，并具体引用两处相互矛盾的事实：出生时间（150 年前 vs. 大崩塌后 50 年）和出生地点（首都 vs. 边陲小镇）。
**断言：**
- [ ] 结论严格为 CONSISTENT / INCONSISTENT 之一——而非自由文本
- [ ] 结论标记格式为 `ND-CONSISTENCY: INCONSISTENT`
- [ ] 理由具体引用了两处矛盾，而非仅仅说"与设定不符"
- [ ] 引用权威来源（世界圣经）作为既定事实的依据

### 用例 4：冲突升级——正确的上级
**场景：** 一名作家在其最新对话中设定远古文明"只用歌声交流"。而 world-builder 现有的设定条目描述同一文明通过书写符文进行沟通。两者都在叙事领域内，两位创作者对哪一个是正史存在分歧。
**预期：** narrative-director 在其领域内做出有约束力的正史裁定。对于叙事内部的冲突，他们无需升级到更高权威——这属于其声明的领域权威范围。他们发布裁定（例如"符文书写为正史中的主要交流方式；歌唱可用于仪式/典礼"），并指示作家和 world-builder 双方将各自工作与裁定对齐。
**断言：**
- [ ] 做出有约束力的正史裁定——不将此叙事内部冲突推给 creative-director
- [ ] 裁定表述清晰，并为双方提供了调和路径
- [ ] 指示双方（作家和 world-builder）更新各自文档以对齐
- [ ] 以可作为正史事实加入世界圣经的方式记录该裁定

### 用例 5：上下文传递——使用提供的上下文
**场景：** 代理收到一个门禁上下文块，其中包含三份现有设定文档：世界圣经（确定大崩塌时间线及起因）、角色注册表（列出正史角色年龄、出身和阵营），以及一份阵营文档（描述沉没档案馆守护者）。提交了一个引入未注册新角色的新故事章节。
**预期：** 评估将新角色与角色注册表交叉核对（无冲突），将章节的时间线引用与世界圣经核对，并根据阵营文档评估章节对档案馆守护者的刻画。评估中使用了来自全部三份所提供文档的具体事实。
**断言：**
- [ ] 将新角色与所提供的角色注册表交叉核对
- [ ] 根据所提供的世界圣经事实检查时间线引用
- [ ] 根据所提供的阵营文档评估阵营刻画
- [ ] 不生成泛泛的叙事反馈——所有断言均可追溯到所提供的文档

---

## 协议合规

- [ ] 仅使用 CONSISTENT / INCONSISTENT 词汇返回结论
- [ ] 保持在声明的叙事领域内
- [ ] 对叙事内部冲突做出有约束力的裁定，不做不必要的升级
- [ ] 在输出中使用门禁 ID（例如 `ND-CONSISTENCY: INCONSISTENT`），而非行内散文式结论
- [ ] 不做出有约束力的视觉设计、技术或制作决定

---

## 覆盖说明
- 对话质量评审（区别于世界观一致性）未覆盖——应增加专门用例。
- 跨完整章节集的多文档一致性检查未覆盖——推迟到与 /review-all-gdds 集成。
- 机制变更对叙事的影响（例如某个破坏故事张力的游戏机制）需要与 game-designer 协调，此处未覆盖。
- 角色弧光评审（随时间推进的成长、动机连贯性）未覆盖。
