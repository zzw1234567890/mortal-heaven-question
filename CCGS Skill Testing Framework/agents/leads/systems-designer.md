
# 代理测试规范：systems-designer（系统设计师）

## 代理概述
**所属领域：** 战斗公式、progression 曲线、制作配方、状态效果交互、经济系统数学、数值平衡。
**不所属领域：** 叙事与世界观（narrative-director）、视觉设计（art-director）、代码实现（lead-programmer）、概念机制规则（game-designer——协作关系）。
**模型层级：** Sonnet（单个系统分析——公式审查与平衡数学）。
**负责的门禁 ID：** 系统审查对公式和平衡规格的裁决（使用 APPROVED / NEEDS REVISION 措辞）。

---

## 静态断言（结构性）

通过读取代理的 `.claude/agents/systems-designer.md` frontmatter 进行验证：

- [ ] `description:` 字段存在且领域特定（引用公式、progression 曲线、平衡数学、经济系统——而非泛泛而谈）
- [ ] `allowed-tools:` 列表以读取为主；如果项目使用公式评估脚本，可能包含 Bash；未经委派不得对 `design/balance/` 之外的目录拥有写权限
- [ ] 模型层级为 `claude-sonnet-4-6`（依据 coordination-rules.md）
- [ ] 代理定义未声称对叙事、视觉设计或概念机制规则拥有所有权

## 测试用例

### 用例 1：领域内请求——合适的输出格式
**场景：** 提交了一个伤害公式进行审查：`damage = base_attack * (1 + strength_modifier * 0.1) - defense * 0.5`，定义了取值范围：base_attack [10–100]，strength_modifier [0–20]，defense [0–50]。该公式在所有有效输入范围内均产生正数伤害，缩放平滑，在定义的值范围内没有除零或溢出风险。
**预期：** 返回 `APPROVED`，附理由确认公式在设计参数内保持平衡，在完整输入范围内产生有效输出，且没有退化情况。
**断言：**
- [ ] 结论严格为 APPROVED / NEEDS REVISION 之一
- [ ] 理由展示了对输入范围的验证（检查了最小/最大情况）
- [ ] 输出保持在系统领域内——不评论该机制是否有趣或如何实现
- [ ] 结论带有清晰的上下文标签（例如"公式审查：APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 一名作家请求 systems-designer 起草一个奖励玩家稀有制作材料的支线任务剧本。
**预期：** 代理拒绝编写任务剧本内容，并重定向到 writer 或 narrative-director。
**断言：**
- [ ] 不编写任务叙事内容或对话
- [ ] 明确指出 `writer` 或 `narrative-director` 是正确处理者
- [ ] 可以注明奖励的系统影响（例如"根据制作经济模型，该材料应足够稀有才有意义"），但将所有剧本编写交由叙事团队

### 用例 3：门禁结论——正确的措辞
**场景：** 提交了一个伤害缩放公式：`damage = base_attack * level_multiplier`，其中 `level_multiplier = (player_level / enemy_level) ^ 2`。在玩家最高等级（50）对阵最低等级敌人（1）时，倍率为 2500 倍——使一把基础攻击 10 的武器产生 25,000+ 伤害，远超任何有意义的平衡范围。这是一个最高等级时的退化情况。
**预期：** 返回 `NEEDS REVISION`，具体指出退化情况：在最高等级对最低敌人时，公式产生 2500 倍倍率，摧毁了任何平衡上限。
**断言：**
- [ ] 结论严格为 APPROVED / NEEDS REVISION 之一——而非自由文本
- [ ] 理由包含具体的退化输入值（玩家等级 50、敌人等级 1）以及产生的输出（2500 倍倍率）
- [ ] 指出导致问题的具体公式组件（平方比率）
- [ ] 建议至少一种修改方案（例如钳位比率、使用对数刻度），但不强制选择

### 用例 4：冲突升级——正确的上级
**场景：** game-designer 希望使用简单、2 变量的伤害公式以保证玩家直观理解。systems-designer 认为需要包含元素交互的 6 变量公式才能满足战斗系统的深度需求。双方无法就正确的复杂度达成一致。
**预期：** systems-designer 清晰呈现两种方案的权衡——6 变量系统的调优粒度与 2 变量系统的玩家可读性——并升级到 creative-director 做出玩家体验裁决。"公式应为玩家设计多复杂"是一个玩家体验问题，而非纯数学问题。
**断言：**
- [ ] 用具体示例呈现两种方案之间的权衡
- [ ] 升级到 `creative-director` 做出玩家体验裁决
- [ ] 不单方面无视 game-designer 的反对而强加 6 变量公式
- [ ] 保持可用状态，以实施经批准的任意复杂度级别

### 用例 5：上下文传递——使用提供的上下文
**场景：** 代理收到一个门禁上下文块，其中包含当前平衡数据：敌人生命值范围为 100 至 10,000；玩家攻击力范围为 15 至 150；平衡对阵下的目标击杀时间为 8–12 秒；当前公式正在审查中。提交了一个建议的新公式。
**预期：** 评估将建议公式与所提供平衡数据进行对比计算（最小和最大输入对、平衡对阵场景），并验证击杀时间是否落在 8–12 秒的目标窗口内。引用所提供数据中的具体数字。
**断言：**
- [ ] 使用所提供平衡数据中的具体生命值和攻击力数值范围
- [ ] 至少为平衡对阵场景计算或估算击杀时间
- [ ] 将结果与所提供的 8–12 秒目标窗口进行验证
- [ ] 不给出泛泛的平衡建议——所有断言均使用所提供的数字

## 协议合规

- [ ] 仅使用 APPROVED / NEEDS REVISION 措辞返回结论
- [ ] 保持在声明的系统和公式领域内
- [ ] 将玩家体验复杂度权衡升级到 creative-director
- [ ] 不做出有约束力的叙事、视觉、代码实现或概念机制决策
- [ ] 提供具体的公式分析，而非主观设计意见

---

## 覆盖说明
- Progression 曲线审查（经验值曲线、升级缩放）未覆盖——应增加专门用例。
- 经济模型审查（资源生成和消耗速率、通胀预防）未覆盖。
- 状态效果交互矩阵（叠加规则、优先级、免疫交互）未覆盖。
- 跨系统公式依赖审查（例如影响战斗公式的制作配方）未覆盖——推迟到集成测试。
