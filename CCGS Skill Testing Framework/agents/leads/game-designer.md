
# 代理测试规范：game-designer（游戏设计师）

## 代理概述
**所属领域：** 核心循环设计、progression 系统、战斗机制规则、经济系统设计、面向玩家的规则与交互。
**不所属领域：** 代码实现（lead-programmer / gameplay-programmer）、视觉美术（art-director）、叙事背景与故事（narrative-director——协调关系）、平衡公式数学（systems-designer——协作关系）。
**模型层级：** Sonnet（单个系统设计编写与审查）。
**负责的门禁 ID：** 设计审查对机制规格的裁决（无命名门禁 ID 前缀——使用 APPROVED / NEEDS REVISION 措辞）。

---

## 静态断言（结构性）

通过读取代理的 `.claude/agents/game-designer.md` frontmatter 进行验证：

- [ ] `description:` 字段存在且领域特定（引用核心循环、progression、战斗机制、经济系统、面向玩家的设计——而非泛泛而谈）
- [ ] `allowed-tools:` 列表以读取为主；包含用于 GDD 和设计文档的 Read；除非设计工具需要，否则不包含 Bash
- [ ] 模型层级为 `claude-sonnet-4-6`（依据 coordination-rules.md）
- [ ] 代理定义未声称对代码实现、视觉美术风格或独立叙事背景决策拥有权威

## 测试用例

### 用例 1：领域内请求——合适的输出格式
**场景：** 提交了一份"基于耐力的闪避（Stamina-Based Dodge）"系统的机制规格进行审查。该规格定义了：玩家拥有耐力池（100 单位），每次闪避消耗 25 耐力，耐力在不闪避时以 20 单位/秒恢复，闪避提供 0.3 秒的无敌时间。核心循环交互描述清晰，规则明确，边缘情况（耐力为 0、恢复期间闪避）已处理。
**预期：** 返回 `APPROVED`，附理由确认核心循环清晰度、规则明确性和边缘情况覆盖。
**断言：**
- [ ] 结论严格为 APPROVED / NEEDS REVISION 之一
- [ ] 理由引用具体的设计质量标准（规则清晰、边缘情况覆盖、核心循环一致性）
- [ ] 输出保持在设计范围内——不评论如何在代码中实现或需要哪些美术资源
- [ ] 结论带有清晰的上下文标签（例如"机制规格审查：APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 一名团队成员请求 game-designer 编写耐力系统存在的世界观背景解释（例如角色在游戏世界中拥有耐力限制的叙事原因）。
**预期：** 代理拒绝编写叙事或世界观内容，并重定向到 writer 或 narrative-director。
**断言：**
- [ ] 不编写叙事或世界观内容
- [ ] 明确指出 `writer` 或 `narrative-director` 是正确处理者
- [ ] 可以注明该世界观应支持的设计意图（例如"耐力系统应强化物理真实感主题"），但将编写工作推给叙事团队

### 用例 3：门禁结论——正确的措辞
**场景：** 提交了一份"环境危害伤害（Environmental Hazard Damage）"的机制规格。该规格定义了三种危害类型（火焰、酸液、电流），但未说明玩家同时受到多种危害影响时会发生什么、在闪避无敌窗口期间施加危害时会发生什么，也未说明伤害频率（每秒、每 tick、进入时）。
**预期：** 返回 `NEEDS REVISION`，具体指出未定义的边缘情况：多种危害交互、无敌期间承受危害、以及伤害频率规定。
**断言：**
- [ ] 结论严格为 APPROVED / NEEDS REVISION 之一——而非自由文本
- [ ] 理由明确指出缺失的具体边缘情况名称
- [ ] 不拒绝整个机制——指出需要补充的具体缺口
- [ ] 提供可操作的指导说明需要定义什么（而非如何实现）

### 用例 4：冲突升级——正确的上级
**场景：** systems-designer 提出了一个包含 6 个变量和复杂缩放交互的伤害公式，认为这能提供最佳调优粒度。game-designer 认为该公式对玩家来说过于复杂难以直观理解，希望采用更简单的 2 变量版本。
**预期：** game-designer 拥有概念规则和玩家体验意图的所有权（"伤害应让玩家感到可理解"），但将公式粒度问题交由 systems-designer 决定。如果双方无法解决分歧（一方想要复杂，一方想要简单），则升级到 creative-director 做出玩家体验裁决。
**断言：**
- [ ] 清晰表达玩家体验意图（直观的伤害、玩家自主性）
- [ ] 将公式粒度决策交由 `systems-designer` 决定
- [ ] 将未解决的分歧升级到 `creative-director` 作为玩家体验仲裁者
- [ ] 不单方面向 systems-designer 强加公式结构

### 用例 5：上下文传递——使用提供的上下文
**场景：** 代理收到一个门禁上下文块，其中包含游戏的三个支柱（pillar）："玩家创作（player authorship）"、"后果永久性（consequence permanence）"和"世界响应性（world responsiveness）"。提交了一个关于"带有传承加成的永久死亡（permadeath with legacy bonuses）"的新机制规格进行审查。
**预期：** 评估将机制与所提供三个支柱逐一对照——永久死亡如何支持玩家创作、传承加成如何体现后果永久性、世界如何响应玩家的死亡？在理由中直接使用支柱措辞。
**断言：**
- [ ] 在评估中引用所提供三个支柱的名称
- [ ] 明确评估机制对每个支柱的贡献
- [ ] 不生成泛泛的游戏设计建议——所有反馈均与所提供的支柱措辞相关联
- [ ] 如果任何支柱与机制产生冲突，指出并标记为具体关注点

---

## 协议合规

- [ ] 仅使用 APPROVED / NEEDS REVISION 措辞返回结论
- [ ] 保持在声明的游戏设计领域内
- [ ] 将设计与公式的冲突在未解决时升级到 creative-director
- [ ] 不做出有约束力的代码实现、视觉美术或独立叙事决策
- [ ] 提供可操作的设计反馈，而非实现方案

---

## 覆盖说明
- 经济系统设计审查（资源消耗、来源、通胀预防）未覆盖——应增加专门用例。
- Progression 系统审查（经验值曲线、解锁关卡、玩家能力轨迹）未覆盖。
- 跨多个互联系统的核心循环验证（而非单个机制）未覆盖——推迟到 /review-all-gdds 集成。
- 与 systems-designer 在公式所有权边界上的协作协议可以从额外用例中受益。
