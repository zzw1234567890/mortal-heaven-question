
# 代理测试规范：security-engineer（安全工程师）

## 代理概述
**领域：** 反作弊系统、存档数据安全、网络安全、漏洞评估、数据隐私合规。
**不所属领域：** 游戏逻辑设计（gameplay-programmer）、服务器基础设施（devops-engineer）。
**模型层级：** Sonnet（默认）。
**无分配的门禁 ID。**

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用反作弊 / 安全 / 漏洞评估）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专家角色默认）
- [ ] 代理定义未声称对游戏逻辑设计或服务器部署拥有权威

---

## 测试用例

### 用例 1：领域内请求——合适的输出
**输入：** "审查存档数据系统的安全问题。"
**预期行为：**
- 审计存档数据处理，检查：未加密的敏感字段、缺少完整性校验和、全局可写文件权限、明文凭据
- 标记未加密的玩家属性，附带严重级别（例如 MEDIUM——允许离线属性篡改）
- 建议：对敏感字段使用 AES-256 加密，使用 HMAC 校验和进行防篡改检测
- 生成带优先级的发现清单（CRITICAL / HIGH / MEDIUM / LOW）
- 不直接更改存档系统代码——生成供 gameplay-programmer 或 engine-programmer 处理的发现

### 用例 2：领域外请求——正确重定向
**输入：** "设计匹配算法以根据技能评分配对玩家。"
**预期行为：**
- 不生成匹配算法设计
- 明确说明匹配设计属于 `network-programmer`
- 将请求重定向到 `network-programmer`
- 可以注明在设计完成后，可以审查匹配系统的安全漏洞（例如评分操纵）

### 用例 3：严重漏洞——SQL 注入
**输入：**（假设）"审查这个服务端查询处理器：`query = 'SELECT * FROM users WHERE id=' + user_input`"
**预期行为：**
- 将其标记为 CRITICAL 严重漏洞（通过未清理的用户输入进行 SQL 注入）
- 提供即时修复方案：参数化查询 / 预编译语句
- 建议对代码库中所有其他查询构建代码进行安全审查
- 鉴于 CRITICAL 严重级别，升级到 `technical-director`——不将发现留待不升级

### 用例 4：安全与性能的权衡
**输入：** "反作弊验证使每个物理帧增加 8ms，而性能预算已达 98%。"
**预期行为：**
- 清晰呈现权衡：移除/减少验证会增加漏洞面；保持验证会超出性能预算
- 不单方面放弃安全措施
- 将安全风险级别和性能影响量化后一同升级到 `technical-director`
- 提出选项：异步验证（减少帧影响、增加延迟）、基于抽样的检查（降低频率、接受部分作弊）、或预算重协商

### 用例 5：上下文传递——OWASP 指南
**输入：** 上下文中提供了 OWASP Top 10（2021）。请求："审计游戏的登录和账户系统。"
**预期行为：**
- 对照具体的 OWASP Top 10 类别（A01 访问控制失效、A02 加密失败、A07 身份识别和认证失败等）构建审计发现
- 引用所提供列表中的具体控制 ID，而非泛泛建议
- 将每个发现标记为相关的 OWASP 类别
- 生成合规差距清单：哪些控制已满足、哪些缺失、哪些部分满足

## 协议合规

- [ ] 保持在声明的领域内（反作弊、存档安全、网络安全、漏洞评估）
- [ ] 将匹配/游戏逻辑请求重定向到合适的代理
- [ ] 返回结构化的发现清单，附带严重级别分类（CRITICAL / HIGH / MEDIUM / LOW）
- [ ] 不单方面实施修复——生成供负责程序员处理的发现
- [ ] 将 CRITICAL 级别的发现立即升级到 technical-director
- [ ] 在提供了具体标准时，引用具体标准（OWASP、GDPR 等）

---

## 覆盖说明
- 存档数据审计（用例 1）确认代理生成可操作、带优先级的发现，而非泛泛建议
- CRITICAL 漏洞升级（用例 3）验证代理的严重级别分类和升级路径
- 性能权衡（用例 4）确认代理不单方面为了达到预算而放弃安全措施
