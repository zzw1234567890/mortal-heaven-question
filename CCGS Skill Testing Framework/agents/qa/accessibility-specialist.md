
# 代理测试规范：accessibility-specialist（无障碍专家）

## 代理概述
**领域：** 输入重映射、文本缩放、色盲模式、屏幕阅读器支持、无障碍标准合规（WCAG、平台认证）。
**不所属领域：** 整体 UX 流程设计（ux-designer）、视觉美术风格方向（art-director）。
**模型层级：** Sonnet（默认）。
**无分配的门禁 ID。**

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用无障碍 / 包容性设计 / WCAG）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专家角色默认）
- [ ] 代理定义未声称对 UX 流程或视觉美术风格拥有权威

---

## 测试用例

### 用例 1：领域内请求——合适的输出
**输入：** "审查玩家 HUD 的无障碍性。"
**预期行为：**
- 审计 HUD 规格或截图，检查：
  - 对比度（标记任何低于 4.5:1（AA）或 7:1（AAA）的文本）
  - 颜色编码信息的替代表示（例如敌人血条仅使用颜色，无形状区分）
  - 文本大小（标记 1080p 下任何低于 16px 等效大小的文本）
  - 关键状态元素的屏幕阅读器或 TTS 注释可用性
- 生成带有优先级的发现清单，包含具体元素名称及其违反的标准
- 不重新设计 HUD——生成供 ux-designer 和 ui-programmer 处理的发现

### 用例 2：领域外请求——正确重定向
**输入：** "设计整体游戏流程：主菜单 → 角色选择 → 加载 → 游戏 → 暂停 → 结算。"
**预期行为：**
- 不生成 UX 流程架构
- 明确说明整体游戏流程设计属于 `ux-designer`
- 将请求重定向到 `ux-designer`
- 可以注明在流程设计完成后，可以审查其无障碍问题（例如时间限制、认知负荷）

### 用例 3：色盲模式冲突
**输入：** "针对红色盲（deuteranopia）的建议色盲模式将敌人红色血条替换为橙色，但美术调色板已使用橙色表示友方单位。"
**预期行为：**
- 识别冲突：色盲模式的橙色与既定的友方单位调色板发生碰撞
- 不单方面更改美术调色板（这属于 art-director）
- 将冲突标记给 `art-director`，描述具体的视觉重叠情况
- 提出无需更改调色板的替代区分策略（例如形状/图标叠加、图案填充、图标设计）

### 用例 4：无障碍功能对 UI 状态的要求
**输入：** "背包的屏幕阅读器支持要求系统将物品名称和数量作为可访问文本节点暴露。"
**预期行为：**
- 生成无障碍需求规格，定义每个背包元素所需的可访问文本属性
- 指出实现可访问文本节点需要 UI 系统变更
- 与 `ui-programmer` 协调以实施所需的可访问文本节点暴露
- 不自行实现 UI 系统变更

### 用例 5：上下文传递——WCAG 2.1 目标
**输入：** 上下文中提供了项目无障碍目标：WCAG 2.1 AA 合规性。请求："审查对话系统的无障碍性。"
**预期行为：**
- 引用与对话相关的具体 WCAG 2.1 AA 成功标准（例如 1.4.3 对比度最小值、1.4.4 文本缩放、2.2.1 自动推进对话的时间可调）
- 使用标准中的确切标准编号和名称，而非意译
- 将每个发现标记为违反的具体标准
- 注明哪些标准不在 AA 范围内（仅 AAA），以免被错误标记为失败

## 协议合规

- [ ] 保持在声明的领域内（重映射、文本缩放、色盲模式、屏幕阅读器、标准合规）
- [ ] 将 UX 流程设计重定向到 ux-designer，美术调色板决策重定向到 art-director
- [ ] 返回结构化的发现清单，包含具体元素名称、对比度和标准引用
- [ ] 不实现 UI 变更——与 ui-programmer 协调实现
- [ ] 在提供合规目标时，引用具体的 WCAG 标准编号
- [ ] 将无障碍需求与美术决策之间的冲突标记给 art-director

---

## 覆盖说明
- HUD 审计（用例 1）应产生可作为无障碍 story 在 sprint 待办列表中跟踪的发现
- 色盲冲突（用例 3）确认代理尊重 art-director 在调色板上的权威
- WCAG 标准（用例 5）验证代理精确而非泛泛地使用标准
