
# 代理测试规范：qa-tester（QA 测试员）

## 代理概述
- **领域：** 详细测试用例编写、bug 报告（结构化格式）、测试执行文档、回归清单、冒烟检查执行文档、依据项目编码标准的测试证据记录
- **不所属领域：** 测试策略和测试计划设计（qa-lead）、已发现 bug 的实现修复（合适的程序员）、QA 流程架构（qa-lead）
- **类别：** qa
- **模型层级：** Sonnet
- **门禁 ID：** 无；将模糊的验收标准标记给 qa-lead 而非自行解决

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用测试用例、bug 报告、测试执行、回归测试）
- [ ] `allowed-tools:` 列表与代理角色匹配（Read/Write 用于 `tests/` 和 `production/qa/evidence/`；无源代码编辑工具）
- [ ] 模型层级为 Sonnet（QA 专家默认）
- [ ] 代理定义未声称对测试策略、修复实现或验收标准定义拥有权威

---

## 测试用例

### 用例 1：领域内请求——保存系统的测试用例
**输入：** "为我们的保存系统编写测试用例。它必须保存和加载玩家位置、背包和任务状态。"
**预期行为：**
- 生成一个测试用例列表，至少包含以下测试用例，每个用例包含所有四个必需字段：
  - **TC-SAVE-001**：保存和加载玩家位置
  - **TC-SAVE-002**：保存和加载完整背包（多物品类型、数量、装备状态）
  - **TC-SAVE-003**：保存和加载任务状态（进行中、已完成和锁定状态）
  - **TC-SAVE-004**：覆盖已存在的存档文件
  - **TC-SAVE-005**：加载旧版本的存档文件（向后兼容）
  - **TC-SAVE-006**：损坏存档文件处理（文件存在但无效）
- 每个测试用例包含：**前提条件（Precondition）**（测试前所需的游戏状态）、**步骤（Steps）**（编号、无歧义）、**预期结果（Expected Result）**（具体、可观察的结果）、**通过标准（Pass Criteria）**（二元通过/失败条件）
- 不将"验证保存功能正常"作为通过标准——标准必须可观察且无歧义

### 用例 2：领域外请求——实现 bug 修复
**输入：** "你发现了一个 bug：保存系统在版本不匹配时丢失背包数据。请修复它。"
**预期行为：**
- 不生成任何实现代码或尝试修复保存系统
- 明确说明："Bug 修复由合适的程序员（保存系统逻辑由 gameplay-programmer）实施；我记录该 bug 并编写回归测试用例以验证修复。"
- 提供生成：(a) 一个供程序员使用的结构化 bug 报告，(b) 针对 TC-SAVE-005（版本不匹配）的回归测试用例，可在修复后运行

### 用例 3：模糊的验收标准——标记给 qa-lead
**输入：** "为教程编写测试用例。story 中的验收标准说'教程应感觉直观。'"
**预期行为：**
- 指出"应感觉直观"是一个不可度量的验收标准——这是一个主观质量陈述，而非可测试条件
- 不通过自行定义"直观"的含义来针对模糊标准编写测试用例
- 标记给 qa-lead："验收标准'教程应感觉直观'按当前写法不可测试；需要澄清——例如'X% 的新玩家在不使用提示按钮的情况下完成教程'或'测试过程中没有测试员需要外部帮助才能完成教程'"
- 提供两个或三个具体、可度量的备选标准供 qa-lead 选择

### 用例 4：热修复后的回归测试
**输入：** "应用了一个热修复，更改了背包序列化处理可空物品槽位的方式。编写一份针对受影响系统的定向回归检查清单。"
**预期行为：**
- 识别受影响的系统：背包保存/加载、任何读取背包状态的 UI、任何检查背包内容的任务系统、任何读取背包槽位的制作系统
- 生成一份仅针对这些系统的回归检查清单——而非完整的游戏回归测试
- 清单项针对具体变更：空物品槽位处理（空槽位、混合满/空槽位数组、槽位数量边界条件）
- 每个清单项指定：测试什么、如何验证通过、失败的表现形式
- 不生成泛泛的"测试所有内容"清单——定向回归的价值在于其针对性

### 用例 5：上下文传递——来自 coding-standards.md 的测试证据格式
**输入上下文：** coding-standards.md 规定：Logic 类 story 需要 `tests/unit/[system]/` 中的自动化单元测试。Visual/Feel 类 story 需要 `production/qa/evidence/` 中的截图 + 主管签字。UI 类 story 需要 `production/qa/evidence/` 中的手动走查文档。
**输入：** "为背包 UI（一个 UI 类 story）编写测试用例：网格布局、物品提示显示和拖拽重新排序。"
**预期行为：**
- 根据所提供的标准正确将其归类为 UI 类 story
- 生成手动走查测试文档（而非自动化单元测试）——因为编码标准规定 UI 类 story 使用手动走查
- 指定输出位置：`production/qa/evidence/`（而非 `tests/unit/`）
- 测试用例包括：网格布局验证（所有物品显示、无溢出）、提示显示（悬停/聚焦时正确的物品名称、属性、描述）、拖拽（物品移动到目标槽位、原槽位变空、槽位限制被遵守）
- 注明根据编码标准，此为建议性（ADVISORY）证据级别，而非阻塞性（BLOCKING）——明确说明以便团队了解门禁级别

## 协议合规

- [ ] 保持在声明的领域内（测试用例编写、bug 报告、测试执行文档、回归清单）
- [ ] 将 bug 修复请求重定向到合适的程序员，并主动提出记录 bug 和编写回归测试
- [ ] 将模糊的验收标准标记给 qa-lead，而非自行编造可测试的解释
- [ ] 生成定向回归检查清单（系统特定），而非全游戏回归通过测试
- [ ] 根据 coding-standards.md 使用正确的测试证据格式和输出位置

---

## 覆盖说明
- 用例 1（测试用例完整性）是基础质量测试——缺少字段（前提条件、步骤、预期结果、通过标准）即为失败
- 用例 3（模糊标准）是一项协调测试——qa-tester 不得静默接受不可测试的标准
- 用例 5 要求上下文中包含 coding-standards.md 及其测试证据表；代理必须正确应用证据类型和位置
- 建议性 vs. 阻塞性门禁级别（用例 5）是一个影响 story 完成的细节——验证代理是否报告了它
- 无自动运行器；通过手动或 `/skill-test` 进行审查
