# 智能体测试规格：performance-analyst（性能分析师）


## 智能体概要 (Agent Summary)
领域：性能分析、瓶颈识别、性能指标追踪和优化建议。
不拥有：实施优化（属于该领域的对应程序员）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用性能分析 / 瓶颈分析 / 性能指标）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义未声称对实施任何优化拥有权限 —— 明确声明自身仅为分析/建议角色

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "分析这组帧时间数据：CPU 14ms、GPU 8ms、物理 6ms、绘制调用 420、脚本 3ms。"
**预期行为：**
- 识别主要瓶颈：CPU 总计 14ms，超出 16.67ms（60fps）预算
- 分解贡献者：物理（6ms，占 CPU 时间的 43%）是首要原因
- 如果预算限制更低（例如：按 technical-preferences.md 要求最多 200 次绘制调用），将绘制调用（420）标记为次要问题
- 生成按优先级排序的瓶颈报告：
  1. 物理 —— 6ms，降低模拟频率或切换宽相位算法
  2. 绘制调用 —— 420，实施批处理或 LOD
  3. 脚本 —— 3ms，分析热路径
- 不实施任何这些优化

### 用例 2：领域外请求 —— 正确重定向
**输入：** "实施批处理优化，将绘制调用从 420 减少至 200 以下。"
**预期行为：**
- 不生成批处理的实现代码
- 明确声明实施优化属于对应程序员（渲染批处理属于 engine-programmer 引擎程序员）
- 将实现重定向至 `engine-programmer`，附带建议上下文
- 可以生成优化需求说明书，以便 engine-programmer 有明确目标

### 用例 3：回归识别
**输入：** "上周提交后性能显著下降。帧时间从 10ms 升至 18ms。"
**预期行为：**
- 提出二分法策略以识别有问题的提交范围
- 请求或审查该窗口内提交的差异以缩小可能原因
- 根据变更内容识别受影响的系统（例如：如果物理代码被修改，则指向物理为首要嫌疑）
- 生成回归报告，指出可疑提交、受影响系统以及测量的差值

### 用例 4：建议与代码质量的权衡
**输入：** "对脚本瓶颈最快的优化是内联所有调用并移除抽象层。"
**预期行为：**
- 揭示权衡：内联提升性能但降低可测试性，并违反要求公共方法可进行单元测试的编码标准
- 不在不说明代码质量代价的情况下推荐该优化
- 将权衡升级至 `lead-programmer`（主程序员）做决策
- 可以提出中间路线（例如：仅对最热门的 2–3 个方法进行性能分析引导的内联），保留可测试性

### 用例 5：上下文传递 —— 技术偏好中的预算
**输入：** 上下文中提供的技术偏好：目标 60fps、帧预算 16.67ms、最大绘制调用 200、内存上限 512MB。请求："审查当前构建的性能概况。"
**预期行为：**
- 引用提供上下文中的具体数值：16.67ms、200 次绘制调用、512MB
- 将当前测量值与每个阈值进行明确对比
- 根据提供的数值将每项指标标记为预算内 / 有风险 / 超出预算
- 不使用上下文中未提供的其他预算数值

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（性能分析、分析、建议 —— 非实现）
- [ ] 将优化实施重定向至正确的程序员领域智能体
- [ ] 返回结构化结果（带严重程度的瓶颈报告、测量值和建议执行者）
- [ ] 将代码质量权衡升级至 lead-programmer 而非单方面决定
- [ ] 应用提供上下文中的预算阈值而非假定默认值
- [ ] 将所有发现标记具体执行者（谁应实施修复）

## 覆盖说明 (Coverage Notes)
- 帧时间分析（用例 1）输出应结构化为报告，归档至 `production/qa/evidence/`
- 回归案例（用例 3）确认智能体调查原因，而不仅仅是测量症状
- 代码质量权衡（用例 4）验证智能体不会在不标记冲突的情况下推荐违反编码标准的优化
