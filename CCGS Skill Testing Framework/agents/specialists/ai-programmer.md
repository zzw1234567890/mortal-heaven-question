# 智能体测试规格：ai-programmer（AI 程序员）


## 智能体概要 (Agent Summary)
领域：NPC 行为、状态机、寻路、感知系统和 AI 决策。
不拥有：玩家机制（属于 gameplay-programmer 玩法程序员）、渲染或引擎内部（属于 engine-programmer 引擎程序员）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用 NPC 行为 / AI 系统）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义未声称对玩家机制或引擎渲染拥有权限

---

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "实现一个守卫 NPC 的巡逻与警戒行为树：在路点之间巡逻，在 10 单位内检测到玩家，然后进入警戒状态并追击。"
**预期行为：**
- 生成行为树规格（节点：Selector 选择器、Sequence 序列、Leaf 叶子行为）及相应代码骨架
- 定义命名清晰的守卫、警戒、追击 (Patrol, Alert, Pursue) 状态
- 将感知/检测检查作为条件节点使用，而非内联在移动代码中
- 路点数据驱动（作为资源或导出变量传递），非硬编码位置
- 输出包含公共 API 的文档注释

### 用例 2：领域外请求 —— 正确重定向
**输入：** "实现 WASD 移动和冲刺能力的玩家输入处理。"
**预期行为：**
- 不生成玩家输入或移动代码
- 明确声明此内容超出其领域（玩家机制属于 gameplay-programmer）
- 将请求重定向至 `gameplay-programmer`
- 可以说明一旦玩家位置通过 API 可用，AI 感知即可引用

### 用例 3：跨领域协调 —— 关卡约束
**输入：** "为仓库关卡设计寻路，但该关卡有狭窄走廊导致导航网格混乱。"
**预期行为：**
- 不单方面修改关卡布局或导航网格资源
- 与 `level-designer`（关卡设计师）协调，明确导航网格需求和走廊尺寸
- 提出基于关卡几何条件的寻路方案（例如：配合智能体半径调整的导航网格、流场）
- 清晰记录假设条件并标记阻塞项

### 用例 4：性能升级 —— 自定义数据结构
**输入：** "寻路优先级队列是性能瓶颈；我需要一个自定义二叉堆实现来提升性能。"
**预期行为：**
- 识别出底层引擎集成的数据结构属于 engine-programmer 的领域
- 向 `engine-programmer` 升级问题，附带清晰的瓶颈描述和所需接口
- 可以提供算法规格（二叉堆接口、预期操作）以指导 engine-programmer
- 如果该底层结构需要引擎内存管理，则不单方面实现

### 用例 5：上下文传递 —— 利用关卡布局设计寻路
**输入：** 上下文中提供关卡布局文档，显示两个咽喉点：(12, 0) 处的门口和 (40, 5) 处的桥梁。请求："设计此关卡中敌人的巡逻路线和威胁响应。"
**预期行为：**
- 引用上下文中提供的特定咽喉点坐标
- 设计利用咽喉点作为战术位置的巡逻路线
- 指定在追击过程中将 NPC 引向已识别咽喉点的警戒状态转换
- 不编造布局文档中未包含的几何体

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（NPC 行为、寻路、感知、状态机）
- [ ] 将领域外请求重定向至正确智能体（gameplay-programmer、engine-programmer、level-designer）
- [ ] 返回结构化结果（行为树规格、状态机图、代码骨架）
- [ ] 未经明确委派不修改玩家机制文件
- [ ] 将性能关键的底层结构升级至 engine-programmer
- [ ] 使用数据驱动的 NPC 配置（路点、检测半径）而非硬编码值

---

## 覆盖说明 (Coverage Notes)
- 行为树输出（用例 1）应由 `tests/unit/ai/` 中的单元测试验证
- 关卡布局上下文（用例 5）验证智能体读取并应用提供的文档而非自行编造
- 性能升级（用例 4）确认智能体能识别 engine-programmer 的边界
