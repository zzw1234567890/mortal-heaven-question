# 智能体测试规格：gameplay-programmer（玩法程序员）


## 智能体概要 (Agent Summary)
领域：游戏机制代码、玩家系统、战斗实现和交互功能。
不拥有：UI 实现（属于 ui-programmer UI 程序员）、AI 行为树（属于 ai-programmer AI 程序员）、引擎/渲染系统（属于 engine-programmer 引擎程序员）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用游戏机制 / 玩家系统）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep —— 不包含仅编排型智能体所需的工具
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义未声称对 UI、AI 行为或引擎/渲染代码拥有权限

---

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "实现一个近战连击系统，三次连续轻攻击可连锁触发终结技。"
**预期行为：**
- 按照项目语言（GDScript/C#）和编码标准生成代码或代码骨架
- 将连击状态追踪、输入窗口计时和终结技触发逻辑定义为独立、可测试的方法
- 如果上下文中提供了相关 GDD 章节，则引用该章节
- 不实现 UI 反馈（委托给 ui-programmer）或 AI 反应（委托给 ai-programmer）
- 按照编码标准在所有公共方法中包含文档注释

### 用例 2：领域外请求 —— 正确重定向
**输入：** "构建包含暂停和设置面板的主菜单界面。"
**预期行为：**
- 不生成菜单实现代码
- 明确声明此内容超出其领域
- 将请求重定向至 `ui-programmer`
- 可以说明如果暂停菜单需要读取游戏状态，可以提供状态 API 接口

### 用例 3：领域边界 —— 线程标记
**输入：** "连击系统导致帧率卡顿；你能添加线程来分散输入处理吗？"
**预期行为：**
- 不单方面实现线程或异步系统
- 将线程问题标记给 `engine-programmer`，附带热路径的清晰描述
- 可以生成一个非线程的重构方案作为安全的中间步骤，减少每帧工作量
- 记录升级情况以便 lead-programmer 知晓

### 用例 4：与已接受 ADR 冲突
**输入：** "将伤害计算改为直接使用浮点累加，而非 ADR-003 中的定点公式。"
**预期行为：**
- 识别出提议的变更违反了 ADR-003（状态为 Accepted 已接受）
- 不静默地实现该违规变更
- 向 `lead-programmer` 标记冲突，附带 ADR 引用和权衡说明
- 仅在 lead-programmer 或 technical-director 明确覆盖决策后方可实施

### 用例 5：上下文传递 —— 按 GDD 规格实现
**输入：** 上下文中提供关于"玩家战斗 (PlayerCombat)"的 GDD。请求："实现战斗 GDD 中的耐力消耗公式。"
**预期行为：**
- 读取所提供 GDD 的公式章节
- 严格按照书面公式实现 —— 不发明新变量或调整系数
- 将耐力消耗设为数据驱动值（外部配置），而非硬编码常量
- 注意 GDD 边缘情况章节中的任何边界情况并在代码中处理

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（机制、玩家系统、战斗）
- [ ] 将领域外请求重定向至正确智能体（ui-programmer、ai-programmer、engine-programmer）
- [ ] 返回结构化结果（代码骨架、方法签名、内联注释）而非自由形式的意见
- [ ] 未经明确委派不修改 `src/gameplay/` 或 `src/core/` 之外的文件
- [ ] 标记 ADR 违规而非静默覆盖
- [ ] 游戏数值数据驱动，绝不硬编码

---

## 覆盖说明 (Coverage Notes)
- 连击系统测试（用例 1）应在 `tests/unit/gameplay/` 中通过单元测试验证
- 线程升级（用例 3）验证智能体不会越界进入引擎领域
- ADR 冲突（用例 4）确认智能体尊重架构治理流程
- 用例 1 和用例 5 共同验证智能体按规格实现而非即兴发挥
