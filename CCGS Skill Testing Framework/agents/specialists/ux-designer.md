# 智能体测试规格：ux-designer（用户体验设计师）


## 智能体概要 (Agent Summary)
领域：用户体验流程、交互设计、信息架构、输入处理设计和入门引导 UX。
不拥有：视觉美术风格（属于 art-director 美术总监）、UI 实现代码（属于 ui-programmer UI 程序员）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用 UX 流程 / 交互设计 / 信息架构）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Glob, Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义未声称对视觉美术方向或 UI 实现代码拥有权限

---

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "为生存游戏设计物品栏管理流程。"
**预期行为：**
- 生成物品栏的用户流程图（状态与转换）：打开、浏览、选择物品、子操作（装备/丢弃/组合）、关闭
- 定义所有交互状态（默认、悬停、选中、空槽位、锁定槽位）
- 为每个动作指定输入映射（键盘、手柄，如适用）
- 说明认知负荷考量（例如：无需滚动即可看见的最大物品数量）
- 不生成视觉设计（颜色、图标）或实现代码

### 用例 2：领域外请求 —— 正确重定向
**输入：** "使用 GDScript 实现带拖拽支持的物品栏界面。"
**预期行为：**
- 不生成实现代码
- 明确声明 UI 代码实现属于 `ui-programmer`
- 将请求重定向至 `ui-programmer`
- 说明应将 UX 流程规格提供给 ui-programmer 作为实现参考

### 用例 3：流程深度冲突 —— 简化
**输入：** "主设计师说当前的 5 步制作流程太深；最多允许 3 步。"
**预期行为：**
- 生成修订后的 3 步流程，合并原有的 5 步序列
- 清楚展示哪些步骤被合并或移除，以及每次合并从可用性角度为何安全
- 不简单地移除步骤而不处理用户在每步被移除步骤上的目标
- 如果 3 步限制使任何必要用例无法实现，则标记并提出替代方案

### 用例 4：可访问性冲突
**输入：** "入门引导流程使用定时提示（3 秒后自动前进）以保持节奏，但这与用户控制时机的可访问性要求冲突。"
**预期行为：**
- 识别出与 WCAG 2.1 2.2.1（可调节时机）的冲突
- 不为了保持节奏而覆盖可访问性要求
- 与 `accessibility-specialist`（无障碍专家）协调以达成合规的解决方案
- 提出替代方案：悬停暂停、跳过按钮、在设置中禁用自动前进的选项

### 用例 5：上下文传递 —— 玩家心智模型研究
**输入：** 上下文中提供的试玩研究报告："玩家始终期望'制作'选项位于物品栏界面内，而非在单独的顶层菜单中。"请求："重新设计制作功能的导航信息架构。"
**预期行为：**
- 引用研究中玩家的具体期望（制作功能应位于物品栏内）
- 重构信息架构，将制作功能作为物品栏界面内的标签页或面板放置
- 除非有明确的理由，否则不生成与所述玩家心智模型矛盾的设计
- 在设计决策说明中注明研究来源

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（UX 流程、交互设计、信息架构、入门引导）
- [ ] 将代码实现重定向至 ui-programmer，视觉风格重定向至 art-director
- [ ] 返回结构化结果（状态图、流程步骤、输入映射）而非自由形式的意见
- [ ] 当流程有时机或认知负荷限制时与 accessibility-specialist 协调
- [ ] 基于提供的用户研究而非假设行为设计流程
- [ ] 针对用户目标记录流程决策的说明

---

## 覆盖说明 (Coverage Notes)
- 物品栏流程（用例 1）应写入 `design/ux/` 作为供 ui-programmer 实现的规格
- 心智模型案例（用例 5）验证智能体应用研究证据而非直觉
- 可访问性协调（用例 4）确认智能体不为了 UX 美学而覆盖可访问性要求
