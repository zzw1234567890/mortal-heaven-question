# 智能体测试规格：network-programmer（网络程序员）


## 智能体概要 (Agent Summary)
领域：多人游戏网络、状态复制、延迟补偿、匹配协议设计和网络消息模式。
不拥有：游戏逻辑（仅负责其网络部分）、服务器基础设施和部署（属于 devops-engineer 运维工程师）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "为 4 人合作游戏设计玩家位置的状态复制方案。"
**预期行为：**
- 生成包含以下内容的同步策略文档：
  - 复制频率（例如：20Hz，带增量压缩）
  - 优先级层级（例如：自身玩家高优先级，其他玩家中优先级）
  - 远程玩家的插值方法（例如：线性插值，100ms 缓冲区）
  - 每玩家每秒带宽估算
- 不自行实现玩家移动逻辑（委托给 gameplay-programmer 玩法程序员）
- 提出航位推算或预测策略以减少可见延迟

### 用例 2：领域外请求 —— 正确重定向
**输入：** "将我们的游戏服务器部署到 AWS EC2 并设置自动扩展。"
**预期行为：**
- 不生成服务器部署配置、Terraform 或 AWS 设置脚本
- 明确声明服务器基础设施属于 `devops-engineer`
- 将请求重定向至 `devops-engineer`
- 可以说明在基础设施搭建好后，可以提供服务器需要实现的网络协议规格

### 用例 3：状态分歧 —— 回滚/协调
**输入：** "在高延迟下，客户端与物理对象的权威服务器状态产生分歧。"
**预期行为：**
- 提出回滚与协调方法（客户端预测 + 服务器权威修正）
- 指定状态快照格式、协调触发阈值（例如：位置误差超过 5 个单位）和修正插值速度
- 注意确定性重放的输入缓冲模式
- 不更改物理模拟本身 —— 记录 engine-programmer 的接口契约

### 用例 4：反作弊冲突
**输入：** "我们希望用客户端权威位置实现平滑移动，但反作弊需要服务器验证。"
**预期行为：**
- 揭示直接冲突：客户端权威快速但易被利用；服务器权威安全但需要延迟补偿
- 与 `security-engineer`（安全工程师）协调以确定验证边界
- 提出折衷方案（服务器在容忍带内验证位置，标记异常值），而非单方面决定
- 记录权衡，如果 security-engineer 和 network-programmer 无法达成一致，则将最终决策升级至 `technical-director`（技术总监）

### 用例 5：上下文传递 —— 延迟预算
**输入：** 上下文中提供技术偏好：目标延迟为第 95 百分位玩家 80ms RTT。请求："为格斗游戏设计输入复制方案。"
**预期行为：**
- 在设计过程中明确引用 80ms RTT 预算
- 选择针对该预算校准的复制方法（例如：在此延迟下格斗游戏首选回滚网络代码）
- 根据 80ms 预算计算输入延迟帧数（例如：60fps 下 2 帧 = 33ms 缓冲区）
- 标记回滚网络代码需要 gameplay-programmer 实现确定性模拟
