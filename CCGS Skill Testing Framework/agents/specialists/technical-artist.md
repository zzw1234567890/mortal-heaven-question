未分配关卡 ID。

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "为敌人死亡序列创建一个溶解效果着色器。"
**预期行为：**
- 生成适合配置引擎的着色器代码或 Shader Graph 节点规格（Godot 着色语言 / Unity Shader Graph / Unreal 材质蓝图）
- 定义 `dissolve_amount` 统一变量（0.0–1.0）作为动画驱动
- 使用噪声纹理采样确定溶解阈值
- 标注边缘光照技术作为可选增强
- 输出具有引擎版本意识（如需截止日期后的 API 则检查版本参考）

### 用例 2：领域外请求 —— 正确重定向
**输入：** "定义美术圣经调色板：UI 的主色、辅色和强调色。"
**预期行为：**
- 不生成调色板决策或美术方向文档
- 明确声明美术风格决策属于 `art-director`
- 将请求重定向至 `art-director`
- 可以说明一旦调色板确定，随后可以实现色彩分级或调色板 LUT 着色器

### 用例 3：性能警告 —— GPU 粒子数量
**输入：** "VFX 系统在爆炸池中触发 GPU 粒子数量警告 —— 50,000 个粒子。"
**预期行为：**
- 生成针对该警告的优化规格
- 提出具体策略：每个发射器的粒子预算上限、基于 LOD 的粒子削减、GPU 实例化，或为远处效果切换至基于网格的 VFX
- 在可计算的情况下提供修复前后的 GPU 成本估算
- 不改变爆炸的游戏行为（任何游戏性影响委托给 gameplay-programmer）

### 用例 4：引擎版本兼容性
**输入：** "为水面着色器使用新的纹理采样器 API。"
**预期行为：**
- 在建议任何 API 之前检查引擎版本参考（例如：`docs/engine-reference/godot/VERSION.md`）
- 如果请求的 API 在知识截止日期之后（例如：Godot 4.4+ 纹理类型变更），则标记
- 提供项目锁定引擎版本的正确语法
- 如果对截止日期后的行为不确定，明确说明不确定性并指向已验证的文档

### 用例 5：上下文传递 —— 使用性能预算
**输入：** 上下文中提供来自 `technical-preferences.md` 的性能预算：GPU 帧预算 2ms、最大绘制调用 200。请求："优化森林渲染系统。"
**预期行为：**
- 引用提供上下文中的 2ms GPU 预算和 200 次绘制调用限制
- 提出针对这些确切目标校准的优化（例如："批处理将绘制调用从 340 降至约 180，在 200 限制之内"）
- 不提出会在其他维度超出所述预算的优化
- 按预期影响与实施成本的对比生成排序的优化列表
