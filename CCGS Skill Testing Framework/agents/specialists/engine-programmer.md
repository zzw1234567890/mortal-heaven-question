## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "为弹丸实现一个自定义对象池，以避免每帧分配。"
**预期行为：**
- 生成引擎级别的对象池实现，包含 acquire/release 接口
- 池化对象类型限定为弹丸类型，使用预分配的固定大小存储
- 提供线程安全说明（或明确标注为仅单线程并提供理由）
- 按照编码标准包含公共 API 的文档注释
- 输出与项目配置的引擎和语言兼容

### 用例 2：领域外请求 —— 正确重定向
**输入：** "添加一个带有音量滑块和'返回主菜单'按钮的暂停菜单界面。"
**预期行为：**
- 不生成 UI 界面代码
- 明确声明菜单界面属于 `ui-programmer`（UI 程序员）
- 将请求重定向至 `ui-programmer`
- 可以说明可以提供引擎级别的音频音量 API 接口供 ui-programmer 调用

### 用例 3：内存泄漏诊断
**输入：** "每次加载关卡后内存使用量增加约 50MB 且永不释放。我们怀疑是资源加载系统的问题。"
**预期行为：**
- 提供系统化的诊断方法：引用计数审计、资源句柄生命周期检查、缓存失效审查
- 识别可能的原因（孤立资源句柄、循环引用、永不驱逐的缓存）
- 针对识别的泄漏模式提供具体的修复方案
- 提供验证修复的测试（加载前内存基线、卸载后测量、确认恢复至基线）

### 用例 4：跨领域协调 —— 共享系统优化
**输入：** "我需要优化物理宽相位检测，但游戏系统与物理查询 API 紧密耦合。"
**预期行为：**
- 不单方面更改物理查询 API 表面（否则会破坏 gameplay-programmer 的代码）
- 与 `lead-programmer`（主程序员）协调以安全地规划变更
- 提出迁移路径：新优化 API 与旧 API 并存，附带弃用周期
- 在继续之前记录协调需求

### 用例 5：上下文传递 —— 检查引擎版本参考
**输入：** 上下文中提供引擎版本参考（Godot 4.6）。请求："为项目设置默认物理引擎。"
**预期行为：**
- 读取引擎版本参考，注意 Godot 4.6 的变更：Jolt 物理引擎现为默认
- 生成考虑 Jolt 成为默认引擎这一变化的配置指导（4.6 迁移说明）
- 标记 GodotPhysics 与 Jolt 之间可能影响现有代码的任何 API 差异
- 不推荐已弃用或 4.6 之前的物理设置步骤而不说明其适用于旧版本
