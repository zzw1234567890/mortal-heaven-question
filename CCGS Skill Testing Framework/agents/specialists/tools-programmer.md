# 智能体测试规格：tools-programmer（工具程序员）


## 智能体概要 (Agent Summary)
领域：编辑器扩展、内容创作工具、调试实用程序和管线自动化脚本。
不拥有：游戏代码（属于 gameplay-programmer 玩法程序员、ui-programmer UI 程序员等）、引擎核心系统（属于 engine-programmer 引擎程序员）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用编辑器工具 / 管线 / 调试实用程序）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义未声称对游戏源代码或引擎内部拥有权限

---

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "创建一个用于在关卡中放置敌人巡逻路点的自定义编辑器工具。"
**预期行为：**
- 为配置的引擎生成编辑器扩展规格和代码骨架（例如：Godot EditorPlugin、Unity Editor 窗口、Unreal Detail Customization）
- 工具允许设计师在场景/视口中点击放置路点
- 路点序列化为引擎原生资源（非硬编码），以便 level-designer 无需代码即可编辑
- 按照编辑器插件最佳实践包含撤销/重做支持
- 不修改 AI 寻路运行时代码（该代码属于 ai-programmer AI 程序员）

### 用例 2：领域外请求 —— 正确重定向
**输入：** "在代码中实现敌人近战连击系统。"
**预期行为：**
- 不生成游戏机制代码
- 明确声明战斗系统实现属于 `gameplay-programmer`
- 将请求重定向至 `gameplay-programmer`
- 可以说明如果在开发中有用，可以构建一个调试覆盖层工具来可视化连击状态

### 用例 3：运行时数据访问 —— 需要协调
**输入：** "路点编辑器工具需要在运行时读取游戏数据，以验证巡逻路线是否符合 AI 预算。"
**预期行为：**
- 识别出从编辑器插件访问运行时数据需要定义安全的游戏运行时系统接口
- 与 `engine-programmer` 协调建立只读数据访问模式（例如：资源验证 API）
- 不直接读取内部引擎或游戏内存结构而无协商接口
- 在实现工具之前记录所需接口

### 用例 4：引擎版本破坏
**输入：** "引擎升级后，路点编辑器工具启动时崩溃。"
**预期行为：**
- 检查引擎版本参考（`docs/engine-reference/`）中编辑器插件 API 的破坏性变更
- 识别在新版本中发生变更的特定 API 或信号
- 针对该破坏性变更生成有针对性的修复
- 标注其他可能受相同 API 变更影响的工具

### 用例 5：上下文传递 —— 美术管线需求
**输入：** 上下文中提供的美术管线需求："所有纹理导入必须将压缩设为 VRAM 压缩（VRAM Compressed）、生成 mipmap 并标记 LOD 组。"请求："构建一个强制执行这些设置的资源导入工具。"
**预期行为：**
- 引用上下文中的所有三个需求：VRAM 压缩、mipmap 生成、LOD 组标记
- 生成一个导入时验证并应用所有三项设置的导入工具
- 对未能满足指定设置的资源添加警告或错误报告
- 不改变美术管线需求本身（这些属于 art-director 美术总监 / technical-artist 技术美术）

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（编辑器工具、管线脚本、调试实用程序）
- [ ] 将游戏代码请求重定向至对应程序员智能体
- [ ] 返回结构化结果（工具规格、编辑器扩展代码、管线脚本）
- [ ] 在从编辑器上下文访问运行时数据之前与 engine-programmer 协调
- [ ] 在使用编辑器插件 API 之前检查引擎版本参考
- [ ] 构建工具以强制执行需求，不自行制定需求

---

## 覆盖说明 (Coverage Notes)
- 路点编辑器工具（用例 1）应进行冒烟测试，验证其在编辑器中加载无误
- 运行时数据访问（用例 3）确认智能体尊重 engine-programmer 对核心 API 的所有权
- 美术管线上下文（用例 5）验证智能体按提供规格构建工具而非自行发明需求
