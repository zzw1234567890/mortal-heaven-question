# 智能体测试规格：prototyper（原型师）


## 智能体概要 (Agent Summary)
- **领域**：`prototypes/` 目录中的快速一次性原型、概念验证实验、机制可行性测试。标准有意放宽以求速度 —— 原型非生产代码。
- **不拥有**：`src/` 中的生产源代码（属于 gameplay-programmer 玩法程序员）、设计文档（属于 game-designer 游戏设计师）、生产级架构决策（属于 lead-programmer 主程序员 / technical-director 技术总监）
- **模型层级**：Sonnet
- **关卡 ID**：无；在原型完成后生成建议文档；不参与阶段关卡

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用快速原型、prototypes/ 目录、一次性代码）
- [ ] `allowed-tools:` 列表与智能体角色匹配（Read/Write 限于 prototypes/ 目录；无生产环境 src/ 写入权限）
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] 智能体定义明确声明原型代码非生产代码，不得复制到 src/

---

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 原型化抽牌机制
**输入**："在 2 小时内原型化一个抽牌机制。核心问题：每回合抽 3 张牌、手牌上限 7 张感觉好吗？我今天需要在试玩中测试。"
**预期行为**：
- 使用项目引擎脚本语言生成最小可行原型，限定于 `prototypes/card-draw-mechanic/`
- 代码优先考虑速度而非正确性：无需单元测试，无需文档注释，原型可接受全局状态
- 实现最小可行机制：牌组、抽牌函数（抽 N 张牌）、带大小限制的手牌容器，以及简单的 UI 或调试输出以验证状态
- 不实现生产模式（依赖注入、信号、数据驱动配置），除非使用它们比不使用耗时更少
- 在原型文件夹中包含 `README.md`：正在测试的假设、如何运行、在试玩中观察什么

### 用例 2：领域外请求 —— 生产级实现
**输入**："抽牌机制原型效果很好。现在为 src/gameplay/cards/ 编写卡牌系统的生产实现。"
**预期行为**：
- 不向 `src/` 写入生产代码
- 明确说明："原型师在 prototypes/ 中生成一次性代码以验证概念；经验证机制的生产实现由 gameplay-programmer 处理"
- 主动提供过渡文档：原型证明了什么、生产实现应保留什么（机制）、应放弃什么（一次性实现模式）
- 不将原型代码复制到 src/ 中，也不建议将其作为起点而不提醒其非生产质量

### 用例 3：原型验证机制 —— 建议输出
**输入**："抽牌原型试玩效果良好。三次体验都享受每回合抽 3 张牌、手牌上限 7 的设定。未观察到混淆。下一步是什么？"
**预期行为**：
- 在 `prototypes/card-draw-mechanic/conclusion.md`（或等效文件）中生成原型结论文档
- 文档包含：已测试的假设、试玩方法（场次、时长、观察者记录）、结果判定（VALIDATED 已验证）、关键发现（有效点、观察到的任何小问题）、生产建议（应保留的具体机制参数：每回合 3 张牌、手牌上限 7），以及将生产实现请求路由至 gameplay-programmer 的标记
- 不开始编写生产代码
- 输出结构化为可决策的建议，而非叙述性总结

### 用例 4：原型揭示机制不可行 —— 放弃说明
**输入**："基于物理的开锁机制原型已完成。经过 4 次试玩，所有测试者都觉得令人沮丧 —— 需要太高精度，不好玩。一名测试者愤而退出。"
**预期行为**：
- 在 `prototypes/lock-picking-physics/conclusion.md` 中生成原型放弃说明
- 文档包含：已测试的假设、结果判定（ABANDONED 已放弃）、具体原因（精度门槛过高、负面情绪反应、愤而退出事件作为证据），以及值得探索的替代方案建议（简化的钥匙-弹子机制、基于节奏的替代方案、完全移除该机制）
- 不因沉没成本而建议坚持该原型机制
- 不将结果标记为不确定 —— 经过 4 次一致负面反馈的试玩，放弃是正确判定

### 用例 5：上下文传递 —— 使用项目引擎脚本语言
**输入上下文**：项目使用 Godot 4.6 与 GDScript（在 technical-preferences.md 中配置）。
**输入**："原型化一个基本网格移动系统 —— 玩家点击格子，角色移动到该格子。"
**预期行为**：
- 使用 GDScript 生成原型 —— 而非 Python、C# 或伪代码
- 使用适用于网格的 Godot 4.6 节点类型：TileMap 或自定义网格管理器节点，玩家使用 CharacterBody2D 或 Node2D
- 不应用生产编码标准（不需要测试覆盖、不需要文档注释、可接受全局状态）
- 将输出写入 `prototypes/grid-movement/` 而非 `src/`
- 如果 Godot 4.6 API 不确定（考虑到 VERSION.md 中注明的 LLM 知识截止日期），标记该特定 API 并附加说明以验证 Godot 4.6 文档

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（仅限 prototypes/ 目录；用于概念验证的一次性代码）
- [ ] 将生产实现请求重定向至 gameplay-programmer，附带过渡文档提议
- [ ] 在原型评估后生成结构化结论文档（VALIDATED 或 ABANDONED 判定）
- [ ] 不建议保留原型代码的生产形式而不给出明确警告
- [ ] 使用项目配置的引擎和脚本语言；标记版本不确定性

---

## 覆盖说明 (Coverage Notes)
- 用例 2（生产重定向）至关重要 —— 原型代码泄漏到 src/ 是常见的质量问题
- 用例 4（放弃的诚实性）测试智能体是否避免沉没成本偏见 —— 失败的原型应被果断放弃
- 用例 5 要求 technical-preferences.md 已配置引擎和语言；如未配置则测试不完整
- 故意放宽编码标准是一个特性，而非差距 —— 不在原型输出中将缺少测试或文档注释标记为失败
- 无自动运行器；手动审查或通过 `/skill-test` 进行
