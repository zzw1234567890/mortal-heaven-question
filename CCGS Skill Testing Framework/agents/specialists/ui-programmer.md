# 智能体测试规格：ui-programmer（UI 程序员）

领域：菜单界面、HUD、物品栏界面、对话框、UI 框架代码和数据绑定。
不拥有：用户体验流程设计（属于 ux-designer 用户体验设计师）、视觉风格方向（属于 art-director 美术总监 / technical-artist 技术美术）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

## 测试用例 (Test Cases)

### 用例 1：领域内请求 —— 正确输出
**输入：** "按照 `design/ux/inventory-flow.md` 中的 UX 规格实现物品栏界面。"
**预期行为：**
- 在生成任何代码之前先读取 UX 规格
- 使用项目配置的 UI 框架实现（UI Toolkit、UGUI、UMG 或 Godot Control 节点）
- 实现规格中定义的所有状态（默认、悬停、选中、空槽位、锁定槽位）
- 通过项目数据模型将物品栏数据绑定到 UI 元素，而非硬编码值
- 按照编码标准包含公共 UI API 的文档注释

### 用例 2：领域外请求 —— 正确重定向
**输入：** "设计物品栏交互流程 —— 玩家装备、丢弃或组合物品时会发生什么。"
**预期行为：**
- 不生成交互流程设计或用户流程图
- 明确声明 UX 流程设计属于 `ux-designer`
- 将请求重定向至 `ux-designer`
- 说明一旦流程规格就绪即可进行实现

### 用例 3：自定义动画协调
**输入：** "物品栏中的物品选中时需要自定义弹跳动画。"
**预期行为：**
- 认识到定义动画曲线和感觉属于 technical-artist 的领域
- 在没有规格的情况下不自行拟定动画参数（时机、缓动）
- 与 `technical-artist` 协调获取动画规格（持续时间、缓动曲线、过冲量）
- 一旦提供规格，生成将动画绑定到选中状态的实现

### 用例 4：模糊的 UX 规格 —— 回传标记
**输入：** UX 规格中说"选中时显示物品详情"，但未定义选中空槽位时的行为。
**预期行为：**
- 识别出规格中的模糊之处（空槽位选中状态未定义）
- 不针对未定义状态做出任意的实现决策
- 将模糊之处回传给 `ux-designer`，附带具体问题："当选中空物品栏槽位时，详情面板应显示什么？"
- 可以提出两种常见选项（隐藏面板 / 显示占位符）以帮助 ux-designer 快速决策

### 用例 5：上下文传递 —— 引擎 UI 工具包
**输入：** 提供的引擎上下文：项目使用 Godot 4.6 的 Control 节点 UI。请求："为物品栏实现一个可滚动的物品列表。"
**预期行为：**
- 使用 Godot 的 `ScrollContainer` + `VBoxContainer` + `ItemList`（或等效方式）模式，而非 Canvas 或 UGUI
- 不生成 Unity UGUI 或 Unreal UMG 代码用于 Godot 项目
- 在使用特定 API 之前检查引擎版本参考（4.6）中是否有 Control 节点的 API 变更（相比 4.4/4.5）
- 生成与项目配置语言一致的 GDScript 或 C# 代码

## 协议合规 (Protocol Compliance)

- [ ] 保持在声明领域内（菜单、HUD、UI 框架、数据绑定）
- [ ] 将 UX 流程设计重定向至 ux-designer
- [ ] 在实现动画之前与 technical-artist 协调动画规格
- [ ] 将模糊的 UX 规格回传给 ux-designer，而非做出任意的实现决策
- [ ] 返回结构化输出（实现代码、数据绑定模式、UI 状态的状态机）
- [ ] 对项目使用正确的引擎 UI 工具包 —— 绝不产生跨引擎代码

## 覆盖说明 (Coverage Notes)
- 物品栏实现（用例 1）应在 `production/qa/evidence/` 中包含 UI 交互测试或手动演练文档
- 动画协调（用例 3）确认智能体在没有规格的情况下不自行拟定手感参数
- 模糊规格（用例 4）验证智能体将规格缺口回传给编写智能体而非自行猜测
