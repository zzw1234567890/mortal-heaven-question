
# 智能体测试规格：community-manager（社区经理）

## 智能体概述
- **领域 (Domain)**：面向玩家的沟通 —— 补丁说明 (patch notes)、社交媒体帖子、社区更新、玩家反馈收集、来自玩家的 Bug 报告分类、危机沟通 (crisis communication)
- **不负责**：修复所报告的错误（gameplay-programmer 或对应引擎专家）、游戏设计与平衡决策（game-designer）、商业决策（producer 或领导层）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将危机沟通中敏感话题升级至 producer

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及玩家沟通、补丁说明、社区管理）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持写入沟通内容；无游戏代码或服务器工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对 Bug 修复、游戏设计决策或业务策略具有权威

---

## 测试用例

### 用例 1：领域内请求 —— 补丁说明（小更新）
**输入**："我们需要发布补丁说明，内容是关于修复了一个崩溃问题以及两个平衡性调整：链锯伤害从 25 降低到 20，散弹枪后坐力减少 15%。"
**预期行为**：
- 生成适合发布的结构化补丁说明
- 包含标准元素：版本号、日期、类别标题（Bug 修复、平衡性调整）
- 以玩家友好、专业的方式描述每个变更
- 明确说明修复了哪个崩溃问题（例如"修复了进入 3 号地图时发生的罕见崩溃问题"）
- 保持语调积极且透明

### 用例 2：领域外请求 —— 修复所报告的错误
**输入**："一位玩家报告说，冲刺时喝药水会使药水永久留在屏幕上。请修复这个问题。"
**预期行为**：
- 不编写修复问题的游戏代码
- 确认错误报告已记录，并将其分配给适当的开发团队或专家
- 描述 Bug 分流流程：重现 → 记录 → 分类 → 排期修复
- 如有必要，发送后续信息请求，以帮助复现该问题

### 用例 3：领域边界 —— 社区强烈反对
**输入**："我们刚刚削弱了所有玩家最喜欢的武器，社区在 Reddit 和 Discord 上非常愤怒。
**预期行为**：
- 认识到这是一场需要立即关注的社区危机
- 起草回应：承认反馈、解释变更背后的原因（如果存在合理理由）、确认玩家在意游戏健康度
- 如适用，指明对话将交由 game-designer 或 producer 进行潜在调整
- 不承诺回滚更改（这是游戏设计决策）
- 建议沟通时间表：XX 小时内响应，XX 天内发布后续说明（如适用）

### 用例 4：冲突 —— 补丁说明中的品牌语调冲突
**输入**："这是我们内部关于新货币化功能的说明。我们的品牌语调是 '有趣且随意'，但这个说明读起来像是法律文件。"
**预期行为**：
- 识别语调不一致的问题
- 重写补丁说明，使其与公司既定的品牌语调保持一致
- 不改变所传达的功能 —— 仅改变表达方式
- 添加有关货币化变更的透明声明（如适用）

### 用例 5：上下文传递 —— 使用品牌语调文档
**输入上下文**：项目在品牌文档中定义了品牌语调：'友好、随意、社区第一'。使用玩家社区的昵称（例如将玩家称为 'Survivors'）。
**输入**："编写关于我们新赛季的补丁说明，包括新地图和新武器。"
**预期行为**：
- 在补丁说明中使用定义的语调：友好、随意、社区第一
- 使用 'Survivors' 称呼玩家
- 保持专业的沟通标准，同时确保语调轻松
- 在沟通中体现对该赛季的兴奋感

---

## 协议合规

- [ ] 保持在声明的领域内（玩家沟通、补丁说明、社区管理）
- [ ] 将错误修复请求重定向给相应的开发团队/专家
- [ ] 返回结构化的沟通成果（补丁说明、危机回应），而非模糊的建议
- [ ] 标记危机情况，并在必要时升级至 producer
- [ ] 使用项目定义的品牌语调；不偏离已有的声音风格

---

## 覆盖范围说明
- 用例 3（危机沟通）是价值最高的测试 —— 沟通失误可能造成严重的声誉损害
- 用例 4 和 5 测试语调一致性，这对于品牌完整性至关重要
- 用例 5 是最重要的上下文感知测试；智能体必须使用项目中定义的语调
- 无自动运行器；手动审查或通过 `/skill-test` 进行
