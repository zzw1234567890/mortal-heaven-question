
# 智能体测试规格：localization-lead（本地化主管）

## 智能体概述
- **领域 (Domain)**：国际化架构设计 (internationalization/i18n architecture)、字符串提取与管理工作流、语言环境特定测试策略、翻译管线与工具设计、区域格式（日期、数字、货币）处理
- **不负责**：实际文本翻译（由本地化服务商/译员完成）、UI 布局（ux-designer 或 ui-programmer 处理文本扩展）、游戏逻辑变更以适应本地化
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将破坏性架构变更升级至 lead-programmer

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及国际化、字符串管理、语言环境测试、翻译）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持读写本地化配置和字符串文件；无游戏引擎或设计工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对实际翻译、UI 布局或游戏逻辑具有权威

## 测试用例

### 用例 1：领域内请求 —— 针对 Unity 的字符串提取管道设计
**输入**："为我们的 Unity 项目设计一个字符串提取管道。"
**预期行为**：
- 建议使用 Unity 的本地化包 (Unity Localization Package) 或开源替代方案
- 描述提取流程：从 UI 微件/场景中扫描字符串 → 导入到本地化表格 → 导出为翻译格式
- 说明键命名约定（例如 HUD_HEALTH_LABEL、MENU_PLAY_BUTTON）—— 不要将原始文本作为键
- 提及在本地化后验证没有字符串丢失的自动化检查
- 讨论非英语语言环境的文本扩展（例如"Start"翻译成西班牙语后可能是"Iniciar"或"Inicio"，长度不同）

### 用例 2：领域外请求 —— 对话翻译
**输入**："将以下游戏对话翻译成法语："Hello there, adventurer! Welcome to our village.""
**预期行为**：
- 不提供实际翻译
- 明确说明："翻译工作由我们的本地化服务商或专业译员完成；我负责设计确保翻译能正确集成到游戏中的系统和管道"
- 可选地，说明该字符串将如何通过本地化管道（提取 → 发送给译员 → 导入回游戏）进行处理

### 用例 3：领域边界 —— 缺少俄语复数形式
**输入**："在俄语中，我们的 'You found {n} coins!' 字符串显示为 'You found 1 coins' 和 'You found 3 coins'。复数形式错误。"
**预期行为**：
- 识别根本原因：该字符串未处理俄语复数规则（俄语有单数、少量、复数三种复数形式，可能为 1/2-4/5+ 模式）
- 建议在本地化框架中使用复数感知的消息格式（例如 ICU MessageFormat 或 Unity Localization 的复数选择器）
- 提供正确的键设计：使用 `{n} {n, plural, one {coin} few {coins} other {coins}}` 格式（或框架的等效方法）
- 不硬编码复数逻辑 —— 应在本地化框架内处理

### 用例 4：冲突 —— 字符串键命名冲突
**输入**："我们的两个不同 UI 屏幕 'Settings' 和 'Shop' 都使用键 'TITLE' 来表示各自标题。当语言切换时，其中一个会显示错误的标题。"
**预期行为**：
- 识别根本原因：短而不明确的键名 'TITLE' 在两个屏幕之间冲突
- 建议采用带作用域的键命名约定：'SCREEN_SETTINGS_TITLE' 和 'SCREEN_SHOP_TITLE'
- 提供将所有字符串迁移到带作用域键的迁移计划
- 对于遗留字符串，建议同时保留新旧键并逐步替换，以最小化中断
- 说明字符串键如何在提取管道中进行验证以防止未来的冲突

### 用例 5：上下文传递 —— 为 RTL 语言调整管道
**输入上下文**：项目计划在发布后 6 个月内推出阿拉伯语本地化。
**输入**："从现在起，我们的本地化管道需要做哪些准备来支持阿拉伯语？"
**预期行为**：
- 使用上下文中的信息（6 个月窗口，阿拉伯语支持）
- 讨论 RTL（从右到左）文本渲染的技术要求：UI 镜像、文本对齐反转、数字在 RTL 上下文中的处理方式
- 建议当前的 UI 系统应设计为支持从右到左的布局，而不是事后修改
- 讨论阿拉伯语特定的排版注意事项（连字、变形形式的正确字体）
- 提供从现在到阿拉伯语发布之前的关键里程碑时间表

## 协议合规

- [ ] 保持在声明的领域内（国际化、字符串管理、语言环境测试、管道）
- [ ] 将翻译请求重定向给专业译员/本地化服务商
- [ ] 返回结构化发现（提取管道、键命名方案、语言环境测试策略），而非自由格式意见
- [ ] 主动解决复数规则和 RTL 支持等语言环境特定问题
- [ ] 使用项目提供的语言环境和时间线数据；不凭空假设

---

## 覆盖范围说明
- 用例 3（缺少俄语复数形式）是常见的本地化失败模式 —— 中文和日语工作室经常忽略欧洲语言的复数规则
- 用例 4（键命名冲突）是长期项目中常见的工程问题
- 用例 5 是最重要的上下文感知测试；智能体必须使用 6 个月时间线来提出可行的分阶段计划
- 无自动运行器；手动审查或通过 `/skill-test` 进行
