
# 智能体测试规格：analytics-engineer（分析工程师）

## 智能体概述
- **领域 (Domain)**：遥测系统设计 (telemetry system design)、事件跟踪架构、A/B 测试框架与分析方法、玩家行为分析技术、仪表盘与报告规范
- **不负责**：数据基础架构（devops-engineer 处理数据管道基础设施）、UI 实现（ux-designer 或 ui-programmer）、游戏后端实现（gameplay-programmer）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将数据管道基础设施和部署升级至 devops-engineer

## 测试用例

### 用例 1：领域内请求 —— 教程事件跟踪设计
**输入**："为我们的玩家教程设计遥测跟踪。我们需要知道玩家在哪里卡住以及在哪里流失。"
**预期行为**：
- 生成一个事件跟踪计划：在教程流程的关键节点上报事件（教程开始、步骤完成、步骤 N 退出、教程完成）
- 为每个事件指定事件结构：event_name、timestamp、session_id、player_id、tutorial_step、step_duration、exit_reason（如适用）
- 描述如何分析流失：对比事件计数，识别教程中玩家离开的步骤
- 如已知，参考现有的分析框架/平台；否则建议通用结构
- 输出结构化（事件表、分析计划），而非自由格式意见

### 用例 2：领域外请求 —— 实现跟踪
**输入**："编写将所有教程事件发送到我们分析后端的游戏代码。"
**预期行为**：
- 不生成游戏引擎代码以实现跟踪调用
- 明确说明："我设计事件模式和分析策略；由 gameplay-programmer 在引擎中实现跟踪调用"
- 提供事件规范文档，以便实现者准确知道要发送哪些数据

### 用例 3：领域边界 —— UI 变更的 A/B 测试设计
**输入**："设计一个 A/B 测试，验证新的主菜单布局相较于旧布局是否能提升商店转化率。"
**预期行为**：
- 设计 A/B 测试结构：控制组（旧 UI）与实验组（新 UI），两组同时运行且均分流量
- 定义目标指标：商店转化率（进入商店后购买的玩家百分比）、每个浏览会话的平均收入
- 描述统计要求：样本量估算、显著效果所需的预计玩家数量、最小可检测效应 (Minimum Detectable Effect, MDE)
- 说明需要与 devops-engineer 直接协调，以正确分割玩家流量并确保玩家在会话间分组一致
- 不指定 UI 如何改变 —— 那是 ux-designer 的领域

### 用例 4：冲突 —— 重叠的 A/B 测试片段导致无效数据
**输入**："我们同时在运行三个 A/B 测试，它们都在修改大致相同的玩家群。转化率数据看起来很奇怪。"
**预期行为**：
- 将冲突识别为典型的数据完整性问题：重叠的 A/B 测试创建了交互效应，使结果无法解读
- 建议 A/B 测试时序策略：依次运行测试（一次性一个），或在能确保段完全不重叠的情况下运行测试
- 建议使用测试段标识标记所有事件，以便在分析阶段进行筛选
- 鉴于当前数据可能已存在干扰，建议清理并重启测试

### 用例 5：上下文传递 —— 事件模式一致性
**输入上下文**：项目已有事件模式：`event_name: string`、`timestamp: ISO8601`、`session_id: uuid`、`player_id: string`、`properties: map`。
**输入**："为新的钓鱼小游戏添加事件跟踪。"
**预期行为**：
- 使用现有的事件模式 —— 不发明新的框架结构
- 生成符合现有 schema 的事件：新事件遵循 `event_name` + `timestamp` + `session_id` + `player_id` + `properties` 格式
- 为钓鱼小游戏的具体事件命名（例如 fishing_cast、fish_caught、fish_escaped）
- 为每个事件指定 `properties` 映射中的内容（例如 bait_type、fish_species、cast_distance）
- 将事件模式记录为引用项目中已确立的规范

## 协议合规

- [ ] 保持在声明的领域内（跟踪设计、分析、A/B 测试方法）
- [ ] 将遥测的代码实现请求重定向给 gameplay-programmer
- [ ] 返回结构化发现（事件模式表 + 分析计划），而非自由格式意见
- [ ] 在 A/B 测试设计和冲突上主动与 devops-engineer 协调
- [ ] 使用项目现有的事件模式；不发明新的基础架构

## 覆盖范围说明
- 用例 4（重叠的 A/B 测试）是分析中成本最高的失效模式之一；在测试活跃期优先验证
- 用例 5 是上下文感知测试；智能体必须遵循既定事件模式，不得忽略
- 用例 3 要求了解统计学概念（MDE、样本量、显著性）—— 如不熟悉则预期失败
- 无自动运行器；手动审查或通过 `/skill-test` 进行
