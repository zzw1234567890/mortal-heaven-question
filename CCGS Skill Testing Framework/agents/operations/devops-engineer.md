
# 智能体测试规格：devops-engineer（DevOps 工程师）

## 智能体概述
- **领域 (Domain)**：CI/CD 管线、构建脚本与自动化、版本控制工作流、部署基础设施、分支策略、平台构建目标管理
- **不负责**：游戏实现细节（gameplay-programmer 或对应引擎专家）、UI 设计（ux-designer）、游戏网络协议设计（network-programmer）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将架构变更升级至 lead-programmer 或 technical-director

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及 CI/CD、构建管线、部署、版本控制）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持构建系统、CI 配置文件、部署脚本；无游戏引擎或设计工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对游戏逻辑、UI 或网络协议设计具有权威

## 测试用例

### 用例 1：领域内请求 —— 为 Godot 项目配置 CI
**输入**："为我们的 Godot 4.6 项目设置一个 GitHub Actions CI 流程，用于运行 GDScript 测试。"
**预期行为**：
- 生成 GitHub Actions 工作流程或等效的 CI 配置
- 包含所述引擎（Godot 4.6）的正确 setup 步骤
- 使用 Godot 的 headless 模式触发测试执行：`godot --headless --script`
- 包括适当的 trigger（推送到 main、PR）
- 必要时包含缓存策略，以缩短构建时间

### 用例 2：领域外请求 —— 游戏网络实现
**输入**："编写客户端-服务器网络同步代码，实现玩家的位置更新。"
**预期行为**：
- 不生成游戏网络代码
- 明确说明："网络实现由 network-programmer 负责处理；我支持 CI/CD 对多人游戏服务器的构建和部署"
- 提供构建和部署网络服务器二进制文件的策略，但绝不编写网络逻辑

### 用例 3：领域边界 —— 构建失败诊断
**输入**："我们的 CI 构建在 'Asset Compilation' 步骤失败，错误信息是 'Shader compilation failed: missing include file particle_utils.h'。"
**预期行为**：
- 识别问题根本原因：着色器文件缺少包含依赖（没有在构建环境中正确复制或引用）
- 提供包含错误的文件中搜索缺失文件的建议
- 解决 CI 工作中的依赖缺失，或说明如何追踪缺失的包含路径
- 不尝试修复实际的着色器代码（由 technical-artist 处理）
- 保持清晰的边界：构建脚本 vs. 源内容

### 用例 4：冲突 —— 分支策略冲突
**输入**："我们同时使用 Git Flow（develop → feature → develop → release → main）和 trunk-based 开发（所有人都推送到 main）。团队进度很混乱。"
**预期行为**：
- 将其识别为分支策略冲突的根本原因
- 不建议维持两种策略（会导致混乱和合并问题）
- 针对项目当前状态，推荐单一策略（可能是 trunk-based 开发简化版）
- 解释所选策略的影响：合并频率、发布分支的处理方式
- 向上级提出建议作为待定决策

### 用例 5：上下文传递 —— 特定平台的构建矩阵
**输入上下文**：项目将 Windows、Linux 和 Nintendo Switch 列为目标平台。
**输入**："为我们的项目设置 CI 构建矩阵。"
**预期行为**：
- 为目标列表中的每个平台配置构建矩阵条目
- 使用正确的工具链标志配置每个平台的构建
- 包含每个平台的构建成功/失败报告
- 为 Switch 目标添加注释，说明如果 Switch 开发工具包 (NDK) 在 CI 环境中不可用，则可能需要特殊配置
- 引用项目中声明的平台列表

## 协议合规

- [ ] 保持在声明的领域内（CI/CD、构建管线、版本控制）
- [ ] 将游戏网络代码请求重定向给 network-programmer
- [ ] 返回结构化发现（构建脚本、CI 配置、版本控制策略），而非模糊的建议
- [ ] 诊断构建失败，但不修复游戏代码
- [ ] 在配置构建目标时使用项目定义的目标平台列表

---

## 覆盖范围说明
- 用例 3（构建失败诊断）是日常 DevOps 工作流中最核心的测试
- 用例 4（分支策略冲突）需要了解版本控制策略 —— 对于不熟悉 Git Flow 或 trunk-based 开发的智能体可能失败
- 用例 5 是关于平台配置方面最重要的上下文感知测试
- 无自动运行器；手动审查或通过 `/skill-test` 进行
