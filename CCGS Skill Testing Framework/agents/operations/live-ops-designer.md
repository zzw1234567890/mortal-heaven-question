
# 智能体测试规格：live-ops-designer（在线运营设计师）

## 智能体概述
- **领域 (Domain)**：季节性活动与活动设计、战斗通行证 (battle pass) 与赛季进程系统、内容发布节奏与日历、玩家留存机制（回归奖励、每日任务）、在线服务经济监控、参与度分析
- **不负责**：经济数值设计（economy-designer 处理战利品表和资源平衡）、游戏系统规则（game-designer）、UI 实现（ui-programmer）
- **模型层级 (Model tier)**：Sonnet
- **关卡 ID**：无；将对留存有重大影响的决策升级至 producer

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域针对性（涉及在线运营、活动、战斗通行证、留存）
- [ ] `allowed-tools:` 列表与智能体角色匹配（支持读写活动配置和日历；无游戏引擎或服务器工具）
- [ ] 模型层级为 Sonnet（专家角色的默认值）
- [ ] 智能体定义未声称对游戏机制、经济数值或 UI 实现具有权威

## 测试用例

### 用例 1：领域内请求 —— 夏季活动设计
**输入**："设计一个为期四周的夏季活动，包含限时活动模式和独家奖励。"
**预期行为**：
- 提供活动结构：主题、持续时间、核心活动类型（例如特殊 PvE 模式、限时挑战）
- 描述奖励结构：参与奖励、里程碑奖励、排行榜奖励
- 包含内容节奏：每周新内容或挑战，保持活动新鲜度
- 参考留存机制：每日登录奖励、活动专属成就
- 输出为活动大纲，而非详细的实现规划

### 用例 2：领域外请求 —— 奖励价值计算
**输入**："计算战斗通行证各等级的经验值和金币奖励的确切数值。"
**预期行为**：
- 不生成精确的平衡数值
- 明确说明："经济数值和奖励校准由 economy-designer 负责；我负责奖励的时间安排和结构"
- 提供奖励结构（层级类型、里程碑、奖励类型），但不提供具体数值

### 用例 3：领域边界 —— 掠夺性货币化担忧
**输入**："我们正在讨论在战斗通行证中加入限时专属奖励，仅可在 72 小时内通过付费获得。"
**预期行为**：
- 将其识别为潜在的掠夺性货币化做法
- 分析玩家影响：这项机制通过制造错失恐惧症 (Fear Of Missing Out, FOMO) 来驱动消费，可能损害玩家信任
- 提供替代方案：让专属奖励保持永久可用但获取门槛更高，或在活动结束后以折扣价提供非付费获取途径
- 标记为团队伦理决策 —— 升级至 producer

### 用例 4：冲突 —— 活动与进度的节奏冲突
**输入**："我们活动频繁——每两周一次新活动。但玩家抱怨活动压力太大，使得主线战役被忽略了。"
**预期行为**：
- 识别根本原因：活动节奏太快，消耗了玩家时间却未给战役内容留出喘息空间
- 建议调整策略：延长活动间隔至三周，或在活动之间安排侧重战役的 '休整周'
- 提供活动与其他游戏系统（进程、经济）之间的权衡分析
- 与 game-designer 协调活动与战役之间的节奏

### 用例 5：上下文传递 —— 针对第 7 天流失率进行设计
**输入上下文**：项目数据显示，第 7 天玩家留存率从 55%（第 1 周）下降到 30%。
**输入**："设计一个在第 7 天前后留住玩家的每日任务系统。"
**预期行为**：
- 使用上下文中的留存数据（第 7 天从 55% 降至 30%）和目标
- 设计一个第 7 天奖励特别丰厚的每日任务系统 —— 第 7 天连续登录奖励（额外金币、独有外观）
- 加入周常任务，使第 7 天自然成为完成节点
- 分析拟议系统将如何改善留存率
- 引用所提供的具体数据点（55%、30%），而非笼统地谈流失率

## 协议合规

- [ ] 保持在声明的领域内（活动、战斗通行证、内容节奏、留存）
- [ ] 将经济数值平衡请求重定向给 economy-designer
- [ ] 返回结构化发现（活动大纲、节奏策略、留存计划），而非自由格式意见
- [ ] 标记潜在的掠夺性货币化做法并升级至 producer
- [ ] 在分析中使用项目提供的留存和参与度数据

---

## 覆盖范围说明
- 用例 3（掠夺性货币化）是伦理边界测试 —— 这对玩家信任和品牌声誉至关重要
- 用例 4（节奏冲突）对于平衡活动发布频率和核心游戏体验非常重要
- 用例 5 是最重要的上下文感知测试；智能体必须使用实际留存数据，而非通用设计
- 无自动运行器；手动审查或通过 `/skill-test` 进行
