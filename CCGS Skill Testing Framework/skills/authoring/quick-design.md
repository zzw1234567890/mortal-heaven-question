
# 技能测试规格：/quick-design

## 技能概述

`/quick-design` 为那些规模太小而不需要完整 8 章节 GDD 的功能生成轻量级设计规格。目标范围是单个系统功能的设计时间不超过 4 小时。与完整的 8 章节 GDD 格式不同，快速设计规格采用精简的 3 章节格式：概述 (Overview)、规则 (Rules) 和验收标准 (Acceptance Criteria)。

该技能没有总监关卡 —— 增加关卡开销会违背轻量级设计工具的初衷。在将设计说明写入 `design/quick-notes/[name].md` 之前，技能会询问"我可以写入吗？"。如果功能范围对于快速设计来说过大，技能会转而引导至 `/design-system`。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证 —— 无需测试夹具。

- [ ] 包含必需的前置元数据字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 包含 ≥2 个阶段标题
- [ ] 包含判定关键词：CREATED、BLOCKED、REDIRECTED
- [ ] 包含"我可以写入吗"（May I write）协作协议措辞（用于快速说明文件）
- [ ] 末尾包含下一步交接说明
- [ ] 明确注明：无总监关卡（固有轻量级技能）
- [ ] 提及范围检查：如果范围超过 4 小时阈值，则转向 `/design-system`

---

## 总监关卡检查

无总监关卡 —— 此技能不启动任何总监关卡代理。快速设计的轻量特性意味着固有地不设总监关卡开销。对于 4 小时以内的单系统功能，无需完整的 GDD 审查。

---

## 测试用例

### 用例 1：正常路径 —— 小型 UI 变更产生 3 章节规格

**测试夹具：**
- 不存在针对目标功能的现有快速设计说明
- 功能范围明确：一个不涉及跨系统影响的单 UI 元素变更

**输入：** `/quick-design [feature-name]`

**预期行为：**
1. 技能询问范围界定问题：涉及什么系统、什么变更、验收信号是什么
2. 技能确定范围在 4 小时阈值以内
3. 技能起草 3 章节规格：概述 (Overview)、规则 (Rules)、验收标准 (Acceptance Criteria)
4. 草稿展示给用户
5. 询问"我可以写入 `design/quick-notes/[name].md` 吗？"
6. 批准后写入文件

**断言：**
- [ ] 规格恰好包含 3 个章节：概述、规则、验收标准
- [ ] 在询问"我可以写入吗？"之前，草稿展示给用户
- [ ] 在写入前询问"我可以写入 `design/quick-notes/[name].md` 吗？"
- [ ] 文件写入到正确的路径：`design/quick-notes/[name].md`
- [ ] 成功写入后判定结果为 CREATED

---

### 用例 2：失败路径 —— 范围检查失败；转向 /design-system

**测试夹具：**
- 所描述的功能跨越多个系统，或设计时间超过 4 小时（例如，"重新设计整个战斗系统"或"影响所有职业的新进度机制"）

**输入：** `/quick-design [large-feature]`

**预期行为：**
1. 技能询问范围界定问题
2. 技能确定范围超过 4 小时/单系统阈值
3. 技能输出："此功能对于快速设计来说过大。请使用 `/design-system [name]` 创建完整 GDD。"
4. 技能不写入快速设计说明文件
5. 判定结果为 REDIRECTED

**断言：**
- [ ] 技能检测到范围超限并在起草前停止
- [ ] 消息明确将 `/design-system` 列为正确的替代方案
- [ ] 不写入快速设计说明文件
- [ ] 判定结果为 REDIRECTED（而非 CREATED 或 BLOCKED）

---

### 用例 3：边界情况 —— 文件已存在；提供更新选项

**测试夹具：**
- `design/quick-notes/[name].md` 已从之前的会话中存在

**输入：** `/quick-design [name]`

**预期行为：**
1. 技能检测到现有的快速设计说明文件并读取其当前内容
2. 技能询问："[name].md 已存在。更新它，还是创建新版本？"
3. 用户选择更新
4. 技能显示现有规格并询问要修订哪个章节
5. 显示更新后的规格，询问"我可以写入吗？"，批准后更新文件

**断言：**
- [ ] 在提供更新选项之前，技能检测并读取现有文件
- [ ] 向用户提供更新或创建新文件的选项 —— 不会自动覆盖
- [ ] 仅更新修订后的章节（如果用户选择完全重写，则更新整个规格）
- [ ] 在覆盖现有文件之前询问"我可以写入吗？"

---

### 用例 4：边界情况 —— 未提供参数

**测试夹具：**
- `design/quick-notes/` 目录可能存在也可能不存在

**输入：** `/quick-design`（无参数）

**预期行为：**
1. 技能检测到未提供参数
2. 技能输出用法错误："未指定功能名称。用法：/quick-design [feature-name]"
3. 技能提供示例：`/quick-design pause-menu-settings`
4. 不创建任何文件

**断言：**
- [ ] 未提供参数时，技能输出用法错误
- [ ] 显示一个使用正确格式的用法示例
- [ ] 不写入快速设计说明文件
- [ ] 技能不会静默选择一个功能名称或默认为任何操作

---

### 用例 5：总监关卡 —— 无关卡启动；明确注明适用于 4 小时以内的功能

**测试夹具：**
- 功能在快速设计范围内
- `production/session-state/review-mode.txt` 存在且包含 `full`

**输入：** `/quick-design [feature-name]`

**预期行为：**
1. 技能询问范围界定问题并确定范围在阈值内
2. 技能不读取 `production/session-state/review-mode.txt`
3. 技能不启动任何总监关卡代理
4. 起草规格，询问"我可以写入吗？"，批准后写入文件
5. 输出明确注明："无总监关卡审查 —— 快速设计适用于 4 小时以内的功能"

**断言：**
- [ ] 不启动任何总监关卡代理（无 CD-、TD-、PR-、AD- 前缀的关卡）
- [ ] 技能不读取 `production/session-state/review-mode.txt`
- [ ] 输出包含解释为何无需关卡审查的说明
- [ ] 审查模式对此技能的行为无影响
- [ ] 完整的 GDD 审查路径（`/design-system`）被提及作为较大功能的替代方案

---

## 协议合规性

- [ ] 在起草前运行范围检查（如果范围过大，转向 `/design-system`）
- [ ] 使用 3 章节格式（概述、规则、验收标准）—— 而非 8 章节 GDD 格式
- [ ] 在询问"我可以写入吗？"之前，草稿展示给用户
- [ ] 在写入前询问"我可以写入 `design/quick-notes/[name].md` 吗？"
- [ ] 无总监关卡 —— 不读取 review-mode.txt
- [ ] 以下一步交接结束（例如，继续实施或 `/dev-story`）

---

## 覆盖说明

- 范围阈值启发式规则（4 小时以内、单系统）是一个判断调用 —— 技能内部检查是权威定义，不通过计算小时数进行独立测试。
- `design/quick-notes/` 目录在不存在时自动创建 —— 此文件系统行为在此未独立测试。
- 与故事管线的集成（快速设计能否直接生成故事？）超出了本规格的范围 —— 快速设计是独立的。
