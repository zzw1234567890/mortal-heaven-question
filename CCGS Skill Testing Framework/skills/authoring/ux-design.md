
# 技能测试规格：/ux-design

## 技能概述

`/ux-design` 是一个引导式、逐节编写的用户体验规格 (UX Spec) 创作技能。它为指定的屏幕或 HUD 元素生成用户流程（以文本形式描述）、交互状态定义、线框描述和无障碍说明。该技能遵循"骨架优先"模式：立即创建包含所有章节标题的文件，然后通过讨论填充每个章节，并在用户批准后逐一写入磁盘。

该技能没有内联总监关卡 —— `/ux-review` 是独立的审查步骤。每个章节需要询问"我可以将章节 [N] 写入 [文件路径] 吗？"。如果命名屏幕的用户体验规格已存在，技能提供改造各章节而非替换的选项。当所有章节写入完成时，判定结果为 COMPLETE。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证 —— 无需测试夹具。

- [ ] 包含必需的前置元数据字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 包含 ≥2 个阶段标题
- [ ] 包含判定关键词：COMPLETE
- [ ] 包含逐节的"我可以写入吗"（May I write）措辞
- [ ] 包含下一步交接说明（例如 `/ux-review` 以验证完成的规格）

---

## 总监关卡检查

无。`/ux-design` 没有内联总监关卡。`/ux-review` 是在此技能完成后调用的独立审查技能。

## 测试用例

### 用例 1：正常路径 —— 新 HUD 规格，所有章节编写并写入

**测试夹具：**
- `design/ux/` 中不存在现有的 HUD 用户体验规格
- 引擎和渲染偏好已配置

**输入：** `/ux-design hud`

**预期行为：**
1. 技能创建骨架文件 `design/ux/hud.md`，包含所有章节标题
2. 技能讨论并起草每个章节：用户流程 (User Flows)、交互状态 (Interaction States)（正常/悬停/聚焦/禁用）、线框描述 (Wireframe Description)、无障碍说明 (Accessibility Notes)
3. 每个章节起草且用户确认后，技能询问"我可以将章节 [N] 写入 `design/ux/hud.md` 吗？"
4. 每个章节在批准后按顺序写入
5. 所有章节写入后，判定结果为 COMPLETE
6. 技能建议下一步运行 `/ux-review`

**断言：**
- [ ] 首先创建骨架文件（包含空的章节主体）
- [ ] 逐节询问"我可以写入章节 [N] 吗？"（而非在最后一次性询问）
- [ ] 所有必需章节均存在：用户流程 (User Flows)、交互状态 (Interaction States)、线框描述 (Wireframe Description)、无障碍说明 (Accessibility Notes)
- [ ] 末尾包含指向 `/ux-review` 的交接
- [ ] 判定结果为 COMPLETE

### 用例 2：现有用户体验规格 —— 改造：用户选择要更新的章节

**测试夹具：**
- `design/ux/hud.md` 已存在，所有章节均已填充
- 用户只想更新无障碍说明 (Accessibility Notes) 章节

**输入：** `/ux-design hud`

**预期行为：**
1. 技能读取现有的 `design/ux/hud.md`，检测到所有章节均已填充
2. 技能报告："HUD 的用户体验规格已存在 —— 提供改造选项"
3. 技能列出所有章节并询问要更新哪个
4. 用户选择无障碍说明 (Accessibility Notes)
5. 技能起草更新后的无障碍内容，并询问"我可以将无障碍说明章节写入 `design/ux/hud.md` 吗？"
6. 仅更新该章节；其他章节保持不变；判定结果为 COMPLETE

**断言：**
- [ ] 检测到现有规格并提供改造选项
- [ ] 用户选择要更新的章节
- [ ] 仅更新选定的章节 —— 其他章节不变
- [ ] 为更新的章节询问"我可以写入吗？"
- [ ] 判定结果为 COMPLETE

---

### 用例 3：依赖缺失 —— 规格引用了一个没有设计文档的系统

**测试夹具：**
- 用户正在为物品栏屏幕编写用户体验规格
- `design/gdd/inventory.md` 不存在

**输入：** `/ux-design inventory-screen`

**预期行为：**
1. 技能开始编写物品栏屏幕用户体验规格
2. 在用户流程章节中，技能尝试引用物品栏系统规则
3. 技能检测到："未找到物品栏系统的 GDD —— 用户体验规格存在依赖缺失 (DEPENDENCY GAP)"
4. 依赖缺失在规格中标记（内联注明："DEPENDENCY GAP: inventory GDD"）
5. 技能继续编写，对缺失的规则使用占位说明
6. 判定结果为 COMPLETE，附带关于依赖缺失的建议说明

**断言：**
- [ ] 规格中出现 DEPENDENCY GAP 标签，指向缺失的系统文档
- [ ] 技能不会因缺失的 GDD 而阻塞 —— 它使用占位符继续
- [ ] 依赖缺失也在技能输出中注明（不仅在文件中）
- [ ] 交接建议同时执行 `/ux-review` 和编写缺失的 GDD

### 用例 4：未提供参数 —— 用法错误

**测试夹具：**
- 技能调用时未提供参数

**输入：** `/ux-design`

**预期行为：**
1. 技能检测到未提供屏幕名称或参数
2. 技能输出用法错误："需要屏幕名称。用法：`/ux-design [screen-name]`"
3. 技能提供示例：`/ux-design hud`、`/ux-design main-menu`、`/ux-design inventory`
4. 不创建任何文件；不询问"我可以写入吗？"

**断言：**
- [ ] 明确说明用法错误
- [ ] 提供调用示例
- [ ] 不创建任何文件
- [ ] 技能不会尝试在没有参数的情况下继续

### 用例 5：总监关卡检查 —— 无关卡；ux-review 是独立的审查技能

**测试夹具：**
- 已提供参数的新屏幕规格

**输入：** `/ux-design settings-menu`

**预期行为：**
1. 技能编写设置菜单用户体验规格的所有章节
2. 不启动任何总监代理
3. 输出中不出现关卡 ID

**断言：**
- [ ] 在 ux-design 过程中不调用总监关卡
- [ ] 不出现关卡跳过消息
- [ ] 判定结果为 COMPLETE，不经过任何关卡检查

## 协议合规性

- [ ] 在讨论内容之前，先创建包含所有章节标题的骨架文件
- [ ] 一次讨论并起草一个章节
- [ ] 每个章节批准后询问"我可以写入章节 [N] 吗？"
- [ ] 检测现有规格并提供改造路径
- [ ] 以交接至 `/ux-review` 结束
- [ ] 当所有章节写入完成时，判定结果为 COMPLETE

---

## 覆盖说明

- 交互状态枚举（正常/悬停/聚焦/禁用/错误）是每个规格的核心要求；`/ux-review` 技能检查其完整性。
- 线框描述仅为文本（无图片）；设计师可在之后手动添加图片引用。
- 响应式布局问题（不同屏幕尺寸）被注明为可选内容，在此不做断言测试。
