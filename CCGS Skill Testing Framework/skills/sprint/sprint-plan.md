
# 技能测试规格：/sprint-plan

## 技能概述 (Skill Summary)

`/sprint-plan` 读取当前里程碑 (milestone) 文件和待办列表 (backlog) 故事，然后生成一个新的编号冲刺 (sprint)，故事按实现层级和优先级分数排序。在完整模式下，PR-SPRINT 主管门控 (director gate) 在冲刺草稿编译后运行（制作人 (producer) 审查计划）。在精简和单人模式下，该门控被跳过。该技能在持久化之前会询问"我可以写入 `production/sprints/sprint-NNN.md` 吗？"。判决 (verdict)：COMPLETE（完成——冲刺已生成并写入）或 BLOCKED（受阻——因数据缺失或门控失败无法继续）。

---

## 静态断言（结构）

- [ ] 包含所需 frontmatter 字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`

- [ ] 包含判决关键词：COMPLETE、BLOCKED

- [ ] 具有下一步交接指引（冲刺写入后该做什么）

## 主管门控检查 (Director Gate Checks)

| 门控 ID   | 触发条件           | 模式守卫             |
|-----------|--------------------|------------------------|
| PR-SPRINT | 冲刺草稿构建完成后      | 仅完整模式（非精简/单人）  |

## 测试用例 (Test Cases)

### 用例 1：快乐路径——有待办故事的待办列表生成冲刺

**测试夹具 (Fixture)：**
- `production/milestones/milestone-02.md` 存在，容量为 `10 个故事点`
- 待办列表包含 5 个未开始的故事，分布于 2 个史诗，优先级混合
- `production/session-state/review-mode.txt` 包含 `full`
- 下一个冲刺编号为 `003`（冲刺 001 和 002 已存在）

**输入：** `/sprint-plan`

**预期行为：**
1. 技能读取当前里程碑以获取容量和目标
2. 技能读取待办列表中的所有未开始故事；按层级 + 优先级排序
3. 技能起草 sprint-003，故事符合容量限制
4. 技能在调用门控前向用户展示草稿
5. 技能调用 PR-SPRINT 门控（完整模式）；制作人批准
6. 技能询问"我可以写入 `production/sprints/sprint-003.md` 吗？"
7. 用户批准；文件写入

**断言 (Assertions)：**
- [ ] 故事在按优先级排序前先按实现层级排序
- [ ] 冲刺草稿在任何写入或门控调用前先展示
- [ ] 完整模式下，PR-SPRINT 门控在草稿就绪后被调用
- [ ] 技能在写入冲刺文件前询问"我可以写入吗"
- [ ] 写入的文件路径匹配 `production/sprints/sprint-003.md`
- [ ] 成功写入后判决为 COMPLETE

---

### 用例 2：受阻路径——待办列表为空

**测试夹具 (Fixture)：**
- `production/milestones/milestone-02.md` 存在
- 任何史诗待办列表中均无未开始的故事

**输入：** `/sprint-plan`

**预期行为：**
1. 技能读取待办列表——未发现未开始的故事
2. 技能输出"待办列表中没有未开始的故事"
3. 技能建议运行 `/create-stories` 以填充待办列表
4. 不调用门控；不写入文件

**断言：**
- [ ] 判决为 BLOCKED
- [ ] 输出包含"没有未开始的故事"或类似信息
- [ ] 输出推荐 `/create-stories`
- [ ] PR-SPRINT 门控未被调用
- [ ] 未调用写入工具

---

### 用例 3：门控返回 CONCERNS——冲刺过载，写入前进行修订

**测试夹具 (Fixture)：**
- 待办列表有 8 个故事，总计 16 点；里程碑容量为 10 点
- `review-mode.txt` 包含 `full`

**输入：** `/sprint-plan`

**预期行为：**
1. 技能起草包含全部 8 个故事的冲刺（超容量）
2. PR-SPRINT 门控运行；制作人返回 CONCERNS：冲刺过载
3. 技能向用户展示关注点，并询问哪些故事需要延期
4. 用户选择延期 3 个故事；冲刺修订为 5 个故事 / 10 点
5. 技能使用修订后的冲刺询问"我可以写入吗"，批准后写入

**断言：**
- [ ] PR-SPRINT 门控的 CONCERNS 在任何写入前向用户展示
- [ ] 技能允许根据门控反馈修订冲刺
- [ ] 写入文件的是修订后的冲刺（而非原始版本）
- [ ] 修订和写入后判决为 COMPLETE

---

### 用例 4：精简模式——PR-SPRINT 门控被跳过

**测试夹具 (Fixture)：**
- 待办列表有 4 个故事；里程碑容量为 8 点
- `review-mode.txt` 包含 `lean`

**输入：** `/sprint-plan`

**预期行为：**
1. 技能读取审查模式——确定为 `lean`
2. 技能起草冲刺并向用户展示
3. PR-SPRINT 门控被跳过；输出注明"[PR-SPRINT] 已跳过——精简模式"
4. 技能请求用户直接批准该冲刺
5. 用户批准；冲刺文件写入

**断言：**
- [ ] 精简模式下 PR-SPRINT 门控未被调用
- [ ] 输出中明确注明了跳过
- [ ] 写入前仍需用户批准（门控跳过 ≠ 批准跳过）
- [ ] 写入后判决为 COMPLETE

---

### 用例 5：边缘情况——前一个冲刺仍有未关闭的故事

**测试夹具 (Fixture)：**
- `production/sprints/sprint-002.md` 存在，有 2 个故事的状态仍为 `Status: In Progress`
- 待办列表中有 5 个新的未开始故事
- `review-mode.txt` 包含 `full`

**输入：** `/sprint-plan`

**预期行为：**
1. 技能读取 sprint-002 并检测到 2 个未关闭（进行中）的故事
2. 技能标记："Sprint 002 有 2 个未关闭的故事——请在规划 sprint 003 前确认是否结转"
3. 技能向用户提供选择：结转故事、延期或取消
4. 用户确认结转；结转的故事以 `[CARRY]` 标签前置到新冲刺中
5. 构建冲刺草稿；PR-SPRINT 门控运行；批准后写入冲刺

**断言：**
- [ ] 技能检查最近冲刺文件中是否存在未关闭的故事
- [ ] 在继续冲刺规划之前，用户被要求确认结转
- [ ] 结转的故事出现在新冲刺草稿中，并附有区分的标签
- [ ] 技能不会静默忽略前一个冲刺中的未关闭故事

## 协议合规性 (Protocol Compliance)

- [ ] 在调用 PR-SPRINT 门控或请求写入之前先展示冲刺草稿
- [ ] 在写入冲刺文件前始终询问"我可以写入吗"
- [ ] PR-SPRINT 门控仅在完整模式下运行
- [ ] 精简和单人模式输出中出现跳过消息
- [ ] 技能输出末尾明确声明判决

---

## 覆盖范围说明 (Coverage Notes)

- 没有里程碑文件存在的情况未进行明确测试；行为遵循 BLOCKED 模式，并建议运行 `/gate-check` 以推进里程碑。
- 单人模式行为等同于精简模式（门控跳过，需用户批准），且未单独测试。
- 并行故事选择算法在此处未经测试；这些是 sprint-plan 子智能体的单元关注点。
