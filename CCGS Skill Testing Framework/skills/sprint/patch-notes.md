
# 技能测试规格：/patch-notes

## 技能概述 (Skill Summary)

`/patch-notes` 是一个 Haiku 层级 (Haiku-tier) 的技能，从现有的更新日志 (changelog) 内容生成面向玩家的补丁说明 (patch notes)，去除内部任务 ID 和技术术语，改用通俗语言。它过滤条目，只保留与玩家相关的条目（可见的功能和 Bug 修复；内部重构被排除）。不使用任何主管门控 (director gate)。该技能在持久化之前会询问"我可以写入 `docs/patch-notes-vX.X.md` 吗？"。判决 (verdict) 始终为 COMPLETE（完成）。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证——无需测试夹具 (fixture)。

- [ ] 包含判决关键词：COMPLETE

## 主管门控检查 (Director Gate Checks)

无。补丁说明生成是一项快速编译任务；不调用任何门控。

## 测试用例 (Test Cases)

### 用例 1：快乐路径——更新日志过滤为面向玩家的条目

**测试夹具 (Fixture)：**
- `docs/CHANGELOG.md` 存在，包含 5 个条目：
  - "添加双持近战系统"（功能——面向玩家）
  - "修复关卡切换时的崩溃"（修复——面向玩家）
  - "添加敌人巡逻 AI"（功能——面向玩家）
  - "重构输入处理器以使用事件总线"（修复——仅内部）
  - "更新依赖：Godot 4.6"（仅内部）
- 版本为 `v0.4.0`

**输入：** `/patch-notes v0.4.0`

**预期行为：**
1. 技能读取 `docs/CHANGELOG.md`
2. 技能过滤出 3 个面向玩家的条目；排除 2 个内部条目
3. 技能用通俗语言重写条目（无任务 ID，无技术术语）
4. 技能向用户展示草稿
5. 技能询问"我可以写入 `docs/patch-notes-v0.4.0.md` 吗？"
6. 用户批准；文件写入；判决 COMPLETE

**断言 (Assertions)：**
- [ ] 补丁说明中只出现 3 个条目（排除了 2 个内部条目）
- [ ] 条目以通俗语言编写，不包含内部任务 ID
- [ ] 文件路径匹配 `docs/patch-notes-v0.4.0.md`
- [ ] 文件写入前出现"我可以写入吗"提示
- [ ] 写入后判决为 COMPLETE

### 用例 2：未找到更新日志——引导先运行 /changelog

**测试夹具 (Fixture)：**
- `docs/CHANGELOG.md` 不存在

**输入：** `/patch-notes v0.4.0`

**预期行为：**
1. 技能尝试读取 `docs/CHANGELOG.md`——未找到
2. 技能输出："未找到更新日志——请先运行 /changelog 生成一份"
3. 不生成补丁说明；不写入文件

**断言：**
- [ ] 当更新日志不存在时，技能不会崩溃
- [ ] 输出明确引导用户运行 `/changelog`
- [ ] 不出现"我可以写入吗"提示（没有任何内容可写入）
- [ ] 判决为 BLOCKED（依赖条件未满足）

### 用例 3：来自设计文件夹的语气指南——融入输出

**测试夹具 (Fixture)：**
- `docs/CHANGELOG.md` 存在，包含面向玩家的条目
- `design/community/tone-guide.md` 存在，包含指引："积极、鼓励的语气；避免被动语态"

**预期行为：**
1. 技能读取更新日志
2. 技能检测到 `design/community/tone-guide.md` 中的语气指南
3. 技能在重写条目为通俗语言时应用语气指南
4. 补丁说明使用积极、主动语态的措辞
5. 技能展示草稿、请求写入，在批准后写入

**断言：**
- [ ] 技能检查 `design/` 中是否有社区或语气指南文件
- [ ] 语气指南内容影响补丁说明条目的措辞
- [ ] 输出反映主动语态和积极语气（适用时）
- [ ] 技能注明已应用语气指南

## 协议合规性 (Protocol Compliance)

- [ ] 在生成补丁说明前读取 `docs/CHANGELOG.md`
- [ ] 将条目过滤为仅面向玩家的内容
- [ ] 用通俗语言重写条目，不包含内部 ID
- [ ] 在写入补丁说明文件前始终询问"我可以写入吗"
- [ ] 不调用任何主管门控
- [ ] 在 Haiku 模型层级上运行（快速、低成本）

## 覆盖范围说明 (Coverage Notes)

- 所有更新日志条目均为内部内容（零个面向玩家条目）的情况未进行测试；行为是生成一个空的补丁说明草稿并附带警告。
- 从更新日志页眉中解析版本号是一个实现细节，在此处未经验证。
- 用例 5 中提到的社区经理咨询是建议性的；一个独立的技能或手动审查负责该步骤。
