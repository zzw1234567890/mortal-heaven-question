
# 技能测试规格：/changelog

## 技能概述 (Skill Summary)

`/changelog` 是一个 Haiku 层级 (Haiku-tier) 的技能，通过读取自上一个发布标签以来的 Git 提交历史和已结束的冲刺 (sprint) 故事，自动生成面向开发者的更新日志 (changelog)。它将条目组织为功能、修复和已知问题。不使用任何主管门控 (director gate)。该技能在持久化之前会询问"我可以写入 `docs/CHANGELOG.md` 吗？"。判决 (verdict) 始终为 COMPLETE（完成）。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证——无需测试夹具 (fixture)。

- [ ] 包含判决关键词：COMPLETE

无。更新日志生成是一项快速编译任务；不调用任何门控。

## 测试用例 (Test Cases)

### 用例 1：快乐路径——自上一个发布标签以来多个冲刺

**测试夹具 (Fixture)：**
- Git 历史中有一个三个冲刺前的标签 `v0.3.0`
- 自该标签以来：在冲刺 006、007、008 中有 12 次提交
- 冲刺故事文件引用的任务 ID 与提交消息匹配
- `docs/CHANGELOG.md` 尚不存在

**输入：** `/changelog`

**预期行为：**
1. 技能读取自 `v0.3.0` 标签以来的 Git 日志
2. 技能读取冲刺故事以交叉引用任务 ID
3. 技能将条目编译到功能、修复和已知问题等部分
4. 技能向用户展示草稿
5. 技能询问"我可以写入 `docs/CHANGELOG.md` 吗？"
6. 用户批准；文件写入；判决 COMPLETE

**断言 (Assertions)：**
- [ ] 更新日志覆盖自最近 Git 标签以来的所有提交
- [ ] 条目按功能 / 修复 / 已知问题部分组织
- [ ] 使用冲刺故事引用丰富提交描述
- [ ] 文件写入前出现"我可以写入吗"提示
- [ ] 写入后判决为 COMPLETE

---

### 用例 2：未找到 Git 标签——使用所有提交，注明版本基线

**测试夹具 (Fixture)：**
- Git 仓库有提交但没有任何标签
- 历史中有 20 次提交，跨越 3 个冲刺

**输入：** `/changelog`

**预期行为：**
1. 技能检查 Git 标签——未找到
2. 技能使用全部历史提交作为基线
3. 技能在输出中注明："未找到版本标签——使用全部提历史交；版本基线未设置"
4. 技能仍从可用提交编译出有组织的更新日志
5. 技能询问"我可以写入吗"并在批准后写入

**断言：**
- [ ] 当没有 Git 标签时，技能不会报错
- [ ] 输出明确注明未找到版本基线
- [ ] 使用全部提交历史作为来源
- [ ] 尽管缺少标签，更新日志仍按章节组织

---

### 用例 3：提交消息不含任务 ID——按日期分组并附带说明

**测试夹具 (Fixture)：**
- 自上一个标签以来的 Git 日志中有 8 次提交
- 5 次提交的消息中没有任务 ID（例如，"修复拼写错误"、"调整数值"）
- 3 次提交引用了与冲刺故事匹配的任务 ID

**输入：** `/changelog`

**预期行为：**
1. 技能读取提交和冲刺故事
2. 3 次提交匹配到冲刺故事，并放入相应章节
3. 5 次未标记的提交按日期归入"杂项"或"其他变更"部分
4. 输出注明："5 次提交无任务 ID——已按日期分组"
5. 技能在批准后写入更新日志

**断言：**
- [ ] 带有任务 ID 的提交被放在相应章节（功能或修复）
- [ ] 没有任务 ID 的提交被单独分组并附带说明
- [ ] 输出标记了缺少任务引用的提交数量
- [ ] 没有任何提交被静默丢弃

---

### 用例 4：更新日志已存在——前置新章节，保留旧条目

**测试夹具 (Fixture)：**
- `docs/CHANGELOG.md` 已存在，包含 `v0.2.0` 和 `v0.3.0` 的章节
- 自 `v0.3.0` 标签以来有新的提交

**输入：** `/changelog`

**预期行为：**
1. 技能检测到 `docs/CHANGELOG.md` 已存在
2. 技能为自 `v0.3.0` 以来的期间编译新条目
3. 技能展示草稿，新章节前置在现有内容之上
4. 技能询问"我可以写入 `docs/CHANGELOG.md` 吗？"（确认前置策略）
5. 用户批准；新内容被前置，旧条目保持不变；判决 COMPLETE

**断言：**
- [ ] 技能在写入前读取现有更新日志以检测先前内容
- [ ] 新章节被前置（而非追加或覆盖）到现有条目之上
- [ ] 写入的文件中保留了 v0.2.0 和 v0.3.0 的旧更新日志条目
- [ ] "我可以写入吗"提示反映了前置操作

---

### 用例 5：门控合规——无门控；先读后写，需批准

**测试夹具 (Fixture)：**
- Git 历史中自上一个标签以来有提交
- `review-mode.txt` 包含 `full`

**输入：** `/changelog`

**预期行为：**
1. 技能在完整模式下编译更新日志
2. 不调用任何主管门控（更新日志生成是编译工作，而非交付门控）
3. 技能在 Haiku 模型上运行——快速编译
4. 技能向用户请求批准，确认后写入文件

**断言：**
- [ ] 无论审查模式如何，均不调用主管门控
- [ ] 输出不引用任何门控结果
- [ ] 技能直接从编译进入"我可以写入吗"提示
- [ ] 判决为 COMPLETE

## 协议合规性 (Protocol Compliance)

- [ ] 在编译前先读取 Git 日志和冲刺故事文件
- [ ] 在写入更新日志前始终询问"我可以写入吗"
- [ ] 不调用任何主管门控
- [ ] 判决始终为 COMPLETE
- [ ] 在 Haiku 模型层级上运行（快速、低成本）

---

## 覆盖范围说明 (Coverage Notes)

- 仓库中 Git 未初始化的情况未进行测试；行为将取决于 Git 命令失败的处理方式。
- 在这些测试中，合并提交与压缩提交未进行明确区分；这是 Git 日志解析阶段的实现细节。
- `/patch-notes` 技能应在 `/changelog` 之后运行，以生成面向玩家的输出；该交接在 patch-notes 规格中得到验证。
