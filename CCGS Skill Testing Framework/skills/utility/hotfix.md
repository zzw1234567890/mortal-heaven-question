# 技能测试规格：/hotfix


## 技能概述 (Skill Summary)

`/hotfix` 管理紧急修复工作流：它从 main 分支创建热修复分支，对识别出的文件应用针对性修复，运行 `/smoke-check` 验证修复不会引入回归问题，并提示用户确认合并回 main 分支。每次代码变更都需要询问"May I write to [filepath]?"。Git 操作（创建分支、合并）作为 Bash 命令呈现，在执行前需要用户确认。

该技能对时间敏感——总监审查是可选的事后步骤，而非阻塞关卡。裁决结果为：HOTFIX COMPLETE（修复已应用、冒烟检查通过、已合并）或 HOTFIX BLOCKED（修复引入回归或用户拒绝）。

---

## 静态断言 (Structural)

由 `/skill-test static` 自动验证——无需测试夹具。

- [ ] 拥有必需的前置元字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 有 ≥2 个阶段标题
- [ ] 包含裁决关键词：HOTFIX COMPLETE、HOTFIX BLOCKED
- [ ] 包含用于代码变更的"May I write"语言
- [ ] 有下一步交接（例如，`/bug-report` 记录问题，或版本号提升）

---

## 总监关卡检查 (Director Gate Checks)

无。热修复对时间敏感。总监审查可以随后作为事后步骤单独进行。本技能内不调用任何关卡。

---

## 测试用例 (Test Cases)

### 用例 1：正常路径 (Happy Path) —— 关键崩溃 Bug 修复，冒烟检查通过

**测试夹具：**
- `main` 分支是干净的
- Bug 在 `src/gameplay/arena.gd` 中被识别（进入 Boss 竞技场时崩溃）
- 用户提供了重现步骤

**输入：** `/hotfix`（用户描述崩溃和受影响的文件）

**预期行为：**
1. 技能提议创建热修复分支：`hotfix/boss-arena-crash`
2. 用户确认；创建分支的 Bash 命令被展示并确认
3. 技能识别 `arena.gd` 中的修复位置并起草变更
4. 技能询问"是否可以写入 `src/gameplay/arena.gd`？"并在批准后应用修复
5. 技能运行 `/smoke-check`——结果为 PASS
6. 技能展示合并命令并要求用户确认合并到 `main`
7. 用户确认；合并执行；裁决结果为 HOTFIX COMPLETE

**断言：**
- [ ] 在进行任何代码变更之前创建热修复分支
- [ ] 在修改任何源文件之前询问"May I write"
- [ ] 修复应用后运行 `/smoke-check`
- [ ] 合并需要用户明确确认（非自动）
- [ ] 成功合并后裁决结果为 HOTFIX COMPLETE

---

### 用例 2：冒烟检查失败 (Smoke Check Fails) —— HOTFIX BLOCKED

**测试夹具：**
- 修复已应用于 `src/gameplay/arena.gd`
- `/smoke-check` 返回 FAIL："检测到玩家生命值钳制回归"

**输入：** `/hotfix`

**预期行为：**
1. 技能应用修复并运行 `/smoke-check`
2. 冒烟检查返回 FAIL，标明具体的回归问题
3. 技能报告："HOTFIX BLOCKED——冒烟检查失败：[回归详情]"
4. 技能提供选项：尝试修订后的修复、回滚更改、或在已知回归的情况下合并（用户承认风险）
5. 冒烟检查失败时不会自动合并

**断言：**
- [ ] 裁决结果为 HOTFIX BLOCKED
- [ ] 冒烟检查失败信息按原样显示给用户
- [ ] 冒烟检查失败时不自动执行合并
- [ ] 为用户提供明确的后续处理选项

---

### 用例 3：修复已发布版本 (Fix to Already-Released Build) —— 标注版本标签，提示补丁版本号提升

**测试夹具：**
- 最新的 Git 标签为 `v1.2.0`
- 热修复针对 v1.2.0 发布版中的一个 Bug

**输入：** `/hotfix`

**预期行为：**
1. 技能检测到当前 HEAD 是一个带标签的发布版本（v1.2.0）
2. 技能注明："热修复针对的是带标签的发布版本 v1.2.0"
3. 冒烟检查通过后，技能提示："版本是否应提升到 v1.2.1？"
4. 如果用户确认版本提升：技能询问"是否可以写入 VERSION 或等效文件？"
5. 版本更新和合并后：裁决结果为 HOTFIX COMPLETE，附带版本信息

**断言：**
- [ ] 版本标签上下文被检测到并展示给用户
- [ ] 在合并后建议（而非要求）补丁版本号提升
- [ ] 版本号提升需要自己的"May I write"确认
- [ ] 裁决结果为 HOTFIX COMPLETE

---

### 用例 4：无重现步骤 (No Repro Steps) —— 技能在应用修复前询问

**测试夹具：**
- 用户使用模糊描述调用 `/hotfix`："第三关某些东西坏了"
- 未提供重现步骤

**输入：** `/hotfix`（模糊描述）

**预期行为：**
1. 技能检测到信息不足以确定修复位置
2. 技能询问："请提供重现步骤以及受影响的文件或系统"
3. 在提供重现步骤之前，技能不创建分支或修改任何文件
4. 用户提供重现步骤后：正常热修复流程开始

**断言：**
- [ ] 在没有重现步骤的情况下不创建分支
- [ ] 在没有明确识别修复位置的情况下不进行任何代码变更
- [ ] 重现步骤请求是具体的（不仅仅是"请提供更多信息"）
- [ ] 用户提供重现步骤后恢复正常热修复流程

---

### 用例 5：总监关卡检查 (Director Gate Check) —— 无关卡；热修复对时间敏感

**测试夹具：**
- 已识别关键 Bug 及其重现步骤

**输入：** `/hotfix`

**预期行为：**
1. 技能完成热修复工作流
2. 执行期间未生成任何总监代理
3. 输出中不出现关卡 ID
4. 事后总监审查（如果需要）是手动跟进，此处不调用

**断言：**
- [ ] 未调用任何总监关卡
- [ ] 未出现关卡跳过消息
- [ ] 裁决结果为 HOTFIX COMPLETE 或 HOTFIX BLOCKED——无关卡裁决

---

## 协议合规性 (Protocol Compliance)

- [ ] 在进行任何代码变更之前创建热修复分支
- [ ] 在修改任何源文件之前询问"May I write"
- [ ] 应用修复后运行 `/smoke-check`
- [ ] 合并前需要用户明确确认
- [ ] 冒烟检查失败时标记为 HOTFIX BLOCKED——不会自动合并
- [ ] 裁决结果为 HOTFIX COMPLETE 或 HOTFIX BLOCKED

---

## 覆盖说明 (Coverage Notes)

- 一个修复需要修改多个文件的情况遵循相同的每文件"May I write"模式，未单独测试。
- 热修复后步骤（创建 Bug 报告、更新变更日志）在交接中建议，但不作为本技能执行的一部分进行测试。
- 合并过程中的冲突解决（如果 main 分支已分叉）未测试；技能会展示冲突并要求用户手动解决。
