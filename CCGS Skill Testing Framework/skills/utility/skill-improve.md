# 技能测试规格：/skill-improve


## 技能概要

`/skill-improve` 对技能文件运行自动化的测试-修复-重测改进循环。它调用 `/skill-test static`（以及可选地 `/skill-test category`）来建立基线分数，诊断失败的检查项，对 SKILL.md 文件提出有针对性的修复，询问"我可以将改进写入 [技能路径] 吗？"，应用修复，并重新运行测试以确认改进。

如果提议的修复使技能变得更差（回归），则回退修复（需要用户确认）而非应用。如果技能已经完美无缺（0 个失败），则技能立即退出而不做任何更改。不适用任何主管闸门。判定结果：IMPROVED（分数提升）、NO CHANGE（无法改进或用户拒绝）或 REVERTED（修复已应用但导致回归并已回退）。

---

## 静态断言（结构性）

由 `/skill-test static` 自动验证——无需测试夹具。

- [ ] 包含必需的元数据字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 包含 ≥2 个阶段标题
- [ ] 包含判定关键字：IMPROVED、NO CHANGE、REVERTED
- [ ] 在应用修复前包含"我可以写入"协作协议语言
- [ ] 包含下一步交接（例如，运行 `/skill-test spec` 以验证行为合规性）

---

## 主管闸门检查

无。`/skill-improve` 是一项元实用技能。不适用任何主管闸门。

---

## 测试用例

### 用例 1：快乐路径 —— 技能有 2 个静态失败，均已修复，IMPROVED

**测试夹具：**
- `.claude/skills/some-skill/SKILL.md` 有 2 个静态失败：
  - 检查 4：尽管 allowed-tools 中有 Write，但没有"我可以写入"措辞
  - 检查 5：末尾没有下一步交接

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 技能运行 `/skill-test static some-skill` —— 基线：5/7 检查通过
2. 技能诊断 2 个失败的检查项（4 和 5）
3. 技能提出修复方案：
   - 在适当的阶段添加"我可以写入"措辞
   - 在末尾添加下一步交接章节
4. 技能询问"我可以将改进写入 `.claude/skills/some-skill/SKILL.md` 吗？"
5. 修复已应用；重新运行 `/skill-test static some-skill` —— 现在 7/7 检查通过
6. 判定结果为 IMPROVED（5→7）

**断言：**
- [ ] 在任何更改前建立基线分数（5/7）
- [ ] 两个失败的检查项均被诊断并在提议的修复中得到处理
- [ ] 在应用修复前询问了"我可以写入"
- [ ] 重测确认改进（7/7）
- [ ] 判定结果为 IMPROVED，显示前后分数对比

---

### 用例 2：修复导致回归 —— 分数对比显示回归，REVERTED

**测试夹具：**
- `.claude/skills/some-skill/SKILL.md` 有 1 个静态失败（缺少交接）
- 提议的修复无意中移除了判定关键字章节（引入了新失败）

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 基线：6/7 检查通过（1 个失败：缺少交接）
2. 技能提出修复并询问"我可以写入改进吗？"
3. 修复被应用；重新运行测试
4. 重测结果：5/7（修复了交接但破坏了判定关键字）
5. 技能检测到回归：分数下降
6. 技能询问用户："修复导致了回归（6→5）。我可以回退更改吗？"
7. 用户确认；更改被回退；判定结果为 REVERTED

**断言：**
- [ ] 重测分数在最终确定前与基线进行比较
- [ ] 当分数下降时检测到回归
- [ ] 用户被要求确认回退（非自动）
- [ ] 用户确认后文件被回退
- [ ] 判定结果为 REVERTED

---

### 用例 3：技能有类别分配 —— 基线捕获两项分数

**测试夹具：**
- `.claude/skills/gate-check/SKILL.md` 是一项闸门技能，有 1 个静态失败和 2 个类别（G 标准）失败
- `tests/skills/quality-rubric.md` 包含闸门技能章节（Gate Skills）

**输入：** `/skill-improve gate-check`

**预期行为：**
1. 技能运行静态和类别测试以建立基线：
   - 静态：6/7 检查通过
   - 类别：3/5 个 G 标准通过
2. 合并基线：9/12
3. 技能诊断所有 3 个失败并提议修复
4. "我可以将改进写入 `.claude/skills/gate-check/SKILL.md` 吗？"
5. 修复已应用；两种测试类型重新运行
6. 重测：静态 7/7，类别 5/5 = 12/12
7. 判定结果为 IMPROVED（9→12）

**断言：**
- [ ] 静态和类别分数均被捕获在基线中
- [ ] 合并分数用于比较（不仅仅是单一类型）
- [ ] 所有 3 个失败均在提议的修复中得到处理
- [ ] 重测确认两种分数类型的改进
- [ ] 判定结果为 IMPROVED，显示合并的前后分数

---

### 用例 4：技能已完美 —— 无需改进

**测试夹具：**
- `.claude/skills/brainstorm/SKILL.md` 没有静态失败
- 类别分数也是 5/5（如适用）

**输入：** `/skill-improve brainstorm`

**预期行为：**
1. 技能运行 `/skill-test static brainstorm` —— 7/7 检查通过
2. 如类别适用：5/5 标准通过
3. 技能输出："无需改进——brainstorm 完全合规"
4. 技能退出，不提议任何更改
5. 不询问"我可以写入"；不修改任何文件
6. 判定结果为 NO CHANGE

**断言：**
- [ ] 技能在确认 0 个失败后立即退出
- [ ] 显示"无需改进"消息
- [ ] 不提议任何更改
- [ ] 不询问"我可以写入"
- [ ] 判定结果为 NO CHANGE

---

### 用例 5：主管闸门检查 —— 无闸门；skill-improve 是元实用技能

**测试夹具：**
- 至少有 1 个静态失败的技能

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 技能运行测试-修复-重测循环
2. 不生成任何主管智能体
3. 输出中不出现闸门 ID

**断言：**
- [ ] 未调用主管闸门
- [ ] 无闸门跳过消息出现
- [ ] 判定结果为 IMPROVED、NO CHANGE 或 REVERTED —— 无闸门判定

---

## 协议合规性

- [ ] 在提议任何更改前始终建立基线分数
- [ ] 在输出中显示前后分数对比
- [ ] 在应用任何修复前询问"我可以写入"
- [ ] 通过比较重测分数与基线来检测回归
- [ ] 在回退前要求用户确认（非自动）
- [ ] 以 IMPROVED、NO CHANGE 或 REVERTED 判定结束

---

## 覆盖说明

- 改进循环设计为每次调用仅运行一个修复-重测周期；运行多次迭代需要重新调用 `/skill-improve`。
- 行为合规性（规格模式测试结果）不包括在改进循环中——只有结构性（静态）和类别分数是自动化的。
- 技能文件无法读取的情况（权限错误或文件缺失）未进行测试；这将在基线建立之前导致错误。
