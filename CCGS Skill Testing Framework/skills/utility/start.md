# 技能测试规格：/start


## 技能概要

`/start` 是新项目的首次引导上手技能。它引导用户完成项目命名、选择游戏引擎以及设置初始目录结构。它创建存根配置文件（CLAUDE.md、technical-preferences.md），然后路由到以所选引擎为参数的 `/setup-engine`。每个创建的文件或目录都需经过"我可以写入"的确认请求，遵循协作协议。

该技能检测项目是否已配置以及是否存在部分设置，并提供恢复或重新启动的适当选项。它没有主管闸门——它是一项实用设置技能，在任何智能体层级存在之前运行。

无。`/start` 是一项实用设置技能。在该技能运行时，尚不存在任何主管智能体。

## 测试用例

### 用例 1：快乐路径 —— 新仓库，无引擎，完整上手流程

**测试夹具：**
- 空仓库：无 CLAUDE.md 覆盖、无 `production/stage.txt`、`technical-preferences.md` 仅有占位符内容
- 无现有设计文档或源代码

**输入：** `/start`

**预期行为：**
1. 技能检测到无现有配置并开始全新的上手流程
2. 技能询问项目名称
3. 技能展示 3 个引擎选项：Godot 4、Unity、Unreal Engine 5
4. 用户选择一个引擎
5. 技能询问"我可以写入初始目录结构吗？"
6. 技能创建 `directory-structure.md` 中定义的所有目录
7. 技能询问"我可以写入 CLAUDE.md 存根吗？"并在批准后写入
8. 技能路由到 `/setup-engine [所选引擎]` 以完成技术配置

**断言：**
- [ ] 在任何文件写入前捕获项目名称
- [ ] 恰好展示 3 个引擎选项
- [ ] 每个配置文件单独询问"我可以写入"
- [ ] 未经用户明确批准不写入任何文件
- [ ] 末尾以所选引擎参数进行到 `/setup-engine` 的交接
- [ ] 所有文件写入并发出交接后判定结果为 COMPLETE

---

### 用例 2：已配置 —— 检测现有配置，提供跳过或重新配置选项

**测试夹具：**
- `technical-preferences.md` 的引擎已设置（非占位符）
- `production/stage.txt` 存在，内容为 `Concept`

**输入：** `/start`

**预期行为：**
1. 技能读取 `technical-preferences.md` 并检测到已配置的引擎
2. 技能报告："此项目已配置为 [引擎]"
3. 技能提供选项：跳过（退出）、重新配置引擎、或重新配置特定章节
4. 如果用户选择跳过：技能干净地退出，并显示当前配置摘要
5. 如果用户选择重新配置：技能进入引擎选择步骤

**断言：**
- [ ] 在用户选择重新配置前，技能不会覆盖现有配置
- [ ] 检测到的引擎名称在状态消息中向用户显示
- [ ] 用户被提供至少 2 个选项（跳过或重新配置）
- [ ] 无论用户跳过还是重新配置，判定结果均为 COMPLETE

---

### 用例 3：引擎选择 —— 用户选择 Godot 4，路由到 /setup-engine godot

**测试夹具：**
- 新仓库——无现有配置

**输入：** `/start`

**预期行为：**
1. 技能展示引擎选项，用户选择 Godot 4
2. 技能在批准后写入初始存根（目录结构、CLAUDE.md）
3. 技能明确路由到 `/setup-engine godot` 作为下一步
4. 交接消息清楚地指明引擎名称和下一次技能调用

**断言：**
- [ ] 交接命令为 `/setup-engine godot`（非泛指的 `/setup-engine`）
- [ ] 在所有初始存根写入后发出交接，而非之前
- [ ] 在写入开始前向用户回显引擎选择

---

### 用例 4：中断的设置 —— 检测到部分配置，提供恢复或重新启动选项

**测试夹具：**
- 目录结构存在（已创建），但 `technical-preferences.md` 仍为全占位符（引擎从未选择——设置被中断）
- 无 `production/stage.txt`

**输入：** `/start`

**预期行为：**
1. 技能检测到部分状态：目录存在但引擎未配置
2. 技能报告："检测到部分设置——目录存在但引擎未配置"
3. 技能提供选项：从引擎选择恢复，或从头重新开始
4. 如果选择恢复：技能跳过目录创建，直接进入引擎选择
5. 如果选择重新开始：技能在继续前询问"我可以覆盖现有结构吗？"

**断言：**
- [ ] 部分状态被正确识别（目录存在，引擎缺失）
- [ ] 用户被提供恢复 vs. 重新开始的选择——而非强制进入单一路径
- [ ] 恢复路径跳过重新创建目录（不会对结构冗余询问"我可以写入"）
- [ ] 重新开始路径在接触任何文件前请求覆盖许可

---

### 用例 5：主管闸门检查 —— 无闸门；start 是实用设置技能

**测试夹具：**
- 任何夹具

**输入：** `/start`

**预期行为：**
1. 技能完成完整的上手流程
2. 任何时候都不生成主管智能体
3. 输出中不出现闸门 ID（CD-*、TD-*、AD-*、PR-*）

**断言：**
- [ ] 技能执行期间未调用主管闸门
- [ ] 无闸门跳过消息出现（闸门不存在，而非被抑制）
- [ ] 技能在未进行任何闸门判定的情况下达到 COMPLETE

## 协议合规性

- [ ] 在任何文件写入前询问项目名称
- [ ] 以结构化选择形式（非自由文本）展示引擎选项
- [ ] 分别对目录结构和 CLAUDE.md 存根询问"我可以写入"
- [ ] 以带引擎名称参数的 `/setup-engine` 交接结束
- [ ] 输出末尾明确说明判定结果（COMPLETE 或 BLOCKED）

---

## 覆盖说明

- 用户拒绝所有引擎选项并提供自定义引擎名称的情况未测试——该技能仅设计用于三种支持的引擎。
- Git 初始化（如有）未在此测试；这是技能边界之外的基础设施问题。
- 单人模式（solo）与精简模式（lean）行为不适用——此技能没有闸门，模式选择无关紧要。
