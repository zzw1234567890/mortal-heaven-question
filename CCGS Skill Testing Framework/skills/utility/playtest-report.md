
# 技能测试规格：/playtest-report

## 技能概述 (Skill Summary)

`/playtest-report` 根据会话笔记或用户输入生成结构化的试玩测试报告。报告分为四个章节：手感/无障碍（Feel/Accessibility）、观察到的 Bug（Bugs Observed）、设计反馈（Design Feedback）和下一步（Next Steps）。当有多名测试人员参与时，技能会汇总反馈并区分多数意见与少数意见。当报告的 Bug 与 `production/bugs/` 中的现有文件匹配时，技能会链接到已有的 Bug 报告。

报告在征询"May I write"后写入 `production/qa/playtest-[date].md`。此处不适用主管关口——CD-PLAYTEST 主管关口（如需要）是单独的调用。判定结果为 COMPLETE（报告已写入）。

---

## 静态断言（结构性）

由 `/skill-test static` 自动验证——无需测试夹具。

- [ ] 包含必需的 frontmatter 字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 包含 ≥2 个阶段标题
- [ ] 包含判定关键词：COMPLETE
- [ ] 在写入报告前包含"May I write"协作协议用语
- [ ] 包含下一步交接（例如，对新发现的问题使用 `/bug-report`，对反馈使用 `/design-review`）

---

## 主管关口检查

无。`/playtest-report` 是一个文档工具。CD-PLAYTEST 关口是单独的调用，不属于本技能的一部分。

## 测试用例

### 用例 1：正常路径——用户提供试玩笔记，生成结构化报告

**测试夹具：**
- 用户提供来自单次试玩会话的打字笔记
- 笔记涵盖：游戏手感、一个 Bug（帧率下降）和一个设计问题（教程过长）
- `production/bugs/` 存在但为空（Bug 尚未报告）

**输入：** `/playtest-report`（用户粘贴会话笔记）

**预期行为：**
1. 技能读取提供的笔记并将其结构化为 4 章节模板
2. 手感/无障碍：提取手感观察结果
3. Bug：记录帧率下降及其可用的复现细节
4. 设计反馈：记录教程时长问题
5. 下一步：建议针对帧率问题使用 `/bug-report`，针对教程反馈使用 `/design-review`
6. 技能询问"我可以写入 `production/qa/playtest-2026-04-06.md` 吗？"
7. 批准后写入报告；判定结果为 COMPLETE

**断言：**
- [ ] 报告中包含所有 4 个章节
- [ ] Bug 列在 Bug 章节中（而非设计反馈章节）
- [ ] 下一步建议适当（崩溃问题使用 bug report，反馈使用 design review）
- [ ] 写入前询问"May I write"
- [ ] 判定结果为 COMPLETE

---

### 用例 2：空输入——引导式逐章节提问

**测试夹具：**
- 调用时用户未提供任何笔记

**输入：** `/playtest-report`

**预期行为：**
1. 技能检测到空输入
2. 技能逐一提示每个章节：
   a. "请描述整体手感及任何无障碍方面的观察"
   b. "是否观察到任何 Bug？请描述"
   c. "测试人员提供了哪些设计反馈？"
3. 用户逐一回答每个提示
4. 技能根据回答汇编报告并询问"May I write"
5. 批准后写入报告；判定结果为 COMPLETE

**断言：**
- [ ] 至少提出 3 个引导性问题（每个主要章节一个）
- [ ] 在所有章节都有输入（或用户明确跳过某个章节）之前，不创建报告
- [ ] 文件写入后判定结果为 COMPLETE

---

### 用例 3：多名测试人员——带有多数/少数意见标记的汇总反馈

**测试夹具：**
- 用户提供来自 3 名测试人员的笔记
- 2/3 的测试人员认为操作"直观"
- 1/3 的测试人员发现 UI 字体过小
- 所有 3 人都注意到了同一个 Bug（玩家卡在平台上）

**输入：** `/playtest-report`（3 人试玩会话）

**预期行为：**
1. 技能识别输入中 3 个不同的测试人员视角
2. 操作直观性 → 标记为"多数意见 (2/3)：操作直观"
3. 字体大小 → 标记为"少数意见 (1/3)：UI 字体大小问题"
4. 卡在平台上 → 标记为"所有测试人员：玩家卡在平台上（已确认）"
5. 技能生成带有多数/少数意见标签的汇总报告
6. 在"May I write"批准后写入报告；判定结果为 COMPLETE

**断言：**
- [ ] 多数意见 (2/3) 被标记为多数
- [ ] 少数意见 (1/3) 被标记为少数
- [ ] 全员报告的 Bug 被注明已由所有测试人员确认
- [ ] 判定结果为 COMPLETE

---

### 用例 4：Bug 与现有报告匹配——链接到现有文件

**测试夹具：**
- `production/bugs/bug-2026-03-30-player-stuck-ledge.md` 存在
- 用户的试玩笔记描述"玩家在靠近墙壁的平台上被卡住"

**输入：** `/playtest-report`

**预期行为：**
1. 技能结构化报告并识别出"卡在平台上"的 Bug
2. 技能扫描 `production/bugs/` 并找到 `bug-2026-03-30-player-stuck-ledge.md`
3. 在 Bug 章节中，报告包含："参见现有报告：production/bugs/bug-2026-03-30-player-stuck-ledge.md"
4. 技能不为此问题建议创建新的 Bug 报告
5. 写入报告；判定结果为 COMPLETE

**断言：**
- [ ] 找到现有 Bug 报告并在试玩测试报告中链接
- [ ] 对于已报告的问题，不建议使用 `/bug-report`
- [ ] 在 Bug 章节中出现了对现有文件的交叉引用
- [ ] 判定结果为 COMPLETE

---

### 用例 5：主管关口检查——无关卡；CD-PLAYTEST 是单独的调用

**测试夹具：**
- 提供了试玩笔记

**输入：** `/playtest-report`

**预期行为：**
1. 技能生成并写入试玩测试报告
2. 不生成任何主管智能体（此处不调用 CD-PLAYTEST）
3. 输出中不出现关口 ID

**断言：**
- [ ] 未调用主管关口
- [ ] 不出现 CD-PLAYTEST 关口跳过消息
- [ ] 判定结果为 COMPLETE，不附带任何关口检查

---

## 协议合规性

- [ ] 将输出结构化为全部 4 个章节（手感、Bug、设计反馈、下一步）
- [ ] 有多名测试人员时标记多数意见与少数意见
- [ ] 当 Bug 匹配时交叉引用现有 Bug 报告
- [ ] 在写入前询问"我可以写入 `production/qa/playtest-[date].md` 吗？"
- [ ] 报告写入后判定结果为 COMPLETE

---

## 覆盖说明

- CD-PLAYTEST 主管关口（创意总监审查试玩洞察的设计影响）是单独的调用，此处未测试。
- 未测试视频录制或截图附件；报告为纯文本文档。
- 测试人员身份不明（匿名反馈）的情况遵循与用例 3 相同的汇总模式，但不标记测试人员标签。
