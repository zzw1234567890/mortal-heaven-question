# 技能测试规格：/bug-report


## 技能概述 (Skill Summary)

`/bug-report` 根据用户描述创建结构化的 Bug 报告文档。它生成的报告包含以下必需字段：标题 (Title)、重现步骤 (Repro Steps)、预期行为 (Expected Behavior)、实际行为 (Actual Behavior)、严重程度 (Severity, CRITICAL/HIGH/MEDIUM/LOW)、受影响系统 (Affected System(s)) 和构建版本 (Build/Version)。如果用户初始描述缺少任何必需字段，技能会提出后续问题来填补空白，然后再生成草稿。

技能会检查可能重复的报告（通过与 `production/bugs/` 中的现有文件进行比较），并提议链接而非创建新报告。每份报告在"May I write"询问后写入 `production/bugs/bug-[date]-[slug].md`。不使用总监关卡 (director gates)——Bug 报告是一种运营工具。

## 测试用例 (Test Cases)

### 用例 1：正常路径 (Happy Path) —— 用户描述崩溃，生成完整报告

**测试夹具：**
- `production/bugs/` 目录存在且为空
- 没有类似的现有报告

**输入：** `/bug-report`（用户描述："玩家进入 Boss 竞技场时游戏崩溃"）

**预期行为：**
1. 技能提取：标题 = "进入 Boss 竞技场时游戏崩溃"
2. 技能识别崩溃报告为 CRITICAL 严重程度
3. 技能与用户确认重现步骤、预期行为（不崩溃）、实际行为（崩溃）、受影响系统（竞技场/Boss）和构建版本
4. 技能起草完整的结构化报告
5. 技能询问"是否可以写入 `production/bugs/bug-2026-04-06-game-crashes-boss-arena.md`？"
6. 批准后写入文件；裁决结果为 COMPLETE

**断言：**
- [ ] 报告中包含所有 7 个必需字段
- [ ] 崩溃报告的严重程度为 CRITICAL
- [ ] 文件名遵循 `bug-[date]-[slug].md` 约定
- [ ] "May I write"询问时包含完整文件路径
- [ ] 裁决结果为 COMPLETE

### 用例 2：最小化输入 (Minimal Input) —— 技能对缺失字段提出后续问题

**测试夹具：**
- 用户提供："音频有时会中断"
- 没有现有报告

**输入：** `/bug-report`

**预期行为：**
1. 技能识别缺失的必需字段：重现步骤、预期与实际行为、严重程度、受影响系统、构建版本
2. 技能针对每个缺失字段提出有针对性的后续问题（逐一或以结构化提示形式）
3. 用户提供答案
4. 技能根据答案编制完整报告
5. 技能询问"May I write?"并在批准后写入

**断言：**
- [ ] 至少提出 3 个后续问题来填补缺失字段
- [ ] 每个必需字段在报告定稿之前均已填写
- [ ] 在所有必需字段填写完毕之前不写入报告
- [ ] 所有字段填写完毕且文件写入后，裁决结果为 COMPLETE

### 用例 3：可能重复 (Possible Duplicate) —— 提议链接而非创建新报告

**测试夹具：**
- `production/bugs/bug-2026-03-20-audio-cut-out.md` 已存在，标题相似且严重程度为 MEDIUM

**输入：** `/bug-report`（用户描述："音频随机停止工作"）

**预期行为：**
1. 技能扫描现有报告并发现类似的音频 Bug
2. 技能报告："存在类似的 Bug 报告：bug-2026-03-20-audio-cut-out.md"
3. 技能提供选项：链接为重复项（在现有报告中添加备注），或无论如何创建新报告
4. 如果用户选择链接：技能向现有文件添加交叉引用备注（询问"是否可以更新现有报告？"）
5. 如果用户选择创建新报告：正常报告创建流程继续

**断言：**
- [ ] 在创建新报告之前展示类似的现有报告
- [ ] 给予用户选择权（不强制链接或创建）
- [ ] 如果选择链接：在修改现有文件之前询问"May I update"
- [ ] 两种路径的裁决结果均为 COMPLETE

---

### 用例 4：多系统 Bug (Multi-System Bug) —— 报告包含多个系统标签

**测试夹具：**
- 没有现有报告

**输入：** `/bug-report`（用户描述："完成关卡后，存档系统冻结，UI 无法显示完成界面"）

**预期行为：**
1. 技能从描述中识别出 2 个受影响系统：存档系统 (Save System) 和 UI
2. 报告草稿中列出两个系统作为受影响系统
3. 评估严重程度（可能为 HIGH——存档冻结存在数据丢失风险）
4. 技能以适当的文件名询问"May I write"
5. 报告写入时包含两个系统标签；裁决结果为 COMPLETE

**断言：**
- [ ] 报告中列出两个受影响系统
- [ ] 创建单份报告（而非每个系统一份）
- [ ] 严重程度反映影响最大的组件（存档冻结 → HIGH 或 CRITICAL）
- [ ] 裁决结果为 COMPLETE

- [ ] 在创建新报告之前检查是否有类似的现有报告
- [ ] 在写入之前询问"是否可以写入 `production/bugs/bug-[date]-[slug].md`？"

## 覆盖说明 (Coverage Notes)

- 用户提供的严重程度对于所述影响来说可能过低的情况（例如，崩溃标注为 LOW）未进行测试；技能可能会建议更高的严重程度，但最终尊重用户输入。
- 构建/版本字段是必需的，但如果用户不知道可以填写"unknown"——这被接受为有效值，未单独测试。
- 报告 slug 生成（将标题净化为文件名）是实现细节，此处未进行断言测试。
